==== 2.13.3. Масштабирование и отказоустойчивость хранения данных

===== 2.13.3.1. Подходы к масштабированию

При проектировании использовалась *стратегия независимого масштабирования отдельных хранилищ в соответствии с их профилем нагрузки*. Такой подход позволяет увеличивать производительность только тех компонентов, которые становятся узким местом, без необходимости масштабирования всей системы хранения целиком.

*Вертикальное масштабирование* используется для компонентов, где критичны:

* производительность отдельных операций,
* скорость работы индексов,
* минимизация задержек транзакций.

В первую очередь данный подход применяется для:

* БД резервирования,
* БД конфликтов,
* Метаданных, справочников,
* БД реестра схем.

Для этих компонентов увеличение вычислительных ресурсов узла позволяет повысить производительность без усложнения архитектуры хранения.

*Горизонтальное масштабирование* используется для компонентов, испытывающих высокую нагрузку на чтение, обработку событий или хранение больших объемов данных. В архитектуре Сервиса данный подход применяется для:

* Распределенного кеша,
* Хранилища модели чтения,
* Гео-индекса, координат,
* Event Bus,
* Очереди ошибок.

Горизонтальное масштабирование обеспечивает:

* увеличение пропускной способности,
* распределение нагрузки между узлами,
* повышение отказоустойчивости,
* возможность независимого наращивания ресурсов.

Для PostgreSQL-хранилищ применяется *секционирование крупных таблиц*. В первую очередь данный подход используется для:

* Хранилища событий,
* БД резервирования,
* БД конфликтов,
* Хранилища временных рядов.

Секционирование выполняется по:

* временным диапазонам,
* идентификаторам магазинов,
* регионам,
* диапазонам SKU.

Использование секционирования позволяет:

* сократить объем обрабатываемых данных в запросах,
* ускорить обслуживание индексов,
* упростить архивирование,
* повысить эффективность удаления устаревших данных.

*Для разгрузки* основных PostgreSQL-хранилищ используются *реплики чтения*. Реплики применяются для:

* Сервиса сверки,
* аналитических запросов,
* административных операций,
* формирования отчетности,
* фоновых задач.

При этом все операции изменения данных выполняются только через основной узел соответствующего хранилища.

Распределенный кеш масштабируется путем *добавления новых узлов кластера*. Данные автоматически распределяются между узлами на основе механизма распределения слотов.

Такой подход позволяет:

* масштабировать производительность чтения практически линейно,
* локализовать горячие ключи,
* распределять нагрузку между узлами,
* выполнять масштабирование без остановки Сервиса.

*Основную нагрузку* Redis Cluster формируют:

* запросы доступности товаров,
* операции поиска данных модели чтения,
* массовые операции MGET,
* запросы популярных SKU.

Для Event Bus и Очереди ошибок масштабирование выполняется за счет *увеличения количества партиций топиков*.

Партиционирование используется для:

* распределения нагрузки между брокерами,
* параллельной обработки сообщений,
* увеличения пропускной способности,
* балансировки групп потребителей.

В качестве ключей партиционирования используются:

* SKU,
* комбинация SKU и магазина,
* идентификатор резерва,
* идентификатор операции сверки.

Такой подход позволяет сохранить порядок обработки событий для одного бизнес-объекта при одновременном масштабировании обработки на уровне всего Сервиса.

Для Хранилища модели чтения и Гео-индекса, координат масштабирование достигается за счет *распределения коллекций между узлами MongoDB Cluster*. Основными критериями распределения данных являются:

* регион,
* магазин,
* диапазон SKU.

Это позволяет равномерно распределять объем хранения и нагрузку между узлами кластера по мере роста количества магазинов, товаров и пользовательских запросов.

Выбранные подходы позволяют независимо масштабировать:

* транзакционный контур резервирования,
* модель чтения,
* распределенный кеш,
* потоковую обработку событий,
* геопространственные данные,
* технические и архивные хранилища,

что соответствует требованиям Сервиса по обработке до 50 000 RPS пользовательских запросов и до 20 000 EPS событий изменения остатков без необходимости масштабирования всей подсистемы хранения данных как единого монолита.

===== 2.13.3.2. Репликация и высокая доступность

Для обеспечения непрерывной работы Сервиса и минимизации риска потери данных *все критичные хранилища разворачиваются в многоузловой конфигурации* с *репликацией данных и автоматическим переключением на резервные узлы*.

Подходы к репликации различаются в зависимости от назначения хранилища и требований к консистентности данных.

Для БД резервирования, БД конфликтов, Метаданных, справочников, БД реестра схем и Хранилища событий используется схема : 1 основной узел записи + 1 резервный узел ожидания + 1 или 2е (или несколько) реплик чтения.

В данной конфигурации:

* все операции записи выполняются через основной узел,
* резервный узел поддерживает актуальную копию данных,
* реплики чтения используются для фоновых и аналитических операций.

Для данных резервирования используется синхронная репликация между основным и резервным узлом. Подтверждение транзакции выполняется только после успешной записи данных на оба узла.

Такой подход необходим для исключения потери:

* активных резервов,
* подтвержденных операций резервирования,
* результатов сверки,
* критичных справочных изменений.

Для реплик чтения используется асинхронная репликация. Это позволяет снизить задержки транзакций и уменьшить нагрузку на основной узел.

Для БД резервирования допустимое отставание реплик чтения не должно превышать:

* 1 сек в штатном режиме,
* 5 сек в пиковых сценариях.

Для Хранилища модели чтения и Гео-индекса, координат используется *встроенная репликация MongoDB Cluster*.

Каждый набор реплик включает : 1 основной узел + несколько вторичных узлов.

Основной узел выполняет операции записи, а вторичные узлы:

* поддерживают актуальные копии данных,
* используются для отказоустойчивости,
* могут обслуживать часть операций чтения.

Переключение на новый основной узел выполняется автоматически при потере текущего ведущего узла.

Для Распределенного кеша используется Redis Cluster с *репликацией каждого основного узла на отдельный резервный узел*.

При отказе узла:

* выполняется автоматическое назначение реплики новым основным узлом,
* производится перераспределение слотов,
* восстановление выполняется без остановки пользовательского обслуживания.

Поскольку Распределенный кеш не является источником истины, допускается потеря части временных данных кеша при аварийных сценариях. После восстановления данные автоматически восстанавливаются из Хранилища модели чтения.

Для Event Bus и Очереди ошибок используется *репликация партиций между брокерами Kafka*.

Для каждого топика:

* коэффициент репликации равен 3,
* минимальное количество синхронных реплик равно 2.

Подтверждение записи критичных сообщений выполняется только после сохранения сообщения в синхронных репликах.

Такой подход позволяет сохранить сообщения при отказе одного брокера и исключить потерю подтвержденных событий.

Для выбора ведущего узла в кластерах используется механизм кворума. Изменение роли узлов возможно только при наличии большинства доступных участников кластера.

Использование кворума позволяет:

* избежать появления нескольких ведущих узлов одновременно,
* предотвратить расхождение данных,
* сохранить целостность репликации при сетевых разделениях.

Автоматическое переключение на резервный узел применяется для : БД резервирования, конфликтов, реестра схем, Метаданных, справочников, Хранилища событий, модели чтения, Гео-индекса, координат, распределенного кеша.

Основными контролируемыми показателями репликации являются:

* отставание реплик,
* состояние резервных узлов,
* время переключения,
* количество синхронных реплик,
* количество недоступных узлов,
* скорость восстановления после отказа.

Целевые показатели высокой доступности приведены в таблице.

| БД резервирования | 99,95 % | До 30 секунд
| Хранилище событий | 99,95 % | До 30 секунд
| Хранилище модели чтения | 99,9 % | До 30 секунд

Использование многоузлового развертывания, репликации, автоматического переключения и кворумного выбора ведущих узлов позволяет обеспечить непрерывную работу Сервиса при отказе отдельных серверов, сетевых сегментов или экземпляров хранилищ без потери критичных бизнес-данных.

===== 2.13.3.3. Обеспечение консистентности данных

В архитектуре Сервиса используются различные механизмы обеспечения консистентности данных, поскольку операции резервирования, потоковая обработка событий и обновление модели чтения выполняются в разных компонентах и используют различные хранилища данных.

Наиболее строгие требования к консистентности предъявляются к:

* БД резервирования,
* Хранилищу событий,
* БД конфликтов.

Для данных компонентов недопустимы:

* потеря подтвержденного резерва,
* двойное резервирование товара,
* отрицательные остатки,
* нарушение порядка изменения состояния.

Для *защиты от повторного выполнения* бизнес-операций используется *идемпотентность запросов*. Каждая операция резервирования получает уникальный идентификатор идемпотентности, который сохраняется в БД резервирования.

При повторном поступлении запроса с тем же идентификатором:

* повторное резервирование не выполняется,
* клиенту возвращается результат ранее завершенной операции.

Дополнительно в Сервисе используется *дедупликация событий*. Для каждого события сохраняются:

* идентификатор события,
* идентификатор источника,
* версия состояния.

Повторно полученные события не приводят к повторному изменению состояния и отбрасываются на этапе обработки.

Для предотвращения конфликтов конкурентного изменения данных в БД резервирования используется *оптимистическая блокировка*.

При изменении записи контролируется версия состояния объекта. Если версия изменилась после чтения данных и до момента записи, операция отклоняется и выполняется повторная обработка.

*Оптимистическая блокировка* используется для:

* резервирования товара,
* изменения состояния резерва,
* обработки конфликтных операций,
* корректировки остатков.

*Механизм блокировки* применяется для:

* создания резервов,

В рамках одной транзакции:

* фиксируются изменения в БД резервирования,
* сохраняется запись исходящего события,
* подтверждается транзакция.

* резерв был сохранен, но событие не опубликовано,

===== 2.13.3.4. Обеспечение отсутствия потерь данных

Для критичных данных Сервиса применяется многоуровневая защита от потери информации. Защита строится на сочетании *надежной записи данных в хранилища, репликации, резервного копирования и возможности повторного восстановления состояния из сохраненных событий*.

Наиболее критичными данными являются:

* данные резервирования,
* события изменений остатков,
* события резервирования,
* данные Хранилища событий,
* данные БД конфликтов,
* Метаданные, справочники.

Для всех PostgreSQL-хранилищ используется *журнал предварительной записи*. Изменения сначала фиксируются в журнале предварительной записи и только после этого считаются подтвержденными.

Журнал предварительной записи обеспечивает:

* сохранность подтвержденных транзакций,
* восстановление после аварийного завершения работы,
* репликацию изменений на резервные узлы,
* восстановление на произвольный момент времени.

Для Event Bus защита данных обеспечивается *репликацией сообщений между брокерами*.

Основные параметры хранения событий:

* коэффициент репликации : 3,
* минимальное количество синхронных реплик : 2,
* подтверждение записи критичных событий : после записи на синхронные реплики,
* потеря подтвержденного события : не допускается.

Событие считается успешно записанным только после подтверждения от необходимого количества реплик. Такой подход исключает потерю подтвержденных сообщений при отказе отдельного брокера.

Для всех операций публикации событий используется механизм повторных попыток отправки. При временной недоступности получателя выполняется автоматическая повторная передача сообщения.

Если сообщение не может быть обработано после исчерпания установленного количества попыток, оно помещается в *Очередь ошибок*.

Очередь ошибок используется для:

* сообщений с ошибками обработки,
* сообщений с поврежденной структурой,
* сообщений, нарушающих правила валидации,
* сообщений, которые не удалось обработать после повторных попыток.

Использование Очереди ошибок позволяет:

* исключить потерю проблемных сообщений,
* предотвратить остановку основного потока обработки,
* обеспечить последующий анализ и повторную обработку.

Для всех критичных хранилищ *выполняется резервное копирование*.

.Политики резервного копирования
[cols="2,1,1",options="header"]
|===
| Компонент | Полная резервная копия | Журнал изменений

| Хранилище событий, БД резервирования, БД конфликтов, Метаданные, справочники, БД реестра схем | Ежесуточно | Непрерывно
| Хранилище модели чтения, Гео-индекс, координаты | Ежесуточно | Нет
|===

Резервные копии и архивы журналов изменений размещаются в Архивном хранилище и используются для:

* аварийного восстановления,
* восстановления после логических ошибок,
* восстановления после ошибочных изменений данных,
* восстановления отдельных объектов.

Дополнительно в Сервисе используются *снимки состояния производных хранилищ*.

Снимки состояния создаются для:

* Хранилища модели чтения,
* Распределенного кеша,
* Гео-индекса, координат.

Использование снимков позволяет сократить время восстановления после отказа и уменьшить объем последующей синхронизации.

Для восстановления данных также используется *повторное воспроизведение событий из Хранилища событий*.

Повторное воспроизведение применяется в следующих случаях:

* перестроение Хранилища модели чтения,
* восстановление Распределенного кеша,
* восстановление после ошибок обработки,
* пересчет производных данных,
* устранение последствий программных дефектов.

.Механизмы защиты от потери данных
[cols="1,3",options="header"]
|===
| Компонент | Основные механизмы защиты

| БД резервирования | Журнал предварительной записи, синхронная репликация, резервное копирование
| Хранилище событий | Журнал предварительной записи, синхронная репликация, резервное копирование
| Хранилище модели чтения | Репликация, резервные копии, повторное построение из событий
| Распределенный кеш | Репликация, снимки состояния, повторное наполнение из модели чтения
| Event Bus | Репликация сообщений, подтверждение записи, повторная отправка
| Очередь ошибок | Длительное хранение проблемных сообщений до устранения причины ошибки
|===
