=== 2.12. Форматы взаимодействия

## 2.12.1. Назначение раздела

Описать, что раздел фиксирует:

* форматы внешних API;
* форматы внутренних вызовов;
* структуру событий;
* правила версионирования;
* правила обработки ошибок;
* требования к идентификаторам, времени и трассировке запросов.

---

==== 2.12.2. Общие принципы проектирования API

Описать единые правила:

* все внешние API публикуются через API Gateway;
* формат обмена для REST API — JSON;
* все ответы содержат единый идентификатор запроса;
* ошибки возвращаются в едином формате;
* версии API указываются в пути;
* внутренние события содержат технические метаданные;
* операции резервирования должны быть идемпотентными.

---

## 2.12.3. REST API Сервиса запросов остатков

Этот подраздел закрывает требование по проектированию RESTful API для одного микросервиса.

Лучше выбрать именно **Сервис запросов остатков**, потому что он является центральным для пользовательского сценария.

---

### 2.12.3.1. Назначение API

Описать, что API используется для:

* проверки доступности товара;
* поиска магазинов с товаром;
* получения краткой информации об остатке;
* поддержки клиентских приложений, сотрудников магазина и служб доставки.

---

### 2.12.3.2. Перечень методов API

Рекомендуемая таблица:

[cols="1,2,2", options="header"]
|===
| Метод
| Путь
| Назначение

| GET
| /api/v1/availability
| Проверить доступность товара в заданном магазине или зоне поиска

| GET
| /api/v1/stores/nearby
| Найти ближайшие магазины с доступным товаром

| GET
| /api/v1/items/{item_id}/availability
| Получить доступность конкретного товара

| GET
| /api/v1/health
| Проверить техническое состояние сервиса
|===

## 2.12.10. Версионирование форматов

Описать правила:

* версия REST API указывается в пути;
* версия события указывается в поле event_version;
* изменения должны быть обратно совместимыми;
* удаление полей допускается только через новую основную версию;
* потребители событий должны игнорировать неизвестные поля.

---

## 2.12.11. Идентификаторы, время и трассировка

Описать обязательные технические поля:

* request_id — идентификатор внешнего запроса;
* correlation_id — идентификатор бизнес-цепочки;
* causation_id — идентификатор причины события;
* event_id — уникальный идентификатор события;
* created_at / occurred_at — время создания или наступления события.

Здесь важно связать раздел с мониторингом и разбором инцидентов.
