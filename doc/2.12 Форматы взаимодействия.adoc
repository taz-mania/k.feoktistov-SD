=== 2.12. Форматы взаимодействия

Настоящий раздел описывает форматы взаимодействия между компонентами Сервиса и правила передачи данных во внешнем и внутреннем контурах обработки.

Раздел фиксирует единые требования к структуре запросов, ответов, событий и служебных данных, используемых при взаимодействии компонентов распределённой архитектуры. Использование унифицированных форматов взаимодействия позволяет обеспечить совместимость компонентов, упростить сопровождение интеграций и снизить вероятность ошибок обработки данных.

В разделе определяются:

* форматы внешних REST API,
* форматы внутренних межсервисных вызовов,
* структура сообщений Event Bus,
* форматы потоковых уведомлений,
* правила версионирования контрактов взаимодействия,
* правила формирования ошибок,
* требования к идентификаторам запросов и событий,
* правила передачи временных меток,
* требования к трассировке распределённых запросов.

Отдельное внимание уделяется единообразию контрактов взаимодействия между компонентами Сервиса. Для всех типов взаимодействий используются согласованные подходы к:

* именованию полей,
* форматам идентификаторов,
* представлению времени,
* кодам ошибок,
* версиям контрактов,
* структуре служебных метаданных.

Дополнительно раздел определяет правила передачи технической информации, используемой для мониторинга, трассировки и анализа распределённых цепочек обработки запросов. Это необходимо для диагностики ошибок и контроля согласованности взаимодействия между компонентами Сервиса.

Описание внешних REST API в рамках данного раздела используется как основа для публикации OpenAPI-спецификаций и сопровождения интеграционных контрактов клиентских приложений и внешних систем.

==== 2.12.1. Общие принципы проектирования API
 
При проектировании интерфейсов взаимодействия в Сервисе используются единые правила формирования API и обмена данными между компонентами архитектуры. Применение общих принципов позволяет обеспечить единообразие контрактов взаимодействия, упростить сопровождение интеграций и уменьшить вероятность ошибок при развитии Сервиса.

Для внешнего REST API используется формат обмена JSON. Выбор JSON обусловлен его совместимостью с клиентскими платформами, удобством интеграции и простотой обработки в мобильных приложениях, web-клиентах и внешних системах.

Для обеспечения единообразия взаимодействия все ответы REST API содержат идентификатор запроса `request_id`. Данный идентификатор используется для:

* трассировки распределённых запросов,
* анализа ошибок,
* поиска операций в журналах обработки,
* сопоставления запросов и событий между компонентами.

Ошибки обработки запросов возвращаются в едином формате независимо от типа операции или внутреннего компонента, сформировавшего ошибку. Унификация структуры ошибок упрощает обработку исключительных ситуаций на стороне клиентских приложений и внешних интеграционных систем.

Версии REST API указываются непосредственно в пути запроса: `/api/*v1*/...`. Такой подход позволяет одновременно поддерживать несколько версий API без изменения маршрутизации клиентских приложений.

Использование явного версионирования позволяет одновременно поддерживать несколько поколений API и снижает риск нарушения совместимости при изменении контрактов взаимодействия.

Для внутренних событий и сообщений Event Bus дополнительно используются технические метаданные, обеспечивающие контроль обработки и трассировку распространения событий между компонентами Сервиса. В состав метаданных включаются:

* идентификатор события,
* тип события,
* версия схемы события,
* время формирования события,
* correlation_id,
* информация об источнике события.

Наличие технических метаданных позволяет контролировать последовательность обработки событий, обнаруживать дублирующие сообщения и выполнять анализ распределённых цепочек изменения состояния.

Отдельные требования предъявляются к операциям резервирования товара. Поскольку клиентские приложения и внешние системы могут повторно отправлять запросы при временных ошибках соединения или превышении времени ожидания ответа, операции резервирования должны быть идемпотентными.

Повторная отправка одного и того же запроса резервирования не должна приводить к:

* повторному созданию резерва,
* повторному уменьшению доступного остатка,
* дублирующей публикации событий изменения состояния.

Для этого в запросах резервирования используются уникальные идентификаторы операций, позволяющие обнаруживать повторную обработку одного и того же запроса.

==== 2.12.2. REST API Сервиса запросов остатков

В рамках Сервиса основным пользовательским сценарием является проверка доступности товара в магазинах торговой сети. Центральным компонентом, обеспечивающим выполнение данного сценария, является Сервис запросов.

По этой причине в качестве примера проектирования RESTful API в рамках настоящей работы рассматривается API Сервиса запросов. Данный подраздел фиксирует структуру REST API, используемого для получения информации о доступности товаров и поиска магазинов.

API Сервиса запросов проектируется с учётом следующих требований:

* высокая интенсивность пользовательских запросов,
* минимальное время ответа,
* поддержка мобильных клиентов,
* возможность горизонтального масштабирования,
* совместимость с внешними интеграциями,
* поддержка версионирования контрактов.

Основным ресурсом API является информация о доступности товара (`availability`). REST API строится вокруг операций получения состояния ресурса без хранения пользовательского состояния между запросами.

Для получения данных о доступности товара используется следующий маршрут:

`GET /api/v1/availability`

Запрос предназначен для получения информации о наличии товара в магазинах с учётом географического положения пользователя и текущего состояния модели чтения.

В запросе передаются:

[cols="1,1,3", options="header"]
|===
| Параметр | Тип | Назначение

| sku | string | Идентификатор товара
| latitude | number | Географическая широта
| longitude | number | Географическая долгота
| radius | integer | Радиус поиска магазинов
| limit | integer | Максимальное количество магазинов в ответе
|===

Пример запроса:

```http
GET /api/v1/availability?sku=123456&latitude=55.751&longitude=37.617&radius=5&limit=10
```

В ответе возвращается агрегированная информация о доступности товара в найденных магазинах.

Пример ответа:

```json
{
  "request_id": "req-10001",
  "sku": "123456",
  "stores": [
    {
      "store_id": "1001",
      "available_qty": 12,
      "distance_km": 1.8,
      "updated_at": "2026-05-10T10:15:00Z"
    },
    {
      "store_id": "1002",
      "available_qty": 4,
      "distance_km": 2.4,
      "updated_at": "2026-05-10T10:14:42Z"
    }
  ]
}
```

Для поиска ближайших магазинов без проверки конкретного товара используется отдельный маршрут:

`GET /api/v1/stores/nearby`

Такое разделение позволяет независимо использовать:

* географический поиск магазинов,
* проверку доступности товара,
* комбинированные сценарии пользовательского поиска.

При проектировании API используются стандартные принципы RESTful-взаимодействия.

Метод `GET` применяется для операций получения данных и не изменяет состояние Сервиса.

Параметры поиска передаются через строку запроса, поскольку операция не создаёт новые ресурсы и не требует хранения состояния между запросами.

Структура URL отражает тип ресурса и назначение операции:

* `/availability` — доступность товара,
* `/stores` — магазины,
* `/reservations` — операции резервирования.

Для операций получения данных используется постраничное или ограниченное получение результатов. Это необходимо для предотвращения формирования чрезмерно больших ответов при поиске товаров с широкой географией доступности.

Дополнительно API поддерживает передачу технических идентификаторов трассировки в заголовках запросов. Это позволяет сопоставлять пользовательские обращения с внутренними событиями обработки и упрощает анализ распределённых запросов.

Ошибки обработки запросов возвращаются в едином формате независимо от типа операции.

Пример ответа с ошибкой:

```json
{
  "request_id": "req-10001",
  "error_code": "SKU_NOT_FOUND",
  "message": "Товар не найден"
}
```

Для REST API Сервиса запросов используются следующие основные коды ответа:

* 200   Запрос успешно обработан,
* 400   Некорректные параметры запроса,
* 401   Ошибка аутентификации,
* 403   Доступ запрещён,
* 404   Ресурс не найден,
* 429   Превышение допустимой частоты запросов,
* 500   Внутренняя ошибка обработки.

===== 2.12.2.1. Назначение API

REST API Сервиса запросов предназначен для предоставления клиентским приложениям и внешним системам информации о доступности товаров в магазинах торговой сети.

Основной задачей API является обеспечение быстрого получения актуальных данных о наличии товара с минимальным временем ответа и поддержкой высокой интенсивности пользовательских запросов.

API используется для выполнения следующих операций:

* проверка доступности товара,
* поиск магазинов, в которых товар доступен,
* получение краткой информации об остатке товара,
* получение информации о расстоянии до магазина,
* отображение доступности товара в пользовательских интерфейсах.

Основным пользовательским сценарием является поиск товара рядом с текущим местоположением клиента. В рамках данного сценария API позволяет определить:

* наличие товара,
* ближайшие магазины,
* примерный доступный остаток,
* расстояние до точки продажи.

API используется несколькими категориями потребителей:

* МП My eDA, Web-клиентом My eDA, ПМ ТСД eDA,
* службами доставки,
* внешними интеграционными системами.

Для мобильных клиентских приложений API используется при отображении карточек товаров, поиске ближайших магазинов и выборе точки получения заказа.

Для сотрудников магазинов API используется при проверке доступности товара, обработке резервов и уточнении состояния остатков.

Для служб доставки и внешних интеграционных систем API используется как источник оперативной информации о возможности выполнения заказа в конкретном магазине.

Поскольку API ориентирован на операции чтения, его проектирование выполняется с учётом следующих требований:

* минимальное время ответа,
* высокая пропускная способность,
* поддержка массовых запросов,
* устойчивость при пиковых нагрузках,
* независимость от операций записи и резервирования.

Все операции API являются stateless и не требуют хранения пользовательского состояния между запросами. Это упрощает горизонтальное масштабирование Сервиса запросов и позволяет равномерно распределять нагрузку между экземплярами компонентов.

===== 2.12.2.2. Перечень методов API

REST API Сервиса запросов предоставляет набор операций чтения, используемых для получения информации о доступности товаров и поиска магазинов.

Основные методы API приведены в таблице ниже.

[cols="1,2,3", options="header"]
|===
| Метод | Путь | Назначение

| GET | `/api/v1/availability` | Получение информации о доступности товара в магазинах
| GET | `/api/v1/stores/nearby`| Поиск ближайших магазинов относительно местоположения пользователя
| GET | `/api/v1/stores/{store_id}/availability`| Получение доступности товара для конкретного магазина
| GET | `/api/v1/skus/{sku}` | Получение краткой информации о товаре и его доступности
| GET | `/api/v1/health` | Проверка доступности API и состояния обработки запросов
|===

Методы API ориентированы исключительно на операции чтения и не изменяют состояние Сервиса. Для всех операций получения данных используется метод `GET`.

Структура маршрутов строится вокруг логических ресурсов Сервиса:

* `availability` — информация о доступности товара,
* `stores` — магазины,
* `skus` — данные о товарах,
* `health` — техническое состояние API.

Использование отдельных маршрутов для различных сценариев позволяет упростить маршрутизацию запросов, ограничение нагрузки и сопровождение контрактов взаимодействия.

==== 2.12.3. OpenAPI-спецификация

Для формализации контрактов REST API в Сервисе используется спецификация OpenAPI. Использование OpenAPI позволяет зафиксировать структуру интерфейсов взаимодействия в машиночитаемом формате и обеспечить единообразное описание API для клиентских приложений, внешних интеграций и внутренних команд разработки.

Спецификация OpenAPI используется для:

* описания маршрутов API,
* фиксации параметров запросов,
* описания форматов ответов,
* документирования кодов ошибок,
* описания схем передаваемых данных,
* публикации технической документации API,
* автоматизации тестирования контрактов взаимодействия.

В рамках Сервиса OpenAPI-спецификация формируется для внешнего REST API, публикуемого через API Gateway.

Укрупнённая структура спецификации включает:

* общую информацию о сервисе,
* описание версий API,
* перечень маршрутов,
* параметры запросов,
* схемы объектов,
* описание кодов ответа,
* описание ошибок,
* требования к аутентификации,
* технические заголовки запросов.

В качестве базовой версии спецификации используется OpenAPI 3.x.

Ниже приведён пример укрупнённой структуры спецификации REST API Сервиса запросов:

```yaml
openapi: 3.0.0

info:
  title: Inventory Availability API
  version: 1.0.0
  description: API проверки доступности товаров

servers:
  - url: https://api.eda.example

paths:
  /api/v1/availability:
    get:
      summary: Проверка доступности товара

  /api/v1/stores/nearby:
    get:
      summary: Поиск ближайших магазинов

components:
  schemas:
    AvailabilityResponse:
      type: object

    ErrorResponse:
      type: object
```

Детализированное описание маршрутов, параметров запросов, схем объектов и форматов ответов приводится в последующих подразделах настоящего раздела.

===== 2.12.3.1. Общие сведения

Основные параметры API приведены в таблице ниже.

[cols="1,3", options="header"]
|===
| Параметр | Значение

| Название API | Inventory Availability API
| Назначение | Проверка доступности товаров и поиск магазинов
| Текущая версия API | v1
| Базовый путь API | `/api/v1`
| Формат обмена данными | JSON
| Транспортный протокол | HTTPS
| Способ авторизации | Bearer Token
| Точка публикации API | API Gateway
|===

Для всех запросов используется формат данных JSON с кодировкой UTF-8.

Запросы к API выполняются через HTTPS. Использование защищённого соединения необходимо для предотвращения перехвата пользовательских данных и технических идентификаторов запросов.

Авторизация выполняется с использованием токенов доступа, передаваемых в HTTP-заголовке `Authorization`.

Пример заголовка авторизации:

```http
Authorization: Bearer eyJhbGciOiJIUzI1NiIsI...
```

Здесь используется Bearer Token, обычно JWT (JSON Web Token). 

Структура JWT:

```text
header.payload.signature
```

1. *Заголовок токена `header`*:

```json
{
  "alg": "HS256",   // алгоритм подписи
  "typ": "JWT"      // тип токена
}
```
[start=2]
2. *Тело токена `payload`*: 

```json
{
  "sub": "user-123",   //идентификатор пользователя
  "role": "customer",  //роль пользователя
  "exp": 1778496000    //срок действия токена
}
```
[start=3]
3. *Подпись токена `signature`*

```text
Это просто бинарный результат криптографического алгоритма над заголовком и телом токена
```

  Все три части токена кодируются в Base64URL.

После успешной проверки токена запрос получает доступ к разрешённым операциям API.

Для трассировки распределённых запросов клиентское приложение может передавать идентификатор запроса в заголовке:

```http
X-Request-Id: req-10001
```

Если идентификатор отсутствует, он формируется на уровне API Gateway автоматически.

===== 2.12.3.2. Модели данных REST API

В REST API Сервиса запросов используются единые модели данных для описания входных параметров, успешных ответов, сведений о доступности товара и ошибок обработки. Применение общих моделей позволяет унифицировать ответы разных методов API и упростить сопровождение внешних контрактов.

Ниже приведены основные модели REST API.

*AvailabilityRequest*

Модель `AvailabilityRequest` описывает параметры запроса проверки доступности товара.

```json
{
  "sku": "123456",
  "latitude": 55.751,
  "longitude": 37.617,
  "radius": 5,
  "limit": 10
}
```

Поля модели:

[cols="0,0,1", options="header"]
|===
| Поле | Тип | Назначение

| sku | string | Идентификатор товара
| latitude | number | Географическая широта клиента
| longitude | number | Географическая долгота клиента
| radius | integer | Радиус поиска магазинов в километрах
| limit | integer | Максимальное количество магазинов в ответе
|===

*AvailabilityResponse*

Модель `AvailabilityResponse` описывает результат проверки доступности товара.

```json
{
  "request_id": "req-10001",
  "sku": "123456",
  "stores": []
}
```

==== 2.12.4. Формат внутренних gRPC-вызовов

Для внутреннего межсервисного взаимодействия в Сервисе используется gRPC поверх HTTP/2, который применяется для обмена данными между компонентами распределённой архитектуры, требующими минимальных задержек и высокой интенсивности вызовов.

gRPC используется для взаимодействия между:

* Сервисом запросов,
* Сервисом резервирования,
* Гео-сервисом,
* Загрузчиком моделей чтения,
* Сервисом сверки,
* компонентами обработки событий,
* инфраструктурными сервисами.

Основной причиной использования gRPC является необходимость эффективной передачи небольших сообщений между внутренними сервисами при высокой частоте вызовов. В отличие от внешнего REST API, внутренние вызовы ориентированы не на совместимость с клиентскими платформами, а на производительность и минимизацию сетевых накладных расходов.

Для сериализации данных используются protobuf-схемы. Применение protobuf позволяет:

* уменьшить объём передаваемых данных,
* ускорить сериализацию сообщений,
* формализовать внутренние контракты взаимодействия,
* обеспечить контроль версий схем.

Внутренние gRPC-вызовы используются для:

* получения данных о магазинах,
* проверки состояния резервирования,
* загрузки данных модели чтения,
* передачи технических команд,
* выполнения служебных операций синхронизации,
* обмена техническими метаданными.

Детализированное описание protobuf-схем, внутренних методов и форматов сообщений приводится в последующих подразделах настоящего раздела.

===== 2.12.4.1. Общие правила

Внутренние gRPC-вызовы в Сервисе строятся на основе строго формализованных контрактов взаимодействия. Для описания структуры сообщений используются protobuf-схемы, фиксирующие состав полей, типы данных и правила сериализации сообщений.

Использование строгих контрактов позволяет:

* исключить неоднозначность форматов данных,
* контролировать совместимость версий,
* автоматически проверять корректность структуры сообщений,
* упростить сопровождение межсервисного взаимодействия.

Передача данных между сервисами выполняется в бинарном формате protobuf. По сравнению с текстовыми форматами обмена бинарная сериализация уменьшает объём передаваемых сообщений и снижает сетевые накладные расходы при высокой интенсивности внутренних вызовов.

Каждый внутренний вызов обязан содержать технические идентификаторы трассировки. Передача идентификаторов трассировки необходима для восстановления полной цепочки обработки распределённых запросов и анализа ошибок взаимодействия между компонентами.

Все внутренние вызовы содержат технические идентификаторы трассировки, включая:

* идентификатор запроса `request_id`,
* идентификатор сквозной трассировки запросов и событий между компонентами системы `correlation_id`,
* идентификатор источника запроса,
* временную метку формирования вызова.

Это позволяет связывать внутренние вызовы с внешними пользовательскими запросами и событиями Event Bus.

Для всех внутренних вызовов используется контроль времени ожидания ответа. Каждый gRPC-запрос выполняется с заранее установленным timeout-интервалом, ограничивающим максимально допустительное время ожидания ответа от зависимого сервиса.

Контроль времени ожидания используется для:

* предотвращения зависания вызовов,
* ограничения накопления очередей запросов,
* уменьшения вероятности каскадной деградации,
* ускорения обнаружения проблемных компонентов.

При превышении допустимого времени ожидания вызов завершается ошибкой, а вызывающий сервис принимает решение о повторной попытке, переключении на резервный сценарий обработки или возврате ошибки вызывающему компоненту.

Для внутренних gRPC-контрактов используются единые правила:

* явное описание схем protobuf,
* обязательное версионирование protobuf-схем,
* обязательное версионирование контрактов,
* единые правила именования сервисов и сообщений,
* запрет несовместимого изменения обязательных полей,
* единый формат технических метаданных,
* единые коды ошибок внутренних вызовов.

===== 2.12.4.2. Примеры внутренних операций

Ниже приведены примеры основных внутренних операций.

*Получение доступности по списку товаров*

Операция используется Сервисом запросов для пакетного получения информации о доступности нескольких товаров одновременно.

Пример protobuf-описания:

```protobuf
service AvailabilityService {
  rpc GetAvailabilityBatch (AvailabilityBatchRequest)
      returns (AvailabilityBatchResponse),
}

message AvailabilityBatchRequest {
  repeated string sku = 1,
  string store_id = 2,
  string request_id = 3,
}

message AvailabilityBatchResponse {
  repeated StoreAvailability availability = 1,
}
```

Операция используется для:

* ускоренной загрузки карточек товаров,
* отображения результатов поиска,
* пакетной обработки пользовательских запросов.

*Получение доступности по списку магазинов*

Операция используется после выполнения географического поиска магазинов для получения информации о наличии товара в найденных точках продаж.

Пример protobuf-описания:

```protobuf
service StoreAvailabilityService {
  rpc GetAvailabilityByStores (StoreAvailabilityRequest)
      returns (StoreAvailabilityResponse),
}

message StoreAvailabilityRequest {
  string sku = 1,                 // товар 
  repeated string store_ids = 2,  // идентификатор[ы] магазина[ов]
  string request_id = 3,          // идентификатор запроса 
}

message StoreAvailabilityResponse {
  repeated StoreAvailability stores = 1,  // идентификатор[ы] магазина[ов], где товар присутствует
}
```

Операция позволяет сократить количество отдельных внутренних вызовов при обработке запросов поиска ближайших магазинов.

*Проверка технического состояния зависимого компонента*

Операция используется для контроля доступности внутренних сервисов и проверки готовности компонентов к обработке запросов.

Пример protobuf-описания:

```protobuf
service HealthService {
  rpc CheckHealth (HealthRequest)
      returns (HealthResponse),
}

message HealthRequest {
  string request_id = 1,   // идентификатор запроса 
}

message HealthResponse {
  string status = 1,       // статус общего состояния API
  string timestamp = 2,    // штамп времени статуса
}
```

Результаты проверки используются:

* серерами API Gateway,
* балансировщиками нагрузки,
* средствами мониторинга,
* механизмами обнаружения сервисов.

*Получение статуса резерва*

Операция используется для проверки текущего состояния ранее созданного резерва товара.

Пример protobuf-описания:

```protobuf
service ReservationService {
  rpc GetReservationStatus (ReservationStatusRequest)
      returns (ReservationStatusResponse),
}

message ReservationStatusRequest {
  string reservation_id = 1,   // идентификатор резерва
  string request_id = 2,       // идентификатор запроса
}

message ReservationStatusResponse {
  string reservation_id = 1,   // идентификатор резерва
  string status = 2,           // статус резерва
  string updated_at = 3,       // время обновления статуса резерва
}
```

Операция применяется:

* при подтверждении заказа,
* при повторной проверке состояния резервирования,
* при обработке ошибок пользовательских операций,
* при синхронизации состояния резервов между компонентами архитектуры.

==== 2.12.5. Формат WebSocket-сообщений

WebSocket-соединения используются в Сервисе для сценариев, требующих оперативной доставки изменений состояния без постоянного повторного опроса REST API.

Использование потоковых соединений необходимо в ситуациях, когда клиентское приложение должно получать изменения состояния практически в режиме реального времени.

В рамках Сервиса WebSocket-канал применяется для следующих сценариев:

* изменение состояния резервирования,
* обновление статуса обработки заказа,
* уведомление об истечении времени резерва,
* изменение доступности товара в процессе оформления заказа,
* доставка технических уведомлений клиентскому приложению.

Наиболее важным сценарием является обновление состояния резервирования товара. После создания резерва клиентское приложение должно оперативно получать информацию о следующих событиях:

* подтверждение резерва,
* отклонение операции,
* изменение статуса обработки,
* автоматическое освобождение резерва,
* ошибки обработки операции.

Использование WebSocket позволяет избежать постоянного повторного выполнения REST-запросов проверки состояния резерва и уменьшает нагрузку на Сервис запросов и Сервис резервирования.

Дополнительно WebSocket-соединения используются в ситуациях, когда состояние доступности товара может изменяться во время пользовательского сценария. Например, остаток товара может измениться в процессе оформления заказа другим пользователем. В подобных случаях клиентское приложение получает уведомление об изменении состояния без необходимости повторной проверки через REST API.

Детализированное описание назначения WebSocket-канала и структуры сообщений приводится в последующих подразделах настоящего раздела.

===== 2.12.5.1. Назначение WebSocket-канала

WebSocket-канал используется в Сервисе для передачи потоковых уведомлений клиентским приложениям и внутренним интерфейсам мониторинга без необходимости постоянного повторного выполнения REST-запросов.

Основной областью применения WebSocket-взаимодействия являются сценарии, в которых изменение состояния должно отображаться клиенту практически в режиме реального времени.

Одним из ключевых сценариев является обновление статуса резерва товара. После создания резерва пользовательское приложение должно получать уведомления о:

* подтверждении резерва,
* изменении состояния обработки,
* отклонении операции,
* автоматическом истечении времени резерва,
* освобождении зарезервированного остатка.

Использование WebSocket-канала позволяет передавать изменения состояния сразу после обработки события резервирования без необходимости периодического опроса API.

Дополнительно WebSocket-соединения используются в операторских интерфейсах и ПМ ТСД eDA для отображения изменений доступности товара.

В подобных сценариях WebSocket-канал используется для:

* отображения изменения остатка,
* обновления доступности товара,
* уведомления о поступлении товара,
* отображения изменений состояния резервирования,
* обновления состояния заказа.

Это особенно важно для популярных SKU, состояние которых может изменяться с высокой частотой.

Отдельным сценарием использования WebSocket-канала является отображение технических событий на панелях мониторинга и операционного контроля.

Через потоковые уведомления могут передаваться:

* события деградации компонентов,
* изменение состояния Event Bus,
* превышение времени обработки,
* накопление очередей сообщений,
* ошибки обработки резервирования,
* изменение состояния сервисов.

Использование потоковых уведомлений позволяет оперативно отображать изменения состояния инфраструктуры без постоянного опроса технических API мониторинга.

Для различных категорий сообщений используются отдельные типы событий и независимые каналы подписки. Это позволяет разграничить:

* пользовательские уведомления,
* операторские события,
* технические сообщения мониторинга,
* уведомления резервирования.

Подключение к WebSocket-каналу выполняется через API Gateway после успешной проверки доступа и идентификации клиента. Тип доступных уведомлений определяется категорией клиента и разрешёнными сценариями использования API.

===== 2.12.5.2. Общая структура сообщения

Для WebSocket-взаимодействия в Сервисе используется единый формат сообщений, обеспечивающий совместимость клиентских приложений с различными типами потоковых уведомлений.

Все WebSocket-сообщения передаются в формате JSON и содержат:

* тип сообщения,
* идентификатор запроса,
* время формирования сообщения,
* технические идентификаторы трассировки,
* полезную нагрузку события.

Кроме основных, сообщения содержат набор обязательных технических полей, необходимых для: 

* идентификации события, 
* трассировки обработки, 
* контроля версии формата сообщения.

Общая структура WebSocket-сообщения приведена ниже.

```json
{
  "event_type": "reservation_updated",
  "event_id": "evt-50001",
  "request_id": "req-10001",
  "timestamp": "2026-05-10T10:15:00Z",
  "version": "v1",
  "payload": {}
}
```

Описание полей сообщения приведено в таблице ниже.

[cols="0,0,1", options="header"]
|===
| Поле | Тип | Назначение

| event_type | string | Тип события или уведомления. Для определения типа поступившего уведомления
| event_id | string | Уникальный идентификатор события. Позволяет обнаруживать повторную доставку событий
| request_id | string | Идентификатор исходного запроса. Используется для связывания потокового уведомления с исходным пользовательским запросом или операцией резервирования
| timestamp | string | Время формирования сообщения в формате UTC
| version | string | Версия формата сообщения. Контроль совместимости клиентских приложений с различными версиями формата сообщений
| payload | object | Полезные данные конкретного события. Структура данного поля зависит от типа сообщения
|===

В зависимости от значения `event_type` клиентское приложение выбирает соответствующий сценарий обработки сообщения.

Примеры типов событий:

* `reservation_created` -- резерв создан,
* `reservation_updated` -- резерв обновлен,
* `reservation_expired` -- резерв закончился,
* `availability_changed` -- доступность изменилась,
* `order_status_updated` -- статус заказа обновлен.

Пример уведомления об изменении состояния резерва:

```json
{
  "event_type": "reservation_updated",   // тип сообщения : обновление резерва
  "event_id": "evt-50001",               // идентификатор события
  "request_id": "req-10001",             // идентификатор запроса
  "timestamp": "2026-05-10T10:15:00Z",   // штамп времени сообщения
  "version": "v1",                       // версия формата сообщения
  "payload": {                           // состав передаваемых данных сообщения
    "reservation_id": "res-70001",       // идентификатор резерва
    "status": "CONFIRMED",               // статус резерва : резерв подтвержден
    "store_id": "1001",                  // идентификатор магазина
    "sku": "123456"                      // зарезервированный товар
  }
}
```

Использование единой структуры сообщений позволяет унифицировать обработку WebSocket-уведомлений в клиентских приложениях и упростить сопровождение потокового взаимодействия между компонентами Сервиса.

==== 2.12.6. Форматы событий Kafka

Взаимодействие между компонентами Сервиса в асинхронном контуре обработки выполняется через Event Bus на основе Apache Kafka. Передача событий используется для распространения изменений состояния между независимыми компонентами архитектуры без необходимости прямого синхронного взаимодействия.

События Kafka являются основным механизмом распространения изменений:

* остатков товаров,
* состояния резервирования,
* результатов сверки,
* технических уведомлений,
* изменений модели чтения.

Использование событийной модели позволяет разделить процессы фиксации изменений и их последующей обработки различными компонентами Сервиса.

В рамках Сервиса события Kafka используются:

* Загрузчиком моделей чтения,
* Сервисом резервирования,
* Сервисом сверки,
* компонентами мониторинга,
* обработчиками технических операций,
* сервисами аналитической обработки.

Для сериализации сообщений используются Avro-схемы. Применение Avro позволяет:

* уменьшить размер сообщений,
* формализовать структуру событий,
* обеспечить контроль совместимости схем,
* поддерживать версионирование контрактов,
* упростить обработку событий различными потребителями.

Каждое событие Kafka содержит:

* бизнес-данные изменения состояния,
* технические идентификаторы,
* временные метки,
* информацию об источнике события,
* версию схемы события,
* данные трассировки распределённых операций.

Единый подход к структуре событий необходим для:

* контроля последовательности обработки,
* повторной обработки сообщений,
* обнаружения дублирующих событий,
* восстановления модели чтения,
* анализа распределённых цепочек обработки.

События публикуются в отдельные топики Kafka в зависимости от типа изменения состояния и назначения обработки.

В рамках Сервиса используются отдельные категории событий:

* события изменения остатков,
* события резервирования, отмены резервов,
* события результатов сверки,
* технические события мониторинга,
* события восстановления состояния.

Для предотвращения нарушения последовательности обработки события секционируются по комбинации SKU + магазин. Такой подход позволяет сохранить корректный порядок изменения состояния конкретного товара в рамках отдельного магазина.

Контракты событий Kafka являются частью интеграционного контура Сервиса и подлежат обязательному версионированию. Изменение структуры событий выполняется с сохранением совместимости существующих обработчиков.

Детализированное описание структуры событий и примеры сообщений приводятся в последующих подразделах настоящего раздела.

===== 2.12.6.1. Общая структура события

Для всех событий Kafka в Сервисе используется единая структура сообщения, содержащая как бизнес-данные изменения состояния, так и технические метаданные обработки.

Единый формат событий необходим для:

* унификации обработки сообщений,
* контроля совместимости схем,
* трассировки распределённых операций,
* повторной обработки событий,
* восстановления модели чтения,
* анализа причин возникновения изменений состояния.

Все события передаются в формате Avro и содержат обязательный набор технических полей.

Общая структура события приведена ниже.

```json
{
  "event_id": "evt-10001",
  "event_type": "inventory_updated",
  "event_version": "v1",
  "occurred_at": "2026-05-10T10:15:00Z",
  "source": "reservation_service",
  "correlation_id": "corr-50001",
  "causation_id": "cmd-70001",
  "payload": {}
}
```

Описание основных полей события: 

[cols="0,0,1", options="header"]
|===
| Поле | Тип | Назначение

| event_id | string | Уникальный идентификатор события. Применяется для обнаружение повторной обработки сообщений
| event_type | string | Тип события. Для выбора соответствующего сценария обработчиками 
| event_version | string | Версия схемы события. Контроль совместимости Avro-схем и поддержки нескольких версий контрактов событий
| occurred_at | string | Время формирования события. Время возникновения бизнес-события в формате UTC
| source | string | Компонент-источник события. Определяет сервис или компонент, сформировавший событие
| correlation_id | string | Идентификатор распределённой цепочки обработки. Связывает несколько событий и внутренних операций в рамках одной распределённой цепочки обработки
| causation_id | string | Идентификатор исходной операции или команды. Структура поля зависит от типа события и определяется соответствующей Avro-схемой
| payload | object | Полезные данные события. Содержит бизнес-данные конкретного события
|===

Примеры типов событий:

* `inventory_updated`
* `reservation_created`
* `reservation_expired`
* `reservation_cancelled`
* `reconciliation_completed`

Использование единых технических полей позволяет выполнять централизованную обработку событий, упрощает мониторинг потоков данных и обеспечивает сквозную трассировку изменений состояния между компонентами Сервиса.

===== 2.12.6.2. Событие изменения остатка

Событие изменения остатка используется для распространения информации об изменении доступного количества товара между компонентами Сервиса.

Данный тип события является одним из наиболее интенсивных потоков обработки, поскольку формируется при:

* поступлении новых данных из кассовых систем,
* изменении складских остатков,
* создании резервов,
* отмене резервирования,
* корректировке состояния товара,
* выполнении операций сверки.

Событие публикуется в топик `inventory.events` и используется:

* Загрузчиком моделей чтения,
* Сервисом сверки,
* компонентами мониторинга,
* обработчиками аналитики,
* сервисами обновления Распределенного кэша.

Основная задача события — передача актуального состояния доступности товара для конкретного магазина.

Пример события изменения остатка:

```json
{
  "event_id": "evt-10001",
  "event_type": "inventory_updated",
  "event_version": "v1",
  "occurred_at": "2026-05-10T10:15:00Z",
  "source": "inventory_adapter",
  "correlation_id": "corr-50001",
  "causation_id": "cmd-70001",
  "payload": {
    "sku": "123456",
    "store_id": "1001",
    "available_qty": 12,
    "reserved_qty": 3,
    "change_source": "pos",
    "updated_at": "2026-05-10T10:14:58Z"
  }
}
```

Описание бизнес-полей события:

[cols="0,0,1", options="header"]
|===
| Поле | Тип | Назначение

| sku | string | Идентификатор товара
| store_id | string | Идентификатор магазина
| available_qty | integer | Доступное количество товара для продажи или резервирования
| reserved_qty | integer | Количество товара, уже участвующего в активных операциях резервирования
| change_source | string | Определение источника изменения состояния товара
| updated_at | string | Фактическое время изменения остатка. Используется для контроля порядка обработки событий и разрешения конфликтов синхронизации
|===

Примеры значений поля `change_source`:

* `pos` -- изменение после кассовой операции,
* `warehouse` -- изменение после складской операции,
* `reservation` -- изменение после резервирования,
* `reconciliation` -- изменение после сверки данных.

События изменения остатка секционируются по комбинации SKU + магазин для сохранения корректной последовательности обработки изменений состояния конкретного товара.

===== 2.12.6.3. Событие создания резерва

[cols="0,0,1", options="header"]

| sku | string | Идентификатор товара

| status | string | Текущее состояние резерва
| request_source | string | Источник пользовательского запроса
|===

Примеры значений поля `status`:

* `CREATED`
* `CONFIRMED`

* `EXPIRED`

Примеры значений поля `request_source`:

События резервирования используются для последующего обновления модели чтения и перерасчёта доступного остатка товара в конкретном магазине.

===== 2.12.6.5. Событие результата сверки

Событие формируется Сервисом сверки после выполнения проверки состояния остатков и используется:

* компонентами мониторинга,

* сервисами аналитики,
* техническими панелями контроля,
* обработчиками повторной синхронизации модели чтения.

Публикация событий сверки необходима для обнаружения:

* расхождений между моделью чтения и источниками данных,
* ошибок обработки событий,
* пропущенных обновлений,
* конфликтов резервирования,
* нарушения последовательности обработки изменений состояния.

{
  "event_id": "evt-40001",

Описание бизнес-полей события:

[cols="0,0,1", options="header"]

==== 2.12.7. Форматы задач для обработки одним потребителем

Для сценариев P2P (Point-to-Point) в Сервисе используются отдельные форматы задач, предназначенные для обработки одним экземпляром потребителя.

В отличие от событий публикации и подписки, такие сообщения не должны одновременно обрабатываться несколькими компонентами. Каждая задача назначается только одному обработчику и выполняется в рамках отдельного фонового процесса.

Формат задач используется для:

* операций сверки остатков,
* пересчёта модели чтения,
* повторной обработки ошибочных событий,
* технических проверок качества данных,
* восстановления состояния после деградации компонентов,
* фоновых служебных операций.

Задачи публикуются в отдельные очереди обработки и обрабатываются независимыми группами потребителей.

Для всех типов задач используется единая структура сообщения.

Пример задачи:

```json
{
  "task_id": "task-50001",
  "task_type": "rebuild_read_model",
  "created_at": "2026-05-10T11:00:00Z",
  "priority": "HIGH",
  "attempt_number": 1,
  "timeout": 300,
  "status": "NEW",
  "payload": {
    "sku": "123456",
    "store_id": "1001"
  }
}
```

Описание полей задачи приведено в таблице ниже.

[cols="0,0,1", options="header"]
|===
| Поле | Тип | Назначение

| task_id | string | Уникальный идентификатор задачи
| task_type | string | Тип задачи обработки определяет сценарий обработки задачи
| created_at | string | Время создания задачи
| priority | string | Приоритет обработки задачи используется для управления порядком обработки задач
| attempt_number | integer | Номер текущей попытки выполнения задачи используется для контроля повторных запусков
| payload | object | Полезные данные задачи зависят от типа задачи и содержат бизнес-данные, необходимые для выполнения конкретной операции обработки
| timeout | integer | Максимальное допустимое время выполнения задачи в секундах. При превышении данного значения задача может быть принудительно остановлена или повторно назначена другому обработчику
| status | string | Текущее состояние задачи
|===

Примеры значений поля `task_type`:

* `inventory_reconciliation`
* `rebuild_read_model`
* `retry_event_processing`
* `data_quality_check`

Примеры значений поля `priority`:

* `LOW`
* `NORMAL`
* `HIGH`
* `CRITICAL`

Примеры значений поля `status`:

* `NEW`
* `IN_PROGRESS`
* `COMPLETED`
* `FAILED`
* `RETRY`
* `TIMEOUT`

Для задач для обработки одним потребителем дополнительно контролируются:

* количество повторных попыток,
* время ожидания обработки,
* количество зависших задач,
* количество задач с ошибками,
* длительность выполнения операций.

Это позволяет локализовать проблемы фоновой обработки и предотвращать накопление необработанных задач в очередях обработки.

==== 2.12.8. Единый формат ошибок

Для REST API, внутренних gRPC-вызовов и асинхронных операций обработки в Сервисе используется единый формат представления ошибок.

Унификация структуры ошибок необходима для:

* единообразной обработки ошибок клиентскими приложениями,
* упрощения межсервисного взаимодействия,
* трассировки распределённых запросов,
* централизованного мониторинга ошибок,
* анализа причин отказов и деградаций.

Единый формат ошибок используется:

* во внешнем REST API,
* во внутренних gRPC-вызовах,
* при обработке задач P2P,
* при публикации технических событий,
* в потоковых WebSocket-уведомлениях.

Общая структура ошибки приведена ниже.

```json
{
  "error_code": "SKU_NOT_FOUND",
  "message": "Товар не найден",
  "technical_details": "SKU 123456 отсутствует в модели чтения",
  "request_id": "req-10001",
  "timestamp": "2026-05-10T10:15:00Z",
  "retryable": false
}
```

Описание полей ошибки приведено в таблице ниже.

[cols="0,0,1", options="header"]
|===
| Поле | Тип | Назначение

| error_code | string | Машиночитаемый код ошибки используется для автоматизированной обработки ошибок клиентскими приложениями и внутренними сервисами
| message | string | Сообщение для клиента содержит краткое описание ошибки, предназначенное для отображения пользователю или оператору
| technical_details | string | Техническое описание ошибки используется для диагностики причин ошибки и содержит дополнительную техническую информацию для журналирования и мониторинга
| request_id | string | Идентификатор запроса позволяет связать ошибку с конкретным запросом или распределённой цепочкой обработки
| timestamp | string | Время возникновения ошибки
| retryable | boolean | Признак возможности повторной отправки запроса определяет возможность повторной отправки запроса без изменения входных параметров
|===

Для Сервиса используются несколько основных групп ошибок.

Примеры различных групп ошибок:

[cols="1,1,1,1", options="header"]
|===
| Пример ошибки | error_code | message | retryable

| Ошибка поиска товара | SKU_NOT_FOUND | Товар не найден | false
| Ошибка поиска магазина | STORE_NOT_FOUND | Магазин не найден | false
| Ошибка недоступности остатка | AVAILABILITY_TEMPORARILY_UNAVAILABLE | Информация о доступности временно недоступна | true
| Ошибки резервирования | RESERVATION_NOT_AVAILABLE | Резервирование невозможно | false
| Ошибки ограничения нагрузки | RATE_LIMIT_EXCEEDED | Превышено допустимое количество запросов | true
| Ошибки временной недоступности компонентов | SERVICE_UNAVAILABLE | Компонент временно недоступен | true
|===

Пример:

```json 
{
  "error_code": "SERVICE_UNAVAILABLE",
  "message": "Компонент временно недоступен",
  "retryable": true
}
```

Для внутренних взаимодействий дополнительно могут использоваться технические коды ошибок, связанные:

* с превышением времени ожидания,
* ошибками сериализации,
* нарушением совместимости схем,
* ошибками обработки событий,
* нарушением порядка обработки сообщений.

Единый формат ошибок используется во всех контурах взаимодействия Сервиса и является частью общих контрактов обмена данными между компонентами архитектуры.

==== 2.12.9. Версионирование форматов

Для всех взаимодействий в рамках Сервиса используются единые правила версионирования форматов взаимодействия.

Версионирование необходимо для:

* развития контрактов без нарушения совместимости,
* одновременной поддержки нескольких поколений клиентов,
* постепенного обновления компонентов,
* безопасного изменения структуры сообщений,
* снижения риска отказов при обновлении сервисов.

Для внешнего REST API версия указывается непосредственно в пути запроса.

Пример:

```http
GET /api/v1/availability
```

Использование версии в URL позволяет:

* одновременно поддерживать несколько версий API,
* выполнять поэтапную миграцию клиентов,
* изолировать несовместимые изменения,
* упростить маршрутизацию запросов.

При внесении изменений в REST API допускается:

* добавление новых необязательных полей,
* добавление новых маршрутов,
* расширение допустимых значений параметров,
* добавление новых кодов ошибок.

Изменения, нарушающие совместимость существующих клиентов, допускаются только в рамках новой основной версии API.

Для событий Kafka версия контракта указывается в поле `event_version`.

Пример:

```json
{
  "event_type": "inventory_updated",
  "event_version": "v2"
}
```

Поле `event_version` используется обработчиками событий для выбора корректной схемы десериализации и логики обработки сообщения.

Для Avro-схем событий используются правила обратной совместимости. Допускается:

* добавление новых необязательных полей,
* расширение структуры payload,
* добавление новых типов событий,
* расширение технических метаданных.

Удаление существующих полей или изменение их типа допускается только при публикации новой основной версии контракта.

Для внутренних protobuf-контрактов gRPC используются аналогичные правила совместимости. Изменение обязательных полей без выпуска новой версии схемы запрещается.

Потребители событий и внутренних сообщений должны корректно обрабатывать неизвестные поля. При получении новой версии сообщения необязательные неизвестные поля должны игнорироваться без завершения обработки ошибкой.

Такой подход позволяет:

* независимо обновлять сервисы,
* выполнять постепенное внедрение новых функций,
* уменьшить связанность компонентов,
* снизить риск отказов при обновлении контрактов.

Для контроля совместимости схем в Сервисе используется централизованная проверка контрактов перед публикацией новых версий сообщений и API.

==== 2.12.10. Идентификаторы, время и трассировка

Для всех типов взаимодействий в Сервисе используются единые технические идентификаторы и временные метки, обеспечивающие сквозную трассировку распределённых операций между компонентами архитектуры.

Использование единых идентификаторов необходимо для:

* анализа распределённых запросов,
* мониторинга межсервисного взаимодействия,
* расследования инцидентов,
* диагностики ошибок обработки,
* восстановления последовательности событий,
* сопоставления пользовательских запросов и внутренних операций.

Технические идентификаторы передаются:

* во внешних REST API,
* во внутренних gRPC-вызовах,
* в событиях Kafka,
* в задачах Point-to-Point,
* в WebSocket-сообщениях,
* в журналах обработки и мониторинга.

В Сервисе используются следующие обязательные технические поля.

*request_id*

Поле `request_id` используется для идентификации конкретного внешнего пользовательского запроса.

Идентификатор формируется:

* клиентским приложением,
* API Gateway,
* внешней интеграционной системой.

`request_id` передаётся между всеми компонентами, участвующими в обработке запроса, и используется для:

* поиска операции в журналах,
* анализа ошибок,
* сопоставления запросов и ответов,
* восстановления цепочки обработки пользовательского сценария.

Пример:

```json
{
  "request_id": "req-10001"
}
```

*correlation_id*

Поле `correlation_id` используется для объединения нескольких связанных операций в рамках одной бизнес-цепочки обработки.

Один пользовательский запрос может инициировать:

* внутренние gRPC-вызовы,
* публикацию нескольких событий Kafka,
* операции резервирования,
* обновление модели чтения,
* потоковые уведомления.

Все связанные операции получают единый `correlation_id`.

Пример:

```json
{
  "correlation_id": "corr-50001"
}
```

Использование `correlation_id` особенно важно при анализе распределённых инцидентов и поиске причин деградации обработки.

*causation_id*

Поле `causation_id` используется для указания причины возникновения текущего события или операции.

Поле содержит идентификатор:

* команды,
* исходного запроса,
* события,
* фоновой задачи.

Это позволяет восстанавливать причинно-следственную цепочку распространения изменений состояния между компонентами Сервиса.

Пример:

```json
{
  "causation_id": "evt-20001"
}
```

*event_id*

Поле `event_id` используется для уникальной идентификации события Kafka или внутреннего технического сообщения.

`event_id` применяется для:

* обнаружения повторной обработки,
* контроля идемпотентности,
* анализа распространения событий,
* поиска сообщений в Event Bus,
* расследования ошибок обработки событий.

Пример:

```json
{
  "event_id": "evt-30001"
}
```

*created_at* / *occurred_at*

Поля `created_at` и `occurred_at` используются для фиксации времени создания запроса, задачи или события.

Поле `created_at` обычно используется для:

* REST-запросов,
* фоновых задач,
* технических сообщений.

Поле `occurred_at` используется для:

* бизнес-событий,
* событий Kafka,
* изменений состояния.

Все временные значения передаются в формате UTC.

Пример:

```json
{
  "created_at": "2026-05-10T10:15:00Z",
  "occurred_at": "2026-05-10T10:15:02Z"
}
```

Использование временных меток необходимо для:

* анализа последовательности обработки,
* контроля задержек распространения событий,
* обнаружения устаревших данных,
* расчёта времени обработки операций,
* анализа деградаций и инцидентов.

Технические идентификаторы и временные метки являются обязательной частью распределённой трассировки Сервиса и используются средствами мониторинга, журналирования и анализа инцидентов для восстановления полной цепочки обработки пользовательских операций.

==== 2.12.11. Требования к безопасности форматов взаимодействия

Для всех форматов взаимодействия в Сервисе используются единые требования информационной безопасности, направленные на защиту пользовательских данных, предотвращение несанкционированного доступа и снижение риска компрометации внутренних компонентов архитектуры.

1. Все внешние и внутренние взаимодействия выполняются только по защищённым каналам связи.

+
Для REST API и WebSocket-соединений используется HTTPS/TLS.
Для внутренних gRPC-вызовов используется защищённое взаимодействие между сервисами с проверкой доверенных компонентов.

+
Использование защищённых соединений необходимо для:

* предотвращения перехвата запросов,
* защиты идентификаторов авторизации,
* защиты технических идентификаторов трассировки,
* предотвращения подмены сообщений,
* снижения риска атак на внутренние сервисы.

2. В событиях Kafka и внутренних сообщениях запрещается передача чувствительных пользовательских данных, не требуемых для обработки бизнес-операции.

+
В частности, в событиях не должны передаваться:

* пароли,
* токены авторизации,
* платёжные данные,
* полные персональные данные клиентов,
* технические секреты инфраструктуры.

3. События Kafka должны содержать только минимально необходимый набор данных, требуемый для обработки изменений состояния между компонентами Сервиса.

4. Для журналов обработки и мониторинга используется маскирование персональных данных и технических идентификаторов ограниченного доступа.

+
Маскированию подлежат:

* номера телефонов,
* адреса электронной почты,
* токены доступа,
* идентификаторы пользовательских сессий,
* служебные ключи и секреты.

+
Это необходимо для предотвращения компрометации данных при анализе журналов, мониторинга и расследовании инцидентов.

5. Проверка прав доступа для внешнего REST API выполняется на уровне API Gateway до передачи запроса внутренним сервисам.

+
На уровне API Gateway выполняются:

* проверка токена доступа,
* проверка срока действия авторизации,
* определение типа клиента,
* контроль разрешённых маршрутов,
* ограничение частоты запросов,
* фильтрация запрещённых операций.

6. Внутренние API и gRPC-интерфейсы недоступны напрямую из внешнего контура и используются только для межсервисного взаимодействия внутри инфраструктуры Сервиса.

+
Доступ к внутренним API ограничивается:

* сетевой изоляцией,
* правилами маршрутизации,
* проверкой доверенных сервисов,
* разграничением сервисных ролей,
* контролем технических сертификатов и идентификаторов.

7. Дополнительно для форматов взаимодействия используются следующие требования безопасности:

* обязательное версионирование контрактов,
* контроль совместимости схем,
* ограничение размера сообщений,
* контроль времени жизни соединений,
* ограничение времени ожидания ответа,
* журналирование ошибок доступа и нарушений политик безопасности.

Требования безопасности распространяются на REST API, gRPC-вызовы, события Kafka, задачи P2P и WebSocket-взаимодействия между всеми компонентами Сервиса.

==== 2.12.12. Требования к наблюдаемости взаимодействий

Для всех типов взаимодействий в Сервисе обеспечивается обязательная регистрация технических данных, необходимых для мониторинга, анализа производительности, диагностики ошибок и расследования инцидентов.

Требования наблюдаемости распространяются на:

* REST API,
* внутренние gRPC-вызовы,
* события Kafka,
* WebSocket-сообщения,
* задачи P2P,
* фоновые операции обработки.

Все компоненты Сервиса обязаны передавать единый набор технических метаданных в журналы обработки, системы мониторинга и средства распределённой трассировки.

Для внешних и внутренних запросов в журналах должны фиксироваться:

* идентификатор запроса (`request_id`),
* идентификатор бизнес-цепочки (`correlation_id`),
* вызываемый метод или маршрут,
* вызываемый или целевой компонент,
* код ответа или результат обработки,
* длительность обработки запроса,
* время начала и завершения обработки,
* количество повторных попыток выполнения,
* технический статус операции.

Для REST API дополнительно фиксируются:

* HTTP-метод,
* код ответа,
* маршрут API,
* тип клиента,
* размер запроса и ответа,
* результат проверки доступа.

Для внутренних gRPC-вызовов регистрируются:

* имя gRPC-метода,
* время выполнения вызова,
* количество повторных подключений,
* превышение времени ожидания,
* размер protobuf-сообщения.

Для событий Kafka и фоновых задач дополнительно фиксируются:

* идентификатор события (`event_id`),
* тип события (`event_type`),
* размер сообщения,
* номер попытки обработки,
* результат обработки события,
* время публикации,
* время фактической обработки,
* отставание обработки от времени публикации.

Для задач P2P контролируются:

* время ожидания обработки,
* длительность выполнения задачи,
* количество повторных запусков,
* причины ошибок обработки,
* переходы состояний задачи.

Результат обработки события или запроса должен содержать информацию о завершении операции:

* успешно,
* ошибка обработки,
* повторная попытка,
* превышение timeout,
* отклонение по ограничению доступа,
* отклонение по ограничению нагрузки.

Дополнительно в Сервисе регистрируются события деградации взаимодействий:

* рост времени ответа,
* накопление очередей обработки,
* увеличение количества ошибок,
* потеря соединения с зависимыми компонентами,
* превышение времени распространения событий.

Собираемые данные используются:

* средствами мониторинга,
* системами централизованного журналирования,
* распределённой трассировкой,
* панелями операционного контроля,
* механизмами анализа инцидентов.

Единый подход к наблюдаемости взаимодействий позволяет восстанавливать полную цепочку обработки пользовательских операций, анализировать причины деградации компонентов и контролировать стабильность распределённой архитектуры Сервиса.
