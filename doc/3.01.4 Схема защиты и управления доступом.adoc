==== 3.1.4 Защита передачи и хранения данных

===== 3.1.4.1 Защита сетевых соединений

Для защиты сетевого взаимодействия в Сервисе используется обязательное шифрование трафика между внешними клиентами, внутренними микросервисами, интеграционными компонентами и инфраструктурными сервисами платформы.

Защита соединений распространяется на:

* REST API,
* WebSocket-подключения,
* gRPC-взаимодействие,
* подключение к Event Bus, PostgreSQL, MongoDB, Redis Cluster,
* взаимодействие с HashiCorp Vault,
* интеграционные подключения внешних сервисов.

Передача *незашифрованного трафика внутри платформы не допускается*.

*1. Использование TLS 1.3*

Для всех внешних и внутренних соединений используется TLS 1.3.

TLS 1.3 применяется:

* между клиентскими приложениями и API Gateway,
* между внутренними микросервисами,
* при gRPC-вызовах,
* при подключении к Event Bus,
* при подключении к HashiCorp Vault,
* при взаимодействии интеграционных адаптеров с внутренними API.

Использование TLS 1.3 обеспечивает:

* шифрование передаваемых данных,
* защиту JWT access-токенов и refresh-токенов,
* защиту сервисных сертификатов,
* защиту служебных gRPC-вызовов,
* защиту потоков событий и интеграционного трафика.

API Gateway выполняет завершение внешних TLS-соединений и обеспечивает проверку корректности защищенного подключения до передачи запросов во внутренний контур платформы.

*2. Запрет устаревших протоколов*

В Сервисе запрещено использование устаревших и небезопасных версий транспортных протоколов.

Не допускается использование:

* SSL,
* TLS 1.0, 1.1,
* устаревших алгоритмов согласования ключей,
* небезопасных режимов шифрования.

Соединения, пытающиеся использовать неподдерживаемые версии TLS или небезопасные параметры подключения, отклоняются API Gateway и внутренними сервисами до передачи данных на уровень приложений.

Для внутренних сервисов и инфраструктурных компонентов применяются единые политики безопасности транспортного уровня.

*3. Использование современных наборов шифров*

Для TLS-соединений используются современные криптографические наборы шифров, обеспечивающие защиту пользовательских и служебных данных, такие как:

* `TLS_AES_128_GCM_SHA256`,
* `TLS_AES_256_GCM_SHA384`,
* `TLS_CHACHA20_POLY1305_SHA256`,
* `TLS_AES_128_CCM_SHA256`,
* `TLS_AES_128_CCM_8_SHA256`.

Используемые наборы шифров обеспечивают:

* защищенный обмен ключами,
* шифрование трафика,
* контроль целостности передаваемых данных,
* устойчивость к атакам на транспортный уровень.

При установлении соединений запрещается использование:

* устаревших симметричных алгоритмов,
* небезопасных режимов блочного шифрования,
* алгоритмов без контроля целостности,
* слабых криптографических параметров.

Политики использования наборов шифров централизованно применяются:

* в API Gateway,
* во внутренних сервисах,
* в Kafka-кластере,
* в PostgreSQL,
* в MongoDB,
* в HashiCorp Vault.

*4. Защита от атак на транспортный уровень*

Архитектура Сервиса предусматривает защиту от атак, направленных на перехват, подмену или нарушение сетевого взаимодействия.

Защита транспортного уровня включает:

* обязательное использование TLS 1.3,
* использование mTLS для внутренних сервисов,
* проверку доверенных сертификатов,
* контроль срока действия сертификатов,
* проверку криптографической подписи JWT access-токенов,
* ограничение небезопасных параметров соединений.

Для внутренних сервисов дополнительно используется взаимная аутентификация через mTLS, предотвращающая подключение неподтвержденных компонентов к внутреннему контуру платформы.

Особое внимание уделяется защите:

* запросов резервирования,
* административных API,
* WebSocket-подключений,
* потоков Event Bus,
* сервисных gRPC-вызовов,
* операций доступа к HashiCorp Vault.

Для предотвращения атак повторной отправки запросов операции резервирования дополнительно используют:

* идентификатор зпроса,
* контроль идемпотентности,
* проверку уникальности операций.

События нарушения TLS-соединений, использования недоверенных сертификатов, ошибок транспортного уровня и попыток использования неподдерживаемых протоколов фиксируются в журналах аудита безопасности и используются для выявления аномальной активности.

===== 3.1.4.2 Защита публичных интерфейсов Сервиса

Сервис предоставляет публичные API для выполнения операций проверки доступности товаров, поиска ближайших магазинов и резервирования продукции. Указанные интерфейсы доступны из внешних сетей и являются потенциальной целью автоматизированных атак и злоупотреблений.

Для защиты внешнего периметра Сервиса перед API Gateway используется специализированный сервис защиты публичных интерфейсов, предоставляемый внешним поставщиком услуг информационной безопасности (MSSP).

В состав средств защиты входят:

* сервис защиты от распределенных атак отказа в обслуживании -- AntiDDoS,
* сервис защиты от автоматизированных программ и вредоносных роботов -- AntiBot,
* межсетевой экран уровня веб-приложений -- WAF.

Сервис *AntiDDoS* используется для защиты публичных API от попыток исчерпания сетевых и вычислительных ресурсов Сервиса за счет большого количества запросов. Использование данного механизма позволяет сохранить доступность функций проверки остатков и поиска магазинов даже при значительном объеме вредоносного трафика.

Сервис *AntiBot* применяется для выявления и ограничения автоматизированной активности, не связанной с нормальным использованием Сервиса. Контроль подобных обращений позволяет снизить нагрузку на инфраструктуру и предотвратить массовый сбор данных о товарах и магазинах.

Сервис *WAF* используется для анализа входящих HTTP-запросов и блокирования обращений, содержащих признаки эксплуатации уязвимостей веб-приложений, обхода механизмов безопасности или передачи вредоносных данных.

Использование указанных средств обеспечивает дополнительный уровень защиты перед API Gateway и снижает риск нарушения доступности публичных функций Сервиса, компрометации данных и злоупотребления открытыми интерфейсами.

===== 3.1.4.3 Дополнительные механизмы защиты канала

Помимо базового шифрования TLS 1.3 в Сервисе используются дополнительные механизмы защиты транспортного уровня, направленные на предотвращение перехвата трафика, подмены соединений и компрометации криптографических ключей.

*1. Использование HSTS*

Для внешних HTTP-интерфейсов Сервиса используется механизм HSTS принудительного использования HTTPS-соединений, предотвращающий использование небезопасного HTTP и атаки понижения уровня защиты соединения.

API Gateway передает клиентским приложениям заголовки HSTS, запрещающие использование незащищенного HTTP-соединения при повторных подключениях.

Использование HSTS позволяет:

* предотвратить принудительное переключение клиента на HTTP,
* снизить риск атак понижения уровня защиты соединения,
* исключить передачу JWT access-токенов и refresh-токенов по незащищенному каналу,
* обеспечить использование HTTPS для всех пользовательских API.

HSTS применяется:

* для пользовательских REST API,
* для административных API,
* для WebSocket-подключений через API Gateway.

Внутренние сервисные соединения используют только HTTPS/gRPC и не поддерживают незашифрованный HTTP-трафик.

*2. Использование Perfect Forward Secrecy*

Для TLS-соединений используются наборы шифров с поддержкой Perfect Forward Secrecy (PFS), обеспечивающий независимость сессионных ключей отдельных соединений и снижающий риск компрометации ранее переданного трафика при утечке долгосрочных криптографических ключей.
PFS использует уникальные временные сеансовые ключе для каждого TLS-соединения, предотвращая расшифровку ранее переданного трафика при компрометации постоянных ключей сервера: сертификата и закрытого ключа сервера.

В Сервисе запрещается использование наборов шифров без поддержки Perfect Forward Secrecy.

*3. Контроль сертификатов*

Во всех защищенных соединениях выполняется обязательная проверка сертификатов.

Проверка сертификатов включает:

. проверку цепочки доверия,
. проверку срока действия,
. проверку доверенного удостоверяющего центра,
. проверку назначения сертификата,
. проверку соответствия имени сервиса,
. проверку статуса отзыва сертификата.

Для внутренних сервисов используются отдельные сервисные сертификаты и доверенная внутренняя инфраструктура сертификатов.
Сервисы доверяют только сертификатам, выпущенным доверенным удостоверяющим центром платформы.

Соединения с сертификатами, которые:

* просрочены, недоверены,
* с нарушенной цепочкой доверия,
* неподходящего назначения,

отклоняются до передачи данных на уровень приложений.

*4. Защита от атак посредника (MitM)*

Архитектура Сервиса предусматривает защиту от атак посредника при внешнем и внутреннем взаимодействии компонентов -- атак Man in the Middle (MitM).

Защита обеспечивается за счет:

* обязательного использования TLS 1.3,
* применения mTLS для внутренних сервисов,
* проверки доверенных сертификатов,
* проверки имени сервиса,
* использования механизмов PFS и HSTS,
* запрета незашифрованных соединений.

Для внутренних gRPC-вызовов используется взаимная аутентификация сервисов через mTLS. Это предотвращает возможность подключения неподтвержденного компонента к внутреннему контуру платформы и исключает подмену внутренних сервисов.

Особое внимание уделяется защите:

* операций резервирования,
* административных API,
* сервисных gRPC-вызовов,
* потоков Event Bus,
* операций доступа к HashiCorp Vault,
* передачи JWT access-токенов и refresh-токенов.

При обнаружении ошибок TLS-проверки, нарушений проверки сертификатов или попыток использования недоверенных соединений запросы отклоняются, а события фиксируются в журналах аудита безопасности.

===== 3.1.4.4 Шифрование и подписание данных

Помимо защиты транспортного уровня в Сервисе используются дополнительные механизмы шифрования, криптографической подписи и контроля целостности данных.

Дополнительная защита применяется для критичных данных, служебных сообщений, токенов доступа и межсервисного взаимодействия, компрометация которых может привести к несанкционированному доступу, подмене операций резервирования или нарушению целостности потоков событий.

*1. Данные, требующие дополнительной защиты*

Дополнительная криптографическая защита применяется для:

* JWT access-токенов, refresh-токенов, административных токенов,
* сервисных сертификатов,
* учетных данных PostgreSQL, MongoDB, Redis Cluster,
* секретов HashiCorp Vault,
* служебных интеграционных ключей,
* конфиденциальных параметров конфигурации.

Дополнительно защищаются:

* события `reservation.events`,
* служебные события Event Bus,
* административные запросы,
* операции управления доступом,
* операции работы с секретами и сертификатами.

Секреты, ключи и учетные данные не размещаются:

* в исходном коде,
* в контейнерных образах,
* в открытых конфигурационных файлах,
* в журналах приложений.

При автоматическом обнаружении подобных исключений в процесе тестирования кода, образов онтейнеров, конфигурационнх файлов или аудите журналов, в журналах аудита безопасности эти события фиксируются.

*2. Электронная подпись сообщений*

JWT access-токены подписываются криптографическим ключом внешнего поставщика удостоверений.
Токены с нарушенной подписью или измененным содержимым отклоняются до передачи запроса внутренним сервисам.

Для Event Bus дополнительно контролируется подлинность поставщика событий и подписчитка через сервисную аутентификацию и доверенные сертификаты.

Критичные служебные сообщения и административные операции проходят проверку подлинности вызывающей стороны до выполнения бизнес-логики.

*3. Защита служебных токенов и секретов*

Служебные токены, сервисные учетные данные и криптографические ключи хранятся в HashiCorp Vault и передаются сервисам только по защищенным каналам.

Для защиты служебных токенов используются:

* ограниченный срок действия,
* сервисная аутентификация,
* ротация секретов,
* отдельные сервисные учетные записи,
* аудит операций доступа.

Внутренние сервисы получают только собственные секреты и не имеют универсального доступа к хранилищу секретов.

Например:

* Сервис резервирования получает только собственные учетные данные подключения,
* сервис запросов не имеет доступа к секретам Kafka producer,
* интеграционные сервисы получают только собственные интеграционные ключи,
* сервисы без необходимости работы с секретами не получают доступа к HashiCorp Vault.

При отзыве сервисной роли или компрометации компонента выполняется отзыв или ротация связанных секретов и сертификатов.

*4. Контроль целостности данных*

Контроль целостности выполняется для пользовательских запросов, межсервисного взаимодействия и потоков событий.

Контроль включает проверку :

* криптографической подписи JWT,
* целостности TLS-соединений,
* подлинности сервисных сертификатов,

и контроль: 

* последовательности обработки событий,
* идемпотентности операций резервирования,
* корректности идентификатора запроса. 

Особое внимание уделяется защите:

* `reservation.events`, `inventory.events`,
* операций резервирования, административных операций,
* межсервисных gRPC-вызовов.

Для предотвращения нарушения целостности потоков данных используется:

* последовательная обработка событий для одного SKU,
* контроль порядка обработки событий,
* проверка допустимости поставщика событий и подписчика,
* ограничение прав публикации в топики Event Bus.

При обнаружении нарушений целостности данных, ошибок подписи, попыток подмены сообщений или некорректных служебных токенов операции отклоняются, а события фиксируются в журналах аудита безопасности.

===== 3.1.4.5 Криптографические алгоритмы

В Сервисе используются современные криптографические алгоритмы, обеспечивающие защиту пользовательских и служебных данных, аутентификацию компонентов платформы, контроль целостности сообщений и безопасность межсервисного взаимодействия.

Криптографические алгоритмы применяются:

* при защите TLS-соединений,
* при подписании JWT access-токенов,
* при работе и защите сервисных и служебных секретов,
* при хранении и передаче конфиденциальных данных,
* при контроле целостности сообщений и событий.

Использование устаревших и *небезопасных алгоритмов в Сервисе запрещено*.

*1. Использование AES-256-GCM*

Для симметричного шифрования данных используется алгоритм AES-256-GCM.

Использование режима GCM обеспечивает :

* контроль целостности,
* защиту от модификации сообщений,
* устойчивость к атакам на транспортный уровень.

Для TLS-соединений используются современные наборы шифров TLS 1.3, ранее определенные в подразделе защиты сетевых соединений.

*2. Использование SHA-256 и SHA-512*

Алгоритмы SHA-256 и SHA-512 используются для:

* контроля целостности данных,
* криптографической подписи JWT access-токенов,
* проверки целостности сообщений,
* формирования и проверки цифровых подписей.

SHA-256 используется как *основной алгоритм* контроля целостности и подписи в пользовательских и межсервисных сценариях.

SHA-512 используется для отдельных *операций повышенной критичности*, связанных с административными и инфраструктурными компонентами платформы.

Использование *устаревших хеш-алгоритмов*, включая MD5 и SHA-1, в Сервисе *запрещено*.

*3. Использование RSA-3072 и ECC*

Для работы сервисных сертификатов, TLS-соединений и криптографической подписи используются алгоритмы RSA-3072 и алгоритмы на эллиптических кривых (ECC).

RSA-3072 применяется:

* для сертификатов внешних API,
* для административных соединений,
* для инфраструктурных сертификатов,
* для совместимости с интеграционными компонентами.

ECC применяется:

* для внутренних сервисных соединений,
* для mTLS,
* для высоконагруженных gRPC-вызовов,
* для мобильных клиентских подключений,
* для снижения вычислительной нагрузки при сохранении криптографической стойкости.

Сервисные сертификаты выпускаются только доверенной инфраструктурой сертификатов платформы и используются для взаимной аутентификации компонентов.

Использование *слабых криптографических ключей* и *неподдерживаемых алгоритмов запрещено*.

*4. Алгоритмы подписи JWT*

JWT access-токены подписываются криптографическими алгоритмами, обеспечивающими проверку подлинности и целостности токена:

* RS256,
* RS512,
* ES256.

Выбор алгоритма зависит от типа клиента, требований производительности и используемой инфраструктуры сертификатов.

Токены:

* с нарушенной подписью,
* с неподдерживаемым алгоритмом,
* с измененным содержимым,
* подписанные недоверенным ключом,

отклоняются до передачи запроса во внутренние сервисы.

*5. Ограничения использования криптографических алгоритмов*

В Сервисе *запрещено использование*:

* SSL, TLS 1.0, 1.1,
* RC4, 3DES, DES,
* MD5, SHA-1,
* CBC-наборов шифров,
* криптографических ключей недостаточной длины,
* неподдерживаемых алгоритмов подписи JWT.

Контроль допустимых алгоритмов выполняется:

* API Gateway,
* внутренними сервисами,
* Kafka-кластером,
* HashiCorp Vault,
* PostgreSQL, MongoDB,
* механизмами сервисной аутентификации.

Попытки использования неподдерживаемых криптографических алгоритмов фиксируются в журналах аудита безопасности и рассматриваются как события нарушения политик безопасности.

==== 3.1.5 Управление секретами и криптографическими материалами

===== 3.1.5.1 Архитектура хранения секретов

Для хранения секретов, сервисных учетных данных, сертификатов и криптографических ключей в Сервисе используется централизованное защищенное хранилище HashiCorp Vault.

HashiCorp Vault используется как единая точка управления секретами платформы и обеспечивает:

* централизованное хранение секретов,
* разграничение доступа к секретам,
* безопасную выдачу учетных данных,
* хранение сервисных сертификатов,
* управление криптографическими ключами,
* аудит операций доступа.

Использование *локального хранения секретов* внутри сервисов или контейнерных образов в Сервисе *запрещено*.

*1. Использование HashiCorp Vault*

HashiCorp Vault используется для хранения:

* учетных данных PostgreSQL, MongoDB, Redis Cluster, Kafka,
* сервисных сертификатов,
* криптографических ключей,
* ключей проверки подписи JWT-токенов,
* интеграционных секретов,
* административных токенов,
* конфиденциальных параметров конфигурации.

Внутренние сервисы получают секреты динамически при запуске или в процессе работы через защищенное TLS/mTLS-взаимодействие.

Секреты не размещаются:

* в исходном коде,
* в Git-репозиториях,
* в контейнерных образах,
* в открытых конфигурационных файлах,
* в переменных окружения без защиты,
* в журналах приложений.

*2. Централизованное хранение секретов*

Централизованное хранение в HashiCorp Vault используется для:

* единообразного управления секретами,
* централизованной ротации ключей,
* ограничения распространения секретов,
* контроля доступа,
* аудита операций безопасности.

Внутренние сервисы *не хранят постоянные локальные копии секретов* и получают только те учетные данные, которые требуются для выполнения их функций.

Например:

* Сервис запросов получает только учетные данные чтения MongoDB и Redis Cluster,
* Сервис резервирования получает только учетные данные БД резервирования и Kafka producer,
* загрузчик модели чтения получает доступ только к MongoDB и разрешенным Kafka consumer group,
* интеграционные сервисы получают только собственные интеграционные ключи.

Универсальный доступ ко всем секретам платформы отсутствует.

*3. Разграничение доступа к секретам*

Доступ к секретам реализуется на основе сервисных ролей и политик безопасности.

Каждый сервис использует:

* собственную сервисную учетную запись,
* собственный сервисный сертификат,
* отдельную политику доступа HashiCorp Vault.

Политики Vault определяют:

* перечень доступных секретов,
* допустимые операции,
* срок действия выдаваемых учетных данных,
* ограничения использования секретов.

Для административных операций доступа к секретам дополнительно применяются:

* ограничение ролей,
* аудит операций,
* сокращенный срок действия административных сессий,
* контроль критичных действий.

*4. Аудит операций с секретами*

Все операции работы с секретами фиксируются в журналах аудита безопасности:

* получение, изменение, отзыв секретов,
* выдача сервисных сертификатов,
* попытки несанкционированного доступа,
* изменение политик Vault,
* операции административного доступа.

Журналы аудита используются для:

* расследования инцидентов безопасности,
* выявления аномальной активности,
* контроля использования сервисных учетных данных,
* контроля соблюдения политик доступа.

Особое внимание уделяется операциям доступа к:

* ключам проверки JWT access-токенов,
* сервисным сертификатам,
* учетным данным Kafka, PostgreSQL,
* административным секретам,
* ключам интеграционных сервисов.

Попытки доступа к неразрешенным секретам фиксируются как *события безопасности повышенного уровня критичности*.

===== 3.1.5.2 Хранение ключей, сертификатов и паролей

Хранение сертификатов, сервисных учетных данных, криптографических ключей и паролей подключения в Сервисе выполняется централизованно через HashiCorp Vault.

Архитектура хранения секретов направлена на:

* предотвращение компрометации учетных данных,
* ограничение распространения секретов между сервисами,
* централизованную ротацию ключей и паролей,
* аудит операций доступа,
* исключение хранения секретов в открытом виде.

Секреты и ключи не размещаются:

* в исходном коде,
* в Git-репозиториях,
* в контейнерных образах,
* в открытых конфигурационных файлах,
* в журналах приложений.

Выявление нарушения этого принципа приводит к компрометации СЕрвиса и его компонентов.

*1. Хранение сертификатов TLS*

В HashiCorp Vault хранятся:

* внутренние сервисные TLS-сертификаты, mTLS,
* доверенные сертификаты внутренних удостоверяющих центров,
* сертификаты для Kafka,
* сертификаты административных API,
* сертификаты интеграционных подключений.

Сервисные сертификаты используются:

* для gRPC-взаимодействия,
* для mTLS между сервисами,
* для подключения к Event Bus,
* для защищенного доступа к HashiCorp Vault,
* для административных соединений.

Каждый сервис использует отдельный сертификат и отдельный закрытый ключ.

Сервисы получают сертификаты через защищенные механизмы доставки секретов после прохождения сервисной аутентификации.

*2. Хранение сервисных учетных записей*

Для каждого внутреннего сервиса используются отдельные сервисные учетные записи и отдельные секреты подключения.

В HashiCorp Vault хранятся:

* сервисные токены,
* учетные данные Kafka поставщико в событий и подписчиков,
* учетные данные интеграционных сервисов,
* сервисные секреты межсервисного взаимодействия,
* административные сервисные токены.

Каждый сервис получает только собственные учетные данные и не имеет доступа к секретам других компонентов платформы.

*3. Хранение паролей подключения к БД*

Пароли и учетные данные подключения к PostgreSQL, MongoDB и Redis Cluster хранятся централизованно в HashiCorp Vault.

Для каждого сервиса используются отдельные учетные записи БД с минимально необходимыми полномочиями.

HashiCorp Vault хранит учетные данные:

* БД резервирования,
* модели чтения,
* Redis Cluster,
* БД прогрева кеша,
* БД конфликтов,
* хранилища событий,
* хранилища временных рядов.

Сервисы не используют единые универсальные учетные записи подключения.

Например:

* Сервис запросов использует только учетную запись чтения,
* Сервис резервирования использует учетную запись записи в БД резервирования,
* загрузчик модели чтения использует отдельную учетную запись обновления MongoDB.

*4. Хранение ключей проверки JWT access-токенов*

Поскольку JWT access-токены выпускаются внешним поставщиком удостоверений, Сервис не хранит приватные ключи подписи JWT.
Закрытые ключи подписи JWT находятся исключительно у внешнего поставщика удостоверений и не размещаются во внутреннем контуре Сервиса.

В HashiCorp Vault хранятся:

* доверенные ключи проверки подписи JWT access-токенов или JWKS-кеш/набор доверенных ключей проверки,
* доверенные сертификаты внешнего поставщика удостоверений,
* параметры проверки JWT,
* данные доверенной инфраструктуры проверки подписи.

API Gateway использует данные ключи и сертификаты для проведения проверок. 

*5. Ограничение доступа к ключам и секретам*

Доступ к секретам регулируется политиками HashiCorp Vault и сервисными ролями.

Политики доступа определяют:

* перечень разрешенных секретов,
* допустимые операции,
* срок действия сервисных токенов,
* ограничения использования сертификатов,
* разрешенные сервисные сценарии.

Особое ограничение применяется для:

* административных сертификатов,
* учетных данных Kafka,
* сертификатов mTLS,
* доверенных ключей проверки JWT access-токенов,
* интеграционных секретов.

Все операции получения, изменения и отзыва секретов фиксируются в журналах аудита безопасности и используются для контроля безопасности платформы и расследования инцидентов.

===== 3.1.5.3 Ротация секретов

В Сервисе реализуется централизованная ротация секретов, сертификатов и сервисных учетных данных через HashiCorp Vault.

Ротация используется для:

* ограничения времени использования секретов,
* снижения риска длительной компрометации учетных данных,
* предотвращения повторного использования утекших секретов,
* обновления криптографических материалов без остановки платформы,
* поддержания актуальности сервисных сертификатов и ключей.

Ротации подлежат:

* сервисные пароли подключения, сертификаты,
* сертификаты mTLS,
* учетные данные PostgreSQL, MongoDB, Redis Cluster, Kafka,
* временные учетные данные,
* сервисные токены, административные токены,
* интеграционные секреты,
* доверенные ключи и сертификаты проверки JWT access-токенов.

*1. Автоматическая ротация секретов*

HashiCorp Vault используется как централизованный механизм автоматической ротации секретов и сервисных учетных данных.

При автоматической ротации выполняется:

. выпуск нового секрета или сертификата,
. обновление записи в HashiCorp Vault,
. предоставление обновленного секрета сервису,
. переключение сервиса на новый секрет,
. отзыв устаревшего секрета.

Сервисы не используют жестко зафиксированные долгоживущие секреты.

*2. Сроки действия секретов*

Для всех типов секретов устанавливаются ограниченные сроки действия.

Ограниченный срок действия применяется для:

* сервисных и административных токенов,
* сертификатов mTLS,
* учетных данных подключения,
* интеграционных секретов.

*Наиболее короткие сроки действия* используются для:

* административных учетных данных,
* сервисных токенов,
* временных учетных записей,
* критичных инфраструктурных компонентов.

Для пользовательских JWT access-токенов используются *короткоживущие токены*, а для поддержания пользовательской сессии применяются refresh-токены.

Доверенные ключи проверки JWT access-токенов и сертификаты внешнего поставщика удостоверений также подлежат периодическому обновлению при изменении криптографических материалов внешнего поставщика.

*3. Отзыв скомпрометированных секретов*

Архитектура Сервиса предусматривает возможность оперативного отзыва секретов и сервисных учетных данных при обнаружении признаков компрометации.

Отзыв выполняется в случаях:

* утечки секретов,
* компрометации сервисного сертификата,
* компрометации учетной записи,
* нарушения политик безопасности,
* выявления аномальной активности,
* вывода сервиса из эксплуатации.

При отзыве выполняется:

. блокировка использования секрета,
. выпуск нового секрета,
. отзыв старых сертификатов,
. завершение активных сервисных сессий,
. обновление сервисных учетных данных,
. фиксация события в журналах аудита безопасности.

Для сервисов Event Bus дополнительно может выполняться:

* отзыв сертификатов и учетных данных Kafka,
* блокировка поставщиков событий,
* блокировка consumer group,
* ограничение доступа подписчиков к топикам.

*4. Обновление секретов без остановки сервисов*

Обновление большинства секретов и сертификатов выполняется без остановки платформы и без полного отключения сервисов.

Для этого используются:

* ограниченный срок действия секретов,
* поэтапная ротация сертификатов,
* поддержка одновременного действия старого и нового секрета,
* повторное получение секретов сервисами,
* обновление соединений без полной остановки приложения.

Без остановки сервисов выполняется обновление:

* сервисных сертификатов, mTLS-сертификатов,
* учетных данных Kafka, PostgreSQL, MongoDB,
* обновление доверенных ключей проверки JWT access-токенов.

Особое внимание уделяется обновлению:

* сервисов резервирования,
* API Gateway,
* Event Bus,
* HashiCorp Vault,
* сервисных gRPC-соединений.

*При ротации* доверенных ключей проверки JWT access-токенов API Gateway поддерживает *одновременную проверку токенов с использованием старых и новых доверенных ключей* до завершения переходного периода.

Все операции ротации, обновления и отзыва секретов фиксируются в журналах аудита безопасности и используются для контроля безопасности платформы и расследования инцидентов.

===== 3.1.5.4 Защита конфигурационных данных

Конфигурационные данные Сервиса рассматриваются как *критичный элемент безопасности платформы* и защищаются от несанкционированного доступа, подмены и утечки.

Защита конфигураций распространяется на:

* параметры подключения сервисов,
* настройки безопасности,
* конфигурации API Gateway,
* параметры Event Bus, PostgreSQL, MongoDB, Redis Cluster,
* конфигурации HashiCorp Vault,
* настройки сервисной аутентификации,
* параметры интеграционных подключений.

Особое внимание уделяется защите конфигураций, содержащих сервисные учетные данные, сертификаты и параметры доступа к инфраструктурным компонентам.

*1. Шифрование конфигураций*

Конфигурационные данные, содержащие чувствительные параметры, хранятся и передаются в защищенном виде.

Шифрованию подлежат содержащиеся в конфигурационных данных:

* сервисные токены,
* учетные данные PostgreSQL,
* учетные данные MongoDB,
* учетные данные Redis Cluster,
* Kafka credentials,
* параметры интеграционных подключений,
* сертификаты mTLS,
* доверенные ключи проверки JWT access-токенов,
* административные секреты.

Конфигурации с чувствительными параметрами *не передаются между сервисами в открытом виде*.

Для хранения критичных параметров используется HashiCorp Vault, обеспечивающий централизованное защищенное хранение и выдачу секретов.

*2. Исключение секретов из репозиториев*

Размещение секретов и чувствительных конфигурационных данных в исходном коде и репозиториях *запрещено*.

В Git-репозиториях не допускается хранение секретов, токенов, паролей, ключей. В репозиториях размещаются только:

* шаблоны конфигураций,
* параметры без чувствительных данных,
* ссылки на используемые секреты,
* идентификаторы конфигурационных профилей.

Получение секретов выполняется сервисами динамически через HashiCorp Vault после прохождения сервисной аутентификации.

*3. Безопасная доставка конфигураций*

Передача конфигурационных данных сервисам выполняется только через защищенные каналы.

Для доставки конфигураций используются:

* TLS 1.3 + mTLS,
* сервисная аутентификация,
* политики доступа HashiCorp Vault.

Конфигурации передаются сервисам при:

* запуске контейнера,
* обновлении конфигурации,
* ротации секретов,
* изменении сервисных сертификатов.

Каждый сервис получает только собственный набор конфигурационных параметров и не имеет доступа к конфигурациям других компонентов платформы.

Например:

* Сервис запросов не получает конфигурации Kafka producer,
* сервис агрегации метрик не получает учетные данные БД резервирования,
* интеграционные сервисы не получают административные параметры,
* клиентские приложения не имеют доступа к внутренним конфигурациям платформы.

*4. Контроль изменений конфигурации*

Изменение конфигурационных данных выполняется централизованно и подлежит обязательному контролю.

Контроль распространяется на изменения:

* политик безопасности,
* сервисных параметров,
* TLS-конфигураций,
* параметров Event Bus,
* сервисных сертификатов,
* интеграционных настроек,
* политик HashiCorp Vault.

Все изменения конфигурации сопровождаются:

* аутентификацией административной учетной записи,
* проверкой полномочий,
* аудитом операций,
* журналированием изменений,
* фиксацией времени и инициатора изменения.

Особое внимание уделяется изменениям:

* конфигурации API Gateway,
* политик авторизации,
* параметров mTLS,
* доверенных сертификатов,
* параметров проверки JWT access-токенов,
* конфигурации HashiCorp Vault.

Попытки *несанкционированного изменения конфигурации* рассматриваются как *события безопасности повышенного уровня критичности* и фиксируются в журналах аудита безопасности.
