=== 2.9. Observability

==== 2.9.1. Обеспечения надежности функционирования Сервиса

Надежность Сервиса обеспечивается за счет:

. сочетания архитектурных решений, 
. механизмов мониторинга,
. регламентированных процессов реагирования на инциденты. 

Архитектура построена с разделением контуров чтения, записи и событийной обработки, что позволяет изолировать нагрузку и локализовать сбои. Критические компоненты, такие как Распределенный кеш, Event Bus и базы данных, развернуты в отказоустойчивых конфигурациях с репликацией и горизонтальным масштабированием.

Сервис спроектирован с учетом возможных деградаций производительности. Для обеспечения доступности используются механизмы:

. кэширования, 
. предварительного прогрева данных, 
. сценарии отката (fallback) с возвратом данных с признаком устаревания. 

В контуре интеграций применяются:

. ограничения нагрузки, 
. повторные попытки с контролем частоты, 
. механизмы изоляции нестабильных внешних систем. 

Для предотвращения конфликтов при резервировании используется *Менеджер распределенных блокировок*.

Наблюдаемость реализована на основе сбора метрик, логов и трассировок со всех ключевых компонентов Сервиса. Для каждой критической метрики заданы целевые значения (SLA) и пороговые значения для срабатывания алертов. Контроль осуществляется по различным показателям:

* задержка обработки запросов, 
* коэффициент попаданий в кеш, 
* отставание потребителей событий, 
* ошибки резервирования, 
* согласованность данных и т.п.

При отклонении метрик от заданных порогов автоматически формируются уведомления, которые поступают в систему оповещения и обрабатываются дежурной командой SRE персонала. Для каждого типа инцидента предусмотрены типовые сценарии диагностики и устранения, включающие: 

. анализ метрик, 
. выявление узкого места 
. выполнение корректирующих действий, таких как: 
.. масштабирование компонентов, 
.. перераспределение нагрузки, 
.. ограничение входящего трафика, 
.. восстановление данных.

Дополнительно применяется: 

* асинхронная обработка,
* *Очередь ошибок* для изоляции проблемных сообщений, 
* *Сервис сверки* для выявления и устранения расхождений между источниками данных. 

Это позволяет системе сохранять работоспособность даже при частичных отказах и постепенно восстанавливать консистентность данных.

Надежность системы достигается за счет комбинации: 

* отказоустойчивой архитектуры, 
* проактивного мониторинга,
* формализованных процессов управления инцидентами, 

что позволяет поддерживать стабильную работу Сервиса при высоких нагрузках и в условиях деградации отдельных компонентов.

==== 2.9.2. Цели мониторинга

Целью наблюдаемости Сервиса является обеспечение:

* контроля производительности (задержки, пропускная способность);
* выявления деградаций и отказов компонентв;
* диагностики узких мест (кеш, шина событий, БД);
* контроля согласованности данных;
* поддержки процессов эксплуатации и инцидент-менеджмента.

Мониторинг строится на сборе **метрик, логов и трассировок** с ключевых компонентов Сервиса.

Основной акцент мониторинга делается на:

* Redis : Распределенный кеш, Кеш лояльности
* Kafka : Event Bus, Очерндь ошибок
* PostgreSQL : транзакционные БД
* API Gateway : входной поток

==== 2.9.4. Инструменты мониторинга и профилирования

Рекомендуемый стек:

1. *Метрики*

* Cбор метрик : `Prometheus`
* визуализация и дашборды : `Grafana`

2. *Логирование*

* `Loki + Grafana`

3. *Трассировка*

* раскрытия цепочки вызовов, поиска медленных операций, анализа распределенных запросов : `OpenTelemetry`

4. *Алертинг*

* `Alertmanager`

5. *Профилирование*

* Для анализа CPU/memory/GC : `async-profiler`

==== 2.9.5. Сценарии диагностики Сервиса и его компонентов

Сценарии работают следующим образом:

. Каждая метрика привязана к SLA.
. При превышении порога срабатывает алерт.
. Алерт через Alertmanager поступает в систему/персоналу SRE.
. SRE анализирует дашборды и запускает runbook.
. Выполняются конкретные действия: масштабирование, оптимизация, деградация функциональности.

В виде формулы сценарий можно представить так:

[source]
----
Метрика → Порог → Алерт → Анализ → Действия
----

===== Сценарий 1 : Рост задержки

* *Метрики* : `p95/p99 latency` +
*Порог SLA* : `p95` < 150 мс, `p99` < 300 мс +
*Алерт* : Превышение более 5и мин +
*Если → то* : рост → деградация пользовательского опыта, нарушение SLA +
*Действия* : выявить узкое место по трассировке, масштабировать перегруженный компонент +
*Алертинг-цепочка* : Alertmanager → уведомление SRE → проверка Grafana дашборда → запуск runbook → масштабирование сервиса +

* *Метрики* : `cache_hit_ratio` +
*Порог SLA* : более 85% +
*Алерт* : менее 80% более 3х мин +
*Если → то* : падение → рост нагрузки на MongoDB, увеличение latency +
*Действия* : увеличить TTL, настроить прогрев кеша, оптимизировать ключи +
*Алертинг-цепочка* : Alertmanager → SRE → анализ Redis → проверка прогрева кеша → запуск прогрева

* *Метрика* : Redis `latency` +
*Порог SLA* : менее 5 мс +
*Алерт* : более 10 мс +
*Если → то* : рост → замедление всех read-запросов +
*Действия* : масштабировать кластер, перераспределить шардирование, снизить hot keys +
*Алертинг-цепочка* : Alertmanager → SRE → проверка нагрузки Redis → ребалансировка шардов → масштабирование

* *Метрика* : `evictions` +
*Порог SLA* : 0 +
*Алерт* : более 0 (устойчиво) +
*Если → то* : рост → потеря кеша, лавинообразный fallback +
*Действия* : увеличить память, оптимизировать политику вытеснения +
*Алертинг-цепочка* : Alertmanager → SRE → анализ памяти → увеличение лимитов → перераспределение ключей

* *Метрика* : `fallback_rate` +
*Порог SLA* : менее 5% +
*Алерт* : более 10% +
*Если → то* : рост → перегрузка read-модели, рост latency +
*Действия* : улучшить прогрев кеша, ограничить fallback +
*Алертинг-цепочка* : Alertmanager → SRE → анализ причин miss → корректировка кеш-стратегии

* *Метрика* : MongoDB `latency` +
*Порог SLA* : менее 20 мс +
*Алерт* : более 30 мс +
*Если → то* : рост → деградация fallback-сценария +
*Действия* : оптимизировать индексы, масштабировать реплики +
*Алертинг-цепочка* : Alertmanager → DBA/SRE → проверка индексов → масштабирование

* *Метрики* : `fan-out` Гео-сервиса +
*Порог SLA* : не более 20 магазинов +
*Алерт* : более 25 магазинов +
*Если → то* : рост → кратное увеличение нагрузки на Redis +
*Действия* : ограничить количество магазинов, оптимизировать гео-фильтрацию +
*Алертинг-цепочка* : Alertmanager → SRE → анализ Geo → изменение параметров поиска

===== Сценарий 2 : Рост consumer lag (Kafka)

* *Метрики* : `consumer lag` +
*Порог SLA* : меньше 10 сек +
*Алерт* : дольше 30 сек или более 10 000 сообщений +
*Если → то* : рост → устаревание данных в кеше и read-модели +
*Действия* : масштабировать consumers, увеличить throughput обработки +
*Алертинг-цепочка* : Alertmanager → SRE → Grafana (lag) → анализ consumers → масштабирование

* *Метрики* : `processing rate` +
*Порог SLA* : не менее 95% от входящего потока +
*Алерт* : менее 80% +
*Если → то* : падение → накопление очереди событий +
*Действия* : оптимизировать обработку, увеличить количество инстансов +
*Алертинг-цепочка* : Alertmanager → SRE → анализ throughput → масштабирование

* *Метрики* : 'CPU/IO consumers' +
*Порог SLA* : менее 70% +
*Алерт* : более 80% +
*Если → то* : рост → ограничение производительности consumers +
*Действия* : добавить ресурсы или масштабировать горизонтально +
*Алертинг-цепочка* : Alertmanager → SRE → анализ ресурсов → перераспределение нагрузки

* *Метрики* : `retry_rate` +
*Порог SLA* : менее 5% +
*Алерт* : более 10% +
*Если → то* : рост → дополнительная нагрузка и замедление обработки +
*Действия* : устранить ошибки downstream, ограничить повторы +
*Алертинг-цепочка* : Alertmanager → SRE → анализ ошибок → корректировка retry

* *Метрики* : `error_rate` consumers +
*Порог SLA* : менее 1% +
*Алерт* : больше 2% +
*Если → то* : рост → потеря событий или направление необработанных событий в очередь ошибок +
*Действия* : исправить ошибки, выполнить повторную обработку +
*Алертинг-цепочка* : Alertmanager → SRE → анализ логов → fix → replay

===== Сценарий 3 : Ошибки резервирования

*Алертинг-цепочка* : Alertmanager → SRE → анализ hot SKU → ограничение нагрузки

* *Метрики* : `active_locks` +
*Порог SLA* : меньше 5 000 +
*Алерт* : больше 7 500 +
*Если → то* : рост → задержки в обработке операций +
*Действия* : оптимизировать TTL блокировок +
*Алертинг-цепочка* : Alertmanager → SRE → анализ TTL → настройка блокировок

*Порог SLA* : менее 20 мс +
*Алерт* : более 30 мс +
*Если → то* : рост → замедление транзакций резервирования +
*Действия* : оптимизировать запросы, масштабировать БД +
