=== 2.9. Observability

==== 2.9.1. Цели мониторинга

Целью наблюдаемости Сервиса является обеспечение:

* контроля производительности (задержки, пропускная способность);
* выявления деградаций и отказов компонентв;
* диагностики узких мест (кеш, шина событий, БД);
* контроля согласованности данных;
* поддержки процессов эксплуатации и инцидент-менеджмента.

Мониторинг строится на сборе **метрик, логов и трассировок** с ключевых компонентов Сервиса.

==== 2.9.2. План мониторинга и ключевые метрики

Метрики сгруппированы по слоям архитектуры.

[cols="1,1,3", options="header"]
|===
| Объекты мониторинга | Метрики | Интегрпретация изменений 

3+| *Входной слой*

| API Gateway, +
Load Balancer 
| RPS, +
p95 / p99 latency, +
4xx / 5xx error rate, +
rate limit triggered
| 
* рост latency → перегрузка сервисов или downstream-компонентов; +
* рост 5xx → деградация backend; +
* срабатывание rate limit → перегрузка входного контура

3+| *Контур чтения*

| Сервис запросов
| RPS, +
p95 / p99 latency, +
cache_hit_ratio (коэффициент попаданий в кеш), +
fallback_rate (доля fallback на read-model), +
error_rate
| 
* cache_hit_ratio < 80% → проблема с кешем или прогревом; +
* рост fallback_rate → деградация кеша; +
* рост latency → проблемы в распределенном кеша / Гео-сервисе / Хранилище модели чтения

|===

---

## 3. Распределенный кеш (Redis)

```text
ops/sec (GET, SET, INCR)
latency
memory usage
evictions
keyspace hits/misses
```

### Интерпретация:

* рост latency → перегрузка ноды
* evictions > 0 → нехватка памяти
* рост misses → деградация hit ratio

---

## 4. Гео-сервис и Гео-индекс

```text
RPS
latency
количество магазинов на запрос (fan-out)
```

### Интерпретация:

* рост fan-out → увеличение нагрузки на Redis
* рост latency → проблемы индекса или CPU

---

## 5. Контур резервирования

### Сервис резервирования

```text
TPS
latency
error_rate
lock_conflicts
```

### Менеджер блокировок

```text
active_locks
lock_acquire_time
lock_reject_rate
```

### Интерпретация:

* рост lock_conflicts → hot SKU / конкуренция
* рост latency → проблемы БД или блокировок

---

## 6. Event Bus (Kafka)

```text
messages/sec
consumer lag
under-replicated partitions
broker CPU / disk
```

### Интерпретация:

* lag растет → потребители не успевают
* under-replicated → риск потери данных
* рост disk → риск saturation

---

## 7. Потребители событий

(Загрузчик моделей чтения, Сервис синхронизации кеша, Сервис сверки)

```text
processing rate
processing latency
error_rate
retry_rate
```

### Интерпретация:

* падение processing rate → отставание
* рост retry → проблемы downstream

---

## 8. Хранилища

### PostgreSQL

```text
TPS
WAL lag
fsync latency
connections
```

### MongoDB

```text
read/write latency
replication lag
```

### Интерпретация:

* WAL lag → отставание реплик
* рост latency → перегрузка диска

---

## 9. Система лояльности

```text
RPS к платформе
latency
error_rate
cache hit ratio (кэш лояльности)
```

### Интерпретация:

* рост latency → проблемы внешней системы
* низкий hit ratio → перегрузка интеграции

---

## 10. Система в целом (бизнес-метрики)

```text
cancel_rate
availability success rate
ошибки резервирования
```

# 2.9.4 Сценарии диагностики

---

## Сценарий 1: Рост задержки (latency)

### Шаги:

1. Проверить API Gateway latency
2. Проверить Сервис запросов
3. Проверить cache_hit_ratio

Если:

```text
cache_hit_ratio падает
→ проблема в Redis или прогреве
```

Если:

```text
Redis latency растет
→ перегрузка кеша
```

Если:

```text
Mongo latency растет
→ fallback нагружает БД
```

---

## Сценарий 2: Рост consumer lag в Kafka

### Шаги:

1. Проверить lag по группам потребителей
2. Проверить processing rate

Если:

```text
lag растет
→ не хватает consumers
```

Действия:

```text
масштабирование потребителей
проверка ошибок
```

---

## Сценарий 3: Ошибки резервирования

### Шаги:

1. Проверить error_rate
2. Проверить lock_conflicts

Если:

```text
lock_conflicts растет
→ hot SKU
```

Действия:

```text
ограничение нагрузки
увеличение TTL
анализ распределения ключей
```

---

## Сценарий 4: Деградация интеграций

### Шаги:

1. Проверить latency Клиента Лояльности
2. Проверить error_rate адаптеров

Если:

```text
внешняя система медленная
```

Действия:

```text
использование кеша
fallback
ограничение вызовов
```

---

## Сценарий 5: Потеря данных или несогласованность

### Шаги:

1. Проверить Сервис сверки
2. Проверить БД конфликтов

Если:

```text
конфликты растут
→ проблемы источников данных
```

---

# 2.9.5 Итог

Система наблюдаемости обеспечивает:

```text
контроль производительности
раннее выявление деградаций
диагностику узких мест
поддержку отказоустойчивости
```

Основной акцент делается на:

```text
Redis (кеш)
Kafka (шина событий)
PostgreSQL (транзакции)
API Gateway (входной поток)
```
