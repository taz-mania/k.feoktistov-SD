=== 3.2 Аудит безопасности

В разделе описываются меха низмы регистрации, мониторинга, хранения и анализа событий безопасности Сервиса.

Подсистема аудита безопасности используется для:

* контроля попыток несанкционированного доступа,
* регистрации операций аутентификации и авторизации,
* контроля административных действий,
* выявления аномальной активности,
* расследования инцидентов безопасности,
* контроля изменений конфигурации и политик доступа,
* контроля использования сервисных учетных записей и секретов.

Аудит безопасности охватывает:

* API Gateway,
* внутренние микросервисы,
* Event Bus,
* PostgreSQL,
* MongoDB,
* Redis Cluster,
* HashiCorp Vault,
* механизмы сервисной аутентификации,
* административные интерфейсы,
* интеграционные подключения внешних систем.

Особое внимание уделяется регистрации событий, связанных с:

* операциями резервирования,
* административными действиями,
* выдачей и использованием JWT access-токенов,
* использованием сервисных сертификатов,
* доступом к секретам HashiCorp Vault,
* подключением интеграционных сервисов,
* публикацией и чтением событий Event Bus,
* изменением политик доступа и конфигурации безопасности.

События безопасности используются для:

* оперативного выявления нарушений политик безопасности,
* контроля попыток обхода механизмов авторизации,
* выявления компрометации учетных записей,
* контроля использования привилегированных ролей,
* анализа действий пользователей и сервисов,
* расследования инцидентов безопасности.

В рамках раздела далее рассматриваются:

* архитектура подсистемы аудита безопасности,
* регистрация и классификация событий безопасности,
* мониторинг и выявление аномальной активности,
* хранение, поиск и расследование событий безопасности.

==== 3.2.1 Архитектура аудита безопасности

===== 3.2.1.1 Цели и задачи аудита безопасности

Подсистема аудита безопасности предназначена для *контроля безопасности* пользовательских, сервисных и интеграционных операций, выполняемых в Сервисе.
Основной задачей аудита является обеспечение возможности оперативного выявления нарушений безопасности, анализа действий пользователей и сервисов, а также последующего расследования инцидентов.

*1. Выявление инцидентов безопасности*

Подсистема аудита используется для выявления событий, указывающих на нарушение политик безопасности или попытки несанкционированного доступа.

Контролируются:

* попытки использования недействительных JWT access-токенов,
* ошибки аутентификации и авторизации,
* попытки обращения к запрещенным API,
* попытки использования неразрешенных ролей,
* попытки доступа к административным операциям,
* ошибки проверки сервисных сертификатов,
* попытки доступа к неразрешенным секретам HashiCorp Vault,
* попытки публикации или чтения неразрешенных топиков Event Bus.

События безопасности анализируются для выявления:

* аномальной активности,
* компрометации учетных записей,
* нарушения сервисных политик,
* попыток обхода механизмов защиты.

*2. Расследование нарушений безопасности*

Журналы аудита используются для последующего расследования инцидентов безопасности и анализа последовательности действий пользователей, сервисов и интеграционных компонентов.

Подсистема аудита позволяет:

* восстановить последовательность операций,
* определить источник запроса,
* установить использованную роль и учетную запись,
* определить затронутые компоненты Сервиса,
* установить факт изменения конфигурации или политик доступа,
* выявить использование скомпрометированных учетных данных.

Для критичных операций дополнительно фиксируются:

* request_id,
* идентификатор пользовательской сессии,
* сервисная учетная запись,
* источник сетевого запроса,
* результат выполнения операции.

3. Контроль действий пользователей и сервисов

Аудит безопасности используется для контроля действий:

* пользователей,
* административных учетных записей,
* внутренних сервисов,
* интеграционных компонентов.

Особое внимание уделяется операциям:

* административным,
* изменения ролей,
* управления секретами,
* изменения конфигурации,
* резервирования,
* публикации событий Event Bus.

Подсистема аудита позволяет выявлять:

* использование избыточных полномочий,
* нарушение принципа минимально необходимых привилегий,
* попытки выполнения неразрешенных операций,
* аномальное поведение сервисных учетных записей.

*4. Подтверждение соблюдения политик безопасности*

Журналы аудита используются для подтверждения соблюдения требований безопасности и контроля корректности работы механизмов защиты Сервиса.

Аудит позволяет подтвердить:

* применение политик авторизации,
* корректность использования ролей,
* использование защищенных соединений,
* применение сервисной аутентификации,
* соблюдение ограничений доступа,
* контроль административных операций,
* использование разрешенных сервисных учетных записей.

События аудита используются при проведении внутренних проверок безопасности, анализе инцидентов и контроле корректности изменения политик доступа и конфигурации компонентов Сервиса.

===== 3.2.1.3 Поток обработки событий безопасности

Подсистема аудита безопасности обеспечивает централизованную обработку событий безопасности, поступающих от компонентов внешнего и внутреннего контуров Сервиса.

Обработка событий выполняется последовательно и включает:

*1. Сбор событий безопасности*

События безопасности *формируются компонентами Сервиса* в процессе обработки пользовательских запросов, межсервисного взаимодействия и выполнения административных операций.

Источники событий формируют записи аудита в едином формате с обязательным указанием:

* времени события,
* типа события,
* источника события,
* идентификатора пользователя или сервиса,
* request_id,
* результата выполнения операции,
* уровня критичности события.

События безопасности передаются из компонентов Сервиса асинхронно, без блокировки пользовательских операций и бизнес-обработки.

Для критичных компонентов Сервиса дополнительно контролируется доставка событий аудита и отсутствие потери записей безопасности.

*2. Нормализация событий*

События безопасности, поступающие от различных компонентов Сервиса, приводятся к единому внутреннему формату аудита.

Нормализация используется для:

* унификации структуры событий,
* приведения идентификаторов пользователей и сервисов к единому формату,
* унификации уровней критичности,
* исключения дублирующихся записей,
* последующего анализа и поиска событий безопасности.

В процессе нормализации события дополнительно обогащаются:

* идентификатором пользовательской сессии,
* сервисной ролью,
* типом вызывающего компонента,
* категорией события безопасности,
* признаками административной операции.

Для событий Event Bus дополнительно фиксируются:

* топик,
* consumer group,
* тип поставщика событий,
* результат обработки события.

*3. Хранение событий безопасности*

Нормализованные события безопасности централизованно сохраняются в подсистеме хранения журналов аудита безопасности.

Хранение обеспечивает:

* поиск событий безопасности,
* анализ последовательности действий,
* расследование инцидентов,
* контроль изменений конфигурации,
* хранение истории административных операций.

Для критичных событий дополнительно обеспечивается защита от удаления и несанкционированного изменения записей аудита.
Срок хранения событий безопасности определяется политиками безопасности Сервиса и зависит от уровня критичности события.

*4. Анализ событий безопасности*

События аудита используются для автоматизированного анализа безопасности Сервиса и выявления аномальной активности.

Анализ выполняется для выявления:

* повторяющихся ошибок аутентификации,
* попыток обхода авторизации,
* аномальной активности сервисных учетных записей,
* нехарактерного использования административных ролей,
* попыток доступа к запрещенным API,
* аномальной активности Event Bus,
* несанкционированного изменения конфигурации Сервиса.

Для анализа событий используются:

* корреляция событий по request_id,
* анализ пользовательских сессий,
* анализ последовательности действий,
* сопоставление событий различных компонентов Сервиса.

*5. Формирование уведомлений*

При выявлении событий повышенного уровня критичности подсистема аудита формирует уведомления безопасности.
Уведомления используются для оперативного реагирования на инциденты безопасности и последующего расследования нарушений.

Уведомления формируются для:

* повторяющихся ошибок аутентификации,
* попыток доступа к запрещенным операциям,
* использования недействительных сервисных сертификатов,
* попыток доступа к запрещенным секретам,
* использования привилегированных ролей,
* изменения критичных параметров конфигурации,
* нарушения политик безопасности Сервиса.

В составе уведомлений фиксируются:

* источник события,
* уровень критичности,
* затронутый компонент,
* идентификатор пользователя или сервиса,
* время события,
* описание нарушения безопасности.

==== 3.2.2 Регистрация событий безопасности

===== 3.2.2.1 События аутентификации

Подсистема аудита безопасности Сервиса регистрирует события, связанные с аутентификацией пользователей, сервисных учетных записей и интеграционных компонентов.

Регистрация событий аутентификации используется для:

* выявления попыток несанкционированного доступа,
* контроля использования учетных записей,
* расследования инцидентов безопасности,
* выявления компрометации учетных данных,
* анализа аномальной активности.

Для событий аутентификации в журналах аудита безопасности фиксируются:

* время события,
* идентификатор пользователя или сервиса,
* тип учетной записи,
* IP-адрес источника,
* идентификатор запроса,
* результат операции,
* причина отказа при ошибке аутентификации.

*1. Успешная аутентификация*

При успешной аутентификации пользователя или сервиса регистрируется событие успешного входа.

Событие формируется:

* API Gateway,
* средствами аутентификации,
* внутренними сервисами при сервисной аутентификации.

Пример записи аудита безопасности:

```json
{
  "event_type": "authentication_success",  // Тип регистрируемого события
  "timestamp": "2026-06-08T10:15:21Z",     // Штамп времени события
  "user_id": "user_18452",                 // Идентаификатор пользователя
  "role": "operator",                      // Роль пользователя
  "source_ip": "194.164.216.227",          // Адрес пользователя
  "authentication_method": "OAuth2",       // Использованный метод аутентификации
  "request_id": "req-7d9a12",              // Идентификатор запроса
  "result": "success"                      // Результат аутентификации
}
```

Для сервисной аутентификации дополнительно фиксируются:

* сервисная роль,
* идентификатор сервиса,
* идентификатор сертификата mTLS.

Пример записи аудита сервисной аутентификации:

```json
{
  "event_type": "service_authentication_success",
  "timestamp": "2026-06-08T10:16:02Z",
  "service_id": "reservation-service",        // Идентификатор сервиса
  "service_role": "reservation_writer",       // Роль сервиса
  "certificate_id": "mtls-cert-18f2",         // Идентификатор mTLS сертификата 
  "request_id": "req-9a2cd1",
  "result": "success"
}
```

*2. Неуспешная аутентификация*

При ошибке аутентификации формируется событие неуспешного входа.

События регистрируются при:

* использовании недействительного JWT access-токена,
* ошибке проверки подписи токена,
* использовании просроченного токена,
* ошибке проверки сервисного сертификата,
* использовании заблокированной учетной записи,
* ошибке проверки refresh-токена.

Пример записи аудита безопасности:

```json
{
  "event_type": "authentication_failed",
  "timestamp": "2026-06-08T10:18:44Z",
  "user_id": "user_18452",
  "source_ip": "194.164.216.227",
  "failure_reason": "invalid_jwt_signature",   // Причина отказа аутентификации : ошибка подписи JWT-токена
  "request_id": "req-c12851",
  "result": "denied"                           // Результата : В доступе отказано 
}
```

Пример записи ошибки сервисной аутентификации:

```json
{
  "event_type": "service_authentication_failed",
  "timestamp": "2026-06-08T10:19:11Z",
  "service_id": "read-model-loader",
```

==== 3.2.3 Мониторинг безопасности

===== 3.2.3.1 Метрики безопасности

Мониторинг безопасности Сервиса использует специализированные метрики для выявления нарушений политик безопасности, контроля административных действий и обнаружения аномального поведения пользователей и сервисов.

Метрики безопасности дополняют технические метрики observability, разработанные в разделе 2.9, и используются совместно с журналами аудита безопасности и трассировкой запросов.

*1. Метрики изменения политик безопасности*

Контролируются изменение политик безопасности Сервиса и параметры административных изменений.

Контролируется количество:

* изменений политик безопасности, политик API Gateway, HashiCorp Vault,
* изменений ограничений Event Bus,
* ошибок применения политик,
* неуспешных административных изменений.

Рост количества изменений политик безопасности рассматривается как потенциальный признак:

* несанкционированной административной активности,
* ошибочной конфигурации,
* попыток обхода ограничений безопасности.

Дополнительно контролируется:

* частота изменений,
* инициатор изменений,
* источник административного запроса,
* доля неуспешных изменений.

*2. Метрики изменения ролей*

Контролируются операции изменения пользовательских и сервисных ролей.

Контролируются количество:

* назначений и удалений ролей,
* изменений привилегированных ролей,
* ошибок изменения ролей,
* операций изменения сервисных полномочий.

Дополнительно анализируются:

* изменение количества привилегированных ролей,
* рост числа административных учетных записей,
* аномальное изменение сервисных полномочий,
* повторяющиеся ошибки изменения ролей.

Для привилегированных ролей дополнительно контролируется:

* частота использования,
* количество административных операций, отказов авторизации, 
* количество попыток выполнения запрещенных операций.

*3. Метрики изменения конфигурации сервисов*

Контролируются изменения конфигурации компонентов Сервиса.

Контролируются количество:

* изменений конфигурации API Gateway, Event Bus, параметров Kubernetes,
* изменений параметров сервисов, сетевых политик,
* ошибок применения конфигурации.

Дополнительно анализируются:

* частота изменений конфигурации,
* количество аварийных изменений, откатов конфигурации,
* изменение параметров ограничения запросов, подключения к хранилищам данных.

Рост количества изменений конфигурации рассматривается как *потенциальный риск нарушения стабильности или безопасности* Сервиса.

*4. Метрики изменения криптографических настроек*

Контролируются изменения параметров TLS, mTLS и криптографических механизмов Сервиса.

Контролируются количество:

* изменений наборов шифров, TLS-конфигурации, параметров mTLS,
* отзывов сертификатов, 
* ошибок проверки сертификатов, JWT access-токенов,
* изменений доверенных ключей проверки JWT access-токенов.

Дополнительно анализируются:

* рост ошибок проверки сертификатов, количества невалидных JWT access-токенов,
* изменение частоты ротации сертификатов,
* количество отказов сервисной аутентификации, ошибок проверки mTLS.

Данные метрики дополнительно сопоставляются с техническими метриками:

* `error_rate`,
* `p95/p99 latency`,
* `consumer lag`,
* количеством ошибок API Gateway, межсервисного взаимодействия.

Это позволяет выявлять *инциденты безопасности, влияющие на стабильность и доступность* Сервиса.

===== 3.2.3.2 Выявление аномальной активности

Мониторинг безопасности Сервиса выполняет автоматизированное выявление аномальной активности пользователей, сервисных учетных записей и интеграционных компонентов.

Выявление аномалий выполняется на основе:

* журналов аудита безопасности, метрик безопасности, корреляции событий,
* анализа пользовательских и сервисных сессий, сетевой активности, обращений к данным и API.

Мониторинг безопасности выявляет события, потенциально указывающие на:

* компрометацию учетных записей,
* попытки обхода механизмов авторизации,
* несанкционированное использование сервисных ролей,
* массовый сбор данных,
* нарушение политик безопасности Сервиса.

*1. Подбор учетных данных*

Мониторинг безопасности выявляет признаки автоматизированного подбора учетных данных пользователей и сервисов.

Контролируются:

* повторяющиеся ошибки аутентификации,
* большое количество запросов аутентификации с одного IP-адреса,
* попытки аутентификации для множества учетных записей,
* высокая частота ошибок проверки JWT access-токенов,
* повторное использование недействительных refresh-токенов,
* аномальная частота запросов аутентификации.

Дополнительно анализируются:

* география запросов,
* изменение IP-адресов,
* время между попытками аутентификации,
* изменение пользовательского агента,
* использование заблокированных учетных записей.

При выявлении признаков подбора учетных данных мониторинг безопасности может инициировать:

* блокировку учетной записи,
* ограничение частоты запросов,
* завершение пользовательской сессии,
* формирование уведомления безопасности повышенного уровня критичности.

*2. Массовые отказы авторизации*

Мониторинг безопасности выявляет аномальное количество отказов авторизации при обращении к API и внутренним сервисам Сервиса.

Контролируются:

* рост количества ошибок доступа, запрещенных операций,
* повторяющиеся обращения к административным API,
* массовые ошибки проверки полномочий,
* рост количества нарушений политик доступа,
* повторяющиеся ошибки доступа к Event Bus.

Дополнительно анализируются:

* тип вызываемых API,
* используемые роли,
* сервисные учетные записи,
* источник запросов,
* последовательность операций пользователя или сервиса.

Массовые отказы авторизации рассматриваются как потенциальный признак:

* компрометации учетной записи,
* ошибочной конфигурации ролей,
* попытки обхода ограничений доступа,
* использования устаревших сервисных полномочий.

*3. Аномальная активность сервисных учетных записей*

Мониторинг безопасности анализирует активность сервисных учетных записей и выявляет отклонения от ожидаемого поведения сервисов.

Контролируются:

* изменение характера межсервисных запросов,
* обращение сервисов к нехарактерным API,
* рост количества отказов сервисной аутентификации,
* использование неразрешенных топиков Event Bus,
* аномальная активность сервисов в HashiCorp Vault,
* резкий рост количества операций доступа к данным,
* изменение частоты публикации событий.

Дополнительно анализируются:

* сервисная роль,
* используемые учетные данные Kafka,
* тип выполняемых операций,
* частота запросов,
* объем передаваемых данных,
* изменение шаблонов сетевого взаимодействия.

Аномальная активность сервисных учетных записей рассматривается как потенциальный признак:

* компрометации сервисного секрета,
* нарушения сервисной политики,
* ошибочной конфигурации сервиса,
* несанкционированного использования сервисной роли.

*4. Необычные обращения к данным*

Мониторинг безопасности выявляет аномальные операции доступа к данным Сервиса.

Контролируются:

* массовое чтение данных,
* резкий рост количества запросов к хранилищам,
* нетипичный экспорт данных,
* обращения к административным данным, к данным вне стандартного сценария работы роли,
* аномальная активность в PostgreSQL, MongoDB, Redis Cluster.

Дополнительно анализируются:

* объем выгружаемых данных,
* тип данных,
* источник запросов,
* используемая роль,
* время выполнения операций,
* последовательность обращений к данным.

Мониторинг безопасности дополнительно сопоставляет аномальные обращения к данным с:

* изменением ролей, политик доступа, конфигурации сервисов, сетевой активности,

Уровень критичности события влияет на:

* нарушении доступности критичных компонентов.

Для событий повышенного уровня критичности дополнительно контролируются:

* время устранения,
* статус обработки инцидента,
* последовательность действий при расследовании.

Механизмы мониторинга безопасности Сервиса обеспечивают сохранение истории эскалации и последующего расследования инцидентов безопасности.

* административный доступ,

* корректности восстановления журналов аудита.

Механизмы мониторинга безопасности Сервиса дополнительно фиксируют:

* Event Bus,
* HashiCorp Vault,
* Kubernetes,
* доступа к хранилищам данных.

Механизмы мониторинга безопасности Сервиса используют результаты проверки достоверности журналов для выявления:

* попыток сокрытия инцидентов,
* удаления событий безопасности,
* нарушения последовательности событий,
* компрометации административных учетных записей,
* повреждения архивных журналов аудита.

===== 3.2.4.3 Поиск и анализ событий безопасности

Журналы аудита безопасности Сервиса используются для поиска, сопоставления и анализа событий безопасности при расследовании инцидентов и выявлении аномальной активности пользователей и сервисов.

Анализ событий безопасности используется для:

* расследования инцидентов безопасности,
* выявления последовательности действий нарушителя,
* выявления попыток обхода механизмов защиты,
* анализа компрометации учетных записей, действий привилегированных ролей, аномальной активности сервисов.

Для поиска и анализа событий безопасности используются:

* журналы аудита безопасности,
* идентификаторы запросов, пользовательских сессий, сервисных операций,
* события API Gateway, сервисной аутентификации, Event Bus, HashiCorp Vault, Kubernetes, хранилищ данных.

*1. Корреляция событий безопасности*

Для анализа инцидентов безопасности выполняется корреляция событий различных компонентов Сервиса.

Корреляция используется для:

* объединения связанных событий,
* анализа последовательности действий,
* выявления связанных нарушений безопасности,
* определения источника инцидента,
* анализа распространения инцидента между компонентами Сервиса.

Сопоставление выполняется по:

* идентификатору запроса,
* идентификатору пользовательской сессии,
* сервисной учетной записи,
* IP-адресу источника,
* временным меткам,
* идентификаторам операций Event Bus.

Дополнительно сопоставляются:

* события аутентификации, авторизации,
* операции доступа к данным, управления секретами, изменения конфигурации,
* административные операции.

Корреляция событий позволяет выявлять *сложные инциденты безопасности, затрагивающие несколько компонентов* Сервиса одновременно.

*2. Расследование инцидентов безопасности*

Журналы аудита безопасности используются для расследования нарушений безопасности и восстановления последовательности действий пользователей и сервисов.

При расследовании инцидентов анализируются:

* источник инцидента,
* последовательность действий,
* использованные учетные записи, роли и полномочия,
* затронутые компоненты Сервиса,
* изменения конфигурации,
* операции доступа к данным,
* действия привилегированных ролей.

Дополнительно анализируются:

* изменения политик безопасности,
* операции с секретами,
* события HashiCorp Vault, Event Bus,
* операции Kubernetes,
* ошибки сервисной аутентификации.

Результаты расследования используются для:

* локализации инцидента,
* блокировки скомпрометированных учетных записей,
* отзыва сертификатов и секретов,
* изменения политик доступа,
* устранения последствий инцидента.

*3. Построение временной линии событий*

Для критичных инцидентов безопасности выполняется построение временной линии событий.

Временная линия используется для:

* восстановления последовательности действий,
* определения момента компрометации,
* анализа распространения инцидента,
* определения затронутых компонентов,
* анализа реакции на инцидент.

Временная линия включает:

* события аутентификации, авторизации,
* операции доступа к данным, изменения конфигурации, административные операции,
* события управления секретами, межсервисного взаимодействия,
* события Event Bus.

Для событий дополнительно учитываются:

* временные метки,
* идентификаторы запросов,
* сервисные роли,
* сетевые источники запросов,
* результаты выполнения операций.

*4. Анализ действий пользователей и сервисов*

Журналы аудита безопасности используются для анализа поведения пользователей, сервисных учетных записей и интеграционных компонентов Сервиса.

Анализируются:

* действия привилегированных ролей,
* использование административных API,
* операции изменения ролей, политик безопасности,
* обращения к HashiCorp Vault, критичным данным,
* использование сервисных учетных записей,
* операции публикации и чтения Event Bus.

Дополнительно выявляются:

* нетипичные последовательности действий,
* аномальное использование полномочий,
* использование ролей вне стандартного сценария,
* аномальная частота операций,
* попытки обхода политик безопасности,
* нехарактерная активность сервисов.

Результаты анализа используются механизмами мониторинга безопасности Сервиса для *выявления повторяющихся инцидентов и корректировки политик безопасности*.

===== 3.2.4.4 Формирование отчетности по безопасности

Журналы аудита безопасности, события мониторинга и метрики безопасности Сервиса используются для формирования отчетности по состоянию безопасности, расследованию инцидентов и контролю соблюдения политик безопасности.

Отчетность по безопасности используется для:

* контроля состояния безопасности Сервиса,
* анализа нарушений политик безопасности,
* расследования инцидентов,
* контроля административной активности,
* анализа использования привилегированных ролей, аномальной активности пользователей и сервисов.

Для формирования отчетности используются:

* журналы аудита безопасности,
* события аутентификации, авторизации, доступа к данным, изменения конфигурации, управления секретами,
* метрики безопасности,
* события Event Bus, HashiCorp Vault, Kubernetes.

*1. Регулярные отчеты по безопасности*

Регулярные отчеты используются для контроля текущего состояния безопасности Сервиса и анализа изменений за отчетный период.

Регулярные отчеты включают статистику:

* событий безопасности,
* ошибок аутентификации,
* отказов авторизации,
* использования привилегированных ролей,
* административных операций,
* операций с секретами,
* изменения политик безопасности, конфигурации компонентов Сервиса.

Дополнительно в регулярные отчеты включаются перечни:

* критичных событий,
* аномальной активности,
* заблокированных учетных записей,
* ошибок сервисной аутентификации,
* операций изменения ролей,
* операций отзыва сертификатов и ключей.

Регулярные отчеты формируются за установленный отчетный период и используются для *анализа динамики событий безопасности* Сервиса.

*2. Отчеты по инцидентам безопасности*

Отчеты по инцидентам формируются для критичных событий безопасности и нарушений политик доступа.

Отчеты по инцидентам включают:

* описание инцидента,
* время возникновения,
* источник инцидента,
* затронутые компоненты Сервиса,
* использованные учетные записи,
* последовательность событий,
* выполненные административные операции,
* результаты расследования,
* меры реагирования.

Дополнительно фиксируются:

* временная линия событий,
* связанные события безопасности,
* изменения конфигурации,
* операции доступа к данным,
* операции с секретами,
* использование привилегированных ролей.

Отчеты по инцидентам используются для:

* расследования нарушений безопасности,
* анализа причин инцидента,
* локализации последствий,
* корректировки политик безопасности,
* предотвращения повторения инцидентов.

*3. Отчеты по нарушениям политик безопасности*

Отдельные отчеты формируются по событиям нарушения политик безопасности Сервиса.

Контролируются:

* попытки обхода авторизации, выполнения запрещенных операций,
* нарушения политик доступа,
* использование запрещенных сервисных ролей,
* обращения к запрещенным API,
* нарушения политик HashiCorp Vault, Event Bus,
* нарушения сетевых ограничений Kubernetes.

Отчеты по нарушениям политик включают:

* идентификатор пользователя или сервиса,
* тип нарушения,
* затронутый компонент,
* используемую роль,
* идентификатор запроса,
* время возникновения события,
* результат выполнения операции.

Дополнительно анализируются:

* повторяемость нарушений,
* последовательность действий,
* частота возникновения событий,
* наличие связанных инцидентов безопасности,
* использование привилегированных полномочий.

4. Статистика событий безопасности

Статистика событий безопасности используется для анализа общей активности механизмов безопасности Сервиса и выявления изменений в характере событий безопасности.

Анализируется количество:

* событий аутентификации,
* отказов авторизации,
* административных операций,
* событий изменения конфигурации,
* операций с секретами, изменения ролей,
* ошибок сервисной аутентификации,
* нарушений политик безопасности,
* аномальных обращений к данным.

Дополнительно контролируются:

* распределение событий по уровням критичности, по компонентам Сервиса,
* динамика изменения количества событий,
* рост количества критичных нарушений,
* повторяемость инцидентов безопасности.

Результаты анализа статистики событий безопасности используются для: 

* корректировки политик безопасности, 
* настройки механизмов мониторинга безопасности Сервиса
* выявления новых сценариев нарушений безопасности.
