=== 3.1 Схема защиты и управления доступом

В разделе описываются механизмы защиты Сервиса, обеспечивающие аутентификацию, авторизацию, защиту каналов передачи данных, управление привилегиями и безопасное хранение секретов.

Обеспечение безопасности разрабатываемого Сервиса должно выполняться не только на уровне пользовательского доступа, но и на уровне взаимодействия внутренних компонентов платформы.

Основной точкой входа внешних запросов в Сервис является API Gateway, выполняющий функции *аутентификации клиентов*, *проверки токенов* доступа, ограничения частоты запросов и применения политик безопасности. Внешние клиентские приложения взаимодействуют с API Сервиса по протоколу HTTPS с использованием *JWT-токенов* доступа, полученных через механизм *OAuth 2.0/OpenID Connect*. Проверка подлинности и срока действия токенов выполняется на стороне *API Gateway* до передачи запросов во внутренний контур микросервисов.

Для взаимодействия внутренних сервисов используется защищенный протокол gRPC поверх HTTP/2 с обязательным применением *mTLS*. Межсервисная аутентификация выполняется на основе сервисных *сертификатов*, выданных доверенным внутренним удостоверяющим центром. Такой подход позволяет исключить неавторизованное подключение компонентов к внутреннему контуру Сервиса и обеспечивает подтверждение подлинности как клиента, так и сервера при каждом соединении.

Передача данных между компонентами Сервиса защищается с использованием *TLS 1.3* и современных криптографических наборов *шифров с поддержкой Perfect Forward Secrecy*. Для внешних HTTP-интерфейсов дополнительно применяется механизм *HSTS*, предотвращающий использование незащищенных соединений. Несмотря на наличие защищенного транспортного канала, критичные данные дополнительно защищаются на уровне приложений с использованием криптографической *подписи токенов доступа*, контроля целостности сообщений и шифрования конфиденциальных конфигурационных параметров.

Управление доступом в Сервисе реализуется на основе *ролевой модели RBAC*. Политики доступа разграничивают права клиентских приложений, административных пользователей, операторов сопровождения и внутренних сервисов. Проверка полномочий выполняется как на уровне API Gateway, так и непосредственно во внутренних сервисах при выполнении критичных операций, связанных с резервированием товаров, административным управлением и доступом к служебным данным.

Для хранения секретов, сертификатов, ключей подписи, учетных данных подключения и конфиденциальных параметров конфигурации используется централизованное защищенное хранилище *HashiCorp Vault*. Секреты не размещаются в исходном коде, контейнерных образах или статических конфигурационных файлах. Архитектура решения предусматривает периодическую ротацию секретов и сертификатов без остановки работы сервисов.

Дополнительно в рамках подсистемы безопасности реализуются механизмы *аудита событий* аутентификации, авторизации, операций управления доступом и действий с критичными данными. *События безопасности* передаются в централизованный *контур мониторинга* и используются для выявления аномальной активности, расследования инцидентов и контроля соблюдения политик безопасности.

==== 3.1.1 Архитектура безопасности Сервиса

===== 3.1.1.1 Цели и принципы обеспечения безопасности

Подсистема безопасности Сервиса *предназначена для*: 

* защиты пользовательских и служебных данных, 
* предотвращения несанкционированного доступа к компонентам платформы, 
* обеспечения целостности потоков событий 
* ограничения возможности распространения инцидентов между внутренними компонентами.

На состав механизмов защиты влияют такие *особенности архитектуры* решения:

* наличие внешнего клиентского контура,
* использование микросервисного взаимодействия,
* применение потоковой передачи событий через Event Bus,
* наличие распределенных хранилищ данных,
* высокая интенсивность межсервисного обмена,
* необходимость обеспечения непрерывной работы Сервиса при пиковых нагрузках.

Основными *целями обеспечения безопасности* являются:

* защита пользовательских и служебных данных от несанкционированного доступа,
* обеспечение подлинности клиентских приложений и внутренних сервисов,
* предотвращение неавторизованного выполнения операций резервирования и административного управления,
* обеспечение целостности потоков событий и служебных сообщений,
* ограничение распространения инцидентов внутри внутреннего контура платформы,
* обеспечение контролируемого доступа к конфиденциальным данным и секретам,
* обеспечение возможности расследования инцидентов безопасности на основе журналов аудита.

*1. Принцип минимально необходимых привилегий*

Во всех компонентах Сервиса реализуется принцип минимально необходимых привилегий. Каждый пользователь, сервис или инфраструктурный компонент получает только тот набор полномочий, который требуется для выполнения его непосредственных функций.

Клиентские приложения получают доступ только к пользовательским API, необходимым для проверки доступности товаров и выполнения операций резервирования. Административные операции, операции сопровождения и доступ к служебным данным из внешнего клиентского контура недоступны.

Внутренние сервисы используют отдельные сервисные учетные записи и сертификаты. Политики доступа ограничивают перечень разрешенных межсервисных вызовов, топиков Event Bus и хранилищ данных, к которым может обращаться конкретный сервис.

Для подключения к PostgreSQL, MongoDB, Redis и HashiCorp Vault используются отдельные учетные данные с ограниченными полномочиями. Доступ к операциям изменения конфигурации, управления схемами данных и административным функциям предоставляется только специализированным административным сервисам.

*2. Принцип нулевого доверия*

Архитектура безопасности Сервиса строится по принципу нулевого доверия (Zero Trust). Ни один запрос, сетевое соединение или внутренний компонент не считаются доверенными автоматически, включая взаимодействия внутри контура платформы.

Каждый внешний запрос проходит обязательную аутентификацию и проверку токена доступа на стороне API Gateway. Внутренние сервисы проходят взаимную аутентификацию с использованием mTLS и сервисных сертификатов.

Даже при размещении компонентов в одном Kubernetes-кластере внутренние сервисы не получают неограниченного сетевого доступа друг к другу. Межсервисное взаимодействие выполняется только по явно разрешенным маршрутам и через контролируемые интерфейсы.

Для всех запросов дополнительно проверяются:

* подлинность вызывающей стороны,
* срок действия токенов и сертификатов,
* полномочия на выполнение операции,
* допустимость доступа к конкретному ресурсу.

*3. Разделение внешнего и внутреннего контуров*

Архитектура Сервиса разделяется на внешний и внутренний контуры безопасности.

Во *внешнем контуре* располагаются клиентские приложения и публичные API, доступные через API Gateway. API Gateway является единственной точкой входа внешнего трафика во внутренний контур платформы.

Во *внутреннем контуре* располагаются:

* микросервисы и их хранилища,
* Event Bus,
* HashiCorp Vault,
* внутренние административные компоненты.

Прямой доступ внешних клиентов к внутренним сервисам, Kafka, базам данных и хранилищам секретов *запрещен*.

Внутренние административные интерфейсы и служебные API изолированы от пользовательского трафика и доступны только из доверенного административного сегмента.

*3. Защита на каждом уровне архитектуры*

Механизмы защиты реализуются на всех уровнях архитектуры Сервиса и не ограничиваются только внешним периметром.

* На уровне сетевого взаимодействия применяется TLS 1.3, mTLS и фильтрация межсервисных соединений.

* На уровне API выполняются:

** аутентификация клиентов,
** проверка JWT-токенов,
** контроль ролей и полномочий,
** ограничение частоты запросов,
** защита от повторной отправки запросов.

* На уровне бизнес-логики реализуется проверка допустимости выполнения операций резервирования, административного управления и доступа к служебным функциям.

* На уровне хранения данных используются:

** разграничение прав доступа,
** отдельные учетные записи сервисов,
** защита секретов,
** контроль доступа к административным операциям,
** аудит операций изменения данных и конфигурации.

* На уровне потоковой обработки событий дополнительно контролируются права публикации и чтения топиков Event Bus, а также выполняется защита служебных сообщений от неавторизованного доступа.

*4. Централизованное управление доступом*

Управление аутентификацией, ролями и политиками безопасности выполняется централизованно.

API Gateway осуществляет централизованную проверку JWT-токенов, контроль срока действия токенов и базовую авторизацию запросов до передачи их во внутренние сервисы.

Политики доступа и роли пользователей управляются из единого контура управления доступом. Внутренние сервисы используют унифицированные механизмы проверки полномочий и не хранят локальные независимые политики авторизации.

Управление секретами, сертификатами и ключами подписи также выполняется централизованно через HashiCorp Vault. Это позволяет обеспечить единый контроль доступа к криптографическим материалам, централизованную ротацию секретов и аудит операций доступа к ним.

===== 3.1.1.2 Общая схема безопасности компонентов

Архитектура безопасности Сервиса построена с учетом разделения компонентов по уровням доверия, изоляции внутренних сервисов от внешнего доступа и централизованного применения механизмов аутентификации и авторизации.

Схема безопасности учитывает следующие особенности архитектуры решения:

* наличие внешнего пользовательского API,
* интенсивное межсервисное взаимодействие,
* использование Event Bus для потоковой передачи событий,
* наличие нескольких распределенных хранилищ данных,
* разделение пользовательских и административных функций,
* необходимость ограничения распространения инцидентов между компонентами.

*Точки входа в Сервис*

Единственной точкой входа внешнего пользовательского трафика в Сервис является API Gateway.

Через API Gateway выполняются:

* запросы проверки доступности товаров,
* операции резервирования,
* геопоиск магазинов,
* получение статусов операций,
* WebSocket-подключения клиентских приложений.

API Gateway завершает TLS-соединения, выполняет аутентификацию клиентов, проверку JWT-токенов, ограничение частоты запросов и первичную авторизацию операций.

Прямой доступ внешних клиентов к внутренним микросервисам, PostgreSQL, MongoDB, Redis Cluster, Event Bus и HashiCorp Vault отсутствует.

Внутренние административные интерфейсы не публикуются во внешний контур и доступны только авторизованным административным пользователям через API Gateway с использованием отдельных ролей и политик доступа.

*Доверенные и недоверенные зоны*

Архитектура Сервиса разделяется на несколько логических зон безопасности.

К недоверенной зоне относится внешний пользовательский контур, включающий:

* мобильные приложения,
* внешние клиентские системы,
* внешние интеграционные подключения.

Все запросы из данной зоны рассматриваются как потенциально недоверенные независимо от используемого клиента.

К пограничной зоне относится API Gateway, который выполняет функции контролируемой точки входа и разделяет внешний и внутренний контуры платформы.

Во внутреннюю доверенную зону входят:

* микросервисы Сервиса,
* Event Bus,
* кластеры БД,
* HashiCorp Vault,
* сервисы обработки событий,
* административные компоненты.

Несмотря на принадлежность компонентов ко внутреннему контуру, взаимодействие между ними не считается доверенным автоматически и требует обязательной взаимной аутентификации.

Отдельно контролируются операции административного управления и сопровождения Сервиса, включая:

* управление справочными данными,
* изменение конфигурации компонентов,
* управление политиками доступа,
* управление сертификатами и секретами,
* выполнение служебных операций сопровождения.

Доступ к административным операциям разрешен только учетным записям с административными ролями после прохождения аутентификации и проверки полномочий.

*Размещение механизмов контроля доступа*

Контроль доступа реализуется на нескольких уровнях архитектуры.

1. API Gateway выполняет:

* аутентификацию внешних клиентов,
* проверку JWT-токенов,
* проверку срока действия токенов,
* ограничение частоты запросов,
* фильтрацию недопустимых запросов,
* базовую проверку ролей.

2. Внутренние микросервисы дополнительно выполняют проверку полномочий на уровне бизнес-операций. Даже после успешного прохождения API Gateway сервис проверяет допустимость выполнения конкретного действия для вызывающей стороны.
+
Например, при создании резерва API Gateway проверяет подлинность JWT-токена, срок его действия и наличие роли, допускающей обращение к API резервирования. После передачи запроса во внутренний контур Сервис резервирования дополнительно проверяет бизнес-полномочия вызывающей стороны: разрешено ли данному типу субъекта создавать резерв, относится ли операция к допустимому каналу, не выполняется ли изменение чужого резерва, соблюдены ли ограничения по идемпотентности и допустимости операции. Если токен является действительным, но субъект не имеет права выполнить конкретное действие над указанным резервом, операция должна быть отклонена внутренним сервисом.

3. Межсервисное взаимодействие контролируется через mTLS и сервисные сертификаты. Каждый сервис идентифицируется по собственному сертификату, а попытки подключения компонентов без доверенного сертификата отклоняются.

4. Для Event Bus используются политики разграничения доступа на публикацию и чтение топиков. Каждый сервис получает доступ только к тем топикам, которые требуются для выполнения его функций.

5. Хранилища данных используют отдельные учетные записи подключения для различных сервисов. Полномочия ограничиваются необходимыми операциями чтения, записи или администрирования.

*Точки применения политик безопасности*

Политики безопасности применяются на нескольких уровнях обработки запросов и данных.

1. На уровне внешнего API применяются:

* политики аутентификации,
* политики авторизации,
* ограничения частоты запросов,
* контроль допустимых методов API,
* проверка JWT-токенов.

2. На уровне межсервисного взаимодействия применяются:

* политики взаимной аутентификации,
* контроль доверенных сертификатов,
* ограничения сетевого взаимодействия между сервисами.

3. На уровне бизнес-логики применяются:

* проверки полномочий для операций резервирования,
* проверки прав административного управления,
* контроль доступа к служебным функциям,
* ограничения на выполнение критичных операций.

4. На уровне хранения данных применяются:

* разграничение прав доступа к БД,
* ограничения доступа к административным операциям,
* контроль доступа к секретам,
* аудит операций изменения конфигурации и данных.

5. На уровне Event Bus применяются:

* контроль публикации событий,
* контроль чтения топиков,
* разграничение доступа consumer group,
* защита служебных потоков событий.

*Взаимодействие API Gateway, сервисов и хранилищ данных*

1. API Gateway является центральной точкой применения внешних политик безопасности и не допускает передачу неаутентифицированных запросов во внутренний контур платформы.

2. После прохождения аутентификации и первичной авторизации запрос передается во внутренние сервисы по защищенному каналу HTTPS/gRPC с использованием внутреннего TLS-взаимодействия.

3. Внутренние сервисы не используют доверие к результатам только внешней проверки и дополнительно выполняют:

* проверку ролей,
* проверку допустимости бизнес-операции,
* проверку сервисных полномочий,
* контроль доступа к данным.

4. При обращении к PostgreSQL, MongoDB, Redis Cluster и HashiCorp Vault сервисы используют отдельные сервисные учетные записи с ограниченными полномочиями.

5. Взаимодействие с Event Bus также выполняется через аутентифицированные сервисные подключения с разграничением прав публикации и чтения топиков.

Такой подход позволяет локализовать инциденты безопасности внутри отдельных компонентов, исключить несанкционированное распространение доступа между сервисами и обеспечить централизованный контроль безопасности на всех уровнях архитектуры Сервиса.

===== 3.1.1.3 Модель угроз и защищаемые активы

Модель угроз Сервиса сформирована с учетом архитектурных особенностей решения, состава обрабатываемых данных, механизмов межсервисного взаимодействия и потоковой обработки событий.

Основное внимание уделяется защите операций резервирования товаров, предотвращению несанкционированного доступа к внутренним сервисам, защите потоков событий и ограничению возможности компрометации внутренних компонентов платформы.

*1. Защищаемые активы*

К защищаемым активам Сервиса относятся: 

* данные, 
* компоненты, 
* операции, 

компрометация которых может привести к: 

* нарушению доступности Сервиса, 
* искажению информации о товарных остатках, 
* ошибкам резервирования 
* получению несанкционированного доступа к внутреннему контуру платформы.

*2. Пользовательские и бизнес-данные*

К защищаемым пользовательским и бизнес-данным относятся:

* информация о доступности товаров,
* данные резервирования,
* идентификаторы заказов и операций,
* данные о магазинах и геопозиционировании,
* состояние активных резервов,
* служебные атрибуты бизнес-операций,
* данные о статусах выполнения запросов.

Наиболее критичными являются данные, влияющие на *корректность расчета доступного остатка и выполнение операций резервирования* товаров.

*3. Потоки событий и служебные сообщения*

К критичным защищаемым активам относятся потоки событий Event Bus:

* `inventory.events`,
* `reservation.events`,
* `admin.events`,
* `reconciliation.events`,
* `error.events`.

Компрометация потоков событий может привести к:

* искажению модели чтения,
* рассинхронизации данных,
* потере актуальности кэша,
* ошибочному резервированию товаров,
* нарушению последовательности обработки событий.

Дополнительно защищаются служебные сообщения межсервисного взаимодействия, включая gRPC-запросы, WebSocket-соединения и операции синхронизации внутренних компонентов.

*4. Учетные данные и криптографические материалы*

К защищаемым активам также относятся:

* JWT-токены доступа, refresh-токены,
* ключи подписи токенов,
* сервисные сертификаты mTLS,
* учетные данные PostgreSQL, MongoDB, Redis Cluster,
* секреты HashiCorp Vault,
* служебные ключи интеграционных адаптеров.

Компрометация данных активов может привести к *получению доступа к внутреннему контуру Сервиса и выполнению несанкционированных операций* от имени доверенных компонентов.

*5. Критичные бизнес-операции*

Наиболее критичными бизнес-операциями Сервиса являются:

* резервирование товара,
* изменение состояния резерва,
* обновление данных доступности товаров,
* обработка событий изменения остатков,
* административное управление справочными данными,
* операции сверки и разрешения конфликтов,
* операции публикации событий в Event Bus.

Нарушение целостности или несанкционированное выполнение указанных операций может привести к:

* продаже недоступного товара (отрицательный остаток),
* потере резервов,
* рассогласованию модели чтения и хранилища событий,
* нарушению корректности отображаемых остатков,
* массовым ошибкам обработки заказов.

*6. Сценарии несанкционированного доступа*

При проектировании Сервиса учитываются следующие наиболее вероятные сценарии несанкционированного доступа:

*6.1) Несанкционированный доступ к пользовательскому API*

Внешний нарушитель может попытаться:

* использовать недействительные или поддельные JWT-токены,
* получить доступ к административным API,
* выполнить операции резервирования без необходимых полномочий,
* обойти ограничения частоты запросов,
* инициировать массовые запросы проверки доступности товаров.

Для предотвращения данных сценариев API Gateway выполняет:

* обязательную проверку JWT,
* проверку подписи и срока действия токенов,
* контроль ролей,
* ограничение частоты запросов,
* фильтрацию недопустимых запросов.

*6.2) Несанкционированное межсервисное взаимодействие*

При компрометации отдельного сервиса нарушитель может попытаться выполнить обращения к другим внутренним компонентам платформы.

Наиболее критичными являются попытки:

* публикации поддельных событий в Event Bus,
* чтения служебных топиков,
* получения доступа к HashiCorp Vault,
* выполнения административных gRPC-вызовов,
* обращения к PostgreSQL и MongoDB с повышенными привилегиями.

Для ограничения распространения доступа используются:

* mTLS,
* сервисные сертификаты,
* разграничение сервисных ролей,
* изоляция сетевого взаимодействия,
* отдельные учетные записи подключения.

*6.3) Компрометация учетных данных*

В рамках модели угроз рассматриваются сценарии компрометации:

* JWT-токенов, refresh-токенов,
* сервисных сертификатов,
* учетных данных подключения к БД,
* секретов интеграционных сервисов.

Особую критичность представляет компрометация сервисных сертификатов и учетных данных компонентов, имеющих доступ к Event Bus и операциям резервирования.

Для снижения рисков реализуются:

* ограниченный срок действия токенов,
* ротация секретов и сертификатов,
* централизованное хранение секретов в HashiCorp Vault,
* аудит операций доступа к секретам,
* запрет хранения секретов в исходном коде и контейнерных образах.

*6.4) Перехват и подмена трафика*

Архитектура Сервиса предусматривает защиту от:

* перехвата пользовательского трафика,
* подмены межсервисных запросов,
* подделки событий Event Bus,
* атак посредника,
* повторной отправки запросов.

Наиболее критичными являются:

* запросы резервирования,
* служебные gRPC-вызовы,
* публикация событий изменения остатков,
* административные операции,
* передача токенов и сервисных сертификатов.

Для защиты каналов связи используются:

* TLS 1.3,
* mTLS,
* HSTS,
* криптографическая подпись JWT,
* контроль доверенных сертификатов,
* ограничение срока действия токенов.

Для операций резервирования дополнительно используется контроль идемпотентности запросов и проверка уникальности идентификатора запроса, что снижает риск повторного выполнения операций при повторной отправке запросов.

*6.5) Компрометация потоков событий*

Отдельным сценарием угроз является нарушение корректности работы Event Bus.

Наиболее критичными считаются:

* публикация ложных событий,
* изменение порядка обработки событий,
* несанкционированное чтение служебных топиков,
* нарушение целостности потоков `reservation.events` и `inventory.events`.

Для защиты потоковой подсистемы используются:

* разграничение прав producer и consumer,
* сервисная аутентификация,
* изоляция consumer group,
* контроль доступа к топикам,
* аудит операций публикации и чтения событий.

Дополнительно архитектура Сервиса предусматривает контроль последовательности обработки событий для одного SKU, что предотвращает нарушение согласованности модели чтения при обработке потоков обновления остатков и резервирования.

==== 3.1.2 Аутентификация и авторизация

===== 3.1.2.1 Механизм аутентификации пользователей

Аутентификация внешних пользователей и клиентских приложений в Сервисе реализуется с использованием *OAuth 2.0* и *OpenID Connect*. Данные механизмы используются для подтверждения подлинности пользователя, получения токенов доступа и централизованного управления пользовательскими сессиями.

Внутренние микросервисы не выполняют самостоятельную аутентификацию пользователей по логину и паролю. Проверка подлинности пользователя выполняется внешним поставщиком удостоверений, после чего клиентское приложение получает JWT-токен доступа и использует его при обращении к API Сервиса.

*1. Внешний поставщик удостоверений*

В архитектуре решения используется внешний поставщик удостоверений, отвечающий за:

* аутентификацию пользователей,
* управление пользовательскими сессиями,
* выпуск JWT-токенов,
* выпуск refresh-токен,
* управление сроком действия пользовательских сессий,
* централизованное управление учетными записями.

Сервис не хранит пользовательские пароли и не выполняет самостоятельную проверку учетных данных пользователей.

API Gateway *доверяет только токенам*, выпущенным *доверенным поставщиком удостоверений* и подписанным *доверенным криптографическим ключом*.

*2. Процесс получения токена доступа*

Процесс аутентификации пользователя выполняется следующим образом:

. Пользователь проходит аутентификацию во внешнем поставщике удостоверений.
. Поставщик удостоверений проверяет учетные данные пользователя.
. После успешной аутентификации клиентскому приложению выдаются:

* JWT-токен доступа,
* refresh-токен.
  
. Клиентское приложение передает JWT-токен в API Gateway при каждом запросе к API Сервиса.
. API Gateway проверяет:

* подпись JWT,
* срок действия токена,
* issuer,
* audience,
* наличие необходимых ролей и атрибутов доступа.
  
. После успешной проверки запрос передается во внутренние сервисы.

Внутренние сервисы не выполняют повторную аутентификацию пользователя, но дополнительно проверяют полномочия вызывающей стороны при выполнении бизнес-операций.

*3. Использование JWT-токенов*

Для доступа к API используются JWT-токены с ограниченным сроком действия.

Токен содержит:

* идентификатор пользователя,
* идентификатор клиентского приложения,
* перечень ролей,
* срок действия токена,
* служебные атрибуты безопасности.

JWT подписывается криптографическим ключом поставщика удостоверений. Проверка подписи выполняется API Gateway до передачи запроса во внутренний контур Сервиса.

Для снижения риска компрометации токенов используется ограниченный срок их действия. Долгоживущие пользовательские сессии поддерживаются с использованием refresh-токен.

*4. Поддерживаемые сценарии аутентификации*

В Сервисе поддерживаются следующие сценарии аутентификации:

*4.1) Аутентификация пользовательских мобильных приложений*

Мобильные приложения используют OAuth 2.0/OpenID Connect для аутентификации пользователя и получения JWT-токена доступа.

После успешной аутентификации мобильное приложение использует токен при обращении к:

* API проверки доступности товаров,
* API резервирования,
* API получения статуса операций,
* WebSocket-подключениям.

*4.2) Аутентификация внешних интеграционных систем*

Внешние интеграционные системы используют отдельные клиентские учетные записи и сервисные токены доступа.

Для интеграционных подключений применяются отдельные политики доступа и ограничения разрешенных API.

*4.3) Аутентификация административных пользователей*

Пользователи, выполняющие административные операции сопровождения, проходят аутентификацию через того же поставщика удостоверений, но получают отдельные административные роли.

Административные полномочия проверяются API Gateway и дополнительно контролируются внутренними сервисами при выполнении критичных операций изменения конфигурации, управления доступом и сопровождения платформы.

*5. Обработка истечения срока действия токенов*

При истечении срока действия JWT-токена API Gateway отклоняет запрос и возвращает ошибку авторизации.

Для продления пользовательской сессии клиентское приложение использует refresh-токен и получает новый JWT-токен без повторного ввода учетных данных пользователя.

При отзыве пользовательской сессии или компрометации учетной записи refresh-токен аннулируется поставщиком удостоверений, после чего дальнейшее обновление токенов становится невозможным.

===== 3.1.2.2 Механизм аутентификации сервисов

Для межсервисного взаимодействия внутри Сервиса используется взаимная аутентификация компонентов на основе mTLS и сервисных сертификатов.

Механизм сервисной аутентификации применяется для:

* gRPC-взаимодействия микросервисов,
* подключения к Event Bus,
* взаимодействия с PostgreSQL, MongoDB, Redis Cluster,
* доступа к HashiCorp Vault,
* взаимодействия интеграционных адаптеров с внутренними сервисами.

Внутренние сервисы *не используют анонимные подключения* и не выполняют межсервисные вызовы *без подтверждения подлинности* вызывающего компонента.

*1. Межсервисная аутентификация*

Каждый внутренний сервис Сервиса использует собственную сервисную учетную запись и собственный сертификат.

При установлении соединения вызывающий сервис подтверждает свою подлинность с использованием клиентского сертификата, а принимающий сервис подтверждает подлинность своего сервера.

В рамках межсервисного взаимодействия проверяются:

* подлинность сертификата вызывающей стороны,
* принадлежность сертификата доверенному удостоверяющему центру,
* срок действия сертификата,
* допустимость использования сертификата для конкретного типа взаимодействия.

Если сертификат не проходит проверку или не относится к доверенной инфраструктуре, соединение отклоняется до передачи запроса на уровень бизнес-логики.

*2. Использование mTLS*

Для внутреннего взаимодействия сервисов используется mTLS поверх HTTPS/gRPC.

mTLS применяется для:

* вызовов между микросервисами,
* подключения поставщиков событий и подписчиков к Event Bus,
* доступа к HashiCorp Vault,
* защищенного подключения к внутренним API.

При использовании mTLS *обе стороны соединения* выполняют взаимную проверку сертификатов, что исключает возможность подключения неподтвержденного компонента к внутреннему контуру платформы.

Даже при нахождении сервисов внутри одного Kubernetes-кластера все межсервисные соединения требуют прохождения взаимной аутентификации.

*3. Выдача сервисных сертификатов*

Каждому сервису выдается отдельный сертификат, содержащий идентификатор сервиса и параметры допустимого использования.

Сертификаты используются:

* для межсервисного взаимодействия,
* для подключения к Event Bus,
* для доступа к HashiCorp Vault,
* для взаимодействия интеграционных адаптеров,
* для подключения к внутренним API.

Сервисные сертификаты имеют ограниченный срок действия и подлежат периодической ротации.

Закрытые ключи сервисов не размещаются в контейнерных образах и не хранятся в исходном коде. Получение сертификатов и закрытых ключей выполняется через защищенные механизмы доставки секретов из HashiCorp Vault.

*4. Доверенная инфраструктура сертификатов (PKI)*

Проверка сервисных сертификатов выполняется на основе доверенной внутренней инфраструктуры сертификатов (корпоративный PKI).

Внутренние сервисы доверяют только сертификатам, выпущенным доверенным удостоверяющим центром платформы.

При установлении соединения выполняется:

* проверка цепочки доверия сертификата,
* проверка срока действия,
* проверка отзыва сертификата,
* проверка допустимого назначения сертификата.

Сертификаты, не принадлежащие доверенной инфраструктуре, не допускаются к взаимодействию с внутренними сервисами и хранилищами данных.

*5. Аутентификация сервисов при работе с Event Bus*

Поставщики событий и подписчики Event Bus используют сервисную аутентификацию при подключении к Kafka-кластеру.

Каждый сервис получает отдельные учетные данные и отдельные права:

* на публикацию событий,
* на чтение топиков,
* на использование consumer group.

Например:

* Сервис резервирования имеет право *публиковать* события в `reservation.events`,
* загрузчик модели чтения имеет право *читать* `reservation.events` и `inventory.events`,
* сервис прогрева кеша имеет право *читать* `inventory.events`.

Сервисы не получают универсального доступа ко всем топикам Event Bus.

*6. Аутентификация при доступе к хранилищам данных*

Для PostgreSQL, MongoDB, Redis Cluster и HashiCorp Vault используются отдельные сервисные учетные записи и отдельные секреты подключения.

Каждый сервис использует только собственные учетные данные и получает минимально необходимые полномочия.

Например:

* Сервис запросов имеет доступ только на *чтение* модели чтения и Redis Cluster,
* Сервис резервирования имеет доступ к операциям *записи* в БД резервирования,
* загрузчик модели чтения имеет доступ к *обновлению* MongoDB,
* сервисы, не работающие с секретами, *не получают* доступа к HashiCorp Vault.

Такой подход ограничивает распространение инцидента безопасности при компрометации отдельного компонента и снижает риск получения несанкционированного доступа к внутреннему контуру Сервиса.

===== 3.1.2.3 Формат и структура токенов доступа

JWT-токены используются API Gateway и внутренними сервисами для:

* идентификации пользователя,
* определения ролей и полномочий,
* проверки допустимости выполнения операций,
* контроля срока действия пользовательской сессии.

Внутренние сервисы не используют серверные пользовательские сессии и не хранят состояние аутентификации пользователя локально. Все необходимые данные для проверки доступа передаются в составе JWT-токена.

*1. Использование JWT*

При успешной аутентификации внешний поставщик удостоверений выдает клиентскому приложению JWT-токен доступа и refresh-токен.

JWT-токен передается клиентским приложением:

* при вызове REST API,
* при установлении WebSocket-соединения,
* при выполнении операций резервирования,
* при обращении к административным API.

Передача токена выполняется через HTTP-заголовок Authorization.

API Gateway выполняет:

* проверку подписи JWT,
* проверку срока действия токена,
* проверку issuer,
* проверку audience,
* проверку допустимости использования токена.

Только после успешной проверки запрос передается во внутренний контур Сервиса.

*2. Состав полезной нагрузки токена*

JWT-токен содержит минимально необходимый набор данных, используемый для аутентификации и авторизации запросов.

В состав полезной нагрузки токена входят:

* уникальный идентификатор пользователя,
* идентификатор клиентского приложения,
* перечень ролей пользователя,
* идентификатор пользовательской сессии,
* срок действия токена,
* идентификатор поставщика удостоверений,
* служебные атрибуты безопасности.

Дополнительно токен может содержать:

* тип клиентского приложения,
* идентификатор интеграционного канала,
* атрибуты административного доступа,
* признаки сервисного взаимодействия.

Конфиденциальные данные пользователя, пароли и секреты *в JWT-токен не включаются*.

*3. Роли и привилегии в токене*

JWT-токен содержит *перечень ролей и атрибутов доступа*, используемых API Gateway и внутренними сервисами при проверке полномочий.

В зависимости от типа субъекта токен может содержать:

* пользовательские роли,
* административные роли,
* роли интеграционных сервисов,
* ограничения доступа к отдельным API.

Роли используются:

* API Gateway — для базовой авторизации запросов,
* внутренними сервисами — для проверки допустимости бизнес-операций,
* административными API — для ограничения критичных операций сопровождения.

Например:

* клиентское приложение получает доступ *только к пользовательским API*,
* интеграционные сервисы получают доступ *только к разрешенным операциям* обмена данными,
* административные учетные записи получают доступ *к операциям сопровождения и управления конфигурацией*.

Наличие валидного JWT-токена не означает автоматическое разрешение всех операций. Внутренние сервисы *дополнительно проверяют допустимость конкретного действия* для вызывающей стороны.

*4. Срок действия токена*

JWT-токены доступа имеют ограниченный срок действия. Использование короткоживущих токенов снижает риск длительного использования скомпрометированного токена при его утечке или перехвате.

После истечения срока действия JWT-токена API Gateway отклоняет запрос и возвращает ошибку авторизации.

Для поддержания пользовательской сессии используется refresh-токен, позволяющий клиентскому приложению получить новый JWT-токен без повторного ввода учетных данных пользователя.

При компрометации учетной записи, завершении пользовательской сессии или отзыве доступа refresh-токен аннулируется поставщиком удостоверений, после чего дальнейшее обновление JWT-токенов становится невозможным.

===== 3.1.2.4 Выдача, проверка и обновление токенов

Выдача, проверка и обновление JWT-токенов в Сервисе выполняются через внешний поставщик удостоверений и API Gateway.

Внутренние микросервисы не выполняют самостоятельную выдачу пользовательских токенов и не хранят локальные пользовательские сессии. Управление жизненным циклом токенов централизовано и выполняется через внешний контур аутентификации.

*1. Процесс выдачи токенов*

Получение JWT-токена выполняется после успешной аутентификации пользователя или интегрируемой системы у внешнего поставщика удостоверений.

Процесс выдачи токенов включает следующие этапы:

. Клиентское приложение инициирует процедуру аутентификации.
. Пользователь проходит проверку учетных данных у поставщика удостоверений.
. После успешной аутентификации поставщик удостоверений формирует:

* JWT-токен доступа,
* refresh-токен,

. Токены передаются клиентскому приложению по защищенному TLS-соединению.
. Клиентское приложение использует JWT-токен при обращении к API Сервиса.

Для интеграционных сервисов и административных учетных записей используются *отдельные политики выдачи токенов* и *отдельные ограничения полномочий*.

*2. Проверка подписи и срока действия токена*

Проверка JWT-токенов выполняется API Gateway до передачи запроса во внутренний контур Сервиса.
При обработке запроса API Gateway проверяет различные параметры безопасности токена. 
Только после успешного завершения проверки запрос передается внутренним сервисам.

API Gateway отклоняет запрос и возвращает ошибку авторизации c случае:

* истечения срока действия токена,
* нарушения подписи,
* использования неподдерживаемого issuer,
* отсутствия необходимых ролей,
* попытки использования отозванного токена,

Внутренние сервисы доверяют результату проверки подписи со стороны API Gateway, однако *дополнительно проверяют допустимость выполнения конкретной бизнес-операции для вызывающей стороны*.

*3. Использование refresh-токена*

Для поддержания пользовательской сессии используются refresh-токен.

Refresh-токен имеет больший срок действия по сравнению с JWT-токеном доступа и используется клиентским приложением для получения нового JWT-токена без повторного ввода учетных данных пользователя.

Процесс обновления токена выполняется следующим образом:

. Клиентское приложение обнаруживает истечение срока действия JWT.
. Клиентское приложение обращается к поставщику удостоверений с refresh-токеном.
. Поставщик удостоверений проверяет:

* валидность refresh-токена,
* срок действия,
* состояние пользовательской сессии,
* отсутствие блокировки учетной записи.

. После успешной проверки выдается новый JWT-токен доступа.
. Старый JWT-токен становится недействительным после истечения срока действия.

Refresh-токен не используется для прямого доступа к API Сервиса и применяется только для процедуры обновления пользовательской сессии.

*4. Отзыв и блокировка токенов*

Архитектура Сервиса предусматривает возможность досрочного прекращения действия пользовательских токенов.

Отзыв токенов выполняется в следующих случаях:

* завершение пользовательской сессии,
* блокировка учетной записи,
* компрометация токена,
* изменение пользовательских полномочий,
* обнаружение подозрительной активности.

При отзыве доступа поставщик удостоверений аннулирует refresh-токен и запрещает дальнейшее обновление пользовательской сессии.

Даже при наличии ранее выданного JWT-токена пользователь теряет возможность получения новых токенов доступа после завершения срока действия текущего токена, так как срок жизни JWT-токена сущственно меньше срока жизни refresh-токена.

Для административных учетных записей и интеграционных сервисов дополнительно используется принудительное завершение активных сессий при изменении ролей, компрометации учетных данных или нарушении политик безопасности.

События выдачи, обновления, отзыва и блокировки токенов фиксируются в журналах аудита безопасности и используются для выявления аномальной активности и расследования инцидентов.

===== 3.1.2.5 Обработка истечения срока действия токенов

Архитектура Сервиса предусматривает *контролируемую обработку истечения срока действия* JWT-токенов и пользовательских сессий.

Механизмы обработки истечения токенов направлены на:

* ограничение времени использования скомпрометированного токена,
* предотвращение доступа с использованием просроченных учетных данных,
* поддержание непрерывности пользовательской сессии,
* предотвращение выполнения операций с недействительными полномочиями.

Контроль срока действия токенов выполняется API Gateway до передачи запроса во внутренний контур платформы.

*1. Сценарии истечения срока действия токенов*

В Сервисе учитываются следующие сценарии завершения действия JWT-токенов:

* штатное истечение срока действия access-токена,
* истечение срока действия refresh-токена,
* принудительное завершение пользовательской сессии,
* отзыв токена при компрометации учетной записи,
* изменение ролей и полномочий пользователя,
* блокировка учетной записи.

При использовании просроченного JWT-токена API Gateway отклоняет запрос и не передает его внутренним сервисам.

Для WebSocket-подключений дополнительно контролируется актуальность пользовательской сессии после установки соединения. При завершении срока действия токена активное соединение подлежит повторной авторизации или закрытию.

*2. Автоматическое обновление токенов*

Для пользовательских мобильных приложений поддерживается автоматическое обновление JWT-токенов с использованием refresh-токена.

Процесс обновления выполняется следующим образом:

. Клиентское приложение обнаруживает приближение срока истечения JWT-токена.
. До истечения срока действия access-токена приложение инициирует запрос обновления.
. Поставщик удостоверений проверяет:

* валидность refresh-токена,
* состояние пользовательской сессии,
* отсутствие блокировки учетной записи,
* отсутствие отзыва refresh-токена.

. После успешной проверки выдается новый JWT-токен доступа.
. Клиентское приложение продолжает работу без повторной аутентификации пользователя.

Refresh-токен используется только для процедуры обновления сессии и не применяется для прямого доступа к API Сервиса.

Автоматическое обновление токенов позволяет *избежать массового завершения пользовательских сессий* при длительной работе мобильного приложения и *снижает вероятность повторной аутентификации пользователя* во время выполнения бизнес-операций.

*3. Принудительная повторная аутентификация*

Повторная аутентификация пользователя требуется в случаях:

* истечения срока действия refresh-токена,
* отзыва пользовательской сессии,
* компрометации учетной записи,
* изменения критичных пользовательских полномочий,
* длительного отсутствия активности,
* обнаружения аномальной активности.

В указанных сценариях поставщик удостоверений прекращает возможность обновления JWT-токенов, после чего клиентское приложение должно повторно выполнить процедуру аутентификации пользователя.

Для административных учетных записей дополнительно применяется принудительное завершение активных сессий при:

* изменении административных ролей,
* компрометации учетных данных,
* нарушении политик безопасности,
* выполнении операций блокировки учетной записи.

*4. Обработка ошибок авторизации*

Ошибки авторизации обрабатываются API Gateway до передачи запроса во внутренние сервисы.

Причинами отказа авторизации являются:

* истечение срока действия JWT-токена,
* использование отозванного токена,
* нарушение подписи JWT,
* отсутствие необходимых ролей,
* использование недоверенного issuer,
* нарушение ограничений доступа.

При обнаружении ошибки API Gateway:

* отклоняет запрос,
* возвращает код ошибки авторизации,
* не передает запрос внутренним сервисам,
* фиксирует событие в журнале аудита безопасности.

Внутренние сервисы дополнительно выполняют проверку полномочий для бизнес-операций и могут возвращать отказ авторизации даже при наличии валидного JWT-токена, если вызывающая сторона не имеет права выполнять конкретное действие.

Например:

* пользователь не имеет права изменять чужой резерв,
* интеграционный сервис пытается выполнить недопустимую операцию,
* клиентское приложение обращается к административному API.

События истечения токенов, ошибок авторизации, повторной аутентификации и блокировки пользовательских сессий передаются в подсистему аудита безопасности и используются для выявления аномальной активности и расследования инцидентов.

==== 3.1.3 Управление доступом

===== 3.1.3.1 Модель разграничения доступа

Для разграничения доступа в Сервисе используется ролевая модель RBAC. Управление доступом строится на основе ролей, определяющих перечень допустимых API, бизнес-операций и административных действий для пользователя, клиентского приложения или внутреннего сервиса.

*1. Причины выбора модели RBAC*

Использование RBAC обусловлено особенностями архитектуры Сервиса и составом поддерживаемых сценариев доступа.

В Сервисе присутствуют несколько категорий субъектов доступа:

* пользовательские мобильные приложения,
* интеграционные сервисы,
* внутренние микросервисы,
* административные учетные записи,
* сервисные учетные записи инфраструктурных компонентов.

Для каждой категории требуется фиксированный и контролируемый набор полномочий.

Использование ролевой модели позволяет:

* централизованно управлять полномочиями,
* ограничивать доступ к критичным операциям,
* разделять пользовательские и административные функции,
* разграничивать доступ внутренних сервисов к компонентам платформы,
* упрощать аудит операций безопасности,
* ограничивать распространение привилегий при компрометации учетной записи или сервиса.

RBAC используется как единая модель разграничения доступа для всех компонентов Сервиса, включая REST API, gRPC-взаимодействие и Event Bus.

*2. Связь ролей и полномочий*

Каждая роль в Сервисе определяет перечень разрешенных операций и доступных ресурсов.

Роли используются:

* API Gateway — для базовой авторизации запросов,
* внутренними сервисами — для проверки допустимости бизнес-операций,
* Kafka — для разграничения публикации и чтения топиков,
* HashiCorp Vault — для ограничения доступа к секретам,
* PostgreSQL и MongoDB — для ограничения доступа к данным.

Полномочия привязываются не к отдельным пользователям напрямую, а к ролям, включаемым в JWT-токен.

Например:

* пользовательские роли позволяют выполнять операции проверки доступности товаров и резервирования,
* интеграционные роли позволяют выполнять только разрешенные операции обмена данными,
* административные роли позволяют выполнять операции сопровождения и управления конфигурацией,
* сервисные роли ограничивают доступ внутренних компонентов к Kafka, Vault и хранилищам данных.

Даже при наличии роли сервис дополнительно проверяет допустимость выполнения конкретной бизнес-операции.

*3. Принципы назначения ролей*

Назначение ролей выполняется централизованно через внешний поставщик удостоверений и политики безопасности Сервиса.

При выдаче access-токена поставщик удостоверений включает в JWT:

* идентификатор пользователя,
* перечень ролей,
* атрибуты доступа,
* ограничения полномочий.

API Gateway и внутренние сервисы используют данные роли при принятии решения о допустимости выполнения операции.

При назначении ролей используются следующие принципы:

* минимально необходимые привилегии,
* разделение пользовательских и административных полномочий,
* отдельные роли для интеграционных сервисов,
* отдельные роли для внутренних микросервисов,
* ограничение административного доступа,
* запрет универсальных полномочий.

Внутренние сервисы используют отдельные сервисные роли и отдельные учетные данные. Сервисные роли не используются для пользовательского доступа.

Например:

* Сервис резервирования имеет права публикации `reservation.events`,
* загрузчик модели чтения имеет права чтения `reservation.events` и `inventory.events`,
* сервис агрегации метрик имеет доступ только к аналитическим потокам данных,
* сервис запросов имеет доступ только на чтение модели чтения и Redis Cluster.

Изменение ролей и политик доступа выполняется централизованно и сопровождается аудитом операций безопасности.

===== 3.1.3.2 Роли и привилегии пользователей

В Сервисе используется разделение ролей по типам субъектов доступа и выполняемым функциям. Роли определяют перечень допустимых API, операций и внутренних ресурсов, к которым может обращаться пользователь, сервис или интеграционный компонент.

Назначение ролей выполняется внешним поставщиком удостоверений и передается в составе JWT access-токена. API Gateway и внутренние сервисы используют роли при принятии решения о допустимости выполнения операций.

Далее представлены расширенные описания обобщенных ролей, отстсвующих в таблице RBAC в предыдущем разделе:

*Роль «Администратор»*

Предназначена для выполнения операций сопровождения и управления компонентами Сервиса.

Пользователи с данной ролью имеют право:

* выполнять операции изменения конфигурации,
* управлять справочными данными,
* просматривать служебную информацию,
* управлять политиками доступа,
* выполнять операции блокировки и отзыва пользовательских сессий,
* просматривать журналы аудита безопасности,
* выполнять регламентные операции сопровождения.

Административные операции доступны только через защищенные административные API после прохождения аутентификации и проверки полномочий.

Для административных учетных записей дополнительно применяются:

* ограничение срока действия сессий,
* аудит действий,
* контроль критичных операций,
* принудительное завершение сессий при изменении ролей или компрометации учетной записи.

Роль «Администратор» не предоставляет универсального доступа ко всем компонентам платформы и не отменяет ограничения, установленные на уровне отдельных сервисов и хранилищ данных.

*Роль «Оператор»*

Используется для выполнения ограниченного набора операций сопровождения и контроля работы Сервиса.

Пользователи с данной ролью могут:

* просматривать состояние операций резервирования,
* просматривать технические статусы компонентов,
* выполнять операции мониторинга,
* просматривать журналы обработки запросов,
* выполнять разрешенные регламентные операции.

Роль «Оператор» не предоставляет прав:

* изменения конфигурации безопасности,
* управления ролями пользователей,
* доступа к секретам и сертификатам,
* выполнения административных операций изменения политик доступа.

Операторы работают только через разрешенные интерфейсы сопровождения и не имеют прямого доступа к внутренним хранилищам данных и Event Bus.

*Роль «Системный сервис»*

Используется внутренними микросервисами платформы для межсервисного взаимодействия.

Данная роль применяется:

* при gRPC-вызовах,
* при подключении к Event Bus,
* при работе с PostgreSQL,
* при работе с MongoDB,
* при доступе к Redis Cluster,
* при обращении к HashiCorp Vault.

Каждый сервис использует отдельную сервисную учетную запись и отдельный набор полномочий.

Например:

* Сервис резервирования имеет право публикации `reservation.events`,
* загрузчик модели чтения имеет право чтения `reservation.events` и `inventory.events`,
* сервис запросов имеет доступ только к чтению модели чтения и Redis Cluster,
* сервис агрегации метрик имеет доступ только к аналитическим потокам событий.

Системные сервисы не используют пользовательские роли и не имеют универсального доступа ко всем компонентам платформы.

*Роль «Интеграционный сервис»*

Используется для внешних интеграционных компонентов:

* адаптера касс (POS),
* адаптера склада (WMS),
* адаптера заказов (OMS),
* адаптера ERP,
* адаптера служб доставки,
* адаптера прогнозов (PPE).

Для каждого интеграционного компонента используются отдельные учетные данные и отдельные политики доступа.

Интеграционные сервисы получают доступ только к тем API, топикам Event Bus и операциям, которые необходимы для их интеграционного сценария.

Например:

* адаптер касс (POS) публикует события изменения остатков,
* адаптер ERP обновляет разрешенные справочные данные,
* адаптер заказов (OMS) взаимодействует с API резервирования,
* адаптер служб доставки получает статусы резервирования.

Интеграционные сервисы не имеют доступа:

* к административным API,
* к внутренним сервисным операциям,
* к секретам других компонентов,
* к неразрешенным топикам Event Bus.

*Роль «Клиентское приложение»*

Используется пользовательскими мобильными приложениями и внешними клиентскими интерфейсами.

Клиентские приложения имеют право:

* выполнять запросы проверки доступности товаров,
* получать список ближайших магазинов,
* выполнять операции резервирования,
* получать статусы операций,
* использовать WebSocket-подключения для получения обновлений.

Клиентские приложения не имеют доступа:

* к внутренним gRPC API,
* к Event Bus,
* к PostgreSQL, MongoDB и Redis Cluster,
* к операциям сопровождения,
* к административным функциям,
* к внутренним сервисным операциям.

Все запросы клиентских приложений проходят через API Gateway и дополнительно проверяются внутренними сервисами на уровне бизнес-полномочий.

Например, наличие валидного JWT access-токена не позволяет пользователю изменять чужие резервы или выполнять административные операции.

===== 3.1.3.3 Политики авторизации API

Политики авторизации API определяют правила доступа к REST API Сервиса, разграничивают пользовательские, интеграционные и административные операции, а также ограничивают выполнение критичных бизнес-действий.

Контроль авторизации реализуется на двух уровнях:

* API Gateway,
* внутренние микросервисы.

API Gateway выполняет первичную проверку токенов и ролей, а внутренние сервисы дополнительно контролируют допустимость выполнения конкретной бизнес-операции.

*1. Контроль доступа к REST API*

Все внешние REST API Сервиса доступны только через API Gateway.

API Gateway выполняет описанный ранее набор проверок и ограничений.

REST API Сервиса разделяются на несколько категорий:

* пользовательские API,
* интеграционные API,
* административные API,
* служебные API сопровождения.

Для каждой категории API определяются собственные политики авторизации и перечень допустимых ролей.

Например:

* API проверки доступности товаров доступны клиентским приложениям,
* API резервирования доступны только аутентифицированным пользователям и разрешенным интеграционным сервисам,
* административные API доступны только учетным записям сопровождения,
* внутренние служебные API недоступны из внешнего контура.

*2. Проверка ролей*

При обработке запроса API Gateway извлекает роли из JWT access-токена и проверяет наличие полномочий для вызова конкретного API.

Роли используются для ограничения:

* доступных HTTP-методов,
* перечня доступных API,
* допустимых бизнес-операций,
* доступа к административным функциям.

Например:

* клиентское приложение может выполнять GET-запросы проверки доступности товаров,
* пользователь может создавать резерв,
* интеграционный сервис OMS может взаимодействовать с API резервирования,
* учетная запись сопровождения может выполнять административные операции.

При отсутствии необходимой роли запрос отклоняется до передачи во внутренний контур Сервиса.

*3. Проверка полномочий на уровне бизнес-операций*

После прохождения API Gateway внутренние сервисы дополнительно проверяют допустимость выполнения конкретного действия для вызывающей стороны.

Проверка бизнес-полномочий включает:

* проверку принадлежности операции разрешенному типу субъекта,
* проверку допустимости изменения конкретного объекта,
* проверку ограничений бизнес-логики,
* проверку идемпотентности операций,
* проверку ограничений интеграционного канала.

Например:

* пользователь не может изменять чужой резерв,
* интеграционный сервис не может выполнять административные операции,
* клиентское приложение не может напрямую изменять модель чтения,
* сервис без соответствующих полномочий не может публиковать события `reservation.events`.

Даже наличие валидного JWT access-токена и разрешенной роли не означает автоматического доступа ко всем бизнес-операциям.

*4. Ограничение административных операций*

Административные API и операции сопровождения изолированы от пользовательского контура и доступны только учетным записям с административными ролями.

К административным операциям относятся:

* изменение конфигурации компонентов,
* управление политиками доступа,
* управление сертификатами и секретами,
* изменение справочных данных,
* просмотр журналов аудита,
* выполнение регламентных операций сопровождения.

Для административных операций дополнительно применяются:

* сокращенный срок действия пользовательской сессии,
* обязательная повторная проверка полномочий,
* аудит операций,
* журналирование изменений,
* возможность принудительного завершения административной сессии.

Административные API недоступны:

* клиентским приложениям,
* интеграционным сервисам,
* внутренним пользовательским сценариям,
* сервисам без административных полномочий.

Попытки обращения к административным операциям без необходимых ролей фиксируются в журналах аудита безопасности и рассматриваются как события повышенного уровня критичности.

*5. Авторизация WebSocket-подключений*

Для WebSocket-подключений используются отдельные политики авторизации.

Перед установлением WebSocket-соединения API Gateway выполняет:

* проверку JWT access-токена,
* проверку срока действия токена,
* проверку допустимости использования WebSocket,
* проверку ролей пользователя.

WebSocket-соединение получает доступ только к пользовательским событиям, относящимся к текущей пользовательской сессии.

Подключение не может использоваться:

* для административных операций,
* для доступа к внутренним сервисным событиям,
* для чтения событий других пользователей,
* для обхода политик REST API.

===== 3.1.3.4 Управление ролями и политиками доступа

Управление ролями и политиками доступа в Сервисе выполняется централизованно через внешний поставщик удостоверений и согласованные политики безопасности платформы.
Изменение ролей и политик доступа выполняется только административными учетными записями с соответствующими полномочиями.

*1. Создание новых ролей*

Создание новой роли выполняется при появлении нового пользовательского, интеграционного или сервисного сценария взаимодействия.

При создании роли определяются:

* перечень разрешенных API,
* допустимые HTTP-методы,
* доступные бизнес-операции,
* разрешенные топики Event Bus,
* допустимые хранилища данных,
* доступ к секретам и сертификатам,
* ограничения административных операций.

Создание универсальных ролей с *неограниченными полномочиями запрещено*.

Для каждой новой роли применяется принцип минимально необходимых привилегий. Роль получает только те полномочия, которые требуются для выполнения конкретного сценария работы.

Например:

* интеграционный сервис получает доступ только к своим API и топикам Event Bus,
* сервис запросов получает только права чтения,
* сервис резервирования получает права записи только в разрешенные компоненты,
* сервис агрегации метрик получает доступ только к аналитическим потокам данных.

*2. Изменение существующих ролей*

Изменение ролей выполняется централизованно и используется при:

* изменении бизнес-требований,
* появлении новых API,
* изменении интеграционных сценариев,
* изменении политик безопасности,
* выявлении избыточных полномочий,
* расследовании инцидентов безопасности.

Изменение роли может включать:

* добавление/удаление полномочий,
* изменение ограничений доступа,
* изменение разрешенных API,
* изменение доступа к Event Bus, хранилищам данных.

Все изменения ролей фиксируются в журналах аудита безопасности.

Для административных ролей дополнительно применяется:

* контроль критичных изменений,
* журналирование операций,
* возможность принудительного завершения активных пользовательских сессий после изменения полномочий.

*3. Жизненный цикл ролей*

Каждая роль в Сервисе имеет управляемый жизненный цикл, который включает:

. создание роли,
. назначение роли субъектам доступа,
. использование роли при выполнении операций,
. изменение полномочий роли,
. аудит использования роли,
. отзыв или удаление роли.

При выводе роли из эксплуатации:

* прекращается выдача новых access-токенов с данной ролью,
* выполняется отзыв активных пользовательских сессий,
* аннулируются связанные сервисные полномочия,
* блокируется дальнейшее использование роли в API Gateway и внутренних сервисах.

Для сервисных ролей дополнительно выполняется отзыв или ротация связанных учетных данных и сертификатов.

*4. Централизованное хранение политик доступа*

Политики доступа хранятся централизованно и используются всеми компонентами платформы.

Централизованное хранение политик позволяет обеспечить:

* единообразную авторизацию,
* согласованность ролей,
* централизованное изменение политик,
* контроль использования полномочий,
* аудит изменений безопасности.

API Gateway использует централизованные политики для проверки доступа к REST API и WebSocket-подключениям.

Внутренние сервисы используют те же роли и атрибуты доступа при проверке бизнес-полномочий.

Для Event Bus политики доступа определяют:

* разрешенные поставщики сообщений,
* разрешенные подписчики,
* доступные топики,
* допустимые consumer group.

Для HashiCorp Vault политики определяют:

* доступ к секретам,
* доступ к сертификатам,
* допустимые операции с ключами,
* ограничения сервисных учетных записей.

Для PostgreSQL, MongoDB и Redis Cluster используются отдельные сервисные учетные записи и отдельные политики доступа, соответствующие ролям конкретных сервисов.

===== 3.1.3.5 Контроль соответствия политик доступа

В Сервисе реализуется постоянный контроль соответствия ролей и политик доступа фактическим полномочиям пользователей, интеграционных компонентов и внутренних сервисов.

Контроль политик доступа направлен на:

* предотвращение накопления избыточных привилегий,
* ограничение распространения доступа между компонентами,
* выявление неиспользуемых или ошибочно назначенных полномочий,
* контроль соблюдения принципа минимально необходимых привилегий,
* снижение риска компрометации критичных компонентов платформы.

Контроль выполняется для:

* пользовательских ролей,
* административных ролей,
* сервисных ролей,
* интеграционных учетных записей,
* политик доступа Event Bus,
* политик доступа HashiCorp Vault,
* учетных записей PostgreSQL, MongoDB и Redis Cluster.

*1. Периодическая проверка прав доступа*

Для всех критичных ролей и учетных записей выполняется регулярная проверка назначенных полномочий.

Проверка включает:

* анализ назначенных ролей,
* проверку актуальности полномочий,
* проверку использования ролей,
* проверку доступа к административным операциям,
* проверку сервисных учетных записей,
* проверку доступа к Event Bus и HashiCorp Vault.

Особое внимание уделяется:

* административным ролям,
* сервисам с доступом к `reservation.events`,
* сервисам с доступом к HashiCorp Vault,
* учетным записям сопровождения,
* интеграционным сервисам с расширенными полномочиями.

При обнаружении неиспользуемых или избыточных прав выполняется корректировка ролей и ограничение полномочий.

*2. Автоматизированный анализ политик доступа*

Политики доступа анализируются автоматически на основе:

* конфигурации API Gateway,
* ролей из внешнего поставщика удостоверений,
* политик Event Bus,
* политик HashiCorp Vault,
* сервисных учетных записей PostgreSQL, MongoDB и Redis Cluster,
* журналов аудита безопасности.

Автоматизированный анализ позволяет выявлять:

* роли с избыточными полномочиями,
* сервисы с неиспользуемым доступом,
* попытки обращения к запрещенным API,
* нарушения разделения ролей,
* расширение полномочий вне утвержденной модели доступа.

Дополнительно анализируются:

* обращения к административным API,
* попытки чтения неразрешенных топиков Event Bus,
* попытки получения неразрешенных секретов,
* обращения сервисов к неразрешенным хранилищам данных.

*3. Выявление избыточных привилегий*

В рамках контроля безопасности выполняется выявление ролей и учетных записей, обладающих полномочиями, не требующимися для выполнения их функций.

К избыточным привилегиям относятся:

* доступ к неиспользуемым API,
* доступ к неразрешенным топикам Event Bus,
* права записи вместо чтения,
* доступ к административным операциям без необходимости,
* доступ к секретам других сервисов,
* использование расширенных ролей интеграционными компонентами.

Например:

* сервис запросов не должен иметь права изменения БД резервирования,
* сервис прогрева кеша не должен публиковать события `reservation.events`,
* интеграционный сервис не должен иметь доступ к административным API,
* пользовательская роль не должна иметь доступ к операциям сопровождения.

При выявлении избыточных привилегий выполняется:

* ограничение роли,
* изменение политики доступа,
* отзыв лишних полномочий,
* аудит связанных операций безопасности.

*4. Контроль принципа минимальных полномочий*

Во всех компонентах Сервиса реализуется контроль соблюдения принципа минимально необходимых полномочий.

Для каждой роли определяется строго ограниченный перечень:

* доступных API,
* допустимых бизнес-операций,
* разрешенных топиков Event Bus,
* доступных хранилищ данных,
* разрешенных секретов и сертификатов.

Сервисы не получают универсального доступа к инфраструктуре платформы.

Например:

* Сервис резервирования имеет доступ только к операциям резервирования и публикации `reservation.events`,
* загрузчик модели чтения имеет доступ только к чтению событий и обновлению MongoDB,
* сервис агрегации метрик не имеет доступа к операциям изменения бизнес-данных,
* клиентские приложения не имеют доступа к внутренним gRPC API,
* интеграционные сервисы не имеют доступа к административным операциям.

Контроль соблюдения принципа минимальных полномочий выполняется:

* при создании новых ролей,
* при изменении политик доступа,
* при подключении новых интеграционных сервисов,
* при выпуске сервисных сертификатов,
* при предоставлении доступа к HashiCorp Vault,
* при настройке прав доступа Event Bus и хранилищ данных.

События изменения политик доступа, расширения полномочий и нарушения ограничений фиксируются в журналах аудита безопасности и используются для расследования инцидентов и контроля соблюдения требований безопасности.

==== 3.1.4 Защита передачи и хранения данных

===== 3.1.4.1 Защита сетевых соединений

Для защиты сетевого взаимодействия в Сервисе используется обязательное шифрование трафика между внешними клиентами, внутренними микросервисами, интеграционными компонентами и инфраструктурными сервисами платформы.

Защита соединений распространяется на:

* REST API,
* WebSocket-подключения,
* gRPC-взаимодействие,
* подключение к Event Bus, PostgreSQL, MongoDB, Redis Cluster,
* взаимодействие с HashiCorp Vault,
* интеграционные подключения внешних сервисов.

Передача *незашифрованного трафика внутри платформы не допускается*.

*1. Использование TLS 1.3*

Для всех внешних и внутренних соединений используется TLS 1.3.

TLS 1.3 применяется:

* между клиентскими приложениями и API Gateway,
* между внутренними микросервисами,
* при gRPC-вызовах,
* при подключении к Event Bus,
* при подключении к HashiCorp Vault,
* при взаимодействии интеграционных адаптеров с внутренними API.

Использование TLS 1.3 обеспечивает:

* шифрование передаваемых данных,
* защиту JWT access-токенов и refresh-токенов,
* защиту сервисных сертификатов,
* защиту служебных gRPC-вызовов,
* защиту потоков событий и интеграционного трафика.

API Gateway выполняет завершение внешних TLS-соединений и обеспечивает проверку корректности защищенного подключения до передачи запросов во внутренний контур платформы.

*2. Запрет устаревших протоколов*

В Сервисе запрещено использование устаревших и небезопасных версий транспортных протоколов.

Не допускается использование:

* SSL,
* TLS 1.0, 1.1,
* устаревших алгоритмов согласования ключей,
* небезопасных режимов шифрования.

Соединения, пытающиеся использовать неподдерживаемые версии TLS или небезопасные параметры подключения, отклоняются API Gateway и внутренними сервисами до передачи данных на уровень приложений.

Для внутренних сервисов и инфраструктурных компонентов применяются единые политики безопасности транспортного уровня.

*3. Использование современных наборов шифров*

Для TLS-соединений используются современные криптографические наборы шифров, обеспечивающие защиту пользовательских и служебных данных, такие как:

* `TLS_AES_128_GCM_SHA256`,
* `TLS_AES_256_GCM_SHA384`,
* `TLS_CHACHA20_POLY1305_SHA256`,
* `TLS_AES_128_CCM_SHA256`,
* `TLS_AES_128_CCM_8_SHA256`.

Используемые наборы шифров обеспечивают:

* защищенный обмен ключами,
* шифрование трафика,
* контроль целостности передаваемых данных,
* устойчивость к атакам на транспортный уровень.

При установлении соединений запрещается использование:

* устаревших симметричных алгоритмов,
* небезопасных режимов блочного шифрования,
* алгоритмов без контроля целостности,
* слабых криптографических параметров.

Политики использования наборов шифров централизованно применяются:

* в API Gateway,
* во внутренних сервисах,
* в Kafka-кластере,
* в PostgreSQL,
* в MongoDB,
* в HashiCorp Vault.

*4. Защита от атак на транспортный уровень*

Архитектура Сервиса предусматривает защиту от атак, направленных на перехват, подмену или нарушение сетевого взаимодействия.

Защита транспортного уровня включает:

* обязательное использование TLS 1.3,
* использование mTLS для внутренних сервисов,
* проверку доверенных сертификатов,
* контроль срока действия сертификатов,
* проверку криптографической подписи JWT access-токенов,
* ограничение небезопасных параметров соединений.

Для внутренних сервисов дополнительно используется взаимная аутентификация через mTLS, предотвращающая подключение неподтвержденных компонентов к внутреннему контуру платформы.

Особое внимание уделяется защите:

* запросов резервирования,
* административных API,
* WebSocket-подключений,
* потоков Event Bus,
* сервисных gRPC-вызовов,
* операций доступа к HashiCorp Vault.

Для предотвращения атак повторной отправки запросов операции резервирования дополнительно используют:

* идентификатор зпроса,
* контроль идемпотентности,
* проверку уникальности операций.

События нарушения TLS-соединений, использования недоверенных сертификатов, ошибок транспортного уровня и попыток использования неподдерживаемых протоколов фиксируются в журналах аудита безопасности и используются для выявления аномальной активности.

===== 3.1.4.2 Дополнительные механизмы защиты канала

Помимо базового шифрования TLS 1.3 в Сервисе используются дополнительные механизмы защиты транспортного уровня, направленные на предотвращение перехвата трафика, подмены соединений и компрометации криптографических ключей.

*1. Использование HSTS*

Для внешних HTTP-интерфейсов Сервиса используется механизм HSTS принудительного использования HTTPS-соединений, предотвращающий использование небезопасного HTTP и атаки понижения уровня защиты соединения.

API Gateway передает клиентским приложениям заголовки HSTS, запрещающие использование незащищенного HTTP-соединения при повторных подключениях.

Использование HSTS позволяет:

* предотвратить принудительное переключение клиента на HTTP,
* снизить риск атак понижения уровня защиты соединения,
* исключить передачу JWT access-токенов и refresh-токенов по незащищенному каналу,
* обеспечить использование HTTPS для всех пользовательских API.

HSTS применяется:

* для пользовательских REST API,
* для административных API,
* для WebSocket-подключений через API Gateway.

Внутренние сервисные соединения используют только HTTPS/gRPC и не поддерживают незашифрованный HTTP-трафик.

*2. Использование Perfect Forward Secrecy*

Для TLS-соединений используются наборы шифров с поддержкой Perfect Forward Secrecy (PFS), обеспечивающий независимость сессионных ключей отдельных соединений и снижающий риск компрометации ранее переданного трафика при утечке долгосрочных криптографических ключей.
PFS использует уникальные временные сеансовые ключе для каждого TLS-соединения, предотвращая расшифровку ранее переданного трафика при компрометации постоянных ключей сервера: сертификата и закрытого ключа сервера.

В Сервисе запрещается использование наборов шифров без поддержки Perfect Forward Secrecy.

*3. Контроль сертификатов*

Во всех защищенных соединениях выполняется обязательная проверка сертификатов.

Проверка сертификатов включает:

. проверку цепочки доверия,
. проверку срока действия,
. проверку доверенного удостоверяющего центра,
. проверку назначения сертификата,
. проверку соответствия имени сервиса,
. проверку статуса отзыва сертификата.

Для внутренних сервисов используются отдельные сервисные сертификаты и доверенная внутренняя инфраструктура сертификатов.
Сервисы доверяют только сертификатам, выпущенным доверенным удостоверяющим центром платформы.

Соединения с сертификатами, которые:

* просрочены, недоверены,
* с нарушенной цепочкой доверия,
* неподходящего назначения,

отклоняются до передачи данных на уровень приложений.

*4. Защита от атак посредника (MitM)*

Архитектура Сервиса предусматривает защиту от атак посредника при внешнем и внутреннем взаимодействии компонентов -- атак Man in the Middle (MitM).

Защита обеспечивается за счет:

* обязательного использования TLS 1.3,
* применения mTLS для внутренних сервисов,
* проверки доверенных сертификатов,
* проверки имени сервиса,
* использования механизмов PFS и HSTS,
* запрета незашифрованных соединений.

Для внутренних gRPC-вызовов используется взаимная аутентификация сервисов через mTLS. Это предотвращает возможность подключения неподтвержденного компонента к внутреннему контуру платформы и исключает подмену внутренних сервисов.

Особое внимание уделяется защите:

* операций резервирования,
* административных API,
* сервисных gRPC-вызовов,
* потоков Event Bus,
* операций доступа к HashiCorp Vault,
* передачи JWT access-токенов и refresh-токенов.

При обнаружении ошибок TLS-проверки, нарушений проверки сертификатов или попыток использования недоверенных соединений запросы отклоняются, а события фиксируются в журналах аудита безопасности.

===== 3.1.4.3 Шифрование и подписание данных

Помимо защиты транспортного уровня в Сервисе используются дополнительные механизмы шифрования, криптографической подписи и контроля целостности данных.

Дополнительная защита применяется для критичных данных, служебных сообщений, токенов доступа и межсервисного взаимодействия, компрометация которых может привести к несанкционированному доступу, подмене операций резервирования или нарушению целостности потоков событий.

*1. Данные, требующие дополнительной защиты*

Дополнительная криптографическая защита применяется для:

* JWT access-токенов, refresh-токенов, административных токенов,
* сервисных сертификатов,
* учетных данных PostgreSQL, MongoDB, Redis Cluster,
* секретов HashiCorp Vault,
* служебных интеграционных ключей,
* конфиденциальных параметров конфигурации.

Дополнительно защищаются:

* события `reservation.events`,
* служебные события Event Bus,
* административные запросы,
* операции управления доступом,
* операции работы с секретами и сертификатами.

Секреты, ключи и учетные данные не размещаются:

* в исходном коде,
* в контейнерных образах,
* в открытых конфигурационных файлах,
* в журналах приложений.

При автоматическом обнаружении подобных исключений в процесе тестирования кода, образов онтейнеров, конфигурационнх файлов или аудите журналов, в журналах аудита безопасности эти события фиксируются.

*2. Электронная подпись сообщений*

JWT access-токены подписываются криптографическим ключом внешнего поставщика удостоверений.
Токены с нарушенной подписью или измененным содержимым отклоняются до передачи запроса внутренним сервисам.

Для Event Bus дополнительно контролируется подлинность поставщика событий и подписчитка через сервисную аутентификацию и доверенные сертификаты.

Критичные служебные сообщения и административные операции проходят проверку подлинности вызывающей стороны до выполнения бизнес-логики.

*3. Защита служебных токенов и секретов*

Служебные токены, сервисные учетные данные и криптографические ключи хранятся в HashiCorp Vault и передаются сервисам только по защищенным каналам.

Для защиты служебных токенов используются:

* ограниченный срок действия,
* сервисная аутентификация,
* ротация секретов,
* отдельные сервисные учетные записи,
* аудит операций доступа.

Внутренние сервисы получают только собственные секреты и не имеют универсального доступа к хранилищу секретов.

Например:

* Сервис резервирования получает только собственные учетные данные подключения,
* сервис запросов не имеет доступа к секретам Kafka producer,
* интеграционные сервисы получают только собственные интеграционные ключи,
* сервисы без необходимости работы с секретами не получают доступа к HashiCorp Vault.

При отзыве сервисной роли или компрометации компонента выполняется отзыв или ротация связанных секретов и сертификатов.

*4. Контроль целостности данных*

Контроль целостности выполняется для пользовательских запросов, межсервисного взаимодействия и потоков событий.

Контроль включает проверку :

* криптографической подписи JWT,
* целостности TLS-соединений,
* подлинности сервисных сертификатов,

и контроль: 

* последовательности обработки событий,
* идемпотентности операций резервирования,
* корректности идентификатора запроса. 

Особое внимание уделяется защите:

* `reservation.events`, `inventory.events`,
* операций резервирования, административных операций,
* межсервисных gRPC-вызовов.

Для предотвращения нарушения целостности потоков данных используется:

* последовательная обработка событий для одного SKU,
* контроль порядка обработки событий,
* проверка допустимости поставщика событий и подписчика,
* ограничение прав публикации в топики Event Bus.

При обнаружении нарушений целостности данных, ошибок подписи, попыток подмены сообщений или некорректных служебных токенов операции отклоняются, а события фиксируются в журналах аудита безопасности.

===== 3.1.4.4 Криптографические алгоритмы

В Сервисе используются современные криптографические алгоритмы, обеспечивающие защиту пользовательских и служебных данных, аутентификацию компонентов платформы, контроль целостности сообщений и безопасность межсервисного взаимодействия.

Криптографические алгоритмы применяются:

* при защите TLS-соединений,
* при подписании JWT access-токенов,
* при работе и защите сервисных и служебных секретов,
* при хранении и передаче конфиденциальных данных,
* при контроле целостности сообщений и событий.

Использование устаревших и *небезопасных алгоритмов в Сервисе запрещено*.

*1. Использование AES-256-GCM*

Для симметричного шифрования данных используется алгоритм AES-256-GCM.

Использование режима GCM обеспечивает :

* контроль целостности,
* защиту от модификации сообщений,
* устойчивость к атакам на транспортный уровень.

Для TLS-соединений используются современные наборы шифров TLS 1.3, ранее определенные в подразделе защиты сетевых соединений.

*2. Использование SHA-256 и SHA-512*

Алгоритмы SHA-256 и SHA-512 используются для:

* контроля целостности данных,
* криптографической подписи JWT access-токенов,
* проверки целостности сообщений,
* формирования и проверки цифровых подписей.

SHA-256 используется как *основной алгоритм* контроля целостности и подписи в пользовательских и межсервисных сценариях.

SHA-512 используется для отдельных *операций повышенной критичности*, связанных с административными и инфраструктурными компонентами платформы.

Использование *устаревших хеш-алгоритмов*, включая MD5 и SHA-1, в Сервисе *запрещено*.

*3. Использование RSA-3072 и ECC*

Для работы сервисных сертификатов, TLS-соединений и криптографической подписи используются алгоритмы RSA-3072 и алгоритмы на эллиптических кривых (ECC).

RSA-3072 применяется:

* для сертификатов внешних API,
* для административных соединений,
* для инфраструктурных сертификатов,
* для совместимости с интеграционными компонентами.

ECC применяется:

* для внутренних сервисных соединений,
* для mTLS,
* для высоконагруженных gRPC-вызовов,
* для мобильных клиентских подключений,
* для снижения вычислительной нагрузки при сохранении криптографической стойкости.

Сервисные сертификаты выпускаются только доверенной инфраструктурой сертификатов платформы и используются для взаимной аутентификации компонентов.

Использование *слабых криптографических ключей* и *неподдерживаемых алгоритмов запрещено*.

*4. Алгоритмы подписи JWT*

JWT access-токены подписываются криптографическими алгоритмами, обеспечивающими проверку подлинности и целостности токена:

* RS256,
* RS512,
* ES256.

Выбор алгоритма зависит от типа клиента, требований производительности и используемой инфраструктуры сертификатов.

Токены:

* с нарушенной подписью,
* с неподдерживаемым алгоритмом,
* с измененным содержимым,
* подписанные недоверенным ключом,

отклоняются до передачи запроса во внутренние сервисы.

*5. Ограничения использования криптографических алгоритмов*

В Сервисе *запрещено использование*:

* SSL, TLS 1.0, 1.1,
* RC4, 3DES, DES,
* MD5, SHA-1,
* CBC-наборов шифров,
* криптографических ключей недостаточной длины,
* неподдерживаемых алгоритмов подписи JWT.

Контроль допустимых алгоритмов выполняется:

* API Gateway,
* внутренними сервисами,
* Kafka-кластером,
* HashiCorp Vault,
* PostgreSQL, MongoDB,
* механизмами сервисной аутентификации.

Попытки использования неподдерживаемых криптографических алгоритмов фиксируются в журналах аудита безопасности и рассматриваются как события нарушения политик безопасности.

==== 3.1.5 Управление секретами и криптографическими материалами

===== 3.1.5.1 Архитектура хранения секретов

Для хранения секретов, сервисных учетных данных, сертификатов и криптографических ключей в Сервисе используется централизованное защищенное хранилище HashiCorp Vault.

HashiCorp Vault используется как единая точка управления секретами платформы и обеспечивает:

* централизованное хранение секретов,
* разграничение доступа к секретам,
* безопасную выдачу учетных данных,
* хранение сервисных сертификатов,
* управление криптографическими ключами,
* аудит операций доступа.

Использование *локального хранения секретов* внутри сервисов или контейнерных образов в Сервисе *запрещено*.

*1. Использование HashiCorp Vault*

HashiCorp Vault используется для хранения:

* учетных данных PostgreSQL, MongoDB, Redis Cluster, Kafka,
* сервисных сертификатов,
* криптографических ключей,
* ключей проверки подписи JWT-токенов,
* интеграционных секретов,
* административных токенов,
* конфиденциальных параметров конфигурации.

Внутренние сервисы получают секреты динамически при запуске или в процессе работы через защищенное TLS/mTLS-взаимодействие.

Секреты не размещаются:

* в исходном коде,
* в Git-репозиториях,
* в контейнерных образах,
* в открытых конфигурационных файлах,
* в переменных окружения без защиты,
* в журналах приложений.

*2. Централизованное хранение секретов*

Централизованное хранение в HashiCorp Vault используется для:

* единообразного управления секретами,
* централизованной ротации ключей,
* ограничения распространения секретов,
* контроля доступа,
* аудита операций безопасности.

Внутренние сервисы *не хранят постоянные локальные копии секретов* и получают только те учетные данные, которые требуются для выполнения их функций.

Например:

* Сервис запросов получает только учетные данные чтения MongoDB и Redis Cluster,
* Сервис резервирования получает только учетные данные БД резервирования и Kafka producer,
* загрузчик модели чтения получает доступ только к MongoDB и разрешенным Kafka consumer group,
* интеграционные сервисы получают только собственные интеграционные ключи.

Универсальный доступ ко всем секретам платформы отсутствует.

*3. Разграничение доступа к секретам*

Доступ к секретам реализуется на основе сервисных ролей и политик безопасности.

Каждый сервис использует:

* собственную сервисную учетную запись,
* собственный сервисный сертификат,
* отдельную политику доступа HashiCorp Vault.

Политики Vault определяют:

* перечень доступных секретов,
* допустимые операции,
* срок действия выдаваемых учетных данных,
* ограничения использования секретов.

Для административных операций доступа к секретам дополнительно применяются:

* ограничение ролей,
* аудит операций,
* сокращенный срок действия административных сессий,
* контроль критичных действий.

*4. Аудит операций с секретами*

Все операции работы с секретами фиксируются в журналах аудита безопасности:

* получение, изменение, отзыв секретов,
* выдача сервисных сертификатов,
* попытки несанкционированного доступа,
* изменение политик Vault,
* операции административного доступа.

Журналы аудита используются для:

* расследования инцидентов безопасности,
* выявления аномальной активности,
* контроля использования сервисных учетных данных,
* контроля соблюдения политик доступа.

Особое внимание уделяется операциям доступа к:

* ключам проверки JWT access-токенов,
* сервисным сертификатам,
* учетным данным Kafka, PostgreSQL,
* административным секретам,
* ключам интеграционных сервисов.

Попытки доступа к неразрешенным секретам фиксируются как *события безопасности повышенного уровня критичности*.

===== 3.1.5.2 Хранение ключей, сертификатов и паролей

Хранение сертификатов, сервисных учетных данных, криптографических ключей и паролей подключения в Сервисе выполняется централизованно через HashiCorp Vault.

Архитектура хранения секретов направлена на:

* предотвращение компрометации учетных данных,
* ограничение распространения секретов между сервисами,
* централизованную ротацию ключей и паролей,
* аудит операций доступа,
* исключение хранения секретов в открытом виде.

Секреты и ключи не размещаются:

* в исходном коде,
* в Git-репозиториях,
* в контейнерных образах,
* в открытых конфигурационных файлах,
* в журналах приложений.

Выявление нарушения этого принципа приводит к компрометации СЕрвиса и его компонентов.

*1. Хранение сертификатов TLS*

В HashiCorp Vault хранятся:

* внутренние сервисные TLS-сертификаты, mTLS,
* доверенные сертификаты внутренних удостоверяющих центров,
* сертификаты для Kafka,
* сертификаты административных API,
* сертификаты интеграционных подключений.

Сервисные сертификаты используются:

* для gRPC-взаимодействия,
* для mTLS между сервисами,
* для подключения к Event Bus,
* для защищенного доступа к HashiCorp Vault,
* для административных соединений.

Каждый сервис использует отдельный сертификат и отдельный закрытый ключ.

Сервисы получают сертификаты через защищенные механизмы доставки секретов после прохождения сервисной аутентификации.

*2. Хранение сервисных учетных записей*

Для каждого внутреннего сервиса используются отдельные сервисные учетные записи и отдельные секреты подключения.

В HashiCorp Vault хранятся:

* сервисные токены,
* учетные данные Kafka поставщико в событий и подписчиков,
* учетные данные интеграционных сервисов,
* сервисные секреты межсервисного взаимодействия,
* административные сервисные токены.

Каждый сервис получает только собственные учетные данные и не имеет доступа к секретам других компонентов платформы.

*3. Хранение паролей подключения к БД*

Пароли и учетные данные подключения к PostgreSQL, MongoDB и Redis Cluster хранятся централизованно в HashiCorp Vault.

Для каждого сервиса используются отдельные учетные записи БД с минимально необходимыми полномочиями.

HashiCorp Vault хранит учетные данные:

* БД резервирования,
* модели чтения,
* Redis Cluster,
* БД прогрева кеша,
* БД конфликтов,
* хранилища событий,
* хранилища временных рядов.

Сервисы не используют единые универсальные учетные записи подключения.

Например:

* Сервис запросов использует только учетную запись чтения,
* Сервис резервирования использует учетную запись записи в БД резервирования,
* загрузчик модели чтения использует отдельную учетную запись обновления MongoDB.

*4. Хранение ключей проверки JWT access-токенов*

Поскольку JWT access-токены выпускаются внешним поставщиком удостоверений, Сервис не хранит приватные ключи подписи JWT.
Закрытые ключи подписи JWT находятся исключительно у внешнего поставщика удостоверений и не размещаются во внутреннем контуре Сервиса.

В HashiCorp Vault хранятся:

* доверенные ключи проверки подписи JWT access-токенов или JWKS-кеш/набор доверенных ключей проверки,
* доверенные сертификаты внешнего поставщика удостоверений,
* параметры проверки JWT,
* данные доверенной инфраструктуры проверки подписи.

API Gateway использует данные ключи и сертификаты для проведения проверок. 

*5. Ограничение доступа к ключам и секретам*

Доступ к секретам регулируется политиками HashiCorp Vault и сервисными ролями.

Политики доступа определяют:

* перечень разрешенных секретов,
* допустимые операции,
* срок действия сервисных токенов,
* ограничения использования сертификатов,
* разрешенные сервисные сценарии.

Особое ограничение применяется для:

* административных сертификатов,
* учетных данных Kafka,
* сертификатов mTLS,
* доверенных ключей проверки JWT access-токенов,
* интеграционных секретов.

Все операции получения, изменения и отзыва секретов фиксируются в журналах аудита безопасности и используются для контроля безопасности платформы и расследования инцидентов.

===== 3.1.5.3 Ротация секретов

В Сервисе реализуется централизованная ротация секретов, сертификатов и сервисных учетных данных через HashiCorp Vault.

Ротация используется для:

* ограничения времени использования секретов,
* снижения риска длительной компрометации учетных данных,
* предотвращения повторного использования утекших секретов,
* обновления криптографических материалов без остановки платформы,
* поддержания актуальности сервисных сертификатов и ключей.

Ротации подлежат:

* сервисные пароли подключения, сертификаты,
* сертификаты mTLS,
* учетные данные PostgreSQL, MongoDB, Redis Cluster, Kafka,
* временные учетные данные,
* сервисные токены, административные токены,
* интеграционные секреты,
* доверенные ключи и сертификаты проверки JWT access-токенов.

*1. Автоматическая ротация секретов*

HashiCorp Vault используется как централизованный механизм автоматической ротации секретов и сервисных учетных данных.

При автоматической ротации выполняется:

. выпуск нового секрета или сертификата,
. обновление записи в HashiCorp Vault,
. предоставление обновленного секрета сервису,
. переключение сервиса на новый секрет,
. отзыв устаревшего секрета.

Сервисы не используют жестко зафиксированные долгоживущие секреты.

*2. Сроки действия секретов*

Для всех типов секретов устанавливаются ограниченные сроки действия.

Ограниченный срок действия применяется для:

* сервисных и административных токенов,
* сертификатов mTLS,
* учетных данных подключения,
* интеграционных секретов.

*Наиболее короткие сроки действия* используются для:

* административных учетных данных,
* сервисных токенов,
* временных учетных записей,
* критичных инфраструктурных компонентов.

Для пользовательских JWT access-токенов используются *короткоживущие токены*, а для поддержания пользовательской сессии применяются refresh-токены.

Доверенные ключи проверки JWT access-токенов и сертификаты внешнего поставщика удостоверений также подлежат периодическому обновлению при изменении криптографических материалов внешнего поставщика.

*3. Отзыв скомпрометированных секретов*

Архитектура Сервиса предусматривает возможность оперативного отзыва секретов и сервисных учетных данных при обнаружении признаков компрометации.

Отзыв выполняется в случаях:

* утечки секретов,
* компрометации сервисного сертификата,
* компрометации учетной записи,
* нарушения политик безопасности,
* выявления аномальной активности,
* вывода сервиса из эксплуатации.

При отзыве выполняется:

. блокировка использования секрета,
. выпуск нового секрета,
. отзыв старых сертификатов,
. завершение активных сервисных сессий,
. обновление сервисных учетных данных,
. фиксация события в журналах аудита безопасности.

Для сервисов Event Bus дополнительно может выполняться:

* отзыв сертификатов и учетных данных Kafka,
* блокировка поставщиков событий,
* блокировка consumer group,
* ограничение доступа подписчиков к топикам.

*4. Обновление секретов без остановки сервисов*

Обновление большинства секретов и сертификатов выполняется без остановки платформы и без полного отключения сервисов.

Для этого используются:

* ограниченный срок действия секретов,
* поэтапная ротация сертификатов,
* поддержка одновременного действия старого и нового секрета,
* повторное получение секретов сервисами,
* обновление соединений без полной остановки приложения.

Без остановки сервисов выполняется обновление:

* сервисных сертификатов, mTLS-сертификатов,
* учетных данных Kafka, PostgreSQL, MongoDB,
* обновление доверенных ключей проверки JWT access-токенов.

Особое внимание уделяется обновлению:

* сервисов резервирования,
* API Gateway,
* Event Bus,
* HashiCorp Vault,
* сервисных gRPC-соединений.

*При ротации* доверенных ключей проверки JWT access-токенов API Gateway поддерживает *одновременную проверку токенов с использованием старых и новых доверенных ключей* до завершения переходного периода.

Все операции ротации, обновления и отзыва секретов фиксируются в журналах аудита безопасности и используются для контроля безопасности платформы и расследования инцидентов.

===== 3.1.5.4 Защита конфигурационных данных

Конфигурационные данные Сервиса рассматриваются как *критичный элемент безопасности платформы* и защищаются от несанкционированного доступа, подмены и утечки.

Защита конфигураций распространяется на:

* параметры подключения сервисов,
* настройки безопасности,
* конфигурации API Gateway,
* параметры Event Bus, PostgreSQL, MongoDB, Redis Cluster,
* конфигурации HashiCorp Vault,
* настройки сервисной аутентификации,
* параметры интеграционных подключений.

Особое внимание уделяется защите конфигураций, содержащих сервисные учетные данные, сертификаты и параметры доступа к инфраструктурным компонентам.

*1. Шифрование конфигураций*

Конфигурационные данные, содержащие чувствительные параметры, хранятся и передаются в защищенном виде.

Шифрованию подлежат содержащиеся в конфигурационных данных:

* сервисные токены,
* учетные данные PostgreSQL,
* учетные данные MongoDB,
* учетные данные Redis Cluster,
* Kafka credentials,
* параметры интеграционных подключений,
* сертификаты mTLS,
* доверенные ключи проверки JWT access-токенов,
* административные секреты.

Конфигурации с чувствительными параметрами *не передаются между сервисами в открытом виде*.

Для хранения критичных параметров используется HashiCorp Vault, обеспечивающий централизованное защищенное хранение и выдачу секретов.

*2. Исключение секретов из репозиториев*

Размещение секретов и чувствительных конфигурационных данных в исходном коде и репозиториях *запрещено*.

В Git-репозиториях не допускается хранение секретов, токенов, паролей, ключей. В репозиториях размещаются только:

* шаблоны конфигураций,
* параметры без чувствительных данных,
* ссылки на используемые секреты,
* идентификаторы конфигурационных профилей.

Получение секретов выполняется сервисами динамически через HashiCorp Vault после прохождения сервисной аутентификации.

*3. Безопасная доставка конфигураций*

Передача конфигурационных данных сервисам выполняется только через защищенные каналы.

Для доставки конфигураций используются:

* TLS 1.3 + mTLS,
* сервисная аутентификация,
* политики доступа HashiCorp Vault.

Конфигурации передаются сервисам при:

* запуске контейнера,
* обновлении конфигурации,
* ротации секретов,
* изменении сервисных сертификатов.

Каждый сервис получает только собственный набор конфигурационных параметров и не имеет доступа к конфигурациям других компонентов платформы.

Например:

* Сервис запросов не получает конфигурации Kafka producer,
* сервис агрегации метрик не получает учетные данные БД резервирования,
* интеграционные сервисы не получают административные параметры,
* клиентские приложения не имеют доступа к внутренним конфигурациям платформы.

*4. Контроль изменений конфигурации*

Изменение конфигурационных данных выполняется централизованно и подлежит обязательному контролю.

Контроль распространяется на изменения:

* политик безопасности,
* сервисных параметров,
* TLS-конфигураций,
* параметров Event Bus,
* сервисных сертификатов,
* интеграционных настроек,
* политик HashiCorp Vault.

Все изменения конфигурации сопровождаются:

* аутентификацией административной учетной записи,
* проверкой полномочий,
* аудитом операций,
* журналированием изменений,
* фиксацией времени и инициатора изменения.

Особое внимание уделяется изменениям:

* конфигурации API Gateway,
* политик авторизации,
* параметров mTLS,
* доверенных сертификатов,
* параметров проверки JWT access-токенов,
* конфигурации HashiCorp Vault.

Попытки *несанкционированного изменения конфигурации* рассматриваются как *события безопасности повышенного уровня критичности* и фиксируются в журналах аудита безопасности.
