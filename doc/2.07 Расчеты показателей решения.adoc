=== 2.7. Расчеты показателей решения

==== 2.7.1. Исходные данные для расчетов

Расчет базируется на нефункциональных требованиях системы и методологии перевода RPS в количество одновременных сессий с учетом типовых паттернов поведения различных категорий пользователей.

===== 2.7.1.1. Ключевые показатели нагрузки

[cols="3,0,0", options="header"]
|===
| Параметр | Базовый режим | Пиковый режим
| Запросы чтения (Query API), RPS | *10 000* | *50 000*
| Транзакции резервирования, TPS | *100* | *500*
| События записи (Write Stream), EPS | *1 000* | *5 000*
| Одновременные сессии (пользователи + системы) | *83 850* | *417 000*

|===

===== 2.7.1.2. Методология расчета количества пользователей

Пересчет RPS в количество сессий выполнен с учетом интенсивности запросов:

* *Клиенты eDA (мобильное приложение/веб)* : *0,1 RPS* на сессию -- один запрос в 10 секунд при просмотре каталога;
* *Службы доставки (API-агенты)* : *2 RPS* на интеграцию -- автоматический опрос статуса;
* *Сотрудники (ТСД)* : *0,2 RPS* на терминал -- выполнение одной операции сверки товара раз в 5 секунд;
* *Сессии резервирования* : *6 секунд* -- среднее время сессии резервирования.

=== 2.7.2. Единая модель потока данных 

В данном разделе используется сквозная модель обработки данных Сервиса, 
связывающая пользовательские запросы, операции резервирования и поток событий.

Модель описывает преобразование нагрузки между уровнями системы:

Пользовательский запрос -> API -> гео-фильтрация -> мультипликация на магазины -> кэш -> backend -> события -> обновление состояния.

Один пользовательский запрос из-за мультипликации превращается в десятки обращений к кэшу. Потом нагрузка делится: чтение идёт в распределенный кеш (основной слой), а записи формируются через события шины событий.

[cols="1,2", options="header"]
|===
| Этап | Описание

| Пользователь | Инициирует запросы к системе, генерирует входящий трафик: поиск товара, проверка наличия
| API слой | Принимает входящий трафик (RPS) и выполняет первичную обработку запросов
| Гео-поиск + мультиплексирование | Для каждого запроса определяется набор магазинов (до N), что приводит к мультипликации нагрузки на нижнеуровневые компоненты
| Распределенный кэш (Redis) | Основной слой чтения, обслуживающий большую часть запросов
| Backend (Модель чтения / БД) | Обрабатывает случаи отсутствия данных в кэше и выполняет операции резервирования
| Event Bus (Kafka) | Принимает поток событий (EPS), отражающих изменения остатков и резервов
| сервисы обработки событий | Обрабатывают события и обновляют кэш и модель чтения
|===

==== 2.7.2.1. Связь метрик нагрузки

В рамках данной модели используются следующие взаимосвязи:

[cols="1,2"]
|===

| 1. Входящая нагрузка | API RPS → определяется пользовательской активностью
| 2. Мультипликация запроса при гео-поиске | Чтение из кеша RPS = API RPS * N_stores +
где N_stores -- количество магазинов в радиусе поиска.
| 3. Нагрузка на backend | Backend RPS = API RPS × (1 - cache_hit_ratio) +
где cache_hit_ratio  -- коэффициент попаданий в кэш
| 4. Поток событий | EPS = f(операции POS, WMS, резервирования)
| 5. Нагрузка на кэш (запись) | Запись в кеш TPS ≈ EPS + Reservation TPS
|===

При этом учитываются три ключевых фактора:

1. *Hot SKU* : 80% нагрузки приходится на популярные товары, что создаёт конкуренцию за ключи и влияет на задержку.

Нагрузка в системе распределена неравномерно: ~ 80% запросов приходится на ~ 20% SKU. 
Это приводит к:

* росту конкуренции за одни и те же ключи (locks),
* увеличению задержки для популярных SKU,
* локальным перегрузкам Redis.

В расчётах это учитывается через коэффициент:

`Effective Load = Base Load * K_hot`, где:

Таким образом, для популярных ключей:

* `K_hot` ≈ *1,5-2* нагрузки для базового режима,
* `K_hot` ≈ *2* нагрузки для пикового режима.

2. *Всплески нагрузки* : кратковременные пики могут увеличивать нагрузку кратно, и именно под них должен проектироваться запас системы.

Пиковая нагрузка не является равномерной и может иметь кратковременные всплески: кратковременное увеличение RPS в 2-3 раза.

Для этого вводится коэффициент:

`Peak Load = Nominal Peak * K_burst`, где `K_burst` ≈ *1,5-3*.

Всплески нагрузки критичны для:

* Redis (высокая нагрузка по операциям чтения),
* Kafka (задержки),
* API (тротлинг).

3. *Сетевая задержка* : растёт с количеством вызовов, поэтому мультипликацию запросов и синхронные цепочки нужно ограничивать.

Минимальная задержка определяется сетевыми задержками:

* в рамках одного ЦОД : 0,5-1 мс,
* мультирегиональный ЦОД : 2-10 мс и более.

Полная задержка запроса составляет :

`Total latency ≈ сумме всех  RTT + processing time`

При мультипликации гео-запросов задержка растет с числом downstream-вызовов.

Это накладывает ограничения на:

* глубину мультипликации,
* синхронные цепочки вызовов.

Таким образом, все дальнейшие расчёты связаны через эту модель и выводятся из RPS, TPS и EPS.

==== 2.7.2.2. Интерпретация модели

* API RPS формирует базовую нагрузку системы.
* Гео-поиск мультиплицирует нагрузку чтения на кэш.
* Коэффициент попадания в кеш определяет долю обращений к backend.
* Резервирование формирует транзакционную нагрузку
* EPS определяет поток обновлений и нагрузку на Event Bus (Kafka)

==== 2.7.2.3. Ограничения модели

Модель является приближённой и предполагает:

* равномерное распределение нагрузки (за исключением hot SKU),
* фиксированное значение коэффициента попадания в кеш,
* отсутствие коротких всплесков-пиков (учитываются отдельно).

Для учёта отклонений используются:

* коэффициенты hot SKU,
* механизмы ограничения потока запросов (тротлинг) и замедления входящего потока при перегрузке (backpressure),
* сценарии деградации.

==== 2.7.2.4. Вывод

Модель показывает, что:

* коэффициент мультипликации является главным мультипликатором нагрузки;
* hot SKU усиливают конкуренцию;
* короткие всплески нагрузки определяют худшие сценарии; 
* задержки ограничивают архитектуру взаимодействий.

==== 2.7.6. Анализ соответствия архитектурных решений нагрузке

Проведена оценка ключевых архитектурных решений.

. Подход CQRS + Event Sourcing считается готовым к использованию. Разделение путей чтения и записи позволяет независимо масштабировать нагрузки чтения и записи. *Соответствует.*

. Использование master-replica кэша в виде одного экземпляра Redis не является достаточным. Ожидаемая нагрузка около 50 000 RPS на чтение и 5 500 операций записи превышает возможности одной master-ноды. Требуется переход к кластеру Redis. *Требует внимания...*

. Использование PostgreSQL в роли Хранилища событий на одном узле допустимо, но связан с риском. Нагрузка около 5 000 TPS на добавление достижима при использовании быстрых NVMe-дисков, однако требует мониторинга отставания реплки от мастера по WAL и параметров репликации. *Требует внимания...*

. Kafka-кластер из трех брокеров оценивается как достаточный. Нагрузка порядка 15 000 MPS с учетом репликации является штатной, однако для обеспечения параллелизма и отказоустойчивости необходимо предусмотреть не менее 6 партиций. *Соответствует.*

. Использование одного экземпляра загрузчика моделей чтения недопустимо. Один consumer не сможет обработать поток в 5 000 EPS. Требуется горизонтальное масштабирование с увеличением числа инстансов и партиций. *НЕ Соответствует!*

==== 2.7.7. Методика сайзинга компонентов

Объем требуемых вычислительных ресурсов (инстансов, нод, партиций) определяется на основе расчетной нагрузки, полученной в разделах 2.7.1–2.7.4.

Методика сайзинга применяется ко всем ключевым компонентам системы:

 * Срвис запросов,
 * Распределенный кеш (Redis),
 * Event Bus (Kafka),
 * Загрузчик моделей чтения,
 * Сервис и БД резервирования,
 * Гео-сервис.

===== 2.7.7.1. Общий подход к сайзингу

Сайзинг выполняется в три этапа:

1. Определение пиковых нагрузок. +
[`Пиковая Нагрузка`] -- пиковая нагрузка на компонент (RPS, TPS, EPS), из раздела 2.7.
2. Определение производительности одной единицы ресурса. +
[`Производительность на Единицу`] -- устойчивая производительность одной единицы ресурса (инстанса, ноды, partition).
3. Расчет количества ресурсов с учетом запаса. +
[`Фактор Резерва`] — коэффициент запаса (учет отказов, неравномерности нагрузки, роста нагрузки).

Базовая формула:

[source]
----
[Требуемая Нагрузка] = [Пиковая Нагрузка] / [Производительность на Единицу] * [Факор Резерва]
----

Такой подход обеспечивает:

* предсказуемость масштабирования компоенетов сериса, 
* устойчивость к пиковым нагрузкам,
* корректный выбор архитектурных решений (шардирование, кластеризация).

==== 2.7.8. Рекомендуемые архитектурные корректировки

===== 2.7.8.1. Масштабирование Распределенного кеша. *Критично*

Проблема: 

* 417 000 одновременных сессий генерируют 50 000 RPS -> 42 500 RPS попадают в кэш (коэффициент попадания в кеш 85%). 
* Один Master-узел Redis ограничен ~20 000 RPS.

Решение:

* Разворачивание *HA-кластера Redis* с 8 мастер-узлами (шардирование по идентификатору SKU) и 8 репликами. +
или
* *Redis Enterprise* с Auto-sharding и Conflict-free Replicated Data Types (CRDB) для multi-master записи.
* Выделение отдельного кластера Redis для Менеджера блокировок (снижение нагрузки и конкуренции на master-ноде).

===== 2.7.8.2. Защита от лавинообразного обращения к источнику данных (cache stampede)

Проблема: 

* Инвалидация популярного SKU (из 80% горячих) приводит к одновременному направлению 8 500 RPS (в пике) к Хранилищу модели чтения.

Решение:

* Использование *Stochastic TTL* : разброс времени жизни ключа ±10% (5 мин → 4,5-5,5 мин) для предотвращения массового истечения TTL в один момент. 
* Применение предохранителя отказов *Circuit Breaker* на Сервисе запросов : при задержке запросов чтения более 500 мс — возвращать ошибку `503 Retry-After` или данные с призаком устаревания (флагом `stale=true`), но не блокировать потоки в ожидании ответа базы.
* *Принудительный прогрев кеша* : при получении события `inventory.updated` -> проактивное обновление - запист нового значения, а не удаление ключа.

===== 2.7.8.3. Горизонтальное масштабирование обработки событий

Проблема: 

* Один экземпляр Загрузчика моделей чтения не справится с 5 000 EPS.

Решение:

* *Партиционирование Kafka* : 12 партиций по идентификатору SKU -> key-based routing.
* *Горизонтально масштабируемые потребители* : 12 экземпляров Загрузчика моделей чтения, по одному на партицию.
* *Регулирование потока входящей нагрузки* : при задержке потока запросов более 10 000 MPS -> автоматическое масштабирование до 24 экземпляров через механизм Horizontal Pod Autoscaler (HPA) Kubernetes.

===== 2.7.8.4. Усиление Сервиса резервирования для пика 500 TPS

Проблема: 

* 500 конкурентных резервов требуют 500 соединений к распределенному кешу и PostgreSQL на блокировку.

Решение:

* Использование *пула соединений* : применение библиотеки HikariCP с размером пула 200 соединений на экземпляр, минимум 3 экземпляра Сервиса резервирования.
* Применеие *Redlock для критичных SKU* : при остатке менее 6 единиц товара использовать алгоритм распределенной блокировки Redlock на 3 независимые Redis-ноды вместо одиночной блокировки на одном ноде.
* *Изоляция ресурсов* : разделение пула потоков запросов на "быстрые" резервы (остаток больше 10) и "медленные" (остаток меньше 11, которые требуют дополнительных проверок наличия).

===== 2.7.8.5. Защита от нагрузки внешних служб доставки

Проблема: 

* 4 000 API-агентов служб доставки генерируют 8 000 RPS, что эквивалентно нагрузке 80 000 клиентов.

Решение:

* Наличие *отдельного API Gateway* : выделенный сегмент шлюза с отдельным rate limiting до 17 RPS на интеграцию и внесение IP источников запросов компаний партнерв в белый список (IP-whitelisting). 
* *Кеширование ответов* : на 30 секунд для идентичных запросов наличия (conditional GET).
* *Фильтрация на выходе* : использовать в ответах только бинарный статус (boolean available/nil), без точных остатков (реализует защиту от скраппинга ассортимента).

===== 2.7.8.6. Оптимизация Гео-сервиса

Проблема: 

* 5 000 RPS геопоиска с проверкой 20 магазинов = 100 000 запросов в кэш за 200 мс. 0.002 мс на запрос нереально получить при реальном времени сетевого отклика от 0,5-2 мс между сервисами до 20-100 мс через интернет.

Решение:

* Добавить в *Гео-индекс битмаски* : хранить в БД Гео-сервиса не только координаты магазинов, но и кешируемый bloom-фильтр наличия top-1000 SKU, как предварительный слой проверки наличия популярных SKU, снижая количество обращений к распределенному кешу и уменьшая нагрузку от мультипликатора запросов на 20 магазинов.
* Использовать *Гео-индекс Redis* : магазины предварительно загружаются в Redis через GEOADD, а поиск выполняется в Redis командой GEORADIUS с последующей фильтрацией по предварительно загруженному множеству магазинов. 
* *Заранее подгружаем данные в фоне* : прогрев по гео-локации -- данные остатков для ближайших 50 магазинов при первом запросе пользователя подгружаются паралельно без прерыва ответа. 

==== 2.7.9. Лимитирование нагрузки 

При пиковой нагрузке 50 000 RPS и 417 000 одновременных сессий единый лимит 100 RPM (≈1.66 RPS) создает противоречие:  

* 400 000 клиентских сессий генерируют ~8 000 RPS (0.02 RPS/сессию), что лимит не нарушает, но  
* 4 000 инстансов служб доставки при 8 000 RPS дают *2 RPS = 120 RPM на инстанс*, что *превышает установленный лимит* еще в базовом режиме, т.е. не в режиме пика.  
* 10 000 терминалов ТСД при 2 000 RPS = 0.2 RPS = 12 RPM на ртерминал вписываются в лимит, но при инвентаризациях могут генерировать  краткосрочный взрывной рост нагрузки.

Без дифференциации лимитов Сервис либо искусственно ограничит легитимную нагрузку от доставки, что приведет к потерям выручки, либо не защитится от DDoS атаки со стороны скомпрометированных клиентских аккаунтов, при таком развитие событий.

2. *Индивидуальные лимиты ресурсов* : для "горячих" SKU (top-20%) применяется отдельный глобальный rate limit 10 000 RPM независимо от клиента + защита от лавинообразного обращения к источнику данных.
3. *Контролируемая деградация* : при достижении 80% лимита формируется ответ с заголовком `Warning: 199 - Rate Limit Proximity`.
4. *Исключения для POS* : потоковые события от касс (более 5 000 EPS) проходят через *Event Bus*, а не через API Gateway, минуя HTTP rate limiting на уровне количества HTTP-запросов в минуту; контроль нагрузки осуществляется на уровне пропускной способности шины событий (throughput, partitioning, producer quotas, consumer lag).

=== 2.7.11. Стратегия масштабирования и эволюции архитектуры

На основании проведенных расчетов нагрузки (см. <<2.7.3>>, <<2.7.6>>) установлено, что текущая архитектура *условно готова* к заявленным показателям 417 000 одновременных сессий и 50 000 RPS при условии реализации критических корректировок : разворачивание Redis Cluster, параллелизация Загрузчика моделей чтения (см. <<2.7.9>>). 

Ниже описаны подходы к горизонтальному масштабированию в рамках текущей концепции, а также критерии перехода к архитектурному развитию для выхода Сервиса на работу на федеральном масштабе.

==== 2.7.11.1. Горизонтальное масштабирование компонентов

Расчетные параметры масштабирования для достижения целевых показателей *пиковой нагрузки* без изменения архитектурного стиля:

|===
| Компонент | Текущая емкость | Пиковая нагрузка | Масштабирование | Расчет инстансов

| *Сервис запросов* | 1 инстанс = ~500 RPS | 50 000 RPS (чтение) | Stateless, N+1 реплики | *100* реплик (+ 20% запас = 120 инстансов)

*Особенности масштабирования Сервиса запросов:*

Поскольку Сервис запросов работает только с распределенным кешем и не содержит сессий пользователей (является stateless), он масштабируется линейно. При росте нагрузки до 100 000 RPS (планируемый рост через 12 месяцев) требуется 200 инстансов. 

Рекомендуется использовать Kubernetes Horizontal Pod Autoscaler (HPA) с метриками CPU/RPS и стабилизационным окном 5 минут для предотвращения частых колебаний масштабирования при пиках.

*Особенности масштабирования Сервиса резервирования:*

Сервис резервирования из-за ACID-транзакций и блокировок является stateful. Масштабирование сервиса ограничено координацией Менеджера блокировок. При превышении 500 TPS (одновременно 3 000 активных резервов) требуется:

* внедрение шардирования по диапазонам идентификаторов SKU (A-M, N-Z) или шардирование на основе хеш-функции;
* разделение пула на "быстрые" (остаток больше 10 единиц) и "медленные" (остаток меньше 11 единиц) резервы -- разделение ресурсов на изолированные группы;
* использование распределенных блокировок (по алгоритму Redlock) с тремя независимыми Redis-нодами для критичных SKU.

| *Поддержка дарксторов (Virtual Stores)* (бэклог, требование <<1.1.5>> п.3) | Отдельная логика доступности для логистических центров исполнения заказов | Выделение отдельного ограниченного *контекста «Доступность для исполнения заказов» с собственным Гео-индексом и Сервисом резервирования*
|===

==== 2.7.11.4. План эволюции (Roadmap)

1. *Фаза 1 : Оптимизация текущей архитектуры — текущий этап. 0-6 месяцев*

** Внедрение Redis Cluster : 8 мастеров вместо single-node.
** Масштабирование Загрузчика моделей чтения до 12 инстансов.
** Партиционирование таблицы Хранилища событий по месяцам.
** *Целевые показатели* : устойчивость к 50 000 RPS, 417 000 сессий.

2. *Фаза 2 : Региональное масштабирование. 6-12 месяцев*

** Развертывание Read реплики Хранилища событий в другом дата-центре (горячий резерв).
** Внедрение CDN не только для статики, но и для динамического контента.
** Geo-routing запросов, направление клиента к ближайшему ЦОД.
** *Целевые показатели* : DAU > 10M, открытие новых регионов присутствия.

3. *Фаза 3 : Федеральный масштаб. 12-24 месяца*

** Реализация active-active мультирегион (CRDT для остатков).
** Выделение Edge Nodes в крупных городах -- пилот в первых 5 городах.
** Замена PostgreSQL Хранилища событий на распределенное хранилище на базе CockroachDB или ScyllaDB.
** *Целевые показатели* : MAU > 50M, сеть более 5 000 магазинов.

4. *Фаза 4 : Интеллектуальная доступность. 24+ месяца*

** Полная реализация Прогнозирование доступности товара с ML.
** Интеграция с электронными ценниками (ESL) из бэклога.
** Поддержка автономной работы без подключения к центральной системе на базе локальной Event Bus.
** *Целевые показатели* : достижение экономического эффекта от снижения доли отмененных заказов до 0.5%.

*Сетевой трафик:*

* Входящий (запросы) : 50 000 RPS * 1 KB = 50 MB/s = 400 Mbps
* Исходящий (ответы) : 50 000 RPS * 2 KB = 100 MB/s = 800 Mbps
* Межсервисный (Event Bus) : ~200 MB/s = 1.6 Gbps между зонами
* Рекомендация : Применение сетевых интерфейсов 10 Gbps на ноды кластера Redis и Kafka.

*Заключение:*

Переход к мультирегиональному ЦОДу и технологии edge computing требуется при выходе на федеральный масштаб 
(MAU >50M) и является ключевым пунктом архитектурного бэклога.
