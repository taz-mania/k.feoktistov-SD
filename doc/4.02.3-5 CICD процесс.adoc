==== 4.2.3 Процесс непрерывной интеграции (CI)

Процесс непрерывной интеграции используется для регулярной проверки совместимости изменений, вносимых в различные компоненты Сервиса. Основной задачей CI является раннее выявление: 

* ошибок сборки, 
* проблем взаимодействия между сервисами, 
* нарушений в обработке пользовательских и событийных сценариев.

Особое внимание уделяется: 

* стабильности взаимодействия компонентов через Event Bus, 
* корректности работы механизмов резервирования и обработки изменений остатков, 
* сохранению совместимости между версиями сервисов и форматами событий. 

Проверки выполняются автоматически при внесении изменений в исходный код и конфигурации решения.

===== 4.2.3.1 Контроль изменений исходного кода

Управление изменениями исходного кода в Сервисе организовано с использованием *централизованной системы контроля версий*, обеспечивающей хранение: 

* исходного кода прикладных сервисов, 
* инфраструктурных конфигураций, 
* шаблонов развертывания, 
* сценариев автоматизации поставки. 

Такой подход позволяет поддерживать единый процесс сопровождения решения и обеспечивает контроль изменений для всех компонентов архитектуры.

Разработка новых функций, исправление дефектов и изменение конфигураций выполняются в отдельных ветках репозитория. Это позволяет изолировать изменения друг от друга и снижает риск нарушения стабильности основной версии Сервиса. Дополнительно разделение изменений по веткам упрощает параллельную работу над несколькими задачами и обеспечивает возможность независимой проверки отдельных изменений до их объединения с основной веткой разработки.

Перед включением изменений в основную ветку все модификации проходят процедуру Pull Request. На данном этапе: 

* выполняется анализ состава изменений, 
* проверяется корректность модификации: 
** пользовательских API, 
** механизмов обработки событий, 
** конфигураций сервисов, 
** инфраструктурных параметров. 

Дополнительно контролируется отсутствие конфликтов между изменениями различных компонентов Сервиса и оценивается влияние изменений на существующие пользовательские сценарии.

Процесс Pull Request интегрируется с механизмами CI, что позволяет *автоматически запускать базовые проверки качества* до объединения изменений. Для каждого запроса выполняется: 

* сборка измененных компонентов, 
* запуск автоматизированных тестов, 
* проверка корректности подготовки контейнерных образов. 

Такой подход позволяет выявлять часть ошибок еще до этапа интеграции изменений в общую кодовую базу.

Дополнительно для критичных изменений используется *процедура код-ревью*. Проверка выполняется другими участниками команды сопровождения и направлена на выявление потенциальных проблем, связанных с: 

* устойчивостью работы Сервиса, 
* корректностью взаимодействия компонентов, 
* влиянием изменений на ключевые бизнес-сценарии. 

Особое внимание уделяется изменениям, затрагивающим механизмы резервирования, обработку событий Event Bus и обновление модели чтения.

Использование код-ревью позволяет дополнительно контролировать единообразие реализации компонентов и снижает вероятность появления ошибок, которые могут быть не выявлены автоматизированными проверками. Результаты проверки учитываются при принятии *решения о дальнейшем продвижении изменений* в последующие стадии CI/CD-конвейера.

===== 4.2.3.2 Автоматическая сборка и проверка

После внесения изменений в исходный код и прохождения первичного контроля через Pull Request в Сервисе автоматически запускается *процесс сборки и проверки измененных компонентов*. Автоматизация данного этапа позволяет сократить время подготовки изменений к тестированию и снизить вероятность ошибок, связанных с ручным выполнением операций сборки и подготовки релиза.

Процесс автоматической сборки включает: 

* подготовку исполняемых артефактов прикладных сервисов, 
* сборку контейнерных образов, 
* формирование конфигураций, необходимых для последующего развертывания компонентов в тестовых окружениях. 

Поскольку архитектура Сервиса состоит из нескольких независимо разворачиваемых компонентов, сборка выполняется отдельно для измененных сервисов и связанных с ними инфраструктурных конфигураций.

Особое внимание уделяется корректности сборки компонентов, участвующих в обработке пользовательских запросов и событий Event Bus. На данном этапе контролируется возможность совместной компиляции сервисов, корректность зависимостей между компонентами и отсутствие ошибок, препятствующих запуску приложений в контейнерной среде.

После завершения сборки автоматически запускаются *базовые проверки качества изменений*. В рамках данных проверок: 

* выполняются модульные тесты, 
* проверка конфигураций развертывания, 
* контроль корректности подготовки контейнерных образов. 

Для компонентов, взаимодействующих через Event Bus, дополнительно проверяется: 

* корректность обработки событий, 
* отсутствие очевидных нарушений совместимости форматов обмена данными.

Результаты автоматизированных проверок используются как один из критериев готовности изменений к дальнейшему продвижению по CI/CD-конвейеру. При выявлении ошибок сборки, неуспешном прохождении тестов или проблемах подготовки контейнерных образов дальнейшее продвижение изменений автоматически блокируется до устранения выявленных проблем.

Дополнительно на данном этапе выполняется *контроль успешности выполнения процедур сборки и запуска проверок*. Информация о результатах сборки, ошибках тестирования и проблемах подготовки окружения фиксируется в журналах CI/CD-процесса и используется для анализа причин неуспешной поставки изменений.

Использование автоматической сборки и проверки позволяет: 

* обеспечить единообразие подготовки компонентов Сервиса, 
* сократить время интеграции изменений, 
* повысить стабильность поставки новых версий прикладных сервисов и инфраструктурных конфигураций.

===== 4.2.3.3 Автоматизированное тестирование

Автоматизированное тестирование в Сервисе используется как один из основных механизмов *контроля корректности изменений на этапах непрерывной интеграции*. Поскольку архитектура решения включает большое количество взаимодействующих компонентов, асинхронную обработку событий и отдельные механизмы работы с моделью чтения и резервированием, автоматизированные проверки* позволяют выявлять часть ошибок до развертывания изменений* в последующих окружениях.

На первом уровне проверок используются *модульные тесты*, направленные на контроль корректности работы отдельных компонентов и внутренних механизмов обработки данных. В рамках данных проверок контролируется: 

* работа пользовательских API, 
* обработка запросов резервирования, 
* базовая логика обработки событий, 
* корректность отдельных сервисных операций. 

Использование модульного тестирования позволяет: 

* выявлять ошибки реализации на ранних стадиях разработки, 
* снижает вероятность передачи нестабильного кода в общую кодовую базу.

Дополнительно в составе CI-процесса выполняется *интеграционное тестирование*, ориентированное на проверку взаимодействия между компонентами Сервиса. На данном этапе контролируется: 

* корректность обмена данными между сервисами, 
* взаимодействие через Event Bus, 
* работа с хранилищами данных, 
* корректность обновления модели чтения после обработки событий. 

Особое внимание уделяется пользовательским сценариям проверки доступности товаров и резервирования, так как данные процессы затрагивают сразу несколько компонентов архитектуры.

Для сервисов, использующих асинхронную обработку сообщений, дополнительно проверяется: 

* корректность публикации и обработки событий, 
* совместимость взаимодействия между компонентами поставщикаи событий и подписчитками. 

Это позволяет снизить вероятность возникновения ошибок, связанных с нарушением форматов обмена данными или некорректной последовательностью обработки событий.

*Помимо функциональных проверок*, в процессе автоматизированного тестирования выполняются базовые проверки качества изменений. Контролируется: 

* успешность выполнения тестов, 
* корректность подготовки контейнерных образов, 
* отсутствие критических ошибок сборки, 
* возможность запуска компонентов в тестовой среде. 

Для отдельных типов изменений могут дополнительно выполняться проверки конфигураций развертывания и совместимости инфраструктурных параметров.

Результаты автоматизированного тестирования используются как *обязательный критерий дальнейшего продвижения изменений по CI/CD-конвейеру*. При неуспешном прохождении тестов или выявлении критических ошибок выполнение последующих этапов поставки автоматически *блокируется до устранения обнаруженных проблем*.

Дополнительно результаты выполнения тестов интегрируются с механизмами журналирования и мониторинга CI/CD-процесса. Это позволяет: 

* анализировать причины неуспешных проверок, 
* выявлять повторяющиеся проблемы, 
* контролировать стабильность процесса поставки изменений между окружениями.

==== 4.2.4 Процесс непрерывной доставки (CD)

Процесс непрерывной доставки обеспечивает: 

* последовательное продвижение изменений между окружениями, 
* контролируемое обновление компонентов Сервиса. 

Архитура решения предполагает наличие большого количества одновременно работающих сервисов и фоновых обработчиков, поэтому *обновление компонентов выполняется поэтапно с контролем состояния среды и результатов развертывания*.

Для пользовательских сервисов предусматривается возможность *постепенного переключения нагрузки между версиями компонентов*, что позволяет снижать риск полной недоступности Сервиса во время релизов. При этом для отдельных типов изменений, затрагивающих структуры хранения данных или обработку событий, *допускается кратковременная деградация отдельных функций* при сохранении доступности ключевых пользовательских операций.

===== 4.2.4.1 Автоматизация развертывания

В Сервисе автоматизация развертывания используется для обеспечения *предсказуемой и воспроизводимой поставки изменений* между тестовыми и промышленными окружениями. Поскольку решение включает несколько прикладных сервисов, Event Bus, распределенное кэширование и отдельные хранилища данных, выполнение развертывания в ручном режиме существенно повышает риск ошибок конфигурации и нарушения совместимости компонентов.

После успешного завершения этапов сборки и автоматизированного тестирования CI/CD-конвейер автоматически инициирует развертывание новых версий компонентов в тестовые среды. В процессе развертывания выполняется:

* загрузка подготовленных контейнерных образов,
* применение конфигураций окружения,
* обновление прикладных сервисов,
* запуск необходимых инфраструктурных зависимостей.

Автоматическое развертывание позволяет обеспечить *единый порядок поставки изменений* между различными окружениями Сервиса и снизить зависимость результата обновления от ручных действий операторов сопровождения.

Особое внимание уделяется *корректности развертывания компонентов*, участвующих в обработке пользовательских запросов и событий Event Bus. После обновления среды автоматически контролируется:

* успешность запуска сервисов,
* доступность пользовательских API,
* корректность подключения к хранилищам данных,
* работоспособность взаимодействия через Event Bus,
* состояние механизмов кэширования и модели чтения.

Результаты данных проверок используются как один из критериев успешности поставки изменений. При выявлении ошибок запуска или деградации работы компонентов дальнейшее продвижение релиза автоматически останавливается до устранения обнаруженных проблем.

Дополнительно в рамках процесса поставки выполняется *контроль успешности развертывания*. Для этого используются:

* проверка состояния контейнеров и сервисов,
* анализ журналов запуска,
* контроль ключевых метрик работоспособности,
* проверка доступности основных пользовательских сценариев.

Информация о результатах развертывания фиксируется в журналах CI/CD-процесса и используется при анализе причин неуспешных обновлений или принятии решения о необходимости отката изменений.

Для управления поставкой изменений в Сервисе применяется *централизованное управление версиями релизов*. Каждая версия прикладных компонентов и контейнерных образов получает собственный идентификатор, связанный с изменениями в системе контроля версий. Это позволяет:

* отслеживать состав поставляемых изменений,
* контролировать совместимость компонентов,
* обеспечивать воспроизводимость развертывания,
* выполнять возврат к предыдущим стабильным версиям.

Использование автоматизированного развертывания и централизованного управления версиями позволяет повысить стабильность процесса поставки, сократить время вывода изменений в эксплуатацию и снизить вероятность возникновения ошибок при обновлении компонентов Сервиса.

===== 4.2.4.2 Продвижение изменений между окружениями

Продвижение изменений между окружениями в Сервисе организовано как *последовательный и контролируемый процесс поставки*, обеспечивающий поэтапную проверку новых версий компонентов перед их выводом в промышленную эксплуатацию. Такой подход позволяет:

* снизить риск распространения ошибок между средами,
* обеспечивает дополнительный контроль стабильности решения на каждом этапе CI/CD-конвейера.

После успешного завершения автоматизированных проверок изменения последовательно продвигаются между несколькими окружениями Сервиса. В процессе поставки используются:

* тестовые окружения для первичной проверки изменений,
* интеграционные окружения для проверки взаимодействия компонентов,
* production-окружение для промышленной эксплуатации решения.

На каждом этапе выполняется развертывание одинаковых версий контейнерных образов и конфигураций, что обеспечивает *воспроизводимость поставки* и снижает вероятность возникновения ошибок, связанных с различиями между средами.

Переход изменений между окружениями выполняется только при выполнении установленных критериев качества и стабильности. Для продвижения релиза контролируется:

* успешность автоматизированных тестов,
* корректность запуска сервисов,
* отсутствие критических ошибок в журналах,
* стабильность пользовательских API,
* корректность взаимодействия компонентов через Event Bus.

Для изменений, затрагивающих критичные механизмы резервирования, обработку событий или обновление модели чтения, дополнительно может выполняться расширенная проверка совместимости компонентов и анализ поведения системы под тестовой нагрузкой.

Особое внимание уделяется *контролю стабильности среды перед дальнейшим продвижением изменений*. После развертывания новой версии компонентов выполняется наблюдение за состоянием сервисов и анализ ключевых показателей работы Сервиса. На данном этапе контролируется:

* доступность пользовательских сценариев,
* стабильность обработки событий,
* отсутствие роста количества ошибок,
* корректность работы механизмов кэширования,
* отсутствие деградации производительности.

Результаты контроля используются для *принятия решения о возможности дальнейшего продвижения релиза* или необходимости остановки поставки. При выявлении признаков нестабильной работы выполнение последующих этапов CI/CD-конвейера блокируется до устранения обнаруженных проблем.

Дополнительно процесс продвижения изменений интегрируется с механизмами мониторинга и журналирования. Информация о состоянии сервисов после обновления, результаты проверок и данные о деградации компонентов используются для анализа успешности поставки и принятия решений о необходимости отката релиза.

Использование поэтапного продвижения изменений между окружениями позволяет: 

* обеспечить более стабильную поставку новых версий компонентов, 
* снизить вероятность возникновения критических ошибок в production-среде, 
* повысить предсказуемость процесса обновления Сервиса.

===== 4.2.4.3 Обновление без простоя

Для пользовательских компонентов Сервиса процесс обновления организуется с учетом необходимости *сохранения доступности ключевых функций во время выпуска новых версий*. Поскольку решение обрабатывает большое количество запросов проверки доступности товаров и операций резервирования, полная остановка всех сервисов во время обновления может привести к нарушению пользовательских сценариев и росту количества ошибок.

Обновление прикладных сервисов выполняется *поэтапно*, без одновременной остановки всех экземпляров компонентов. В процессе обновления:

* новая версия сервиса запускается параллельно с текущей,
* выполняется проверка успешности запуска компонентов,
* контролируется доступность пользовательских API,
* проверяется корректность взаимодействия с Event Bus и хранилищами данных.

После подтверждения корректности работы новой версии выполняется постепенное отключение предыдущих экземпляров сервиса. Такой подход позволяет *снизить риск полной недоступности компонентов* при возникновении ошибок обновления.

Дополнительно в процессе поставки используется *перераспределение нагрузки между версиями сервисов*. На начальном этапе обновления часть пользовательского трафика может направляться на новую версию компонентов, после чего при отсутствии признаков деградации выполняется дальнейшее увеличение нагрузки на обновленные экземпляры.

Использование постепенного переключения трафика позволяет:

* выявлять проблемы производительности на ранних стадиях обновления,
* ограничивать влияние ошибок на пользователей,
* упрощать возврат к предыдущей версии компонентов,
* снижать риск полной деградации пользовательских сценариев.

Особое внимание уделяется сервисам, участвующим в обработке запросов резервирования и потоковых событий. Для таких компонентов дополнительно контролируется:

* корректность обработки сообщений Event Bus,
* отсутствие ошибок синхронизации модели чтения,
* стабильность работы механизмов кэширования,
* отсутствие роста количества конфликтов резервирования.

Несмотря на использование механизмов поэтапного обновления, возможность выполнения обновлений полностью без простоя зависит от типа изменяемого компонента и характера изменений. Для части обновлений ограничения связаны с необходимостью:

* изменения структур хранения данных,
* обновления схем событий,
* изменения конфигураций инфраструктурных компонентов,
* выполнения миграций баз данных.

В подобных случаях допускается *кратковременная деградация отдельных функций* Сервиса при сохранении доступности основных пользовательских операций. Например, в процессе обновления могут временно увеличиваться задержки обновления модели чтения, снижаться скорость обработки событий или ограничиваться выполнение части фоновых операций.

Дополнительно после обновления выполняется контроль состояния сервисов с использованием механизмов мониторинга и журналирования. Анализируются:

* ключевые показатели производительности,
* количество ошибок,
* стабильность пользовательских сценариев,
* корректность обработки событий после обновления.

Результаты контроля используются для подтверждения успешности поставки либо принятия решения о необходимости отката изменений к предыдущей стабильной версии.

===== 4.2.4.4 Откат изменений

В Сервисе предусматривается возможность *оперативного возврата к предыдущей стабильной версии компонентов* в случае выявления ошибок после выпуска релиза. Необходимость реализации механизма отката обусловлена: 

* высокой интенсивностью пользовательских запросов, 
* использованием асинхронной обработки событий, 
* большим количеством взаимосвязанных сервисов, нарушение работы которых может привести к деградации ключевых бизнес-сценариев.

Основным сценарием отката является *возврат к предыдущей версии прикладных сервисов* при обнаружении:

* роста количества ошибок пользовательских API,
* деградации производительности,
* нарушения обработки событий Event Bus,
* ошибок резервирования,
* нестабильной работы модели чтения,
* проблем взаимодействия между компонентами.

Для выполнения отката используются ранее подготовленные версии контейнерных образов и конфигураций, сохраненные в системе управления версиями релизов. Это позволяет выполнять *воспроизводимое восстановление предыдущего состояния среды* без необходимости повторной подготовки компонентов вручную.

Процесс отката интегрируется в общий CI/CD-конвейер и может выполняться как автоматически, так и вручную. *Автоматический откат* применяется в случаях, когда после обновления фиксируется:

* критическая недоступность сервисов,
* массовое завершение контейнеров с ошибками,
* нарушение запуска компонентов,
* резкий рост количества ошибок после поставки релиза.

В подобных сценариях система поставки может автоматически вернуть предыдущую стабильную версию сервисов без ожидания ручного вмешательства операторов сопровождения.

Дополнительно предусматриваются *сценарии аварийного отката*, применяемые при возникновении более сложных проблем, затрагивающих обработку событий, конфигурации инфраструктуры или обновление схем хранения данных. В таких ситуациях решение о возврате изменений принимается после анализа состояния сервисов, журналов работы и результатов мониторинга.

При выполнении аварийного отката особое внимание уделяется компонентам, участвующим в обработке событий и операциях резервирования. Для данных сервисов дополнительно контролируется:

* корректность восстановления обработки сообщений,
* отсутствие потери критичных событий,
* восстановление синхронизации модели чтения,
* стабильность пользовательских сценариев после возврата версии.

Одной из основных задач процесса отката является *минимизация времени восстановления работоспособности Сервиса*. Для сокращения времени возврата используются:

* централизованное хранение версий релизов,
* автоматизированные сценарии развертывания,
* подготовленные конфигурации предыдущих версий,
* контроль состояния сервисов после восстановления.

Дополнительно после выполнения отката осуществляется *проверка стабильности среды*. Контролируется:

* доступность пользовательских API,
* корректность работы механизмов резервирования,
* стабильность обработки событий,
* состояние модели чтения и механизмов кэширования.

Результаты мониторинга и журналирования используются для подтверждения успешности восстановления и последующего анализа причин возникновения проблем, потребовавших отката изменений.

==== 4.2.5 Управление качеством поставки

Управление качеством поставки направлено на *снижение вероятности вывода в промышленную эксплуатацию нестабильных или несовместимых изменений*. Контроль качества выполняется на всех этапах поставки — от проверки исходного кода до контроля состояния сервисов после обновления среды.

В рамках процесса поставки используются как *автоматизированные проверки*, так и *регламентные процедуры* сопровождения. Автоматизация применяется для выполнения типовых операций сборки, тестирования и развертывания, а решения о продвижении критичных изменений принимаются с учетом результатов испытаний, состояния компонентов и показателей стабильности работы Сервиса после обновления.

===== 4.2.5.1 Автоматизированный контроль качества

Автоматизированный контроль качества в Сервисе используется для *раннего выявления ошибок и нестабильных изменений* до их поставки в последующие окружения. Автоматические проверки позволяют снизить риск распространения ошибок между компонентами и повысить стабильность выпускаемых релизов.

Контроль качества интегрирован в CI/CD-конвейер и выполняется автоматически после внесения изменений в исходный код или конфигурации компонентов. На первом этапе выполняются *автоматические проверки сборки*, включающие:

* компиляцию приложений,
* проверку зависимостей компонентов,
* подготовку контейнерных образов,
* контроль корректности конфигураций развертывания.

Данные проверки позволяют выявлять ошибки, препятствующие запуску сервисов или дальнейшему продвижению изменений между окружениями.

После успешного завершения сборки запускаются *автоматизированные тесты*, обеспечивающие проверку корректности работы компонентов Сервиса. В рамках проверок контролируется:

* работа пользовательских API,
* корректность выполнения операций резервирования,
* взаимодействие сервисов через Event Bus,
* обработка событий и обновление модели чтения,
* базовая работоспособность интеграционных сценариев.

Для отдельных изменений дополнительно могут выполняться проверки совместимости форматов сообщений, корректности конфигураций окружений и стабильности взаимодействия с хранилищами данных.

Особое внимание уделяется компонентам, участвующим в обработке пользовательских запросов и потоковых событий. Для данных сервисов дополнительно контролируется:

* успешность запуска после обновления,
* отсутствие критических ошибок обработки сообщений,
* стабильность взаимодействия компонентов-поставщиков событий и подписчиков,
* корректность работы механизмов кэширования.

Помимо функциональных проверок, в составе CI/CD-конвейера выполняется *контроль базовых показателей качества*. На данном этапе анализируется:

* успешность выполнения тестов,
* отсутствие критических ошибок сборки,
* стабильность запуска сервисов,
* отсутствие признаков деградации после обновления,
* корректность прохождения обязательных этапов поставки.

Результаты автоматизированного контроля качества используются как *обязательный критерий дальнейшего продвижения изменений* между окружениями. При выявлении критических ошибок, нестабильности сервисов или неуспешном прохождении тестов выполнение последующих этапов CI/CD-конвейера автоматически блокируется до устранения обнаруженных проблем.

Дополнительно результаты проверок интегрируются с механизмами мониторинга и журналирования процесса поставки. Это позволяет:

* анализировать причины неуспешных релизов,
* выявлять повторяющиеся ошибки,
* контролировать стабильность процесса поставки изменений,
* отслеживать качество обновлений между версиями компонентов.

Использование автоматизированного контроля качества позволяет: 

* сократить количество ошибок при выпуске новых версий Сервиса, 
* повысить предсказуемость процесса поставки, 
* обеспечить более стабильную работу пользовательских и событийных сценариев после обновления компонентов.

===== 4.2.5.2 Регламентные процедуры

Процесс поставки изменений в Сервисе строится на сочетании *автоматизированных операций CI/CD-конвейера* и *регламентных процедур сопровождения*. Такой подход позволяет одновременно обеспечить высокую скорость поставки типовых изменений и сохранить дополнительный контроль для операций, способных повлиять на стабильность пользовательских сценариев и работу критичных компонентов архитектуры.

Максимальная автоматизация применяется для повторяющихся операций, выполняемых в рамках стандартного жизненного цикла поставки изменений. *В автоматическом режиме* выполняются:

* сборка прикладных сервисов,
* подготовка контейнерных образов,
* запуск автоматизированных тестов,
* развертывание изменений в тестовых окружениях,
* контроль базовых показателей успешности поставки.

Использование автоматизации позволяет обеспечить *единый порядок выполнения операций* для всех компонентов Сервиса и снизить вероятность ошибок, связанных с ручным выполнением процедур сопровождения.

Особое значение автоматизация имеет для компонентов, участвующих в обработке потоковых событий и пользовательских запросов. Для данных сервисов *автоматически контролируется*:

* успешность запуска после обновления,
* корректность взаимодействия через Event Bus,
* доступность пользовательских API,
* стабильность работы модели чтения и механизмов кэширования.

Результаты автоматизированных проверок используются как обязательное условие дальнейшего продвижения изменений между окружениями.

Несмотря на высокий уровень автоматизации, часть операций в Сервисе выполняется в рамках *регламентных процедур сопровождения*. Это связано с необходимостью дополнительного контроля изменений, способных оказать влияние на стабильность production-среды или затронуть критичные бизнес-сценарии.

*В ручном режиме* выполняются:

* согласование поставки релизов в production-среду,
* анализ результатов испытаний,
* приемка изменений после интеграционного тестирования,
* подтверждение готовности среды к обновлению.

Дополнительно регламентные процедуры используются для поставки изменений, связанных с:

* обновлением схем хранения данных,
* изменением форматов событий,
* обновлением инфраструктурных компонентов,
* изменением конфигураций безопасности.

Для подобных изменений может требоваться дополнительная проверка совместимости компонентов и оценка возможного влияния обновления на пользовательские сценарии.

Отдельный регламент предусматривается для *аварийных изменений*, выполняемых при необходимости срочного исправления критических ошибок или восстановления работоспособности Сервиса. В подобных ситуациях допускается упрощенный процесс согласования поставки при сохранении обязательного контроля:

* успешности сборки,
* корректности развертывания,
* стабильности сервисов после обновления,
* возможности оперативного отката изменений.

После выполнения аварийных изменений дополнительно *проводится анализ результатов обновления и проверка отсутствия деградации пользовательских сценариев*.

Использование сочетания автоматизированных процедур и регламентного контроля позволяет обеспечить *баланс между скоростью поставки изменений и стабильностью промышленной эксплуатации* Сервиса. Такой подход снижает вероятность возникновения критических ошибок при выпуске новых версий компонентов и повышает управляемость процесса обновления решения.

===== 4.2.5.3 Подходы к повышению надежности поставки

При организации процесса поставки изменений в Сервисе особое внимание уделяется *снижению риска нарушения стабильности пользовательских и событийных сценариев* после выпуска новых версий компонентов. Поскольку решение обрабатывает большое количество запросов проверки доступности товаров и потоковых событий от внешних систем, ошибки поставки могут оказывать влияние одновременно на несколько контуров обработки данных.

Одним из основных механизмов повышения надежности поставки является *поэтапное внедрение изменений*. Новые версии компонентов последовательно проходят:

* тестовые окружения,
* интеграционные проверки,
* ограниченное обновление production-среды,
* полное переключение нагрузки на новую версию.

Такой подход позволяет выявлять проблемы на ранних стадиях поставки и ограничивать распространение ошибок между компонентами Сервиса.

Дополнительно при обновлении пользовательских сервисов используется *постепенное перераспределение нагрузки между версиями компонентов*. На начальном этапе обновления новая версия получает ограниченный объем запросов, после чего при отсутствии признаков деградации нагрузка постепенно увеличивается.

Использование постепенного обновления позволяет:

* ограничить влияние ошибок на пользователей,
* выявлять проблемы производительности до полного обновления среды,
* снизить вероятность полной недоступности сервисов,
* упростить возврат к предыдущей версии компонентов.

Особое внимание уделяется сервисам, участвующим в обработке операций резервирования и событий Event Bus. Для данных компонентов дополнительно контролируется:

* корректность обработки сообщений,
* стабильность синхронизации модели чтения,
* отсутствие роста конфликтов резервирования,
* корректность взаимодействия producer- и consumer-компонентов.

Для ограничения влияния ошибок в процессе поставки используются механизмы автоматического контроля состояния сервисов и блокировки дальнейшего продвижения релиза при выявлении признаков нестабильной работы. В рамках контроля анализируется:

* успешность запуска компонентов,
* количество ошибок после обновления,
* доступность пользовательских API,
* стабильность обработки событий,
* корректность работы механизмов кэширования.

При обнаружении критических проблем предусматривается *возможность быстрого отката изменений* к предыдущей стабильной версии. Для этого используются:

* сохраненные версии контейнерных образов,
* централизованное хранение конфигураций,
* автоматизированные сценарии развертывания,
* подготовленные процедуры восстановления среды.

Наличие подготовленных механизмов отката позволяет сократить время восстановления работоспособности Сервиса и снизить влияние ошибок поставки на пользователей.

Дополнительно после завершения релиза выполняется *контроль стабильности обновленной среды*. В течение определенного периода после поставки анализируются:

* ключевые показатели производительности,
* количество ошибок пользовательских операций,
* стабильность обработки событий,
* состояние сервисов и контейнеров,
* корректность работы пользовательских сценариев.

Для анализа состояния среды используются данные мониторинга и журналирования Сервиса. Это позволяет своевременно выявлять скрытые проблемы, возникающие после обновления, и принимать решение о необходимости дополнительной стабилизации среды или выполнения отката изменений.

Использование поэтапной поставки, автоматизированного контроля состояния сервисов и механизмов быстрого отката позволяет:
* повысить надежность процесса обновления компонентов, 
* обеспечить более стабильную эксплуатацию Сервиса при регулярном выпуске новых версий.
