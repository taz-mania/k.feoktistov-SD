=== 2.2. Описание концептуальной архитектуры

==== 2.2.1. Архитектурный стиль

Система должна реализовывать Event-Driven Microservices Architecture с элементами CQRS и Event Sourcing для потоков изменения остатков.

==== 2.2.2. Обоснование выбора

. Необходимость горизонтального масштабирования независимых сервисов для выдерживания пиковых нагрузок (возрастание нагрузки до 10x в промо-периоды) без деградации всей системы.
. Асинхронная обработка событий обеспечивает отзывчивость без блокировки POS-систем магазинов при временной недоступности центральных сервисов.
. Использование Event Bus как брокера сообщений обеспечивает автономную работу магазинов при разрыве связи с ЦОД с последующей синхронизацией (eventual consistency).
. Применение CQRS позволяет раздельно оптимизировать путь записи (быстрый прием событий) и путь чтения (низкая задержка для клиентского API).
. Использование Domain-Driven Design с изоляцией bounded contexts (наличие, доступность, резервирование) обеспечивает эволюционируемость системы.

=== 2.2.3. Описание компонентов архитектуры

Ниже представлено описание ключевых элементов концептуальной архитектуры сервиса запросов информации о наличии товаров и механизмов их взаимодействия.

==== Клиенты и каналы доступа

Система обслуживает пользователей через несколько каналов взаимодействия: 

* *Веб-сайт* и *МП* предоставляют покупателям функционал поиска товаров по наличию, просмотра остатков в конкретных магазинах и резервирования товаров. 
* *ТСД* используется сотрудниками торговых залов для сканирования штрих-кодов, корректировки фактических остатков при пересорте и оперативной инвентаризации. 
* *Админ-консоль* предназначена для операторов бэк-офиса и администраторов, позволяя управлять справочниками, настройками кэширования и выполнять ручные корректировки данных. 
* *BI-инструмент* обеспечивает доступ категорийных менеджеров к аналитическим данным об оборачиваемости товаров и статистике остатков в режиме только для чтения.

==== Периметр безопасности и маршрутизация

* Все внешние запросы проходят через балансировщик *LB*, который распределяет нагрузку между экземплярами приложений с учетом географической близости. 
* *CDN* обеспечивает доставку статического контента и справочной информации с минимальной задержкой. 
* *API GW* выступает единой точкой входа, выполняя терминацию защищенного соединения, проверку политик доступа и маршрутизацию запросов к соответствующим доменным сервисам. 
* Для интеграции с внешними службами доставки выделен отдельный сегмент шлюза с повышенными ограничениями доступа.

==== Доменные сервисы и их хранилища

* *Сервис запросов* обрабатывает синхронные обращения клиентов на проверку наличия товара, обращаясь к распределенному кэшу для получения текущих остатков с минимальной задержкой. 
* При отсутствии данных в оперативной памяти сервис обращается к *Хранилищу моделей чтения*, где хранятся денормализованные представления остатков, материализованные из потока событий. 
* Для определения ближайших точек продаж используется *Гео-сервис*, который оперирует данными *Гео-индекса*, содержащего пространственные координаты магазинов и зоны покрытия.
* *Сервис резервирования* управляет жизненным циклом бронирований товаров под заказы клиентов, обеспечивая строгую консистентность операций. 
* Для предотвращения одновременного списания одной единицы товара несколькими клиентами используется *Менеджер блокировок*, координирующий распределенные блокировки через механизмы кэша. 
* База данных *БД резервирования* хранит актуальный реестр активных резервов, их временные метки и статусы, обеспечивая атомарность транзакций.
* *Админ-сервис* предоставляет интерфейс для управления порогами безопасного запаса, настройки параметров кэширования и ручной корректировки остатков при обнаружении расхождений. 
* Все справочные данные, конфигурации и учетные записи пользователей хранятся в базе данных *Метаданные, справочники*.

==== Интеграционные адаптеры

* *Адаптер касс*, *Адаптер склада*, *Адаптер заказов* и *Адаптер ERP* обеспечивают взаимодействие с внешними системами предприятия : кассовые терминалы (POS), складское управление (WMS), управление заказами (OMS) и ресурсами (ERP-система). Адаптеры нормализуют форматы входящих данных и публикуют события в центральную шину для дальнейшей обработки. 
* *Адаптер служб доставки* изолирует внутреннюю архитектуру от внешних служб экспресс-доставки, обеспечивая фильтрацию чувствительной информации и жесткий контроль доступа.
* *Клиент Лояльности* обращается к платформе программ лояльности за данными о статусе покупателей и персонализированными правилами ранжирования результатов поиска, кэшируя ответы для снижения нагрузки на внешнюю систему.

==== Вспомогательные сервисы и обработка данных

* *Сервис команд* принимает потоки событий об изменении остатков от интеграционных адаптеров, выполняет валидацию бизнес-правил и фиксирует факты изменений в *Хранилище событий*. Этот журнал является единым источником истины обо всех операциях с товарными запасами.
* *Захват изменений* отслеживает обновления в хранилище событий и инициирует репликацию данных в целевые системы с минимальным лагом. 
* *Загрузчик моделей чтения* асинхронно строит денормализованные представления для быстрого доступа, используя поток событий как источник данных.
* *Сервис сверки* периодически анализирует расхождения между данными различных источников (кассовые продажи и складские остатки), применяет бизнес-правила приоритизации и публикует корректирующие события для устранения несоответствий. 
* История выявленных конфликтов и примененных правил корректировки хранится в *БД конфликтов*.
* *Сервис прогрева кэша* получает триггеры о предстоящих маркетинговых акциях от движка ценообразования и заблаговременно загружает остатки акционных товаров в оперативную память, предотвращая пиковые нагрузки на базу данных в момент старта промо-акций. 
* Информация о планируемых заданиях и календаре акций хранится в *БД прогрева кэша*. 
* *Сервис синхронизации кэша* обеспечивает проактивное обновление или инвалидацию записей в распределенном кэше при получении уведомлений об изменениях остатков.
* *Управление секретами* централизованно хранит ключи доступа, сертификаты и параметры подключения к внешним системам, обеспечивая динамическое управление учетными данными без их жесткой фиксации в конфигурациях микросервисов.

==== Шина данных и очереди

* *Event Bus* выступает центральным брокером сообщений, обеспечивая асинхронный обмен событиями между компонентами Сервиса. Это позволяет доменным сервисам работать автономно при временных разрывах связи, накапливая события локально с последующей синхронизацией. 
* *Очередь ошибок* изолирует сообщения, которые не удалось обработать после исчерпания попыток повторной обработки, предоставляя возможность ручного анализа и разрешения инцидентов администратором.
* *Реестр схем* управляет версионированием форматов данных событий, гарантируя совместимость между производителями и потребителями при эволюции структур данных.

==== Хранилища данных

* *Распределенный кэш* построен по схеме Master-Replica, обеспечивая отказоустойчивость и быстрый доступ к актуальным остаткам. Master-нода принимает операции записи от сервисов обновления данных, в то время как Replica-ноды обслуживают запросы на чтение.
* *Хранилище временных рядов* агрегирует исторические метрики скорости продаж, события отсутствия товаров и системные показатели для последующего анализа и построения прогнозных моделей. 
* *Архивное хранилище* служит для долгосрочного хранения сырых событий и финансовых сводок с высокой степенью сжатия, обеспечивая compliance и возможность восстановления данных за прошлые периоды.

==== Наблюдаемость и безопасность

* *Система мониторинга и трассировки* собирает метрики производительности, распределенные трассы запросов и health-check компонентов. 
* *Система логирования* централизованно агрегирует журналы операций со всех сервисов, обеспечивая корреляционный поиск по идентификаторам запросов. 
* *Оповещение* генерирует уведомления ответственным командам при превышении критических порогов ошибок или задержек обработки. 
* *Логирование событий информационной безопасности* фиксирует факты доступа и изменений критичных параметров для аудита.
