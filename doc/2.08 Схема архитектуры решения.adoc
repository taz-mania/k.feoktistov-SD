==== 2.8.1. Общее описание детальной архитектуры 

Сервис обеспечивает определение доступности товаров и выполнение операций резервирования в торговой сети в условиях высокой нагрузки. Запросы от мобильных приложений, веб-клиента и внутренних устройств поступают через CDN, балансировщик нагрузки и API Gateway входного слоя, после чего обрабатываются прикладными сервисами.

*В контуре чтения* `Сервис запросов` формирует ответ пользователю, используя `Распределенный кеш`, как основной источник данных о доступности товаров. Для определения ближайших магазинов применяется `Гео-сервис`, который обращается к `Гео-индексу`, координатам. При необходимости используются данные из `Хранилища модели чтения` на базе MongoDB и `хранилищу метаданных и справочников`. Для учета особенностей клиента задействуются `Клиент Лояльности` и `Кэш лояльности`.

*Контур записи и резервирования* обеспечивает создание и управление резервами. `Сервис резервирования` выполняет транзакционные операции с использованием `БД резервирования` и `Менеджер блокировок`, предотвращающего одновременное резервирование одного и того же товара. `Сервис команд` и `Админ-сервис` обеспечивают выполнение управляющих операций и изменение данных в кеше.

Все изменения передаются в *Контур событий* через `Event Bus` на базе Kafka. Потребители событий, включая `Загрузчик моделей чтения`, `Сервис синхронизации кеша`, `Сервис прогрева кэша`, `Сервис агрегации метрик` и `Сервис сверки`, обновляют `Хранилище модели чтения`, поддерживают актуальность `Распределенного кеша`, формируют метрики в `Хранилище временных рядов` и обрабатывают расхождения, сохраняя их в `БД конфликтов`.

Для обработки ошибок используется `Очередь ошибок` на базе того же кластера Kafka. Контроль формата событий выполняется через `Реестр схем` и `БД реестра схем`, а `Захват изменений` (CDC) обеспечивает поступление изменений из источников данных.

Интеграция с внешними системами: кассовыми системами (POS), системами управления складом (WMS), системами управления заказами (OMS), системой промо-правил (PPE) и ERP-системой управления ресурсами предприятия выполняется через выделенные самостоятельные `Адаптеры` и `API Gateway внешних интеграций`. События от внешних систем поступают в `Event Bus` и используются для обновления доступности товаров.

Отдельно реализована интеграция со службами доставки через `Адаптер службы доставки` и `API Gateway служб доставки`. Сервис обеспечивает обмен данными о доступности товаров для доставки, статусах заказов и исполнении доставки. Взаимодействие осуществляется с внешними системами доставки, а получаемые события используются для актуализации доступности товаров и статусов исполнения заказов.

*Слой данных* включает специализированные хранилища: `Хранилище событий` (отказоустойчивый кластер) для журналирования событий, `БД резервирования`` для транзакций, `Хранилище модели чтения` для быстрого доступа к данным, `Хранилище временных рядов` для метрик, `БД прогрева кэша` и `Кэш лояльности`. `Архивное хранилище` используется для долговременного хранения исторических данных.

*Инфраструктурный слой* обеспечивает эксплуатацию системы и включает `Управление секретами`, `Систему логирования`, `Систему мониторинга и трассировки`, а также сервис `Оповещение` для реагирования на инциденты.

Представленная архитектура обеспечивает масштабируемость, отказоустойчивость и высокую производительность сервиса за счет разделения контуров, использования распределенного кеша и асинхронной обработки событий.

==== 2.8.2. Описание комопнентов в составе архитектуры

===== 2.8.2.1. Прикладные микросервисы

*Сервис запросов* 
: Отвечает за обработку пользовательских запросов на получение информации о доступности товаров. Формирует ответы на основе данных из `Распределенного кеша`, при необходимости обращается к `Гео-сервису` для определения ближайших магазинов, а также использует `Хранилище модели чтения` и хранилище `Метаданных и справочников`. Сервис является stateless и масштабируется горизонтально для обработки высокой нагрузки.

*Сервис резервирования* 
: Обеспечивает создание, изменение и завершение операций резервирования товаров. Выполняет транзакционные операции с использованием `БД резервирования` и взаимодействует с `Менеджером блокировок` для предотвращения конфликтов при одновременных запросах. Сервис является критичным для обеспечения корректности остатков и предотвращения перепродажи товара.

*Сервис команд*
: Обрабатывает управляющие операции, не связанные напрямую с пользовательскими запросами чтения, такие, как изменения состояния системы, корректировки данных и служебные действия. Публикует соответствующие события в `Event Bus` для дальнейшей обработки другими компонентами системы.

*Сервис сверки*
: Выполняет асинхронную проверку согласованности данных между различными источниками (POS, WMS, резервирование, кеш). При обнаружении расхождений фиксирует их в 'БД конфликтов' и инициирует корректирующие действия через публикацию событий. Сервис не участвует в обработке пользовательских запросов и работает в фоновом режиме.

*Админ-сервис*
: Предоставляет интерфейс управления системой (Сервисом), включая настройку параметров, управление справочниками и контроль состояния сервисов. Используется административным персоналом магазинов и сервиса и не участвует в обработке пользовательского трафика.

*Гео-сервис*
: Отвечает за поиск ближайших магазинов на основе координат пользователя. Использует `Гео-индекс`, координаты для выполнения пространственных запросов и возвращает список магазинов, который затем используется `Сервисом запросов` для проверки доступности товаров.

*Клиент Лояльности*
: Обеспечивает взаимодействие с `Платформой лояльности` для получения информации о клиенте, которая может влиять на доступность товаров или условия их получения. Для ускорения работы используется 'Кэш лояльности'.

*Сервис агрегации метрик*
: Обрабатывает поток событий из `Event Bus` и формирует агрегированные показатели, которые сохраняются в `Хранилище временных рядов`. Эти данные используются для мониторинга, аналитики и построения отчетов.

*Загрузчик моделей чтения*
: Является потребителем событий и отвечает за построение и обновление `Хранилища модели чтения`. Преобразует поток событий в удобную для быстрого чтения структуру данных.

*Сервис синхронизации кеша*
: Обрабатывает события изменения остатков и резервов и обновляет 'Распределенный кеш', обеспечивая его актуальность. Выполняет инвалидацию устаревших данных и запись новых значений.

*Сервис прогрева кэша*
: Заранее загружает данные о популярных товарах и географических зонах в `Распределенный кеш`. Это позволяет снизить задержки и уменьшить количество промахов кеша при пиковых нагрузках.

*Адаптеры POS/WMS/OMS/PPE*
: Обеспечивают интеграцию с внешними (по тоношению к Сервису) системами компании: кассовые системы (POS), системы управления складом (WMS), системы управления заказами (OMS) и движком цен и промо-акций (PPE). Адаптеры принимают внешние данные, нормализуют их и публикуют в `Event Bus`.

*Адаптер ERP*
: Обеспечивает обмен данными с системой управления ресурсами предприятия. Используется для синхронизации справочников, атрибутов товаров и других бизнес-данных.

*Адаптер службы доставки*
: Обеспечивает интеграцию с внешними службами доставки. Обрабатывает данные о доступности товаров для доставки, статусах заказов и ходе выполнения доставки.

*Менеджер блокировок*
: Обеспечивает механизм распределенных блокировок при выполнении операций резервирования. Предотвращает одновременное изменение одного и того же ресурса (товара в магазине), обеспечивая корректность операций.

*Захват изменений (CDC)*
: Отслеживает изменения в операционных базах данных и публикует их в 'Event Bus'. Это позволяет синхронизировать данные без прямого обращения к источникам.

*Реестр схем*
: Хранит описание форматов событий и обеспечивает контроль их совместимости. Позволяет безопасно развивать систему, изменяя структуру сообщений без нарушения работы потребителей.

===== 2.8.2.2. Хранилища данных

*Распределенный кеш*
: Является основным источником данных для операций чтения. Хранит предрасчитанные значения доступности товаров и обеспечивает быстрый доступ с минимальной задержкой.

*Кэш лояльности*
: Хранит данные, полученные из Платформы Лояльности, и используется для ускорения обработки запросов, связанных с клиентскими условиями.

*Event Bus*
: Является центральным компонентом событийной архитектуры. Обеспечивает передачу событий между сервисами, буферизацию нагрузки и возможность асинхронной обработки данных.

*Хранилище событий*
: Используется для записи событий Сервиса и может применяться для восстановления состояния и аудита. Конфигурация с репликацией обеспечивает отказоустойчивость.

*Хранилище модели чтения*
: Содержит агрегированные данные, оптимизированные для быстрого чтения. Формируется на основе событий и используется при промахах кеша или для сложных запросов.

*Метаданные, справочники*
: Хранилище метаданных и справочников содержит справочную информацию о товарах, магазинах и других сущностях. Эти данные используются сервисами для формирования корректных ответов.

*БД конфликтов*
: Хранит информацию о выявленных расхождениях между источниками данных и результатах их обработки. Используется `Сервисом сверки`.

*БД резервирования*
: Хранит данные о резервах товаров и обеспечивает выполнение транзакционных операций с гарантией согласованности.

*БД реестра схем*
: Хранит версии схем событий и обеспечивает работу `Реестра схем`.

*БД прогрева кэша*
: Хранит задания и параметры прогрева, включая список популярных товаров и регионов. Используется `Сервисом прогрева кэша`.

*Хранилище временных рядов*
: Используется для хранения метрик системы и временных данных. Применяется для мониторинга и анализа производительности.

*Архивное хранилище*
: Предназначено для долговременного хранения исторических данных, событий и метрик. Используется для аналитики, аудита и восстановления данных.
