==== 3.2.2 Регистрация событий безопасности

===== 3.2.2.1 События аутентификации

Подсистема аудита безопасности Сервиса регистрирует события, связанные с аутентификацией пользователей, сервисных учетных записей и интеграционных компонентов.

Регистрация событий аутентификации используется для:

* выявления попыток несанкционированного доступа,
* контроля использования учетных записей,
* расследования инцидентов безопасности,
* выявления компрометации учетных данных,
* анализа аномальной активности.

Для событий аутентификации в журналах аудита безопасности фиксируются:

* время события,
* идентификатор пользователя или сервиса,
* тип учетной записи,
* IP-адрес источника,
* идентификатор запроса,
* результат операции,
* причина отказа при ошибке аутентификации.

*1. Успешная аутентификация*

При успешной аутентификации пользователя или сервиса регистрируется событие успешного входа.

Событие формируется:

* API Gateway,
* средствами аутентификации,
* внутренними сервисами при сервисной аутентификации.

Пример записи аудита безопасности:

```json
{
  "event_type": "authentication_success",  // Тип регистрируемого события
  "timestamp": "2026-06-08T10:15:21Z",     // Штамп времени события
  "user_id": "user_18452",                 // Идентаификатор пользователя
  "role": "operator",                      // Роль пользователя
  "source_ip": "194.164.216.227",          // Адрес пользователя
  "authentication_method": "OAuth2",       // Использованный метод аутентификации
  "request_id": "req-7d9a12",              // Идентификатор запроса
  "result": "success"                      // Результат аутентификации
}
```

Для сервисной аутентификации дополнительно фиксируются:

* сервисная роль,
* идентификатор сервиса,
* идентификатор сертификата mTLS.

Пример записи аудита сервисной аутентификации:

```json
{
  "event_type": "service_authentication_success",
  "timestamp": "2026-06-08T10:16:02Z",
  "service_id": "reservation-service",        // Идентификатор сервиса
  "service_role": "reservation_writer",       // Роль сервиса
  "certificate_id": "mtls-cert-18f2",         // Идентификатор mTLS сертификата 
  "request_id": "req-9a2cd1",
  "result": "success"
}
```

*2. Неуспешная аутентификация*

При ошибке аутентификации формируется событие неуспешного входа.

События регистрируются при:

* использовании недействительного JWT access-токена,
* ошибке проверки подписи токена,
* использовании просроченного токена,
* ошибке проверки сервисного сертификата,
* использовании заблокированной учетной записи,
* ошибке проверки refresh-токена.

Пример записи аудита безопасности:

```json
{
  "event_type": "authentication_failed",
  "timestamp": "2026-06-08T10:18:44Z",
  "user_id": "user_18452",
  "source_ip": "194.164.216.227",
  "failure_reason": "invalid_jwt_signature",   // Причина отказа аутентификации : ошибка подписи JWT-токена
  "request_id": "req-c12851",
  "result": "denied"                           // Результата : В доступе отказано 
}
```

Пример записи ошибки сервисной аутентификации:

```json
{
  "event_type": "service_authentication_failed",
  "timestamp": "2026-06-08T10:19:11Z",
  "service_id": "read-model-loader",
  "failure_reason": "invalid_mtls_certificate",   // Причина отказа аутентификации : ошибка mTLS сертификата
  "request_id": "req-31ab8d",
  "result": "connection_rejected"                 // Результат : Соединение разорвано   
}
```

Повторяющиеся ошибки аутентификации используются механизмами мониторинга безопасности Сервиса для выявления *попыток подбора учетных данных и компрометации учетных записей*.

*3. Блокировка учетной записи*

При выявлении подозрительной активности или нарушении политик безопасности выполняется блокировка учетной записи пользователя или сервиса.

Блокировка может выполняться:

* после серии ошибок аутентификации,
* после выявления признаков компрометации,
* при нарушении административных политик,
* при выявлении аномального поведения учетной записи.

Пример записи аудита безопасности:

```json
{
  "event_type": "account_locked",
  "timestamp": "2026-06-08T10:21:03Z",
  "user_id": "user_18452",                                   // Блокируемая учетная запись
  "lock_reason": "multiple_failed_authentication_attempts",  // Причина блокировки : множественные попытки неуспешной аутентификации
  "request_id": "req-a912fd",
  "result": "account_blocked"                                // Результат : Учетная запись пользователя заблокирована
}
```

Пример записи блокировки сервисной учетной записи:

```json
{
  "event_type": "service_account_blocked",
  "timestamp": "2026-06-08T10:22:17Z",
  "service_id": "integration-erp-adapter",
  "block_reason": "suspicious_activity_detected",  // Причина блокировки : подозрительная активность 
  "request_id": "req-f18d72",
  "result": "service_account_disabled"             // Результат : Аккаунт сервиса отключен  
}
```

Для административных учетных записей события блокировки дополнительно рассматриваются как *события повышенного уровня критичности*.

*4. Обновление токенов*

Подсистема аудита регистрирует операции обновления JWT access-токенов с использованием refresh-токенов.

Регистрируются:

* успешное обновление токена,
* ошибки проверки refresh-токена,
* повторное использование отозванного refresh-токена,
* попытки обновления после завершения пользовательской сессии.

Пример записи аудита успешного обновления токена:

```json
{
  "event_type": "token_refresh_success",
  "timestamp": "2026-06-08T10:24:11Z",
  "user_id": "user_18452",
  "session_id": "sess-3f91ac",            // Идентификатор сессии
  "request_id": "req-b77128",
  "result": "success"
}
```

Пример записи ошибки обновления токена:

```json
{
  "event_type": "token_refresh_failed",
  "timestamp": "2026-06-08T10:25:09Z",
  "user_id": "user_18452",
  "failure_reason": "revoked_refresh_token",   // Причина : Refresh-токен отозван 
  "request_id": "req-8d1aa4",
  "result": "denied"                           // Результат : Доступ запрещен
}
```

Для операций обновления токенов дополнительно контролируются:

* частота обновления,
* источник запроса,
* изменение IP-адреса,
* повторное использование refresh-токена.

Аномальная активность операций обновления токенов рассматривается как *признак возможной компрометации пользовательской сессии*.

===== 3.2.2.2 События авторизации

Подсистема аудита безопасности Сервиса регистрирует события, связанные с проверкой полномочий пользователей, сервисных учетных записей и интеграционных компонентов при выполнении операций доступа.

События авторизации используются для:

* выявления попыток обхода политик доступа,
* контроля использования привилегированных ролей,
* выявления избыточных полномочий,
* расследования нарушений безопасности,
* контроля административных операций.

Для событий авторизации в журналах аудита безопасности фиксируются:

* идентификатор пользователя или сервиса,
* роль,
* вызываемый API-метод,
* тип операции,
* request_id,
* результат проверки полномочий,
* причина отказа в доступе.

*1. Отказ в доступе*

Событие отказа в доступе регистрируется при попытке вызова API или выполнения операции без необходимых полномочий.

Отказы формируются:

* API Gateway,
* внутренними сервисами,
* административными компонентами.

Пример записи аудита безопасности:

```json
{
  "event_type": "authorization_denied",
  "timestamp": "2026-06-08T11:03:18Z",
  "user_id": "user_18452",
  "role": "client_application",
  "resource": "/api/v1/admin/cache/reload",      // Запрашиваемый ресурс
  "request_id": "req-28ad17",
  "denial_reason": "insufficient_permissions",   // Причина отказа : Недостаточно прав доступа
  "result": "access_denied"
}
```

Пример отказа сервисной учетной записи:

```json
{
  "event_type": "service_authorization_denied",
  "timestamp": "2026-06-08T11:04:07Z",
  "service_id": "metrics-aggregator",
  "service_role": "analytics_reader",
  "resource": "reservation.events",
  "request_id": "req-f19a81",
  "denial_reason": "topic_access_forbidden",      // Причина отказа : Доступ к топику запрещен
  "result": "operation_rejected"
}
```

*2. Попытка выполнения запрещенной операции*

Событие формируется при попытке выполнения операции, запрещенной назначенной ролью или политиками доступа Сервиса.

Регистрируются:

* попытки административных операций обычными пользователями,
* попытки публикации запрещенных событий,
* попытки изменения конфигурации,
* попытки удаления критичных данных,
* попытки использования внутренних API.

Пример записи аудита безопасности:

```json
{
  "event_type": "forbidden_operation_attempt",
  "timestamp": "2026-06-08T11:07:33Z",
  "user_id": "user_18452",
  "role": "operator",
  "operation": "delete_inventory_snapshot",      // Операция : Удаление снимка состояния остатков
  "request_id": "req-b17d92",
  "result": "operation_blocked"                  // Результат : Операция заблокирована
}
```

Пример попытки запрещенной сервисной операции:

```json
{
  "event_type": "forbidden_service_operation",
  "timestamp": "2026-06-08T11:08:16Z",
  "service_id": "integration-delivery-adapter",
  "service_role": "delivery_integration",
  "operation": "publish_admin_event",
  "request_id": "req-c62a19",
  "result": "operation_blocked"
}
```

*3. Нарушение политик доступа*

События нарушения политик доступа формируются при выявлении действий, противоречащих установленным ограничениям безопасности Сервиса.

Регистрируются:

* использование неразрешенных ролей,
* обращение к запрещенным ресурсам,
* попытки обхода ограничений API Gateway,
* использование запрещенных сервисных учетных записей,
* нарушение ограничений доступа к Event Bus.

Пример записи аудита безопасности:

```json
{
  "event_type": "access_policy_violation",
  "timestamp": "2026-06-08T11:11:04Z",
  "user_id": "admin_02",
  "role": "administrator",
  "policy": "restricted_admin_operation_policy",           // Политика безопасности, применённая при проверке запроешшых действий 
  "request_id": "req-d7128c",
  "violation_reason": "operation_outside_allowed_scope",   // Причина отказа : операция за рамками полномочий
  "result": "security_violation_detected"                  // Результат : Выявлено нарушение политики безопасности
}
```

Пример нарушения сервисной политики доступа:

```json
{
  "event_type": "service_policy_violation",
  "timestamp": "2026-06-08T11:12:28Z",
  "service_id": "read-model-loader",
  "service_role": "read_model_updater",
  "policy": "event_bus_access_policy",
  "request_id": "req-a29c77",
  "violation_reason": "unauthorized_topic_subscription",   // Причина отказа : Попытка несанкционированной подписки на Kafka-топик
  "result": "subscription_rejected"                        // Результат : Отказано в подписке
}
```

*4. Использование привилегированных функций*

Подсистема аудита безопасности регистрирует использование функций, доступных только привилегированным ролям.

К таким операциям относятся:

* изменение ролей, политик доступа,
* управление сертификатами, секретами,
* изменение конфигурации API Gateway,
* операции управления Event Bus,
* административные операции Kubernetes.

Пример записи аудита безопасности:

```json
{
  "event_type": "privileged_operation_executed",
  "timestamp": "2026-06-08T11:15:42Z",
  "user_id": "admin_01",
  "role": "administrator",
  "operation": "update_access_policy",             // Операция : Обновление политики безопасности
  "request_id": "req-e91bc5",
  "result": "success"
}
```

Пример использования привилегированной сервисной функции:

```json
{
  "event_type": "privileged_service_operation",
  "timestamp": "2026-06-08T11:16:27Z",
  "service_id": "vault-management-service",
  "service_role": "security_administration",
  "operation": "rotate_service_certificate",      // Операция : Обновление сертификата сервиса

  "result": "success"

```

События использования привилегированных функций рассматриваются как события *повышенного уровня критичности и подлежат дополнительному контролю* со стороны мониторинга безопасности Сервиса.

===== 3.2.2.3 События доступа к данным

Подсистема аудита безопасности Сервиса регистрирует события, связанные с доступом к данным, изменением критичной информации и выполнением административных операций над хранилищами данных Сервиса.

Регистрация событий доступа к данным используется для:

* контроля доступа к чувствительной информации,
* выявления несанкционированного изменения данных,
* расследования инцидентов безопасности,
* контроля действий привилегированных учетных записей,
* выявления аномального использования данных Сервиса.

Для событий доступа к данным в журналах аудита безопасности фиксируются:

* идентификатор пользователя или сервиса,
* роль,
* тип операции,
* идентификатор хранилища,
* объект доступа,
* идентификатор запроса,
* результат выполнения операции,
* источник запроса.

*1. Доступ к конфиденциальным данным*

События доступа регистрируются при обращении к данным, доступ к которым ограничен политиками безопасности Сервиса.

Контролируется доступ к:

* административным данным,
* сервисным конфигурациям,
* журналам аудита безопасности,
* секретам и сертификатам,
* данным пользовательских сессий,
* данным резервирования.


===== 3.2.2.4 События управления секретами

Подсистема аудита безопасности Сервиса регистрирует события, связанные с использованием секретов, сервисных сертификатов и криптографических ключей, размещенных в HashiCorp Vault.

Регистрация событий управления секретами используется для:

* контроля доступа к секретам Сервиса,
* выявления несанкционированного использования сервисных учетных записей,
* контроля операций выпуска и отзыва сертификатов,
* расследования инцидентов безопасности,
* выявления компрометации сервисных секретов и ключей.

Для событий управления секретами в журналах аудита безопасности фиксируются:

* идентификатор пользователя или сервиса,
* сервисная роль,
* тип секрета или сертификата,
* операция,
* источник запроса,
* идентификатор запроса,
* результат выполнения операции.

*1. Получение секретов*

События получения секретов регистрируются при обращении сервисов к HashiCorp Vault для получения учетных данных и конфигурационных секретов.

Контролируются операции получения:

* учетных данных Kafka, PostgreSQL, MongoDB, Redis Cluster,
* сервисных токенов,
* интеграционных ключей,
* параметров проверки JWT access-токенов.

Пример записи аудита безопасности:

```json
{
  "event_type": "secret_access",
  "timestamp": "2026-06-08T12:03:11Z",
  "service_id": "reservation-service",
  "service_role": "reservation_writer",
  "secret_type": "postgresql_credentials",              // Тип секрета: Учетные данные доступа к PostgreSQL
  "secret_path": "kv/postgresql/reservation-service",   // Путь к секрету в хранилище HashiCorp Vault
  "request_id": "req-2a91df",
  "result": "success"
}
```

Пример попытки доступа к запрещенному секрету:

```json
{
  "event_type": "secret_access_denied",
  "timestamp": "2026-06-08T12:04:37Z",
  "service_id": "metrics-aggregator",
  "service_role": "analytics_reader",
  "secret_type": "vault_administration_secret",   // Секрет административного доступа к хранилищу секретов HashiCorp Vault
  "secret_path": "kv/security/vault-admin",
  "request_id": "req-81fc72",
  "result": "access_denied"
}
```

*2. Изменение секретов*

Подсистема аудита безопасности регистрирует операции изменения или обновления секретов Сервиса.

Регистрируются изменение:

* сервисных токенов,
* интеграционных ключей,
* параметров подключения,
* административных секретов,
* политик доступа HashiCorp Vault.

Пример записи аудита безопасности:

```json
{
  "event_type": "secret_modified",               // Тип события : Изменение секрета
  "timestamp": "2026-06-08T12:08:14Z",
  "user_id": "security_admin_01",
  "role": "administrator",
  "secret_type": "kafka_credentials",            // Учетные данные сервиса Kafka
  "secret_path": "kv/kafka/read-model-loader",
  "request_id": "req-17dc9a",
  "result": "success"
}
```

Пример изменения политики доступа:

```json
{
  "event_type": "vault_policy_updated",
  "timestamp": "2026-06-08T12:09:52Z",
  "user_id": "security_admin_02",
  "role": "administrator",
  "policy": "reservation_service_policy",
  "operation": "update",
  "request_id": "req-f72c1d",
  "result": "success"
}
```

*3. Ротация секретов*

События ротации регистрируются при автоматическом или ручном обновлении сервисных секретов и учетных данных.

Контролируется ротация:

* сервисных токенов,
* учетных данных Kafka, БД,
* интеграционных ключей,
* административных секретов.

Пример записи аудита безопасности:

```json

  "event_type": "secret_rotation",                 // Тип события : Ротация секрета

  "service_role": "cache_updater",
  "secret_type": "redis_credentials",              // Тип секрета : учетные данные для доступа к Redis
```

*4. Получение сертификатов*

Отзыв выполняется при:

===== 3.2.2.5 События изменения конфигурации

Подсистема аудита безопасности Сервиса регистрирует события, связанные с изменением конфигурации компонентов Сервиса, политик безопасности и криптографических параметров.

Регистрация событий изменения конфигурации используется для:

* контроля административных изменений,
* выявления несанкционированной модификации параметров Сервиса,
* расследования инцидентов безопасности,
* контроля использования привилегированных ролей,
* отслеживания изменений параметров безопасности.
