==== 2.11.5. Обнаружение сервисов

===== 2.11.5.1. Назначение механизма обнаружения сервисов

Сервис реализован как распределённая микросервисная архитектура, в которой обработка пользовательских запросов и внутренних операций выполняется несколькими независимыми компонентами. В процессе работы экземпляры сервисов могут динамически запускаться, останавливаться, масштабироваться или перемещаться между вычислительными узлами. В таких условиях использование статически заданных сетевых адресов становится неэффективным и существенно усложняет сопровождение Сервиса.

Для решения данной проблемы в Сервисе используется механизм обнаружения сервисов. Его основное назначение заключается в *автоматическом определении доступных экземпляров сервисов и предоставлении другим компонентам актуальной информации* о том, каким образом с ними необходимо взаимодействовать.

Механизм обнаружения сервисов является одним из *базовых элементов внутренней инфраструктуры* Сервиса и используется для организации взаимодействия между микросервисами. 

Основная причина использования механизма обнаружения сервисов -- необходимость горизонтального масштабирования. +
При использовании фиксированных адресов любое изменение количества экземпляров потребовало бы ручного обновления конфигурации взаимодействия между компонентами. Для высоконагруженного Сервиса такой подход является неприемлемым, поскольку существенно усложняет масштабирование и снижает устойчивость архитектуры.

Наибольшая нагрузка в Сервисе приходится на пользовательские сценарии проверки доступности товара. При росте количества запросов может потребоваться запуск дополнительных экземпляров Сервиса запросов или компонентов обработки событий. Механизм обнаружения сервисов позволяет подключать новые экземпляры автоматически без изменения конфигурации клиентских компонентов.

Дополнительно данный механизм используется для повышения отказоустойчивости Сервиса. +
При отказе отдельного экземпляра сервиса его сетевой адрес автоматически исключается из списка доступных узлов. После этого запросы начинают перенаправляться только на работоспособные экземпляры. Такой подход *уменьшает вероятность каскадной деградации* обработки внешних запросов и повышает устойчивость внутреннего взаимодействия.

Особенно важную роль механизм обнаружения сервисов играет для компонентов контура чтения. Сервис запросов выполняет большое количество внутренних вызовов к Гео-сервису, Распределенному кэшу и другим компонентам. При высокой интенсивности пользовательских запросов количество экземпляров отдельных сервисов может динамически изменяться. Использование автоматического обнаружения сервисов позволяет выполнять балансировку нагрузки между экземплярами без ручного обновления сетевой конфигурации.

Аналогичный подход используется и в контуре обработки событий. При росте интенсивности поступления событий изменения остатков могут запускаться дополнительные экземпляры Загрузчика моделей чтения. После регистрации в механизме обнаружения сервисов новые экземпляры автоматически начинают участвовать в обработке потока событий.

*Дополнительно* механизм обнаружения сервисов используется для:

* централизованного контроля доступности компонентов,
* упрощения маршрутизации внутренних вызовов,
* автоматического исключения недоступных экземпляров,
* сокращения времени восстановления после отказов,
* упрощения развёртывания новых версий сервисов.

Использование механизма обнаружения сервисов особенно важно для Сервиса проверки доступности товара, поскольку архитектура решения ориентирована на работу в условиях переменной нагрузки. В течение суток интенсивность пользовательских запросов и потоков обновления остатков может существенно изменяться. Возможность динамического масштабирования компонентов без изменения конфигурации взаимодействия является необходимым условием обеспечения стабильной работы Сервиса.

Дополнительным преимуществом механизма обнаружения сервисов является снижение связанности компонентов. Внутренние сервисы взаимодействуют не с конкретными сетевыми адресами, а с логическими именами сервисов. Это упрощает сопровождение архитектуры и позволяет изменять внутреннюю инфраструктуру без влияния на прикладную логику компонентов.

Для контроля работы механизма обнаружения сервисов в Сервисе отслеживаются метрики, изложенные в разделе <<section-monitoring-srvdiscovery, 2.11.9>>.

Использование механизма обнаружения сервисов позволяет обеспечить: 

* динамическое масштабирование, 
* автоматическую балансировку нагрузки, 
* устойчивое взаимодействие между компонентами Сервиса,
* исключение ручного управления сетевой конфигурацией.

===== 2.11.5.2. Принцип работы

Механизм обнаружения сервисов обеспечивает автоматическое определение доступных экземпляров компонентов Сервиса и используется для организации внутреннего взаимодействия между сервисами без применения статически заданных сетевых адресов.

После запуска экземпляра сервиса выполняется следующая последовательность действий:

1. Экземпляр сервиса регистрируется в реестре сервисов.
2. В реестр передаются:

* логическое имя сервиса,
* сетевой адрес,
* используемый порт,
* техническая информация о состоянии экземпляра.

3. Сервис начинает периодически отправлять сигналы доступности, подтверждаетя своё состояние.
4. После успешной регистрации экземпляр становится доступным для внутренних вызовов.
5. API Gateway и внутренние сервисы получают информацию о доступных экземплярах из реестра сервисов и используют её для маршрутизации запросов.
6. Запросы направляются только на работоспособные экземпляры.

Например, при поступлении запроса проверки доступности товара API Gateway определяет список доступных экземпляров Сервиса запросов и перенаправляет запрос на один из них. Аналогичным образом Сервис запросов получает информацию о доступных экземплярах Гео-сервиса и выполняет внутренние вызовы без необходимости хранения статических сетевых адресов.

Дополнительно механизм обнаружения сервисов используется для обработки отказов компонентов.

Если экземпляр сервиса:

* перестаёт отправлять сигналы доступности,
* завершает работу аварийно,
* становится недоступным по сети,

то механизм обнаружения сервисов автоматически исключает его из списка доступных экземпляров и маршрутизации.

После этого:

* новые запросы перестают направляться на недоступный экземпляр,
* нагрузка перераспределяется между оставшимися экземплярами,
* API Gateway и внутренние сервисы получают обновлённую информацию о составе сервисов.

Такой подход позволяет уменьшить вероятность каскадной деградации обработки внешних запросов и сократить время восстановления после отказов.

Для предотвращения нестабильной маршрутизации в Сервисе используются дополнительные механизмы контроля состояния экземпляров:

* проверка доступности сервиса,
* контроль времени ответа,
* исключение перегруженных экземпляров,
* ограничение количества неуспешных обращений,
* повторное подключение после восстановления работоспособности.

Механизм обнаружения сервисов используется как для REST-взаимодействия, так и для gRPC-вызовов.

==== 2.11.6. API Gateway

===== 2.11.6.1. Назначение API Gateway

API Gateway используется в Сервисе как единая точка входа для внешних запросов, точка интеграции с внешнии по отношению к Сервису информационными системами и обеспечивает централизованное управление взаимодействием клиентских приложений и интегрируемых систем с внутренними компонентами Сервиса.

Основной задачей API Gateway является изоляция внутренней структуры Сервиса от внешних потребителей и поставщиков. Клиентские приложения взаимодействуют только с единым публичным интерфейсом, не имея прямого доступа к внутренним сервисам и не завися от особенностей внутренней маршрутизации запросов.

Такой подход особенно важен для Сервиса проверки доступности товара, поскольку внутренняя архитектура включает несколько независимых контуров обработки, недоступных снаружи:

* контур чтения,
* контур записи и резервирования,
* контур обработки событий,
* компоненты модели чтения.

API Gateway скрывает данную внутреннюю структуру и обеспечивает единые правила взаимодействия для всех типов клиентов.

Дополнительно API Gateway используется как центральная точка применения инфраструктурных политик. Через API Gateway реализуются:

* маршрутизация запросов,
* аутентификация и авторизация,
* ограничение частоты запросов,
* централизованное журналирование,
* управление версиями API,
* трассировка распределённых запросов.

Централизация данных механизмов позволяет избежать дублирования одинаковой инфраструктурной логики во внутренних сервисах и упрощает сопровождение архитектуры.

API Gateway также выполняет роль точки применения политик безопасности (PEP) во внешнем контуре взаимодействия Сервиса.

В данной роли API Gateway обеспечивает централизованное применение правил доступа и инфраструктурных ограничений до передачи запроса во внутренние сервисы. Это позволяет исключить дублирование одинаковых механизмов контроля в каждом компоненте Сервиса и обеспечить единообразное применение политик безопасности.

На уровне API Gateway выполняются:

* проверка подлинности клиента,
* проверка прав выполнения операции,
* контроль допустимых маршрутов доступа,
* ограничение частоты запросов,
* проверка технических ограничений запросов,
* фильтрация некорректных или потенциально опасных обращений.

Использование API Gateway как точки применения политик позволяет отделить бизнес-логику внутренних сервисов от инфраструктурных механизмов безопасности и централизованно управлять правилами доступа ко всем внешним API Сервиса.

Дополнительно такой подход упрощает изменение политик безопасности, поскольку обновление правил выполняется на уровне API Gateway без необходимости изменения внутренней логики отдельных сервисов.

Для Сервиса особое значение имеет возможность централизованного ограничения нагрузки. Наибольшая интенсивность запросов возникает при массовой проверке доступности популярных товаров. В таких сценариях API Gateway используется как первый уровень защиты внутренних компонентов от перегрузки.

Дополнительно API Gateway обеспечивает единый механизм публикации REST API и позволяет независимо развивать внутреннюю архитектуру Сервиса без изменения внешних контрактов взаимодействия.

Поскольку API Gateway является критически важным компонентом внешнего взаимодействия, его работоспособность и производительность контролируются отдельными средствами мониторинга и трассировки.

На уровне архитектуры Сервиса различают:

* API Gateway *входного слоя* -- для подключеня пользователей,
* API Gateway *внешних интеграций* -- для подключения внутренних ИС компании eDA, интегрируемых с Cервисом,
* API Gateway *служб доставки* -- для подключения к Cервису внешних ИС независимых служб доставки: Яндекс.Еда, Самокат, Купер и др.

===== 2.11.6.2. Правила маршрутизации

API Gateway выполняет маршрутизацию внешних запросов к внутренним компонентам Сервиса на основании набора правил, определяющих целевой контур обработки и соответствующий сервис.

Основной задачей маршрутизации является разделение различных типов операций между специализированными компонентами архитектуры. Такой подход позволяет независимо масштабировать обработку чтения, резервирования и фоновых операций, а также уменьшает влияние отдельных категорий нагрузки друг на друга.

В рамках Сервиса используются следующие основные правила маршрутизации:

[cols="1,1,1,1", options="header"]
|===
| Внешний путь | Тип запроса | Назначение | Целевой компонент

| `/api/v1/availability` | Проверка доступности товара | Получение информации об остатках и доступности товара | Сервис запросов
| `/api/v1/stores/nearby` | Поиск ближайших магазинов | Географический поиск магазинов | Сервис запросов / Гео-сервис
| `/api/v1/reservations` | Создание резерва | Обработка резервирования товара | Сервис резервирования
| `/api/v1/reservations/ +
{reservation_id}` | Получение статуса резерва | Проверка состояния резервирования | Сервис резервирования
| `/api/v1/reservations/ +
{reservation_id}` | Отмена резерва | Освобождение зарезервированного остатка | Сервис резервирования
| `/api/v1/health` | Технические проверки | Контроль доступности компонентов | Соответствующий внутренний сервис
|===

Маршрутизация выполняется с учётом версии API и типа операции. Использование версионирования позволяет одновременно поддерживать несколько версий внешних контрактов взаимодействия без изменения внутренней структуры Сервиса.

Дополнительно правила маршрутизации учитывают текущее состояние внутренних компонентов. При временной недоступности отдельных экземпляров API Gateway автоматически исключает их из обработки запросов и перенаправляет обращения на доступные экземпляры сервиса.

Для операций проверки доступности товара используется маршрутизация в контур чтения, оптимизированный под обработку большого количества пользовательских запросов. Операции резервирования направляются в контур записи и резервирования, где применяются дополнительные механизмы контроля согласованности данных.

Такое разделение маршрутов обработки позволяет:

* уменьшить влияние операций резервирования на производительность чтения,
* независимо масштабировать различные контуры обработки,
* локализовать деградацию отдельных компонентов,
* повысить устойчивость Сервиса при высокой нагрузке.

Отдельное внимание уделяется маршрутизации WebSocket-соединений. После установления соединения API Gateway поддерживает привязку соединения к соответствующему экземпляру компонента обработки потоковых уведомлений. Это необходимо для корректной доставки обновлений состояния резервирования и других событий в режиме реального времени.

Для предотвращения перегрузки внутренних компонентов API Gateway может применять дополнительные правила маршрутизации:

* ограничение количества одновременных запросов,
* ограничение запросов по отдельным SKU,
* приоритизацию критически важных операций,
* временное ограничение части функциональности при деградации отдельных компонентов.

===== 2.11.6.3. Проверка доступа

Проверка доступа в Сервисе выполняется централизованно на уровне API Gateway и применяется ко всем внешним запросам до их передачи во внутренние компоненты.

Основной задачей механизма проверки доступа является разграничение допустимых операций для различных категорий клиентов и исключение несанкционированного использования API Сервиса.

API Gateway выполняет:

* проверку токенов доступа,
* разграничение прав клиентов,
* передачу контекста пользователя во внутренние сервисы,
* фильтрацию недопустимых запросов,
* защиту от несанкционированного доступа.

В рамках Сервиса используются различные категории потребителей, включая:

* клиентские приложения,
* внутренние рабочие места сотрудников,
* внешние интеграционные системы,
* внутренние технические компоненты.

Для каждой категории клиентов определяются:

* доступные маршруты API,
* допустимые операции,
* ограничения на интенсивность обращений,
* доступ к отдельным функциям Сервиса.

После успешной проверки доступа API Gateway передаёт во внутренние сервисы контекст запроса, содержащий:

* идентификатор клиента,
* идентификатор пользователя,
* тип клиента,
* correlation_id,
* информацию о разрешённых операциях.

Внутренние сервисы используют уже проверенный контекст запроса и не выполняют повторную аутентификацию внешнего клиента.

Для внутреннего межсервисного взаимодействия применяется отдельный механизм сервисной аутентификации, позволяющий разделить внешние пользовательские обращения и внутренние технические вызовы между компонентами архитектуры.

Дополнительно механизм проверки доступа используется для ограничения обращений:

* к недопустимым маршрутам,
* к неподдерживаемым операциям,
* к техническим API,
* к операциям, запрещённым для конкретного типа клиента.

События отказа в доступе фиксируются средствами централизованного журналирования и используются для мониторинга попыток некорректного или несанкционированного обращения к API Сервиса.

===== 2.11.6.4. Ограничение частоты запросов

Для защиты Сервиса от перегрузки и неконтролируемого роста нагрузки на внешнем контуре взаимодействия используется механизм ограничения частоты запросов.

Необходимость применения данного механизма обусловлена особенностями предметной области. Проверка доступности товара относится к высокочастотным операциям, количество которых может резко возрастать в периоды повышенной пользовательской активности, проведения маркетинговых кампаний или появления популярных товаров с ограниченным остатком.

Дополнительным фактором риска является возможность многократного повторного обращения клиентских приложений к одним и тем же данным. Например, при обновлении пользовательского интерфейса или автоматическом повторе запросов количество обращений по одному SKU может многократно увеличиваться за короткий промежуток времени.

Без ограничения частоты запросов подобные сценарии могут приводить к таким последствиям:

* перегрузке Сервиса запросов,
* увеличению нагрузки на Распределенный кэш,
* росту количества внутренних вызовов,
* деградации времени ответа,
* вытеснению критически важных операций менее приоритетными запросами.

Ограничение частоты запросов применяется на уровне API Gateway до передачи запроса во внутренние компоненты Сервиса.

При превышении лимитов API Gateway возвращает ошибку ограничения запросов:

* 429   Too Many Requests,
* 509   Bandwidth Limit Exceeded.

В Сервисе используются несколько уровней ограничения нагрузки:

* по клиентскому приложению,
* по пользователю,
* по IP-адресу,
* по внешней интеграционной системе,
* по отдельным маршрутам API.

Дополнительно могут применяться отдельные ограничения для операций, создающих повышенную нагрузку на внутренние компоненты. В первую очередь это относится к сценариям массовой проверки доступности популярных товаров.

Для подобных операций могут использоваться:

* уменьшенные лимиты частоты запросов,
* ограничение количества параллельных обращений,
* временное ограничение повторных запросов,
* ограничение глубины поиска магазинов,
* ограничение максимального размера ответа.

Разделение лимитов по типам операций позволяет более гибко управлять нагрузкой на различные контуры обработки. Например, операции проверки доступности товара могут иметь более высокую интенсивность обращений, чем операции резервирования, для которых важнее сохранение согласованности данных и устойчивость обработки.

При превышении установленных ограничений API Gateway отклоняет запрос до его передачи во внутренние сервисы. Это позволяет предотвратить распространение перегрузки на компоненты контура чтения и контур записи и резервирования.

Дополнительно механизм ограничения частоты запросов используется как элемент защиты от аномальной активности и ошибочного поведения внешних интеграций. Резкий рост количества однотипных запросов может свидетельствовать о таких проблемах, как:

* ошибки клиентского приложения,
* некорректная интеграция,
* попытки массового автоматизированного обращения к API,
* лавинообразный повтор запросов после временной деградации.

Использование централизованного ограничения нагрузки позволяет локализовать подобные ситуации на уровне API Gateway без влияния на внутренние компоненты Сервиса.

Для контроля эффективности механизма ограничения запросов отслеживаются метрики, изложенные в разделе <<section-monitoring-ratelim, 2.11.9>>.

==== 2.11.7. Асинхронное взаимодействие через Apache Kafka

===== 2.11.7.1. Назначение брокера сообщений

В Сервисе проверки доступности товара брокер сообщений в составе Event Bus используется как центральный механизм передачи событий между компонентами распределённой архитектуры. В качестве брокера сообщений применяется *Apache Kafka*, обеспечивающий обработку большого количества событий при высокой интенсивности обновления данных.

Основной задачей брокера сообщений является передача изменений состояния между независимыми компонентами без необходимости их прямого синхронного взаимодействия. Такой подход позволяет компонентам обрабатывать события независимо друг от друга и уменьшает связанность архитектуры.

Использование брокера сообщений особенно важно для Сервиса, поскольку изменения состояния данных происходят непрерывно и одновременно затрагивают несколько контуров обработки. Одно изменение остатка или операция резервирования могут требовать:

* обновления модели чтения,
* обновления Распределенного кэша,
* фиксации события в Хранилище событий,
* запуска проверки согласованности,
* формирования технических метрик,
* передачи данных на уровень мониторинга.

При прямом синхронном взаимодействии между компонентами каждая операция изменения состояния была бы жёстко связана с доступностью и скоростью обработки всех зависимых сервисов. В условиях высокой нагрузки это привело бы к существенному росту времени обработки операций и увеличению вероятности каскадных отказов.

Использование брокера сообщений позволяет разделить момент фиксации изменения и дальнейшую обработку события различными компонентами. После публикации события дальнейшая обработка выполняется независимо отдельными потребителями.

Дополнительной причиной использования Apache Kafka является необходимость устойчивой обработки интенсивного потока событий изменения остатков. Для Сервиса проверки доступности товара характерно постоянное поступление изменений из различных источников, включая:

* кассовые операции через адаптер POS,
* складские системы через адаптер WMS,
* операции резервирования и отмены резервов,
* результаты сверки данных.

Apache Kafka позволяет обрабатывать подобный поток событий с сохранением порядка обработки внутри разделов и возможностью горизонтального масштабирования потребителей.

Брокер сообщений также используется как механизм временной буферизации нагрузки. При кратковременном росте интенсивности событий компоненты обработки могут продолжать получать сообщения из очереди с собственной скоростью без остановки публикации новых событий.

Дополнительно использование Apache Kafka позволяет реализовать повторную обработку событий. Это особенно важно для операций резервирования и обновления модели чтения, поскольку временные ошибки отдельных компонентов не должны приводить к потере изменений состояния.

Для различных категорий событий в Сервисе используются отдельные топики брокера сообщений. Такой подход позволяет независимо масштабировать обработку различных потоков данных и локализовать влияние перегрузки отдельных компонентов.

Поскольку брокер сообщений является критически важным элементом архитектуры, в Сервисе контролируются отдельные показатели его работы, изложенные в разделе <<section-monitoring-kafka, 2.11.9>>.

===== 2.11.7.2. Сценарий публикации и подписки

Сценарий публикации и подписки используется в Сервисе для распространения изменений состояния между несколькими независимыми компонентами. В рамках данного подхода один компонент публикует событие в Event Bus, после чего событие может быть обработано несколькими потребителями одновременно и независимо друг от друга.

Данный механизм применяется в ситуациях, когда одно изменение должно инициировать несколько параллельных процессов обработки. Наиболее характерным примером является изменение остатка товара.

1. После поступления нового значения остатка формируется событие изменения состояния товара, содержащее информацию:

* о SKU,
* магазине,
* новом значении остатка,
* времени изменения,
* источнике обновления.

2. Событие публикуется в топик `inventory.events`.
3. После публикации события его начинают независимо обрабатывать различные компоненты Сервиса:
.. Загрузчик моделей чтения использует событие для обновления Хранилища модели чтения и последующего обновления данных в Распределенном кэше.
.. Сервис сверки использует тот же поток событий для контроля корректности изменения состояния и выявления возможных расхождений между различными источниками данных.
.. Компоненты мониторинга используют события для расчёта технических метрик, включая:

* скорость изменения остатков,
* задержку распространения событий,
* интенсивность обновлений по отдельным магазинам или SKU.

Аналогичный сценарий используется и для операций резервирования. После подтверждения резервирования событие из топика `reservation.events` может одновременно использоваться для:

* обновления модели чтения,
* расчёта зарезервированного остатка,
* фиксации операций аудита,
* формирования технических уведомлений,
* последующего анализа конфликтов резервирования.

Ключевой особенностью модели публикации и подписки является независимость потребителей друг от друга. Ошибка или временная недоступность одного из компонентов не блокирует обработку события остальными сервисами.

Например, временная недоступность Сервиса сверки не влияет на обновление модели чтения или распространение изменений в Распределенный кэш. Такой подход повышает устойчивость распределённой архитектуры и уменьшает вероятность каскадной деградации.

Дополнительно модель публикации и подписки упрощает развитие Сервиса. При необходимости новый компонент может быть подключён к обработке событий без изменения логики сервисов-публикаторов. Для этого достаточно подписать новый компонент на соответствующий топик Event Bus.

Для предотвращения нарушения последовательности обработки событий используется секционирование по комбинации SKU + магазин. Это особенно важно для потоков изменения остатков и резервирования, поскольку изменение порядка обработки может приводить к временному расхождению состояния модели чтения.

Поскольку различные категории событий могут иметь существенно разную интенсивность обработки, в Сервисе используются независимые группы потребителей для:

* обновления модели чтения,
* сверки данных,
* мониторинга,
* аналитической обработки,
* технических задач обработки ошибок.

Такое разделение позволяет масштабировать отдельные направления обработки независимо друг от друга.

Для контроля корректности работы сценария публикации и подписки дополнительно отслеживаются метрики, изложенные в разделе <<section-monitoring-kafka, 2.11.9>>.

===== 2.11.7.3. Сценарий обработки задачи одним потребителем

Помимо модели публикации и подписки в Сервисе используется сценарий обработки задач одним потребителем. Данный подход применяется для операций, которые должны быть выполнены только одним экземпляром обработчика и не допускают параллельного исполнения одной и той же задачи несколькими компонентами одновременно.

Основной целью данного механизма является предотвращение дублирующей обработки и снижение риска возникновения конфликтующих изменений состояния.

Сценарий обработки одним потребителем используется для:

* повторной обработки ошибочных событий,
* фоновой сверки данных,
* пересчёта модели чтения,
* обработки задач восстановления состояния,
* выполнения технических служебных операций.

В отличие от модели публикации и подписки, где событие может независимо обрабатываться несколькими компонентами, в данном сценарии каждая задача должна быть назначена только одному экземпляру обработчика из группы потребителей.

Наиболее характерный пример -- *задача повторной синхронизации состояния модели чтения после обнаружения расхождения данных*. После фиксации проблемы Сервис сверки:

1. Формирует задачу на повторную обработку.
2. Публикует задачу в отдельный поток обработки в соответствующий топик Event Bus.

Далее задача последовательно проходит следующие этапы:

[start=3]
3. Получение задачи одним экземпляром обработчика.
4. Выполнение операции обработки.
5. Подтверждение успешного завершения задачи.
6. Фиксация результата обработки.

Если обработчик завершает работу аварийно до подтверждения выполнения задачи, сообщение может быть повторно передано другому экземпляру обработчика. Такой подход позволяет избежать потери задач при отказе отдельных компонентов.

Отдельное значение данный сценарий имеет для операций, связанных с восстановлением согласованности данных. При обнаружении конфликтующего состояния важно исключить ситуацию, при которой несколько экземпляров сервиса одновременно пытаются изменить одно и то же состояние модели чтения.

Использование модели обработки одним потребителем позволяет:

* избежать дублирования обработки,
* уменьшить вероятность конфликтующих изменений,
* контролировать последовательность выполнения операций,
* локализовать обработку фоновых задач,
* управлять скоростью выполнения ресурсоёмких операций.

Для различных категорий задач в Сервисе могут использоваться отдельные очереди обработки. Это позволяет разделить:

* задачи повторной обработки,
* задачи сверки,
* операции восстановления,
* технические служебные операции.

Разделение потоков обработки уменьшает влияние длительных или ресурсоёмких задач на остальные процессы обработки.

Дополнительно для задач обработки используются:

* ограничение количества повторных попыток,
* контроль времени выполнения,
* отдельные очереди ошибочных задач,
* фиксация состояния обработки,
* контроль зависших операций.

Это особенно важно для операций восстановления модели чтения и обработки конфликтов резервирования, поскольку неконтролируемое повторное выполнение задач может привести к дополнительной деградации состояния данных.

Для контроля работы сценария обработки одним потребителем отслеживаются показатели, изложенные в разделе <<section-monitoring-single, 2.11.9>>.

Использование сценария обработки задач одним потребителем позволяет обеспечить контролируемое выполнение фоновых операций, уменьшить вероятность конфликтующей обработки и повысить устойчивость Сервиса при выполнении технических операций восстановления и синхронизации состояния данных.

===== 2.11.7.4. Надёжность доставки сообщений

Для Сервиса проверки доступности товара потеря событий недопустима, поскольку события используются для обновления модели чтения, изменения состояния резервов и синхронизации данных между компонентами. Ошибки доставки или пропуск событий могут приводить к расхождению фактических остатков, некорректному отображению доступности товара и конфликтам резервирования.

Для обеспечения надёжности обработки событий в Event Bus используются несколько уровней защиты.

1. На уровне публикации сообщений используется *подтверждение успешной записи* события в брокер сообщений. +
Событие считается опубликованным только после подтверждения его сохранения. Это уменьшает вероятность потери сообщений при временных сетевых ошибках или отказе отдельных узлов брокера сообщений.

2. Для критически важных операций, связанных с резервированием товара и изменением остатков, используется *повторная отправка сообщений* при временной недоступности брокера сообщений. +
Повторные попытки публикации выполняются ограниченное количество раз с контролем времени ожидания между попытками.

3. Дополнительно в Сервисе применяется *идемпотентная обработка событий*. +
Один и тот же экземпляр события может быть получен повторно при восстановлении после ошибки или повторной доставке сообщения. Для предотвращения повторного применения изменений каждое событие содержит *уникальный идентификатор*, позволяющий обнаруживать дублирующую обработку.

4. Отдельное внимание уделяется *сохранению порядка обработки изменений состояния*. +
Для потоков изменения остатков и резервирования нарушение последовательности событий может привести к временному неконсистентному состоянию модели чтения. Для предотвращения подобных ситуаций события секционируются по комбинации SKU + магазин, что обеспечивает последовательную обработку изменений одного товара в рамках конкретного магазина.

5. В процессе обработки событий используется *контроль смещений чтения*. +
Подтверждение обработки сообщения выполняется только после успешного завершения операции обработки. Если экземпляр обработчика завершает работу аварийно до подтверждения смещения, событие может быть повторно передано другому экземпляру потребителя.

6. Для обработки нештатных ситуаций используются *отдельные очереди проблемных сообщений*. +
В такие очереди помещаются события, которые не удалось обработать после установленного количества повторных попыток. Это позволяет:

* изолировать ошибочные события от основного потока обработки,
* предотвратить остановку обработки очереди,
* выполнять последующий анализ проблемных сообщений,
* запускать повторную обработку после устранения причины ошибки.

+
Для задач повторной обработки используются отдельные обработчики, не влияющие на основной поток обновления модели чтения.

7. Дополнительно *контролируется время распространения событий* между публикацией изменения и обновлением модели чтения. +
Рост задержки обработки может свидетельствовать о:

* перегрузке потребителей,
* накоплении необработанных сообщений,
* деградации брокера сообщений,
* проблемах отдельных обработчиков событий.
