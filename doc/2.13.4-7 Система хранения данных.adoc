==== 2.13.4. Геораспределенность и резервирование

===== 2.13.4.1. Подход к георезервированию

Сервис проектируется для размещения в одном центре обработки данных. Геораспределенное размещение и схема «основной ЦОД — резервный ЦОД» в составе данного архитектурного решения не применяются. Требования по доступности, отказоустойчивости и восстановлению обеспечиваются за счет кластеризации компонентов внутри одного ЦОДа, репликации данных, резервных узлов ожидания, резервного копирования и возможности повторного восстановления производных хранилищ из событий.

===== 2.13.4.2. Подход к резервированию в рамках одного ЦОД

Требования по доступности, отказоустойчивости и восстановлению обеспечиваются за счет:

* кластеризации компонентов,
* репликации данных,
* резервных узлов ожидания,
* резервного копирования,
* повторного восстановления производных хранилищ из событий.

Для критичных хранилищ используется схема «активный узел — резервный узел ожидания».

Данный подход применяется для:

* Хранилища событий,
* БД резервирования,
* БД конфликтов,
* Метаданных, справочников,
* БД прогрева кеша,
* БД реестра схем,
* Хранилища временных рядов.

В штатном режиме обработка выполняется активным узлом. Резервный узел поддерживает актуальную копию данных и используется для автоматического переключения при отказе основного узла.

Для Хранилища модели чтения, Гео-индекса, координат, Распределенного кеша и Кэша лояльности используются многоузловые кластеры с внутренней репликацией данных. Отказ отдельного узла не приводит к остановке обработки запросов.

.Подходы к резервированию компонентов
[cols="1,3",options="header"]
|===
| Схема резервирования | Компонент 

| Активный узел + +
 резервный узел ожидания | Хранилище событий, БД резервирования, БД конфликтов, Метаданные, справочники, БД прогрева кеша, БД реестра схем, Хранилище временных рядов
| Многоузловой кластер | Хранилище модели чтения, Гео-индекс, координаты, Распределенный кеш, Кэш лояльности
| Кластер брокеров с репликацией партиций | Event Bus, Очередь ошибок
|===

Для восстановления данных используются три уровня резервирования:

* *Горячее резервирование* : резервные узлы постоянно поддерживаются в актуальном состоянии и могут быть использованы сразу после отказа активного узла.
* *Теплое резервирование* : допускается восстановление актуального состояния после переключения за счет повторной синхронизации из Хранилища событий и Event Bus.
* *Холодное резервирование* : резервные узлы находятся в выключенном состоянии или содержат только базовую конфигурацию и вводятся в эксплуатацию после отказа основных компонентов с последующим восстановлением данных из резервных копий и журналов событий.

.Используемые уровни резервирования
[cols="1,3",options="header"]
|===
| Уровень резервирования | Компоненты

| Горячее | Хранилище событий, БД резервирования, БД конфликтов, Метаданные, справочники
| Теплое | Хранилище модели чтения, Гео-индекс, координаты, Распределенный кеш, Кэш лояльности
| Холодное | Архивное хранилище, резервные копии, архивы событий и журналов предварительной записи, снимки состояния, исторические данные Архивного хранилища
|===

Отдельная аварийная площадка не используется. В случае полной недоступности центра обработки данных восстановление выполняется из резервных копий и архивов данных в соответствии с регламентом аварийного восстановления.

Такой подход позволяет выполнить требования проекта по доступности и сохранности данных без усложнения архитектуры многорегиональным развертыванием и межплощадочной синхронизацией данных.

Правда, в этом случае требование *RPO = 0* для сценария полной потери ЦОД фактически невозможно обеспечить без второй площадки. Поэтому здесь нужно принять одно из двух архитектурных решений:

* *Вариант А* : один ЦОД, требование RPO = 0 относится к отказу отдельных узлов и кластеров внутри ЦОД, не рассматривается катастрофическая потеря всего ЦОД,
* *Вариант Б* : основной ЦОД + резервный ЦОД, межплощадочная синхронная репликация для критичных хранилищ. Но тогда придется переделывать значительную часть архитектуры для добавления полноценной геораспределенной архитектуры.

С учетом всей ранее разработанной архитектуры выбран *Вариант А*. Он лучше согласуется с уже принятыми решениями по архитектуре. 
Реализация геораспределенного решения может быть закреплена в беклоге требоываний.

===== 2.13.4.3. Резервное копирование и восстановление

Для обеспечения сохранности данных в Сервисе используется *централизованная стратегия резервного копирования*, охватывающая все критичные хранилища данных. Резервные копии размещаются в *Архивном хранилище* и используются для аварийного восстановления, восстановления после логических ошибок и восстановления данных на произвольный момент времени.

Резервному копированию подлежат все хранилища и БД в составе микросервисов, за исключением приведенных ниже. 

Для PostgreSQL-хранилищ используется *комбинированная схема* резервного копирования:

* периодические полные резервные копии,
* непрерывное архивирование журналов предварительной записи.

Такой подход позволяет восстановить данные *на любой момент времени между двумя полными резервными копиями*.

Для MongoDB-хранилищ используются:

* полные резервные копии,
* снимки состояния коллекций.

Для Распределенного кеша и Кэша лояльности *регулярное резервное копирование не является обязательным* требованием, поскольку данные могут быть повторно восстановлены из Хранилища модели чтения и исходных источников данных.

Для различных типов данных используются разные сроки хранения резервных копий.

.Сроки хранения резервных копий
[cols="2,1",options="header"]
|===
| Категория данных | Срок хранения

| Ежесуточные резервные копии | 30 суток
| Еженедельные резервные копии | 90 суток
| Ежемесячные резервные копии | 1 год
| Архивы журналов предварительной записи | 30 суток
| Снимки состояния производных хранилищ | 90 суток
|===

Для критичных хранилищ поддерживается восстановление на произвольный момент времени. Это позволяет восстановить данные непосредственно перед ошибочной операцией, программным сбоем или повреждением данных.

Восстановление производных хранилищ выполняется по упрощенной схеме:

* восстановление последней резервной копии,
* синхронизация актуального состояния из Хранилища событий и Event Bus,
* повторное наполнение Распределенного кеша.

Для подтверждения работоспособности процедуры восстановления выполняются регулярные проверки резервных копий.

Проверка включает:

* контроль успешности создания резервных копий,
* контроль целостности архивов,
* тестовое восстановление данных,
* контроль времени восстановления,
* проверку полноты восстановленного состояния.

===== 2.13.4.4. Геораспределенная обработка данных

В рамках архитектуры Сервиса геораспределенная обработка данных не используется. Все основные компоненты размещаются в одном центре обработки данных и работают в рамках единого контура обработки.
Отказ от геораспределенного размещения обусловлен характером обрабатываемых данных и требованиями к консистентности операций резервирования.

Наиболее критичными операциями являются:

* создание резервов,
* отмена резервов,
* изменение доступного остатка,
* обработка конфликтов сверки.

Для указанных операций требуется единое согласованное состояние данных. Размещение компонентов в нескольких центрах обработки данных привело бы к необходимости организации межплощадочной синхронизации и разрешения конфликтов конкурентного изменения данных.

При геораспределенном размещении неизбежно возникает компромисс между:

* доступностью,
* устойчивостью к сетевым разделениям,
* строгой консистентностью данных.

Для Сервиса *приоритетом является сохранение корректности операций резервирования и отсутствие расхождений в остатках товаров*. По этой причине архитектурное *решение смещено в сторону консистентности данных*, а не в сторону независимой многорегиональной обработки.

Дополнительным фактором является задержка передачи данных между площадками. Для операций резервирования даже небольшое увеличение задержек приводит к:

* росту времени обработки транзакций,
* увеличению времени удержания блокировок,
* увеличению вероятности конфликтов,
* снижению пропускной способности Сервиса.

Размещение компонентов в одном центре обработки данных позволяет обеспечить:

* минимальные сетевые задержки между сервисами,
* минимальные задержки репликации,
* стабильную работу распределенных блокировок,
* быстрое обновление модели чтения,
* прогнозируемое время обработки пользовательских запросов.

Отказ от многорегиональной архитектуры на данном этапе снижает эксплуатационные затраты.

При использовании нескольких площадок дополнительно потребовались бы:

* межплощадочные каналы связи,
* синхронная или асинхронная репликация между регионами,
* резервирование вычислительных ресурсов на каждой площадке,
* дополнительные механизмы разрешения конфликтов данных,
* усложненные процедуры сопровождения и аварийного восстановления.

==== 2.13.5. Архитектурные паттерны хранения данных

Подсистема хранения данных Сервиса построена с использованием ряда архитектурных паттернов, позволяющих обеспечить высокую производительность обработки запросов, надежное хранение событий, масштабируемость и согласованность данных при высоких нагрузках.

Выбранные паттерны учитывают особенности предметной области, в которой одновременно присутствуют:

* интенсивный поток изменений остатков товаров,
* операции резервирования с высокими требованиями к консистентности,
* значительное преобладание операций чтения над операциями записи,
* необходимость быстрого поиска данных о наличии товаров,
* требования к восстановлению состояния на основе ранее зарегистрированных событий.

Для разделения нагрузок чтения и записи используется *паттерн разделения контуров команд и запросов*. Его применение позволяет независимо масштабировать транзакционные операции и операции чтения данных.

Для обеспечения надежного хранения истории изменений и возможности повторного восстановления состояния применяется *событийный подход к хранению данных*, при котором изменения фиксируются в виде последовательности событий.

Передача изменений между транзакционными и производными хранилищами реализуется через *механизм захвата изменений данных*, обеспечивающий распространение подтвержденных изменений между компонентами Сервиса.

Для снижения нагрузки на основные хранилища и обеспечения минимального времени ответа пользовательских запросов применяется *многоуровневое кеширование данных*.

Дополнительно в архитектуре используется *распределение данных между узлами хранения*, что позволяет сохранять линейную масштабируемость подсистемы хранения при росте количества товаров, магазинов, резервов и обрабатываемых событий.

Подробное описание применяемых архитектурных паттернов приведено в следующих подразделах.

===== 2.13.5.1. Применение CQRS

В архитектуре Сервиса используется *паттерн разделения команд и запросов* (*CQRS*), при котором контур изменения данных отделен от контура чтения данных.

Контур записи включает:

* Сервис резервирования,
* БД резервирования,
* Хранилище событий,
* Захват изменений (CDC),
* Event Bus.

В данном контуре выполняются:

* создание и отмена резервов,
* изменение доступного остатка,
* обработка результатов сверки,
* регистрация событий изменения состояния.

Контур чтения включает:

* Загрузчик моделей чтения,
* Хранилище модели чтения,
* Гео-индекс, координаты,
* Распределенный кеш,
* Сервис запросов.

В данном контуре выполняются:

* поиск доступности товаров,
* поиск ближайших магазинов,
* получение сведений о товарах,
* обработка пользовательских запросов на чтение.

Передача изменений между контурами осуществляется через Event Bus. После фиксации изменений в контуре записи соответствующие события последовательно обрабатываются Загрузчиком моделей чтения, который обновляет Хранилище модели чтения и связанные производные структуры данных.

*Использование CQRS позволяет*:

* независимо масштабировать операции чтения и записи,
* изолировать транзакционные операции резервирования от пользовательской нагрузки,
* использовать различные технологии хранения для контуров чтения и записи,
* снизить нагрузку на БД резервирования,
* обеспечить высокую производительность Сервиса запросов.

В рассматриваемом Сервисе *применение CQRS обусловлено* существенной асимметрией нагрузки, при которой количество операций чтения многократно превышает количество операций изменения данных. Это позволяет оптимизировать каждое хранилище под свой профиль нагрузки и избежать конкуренции между пользовательскими запросами и транзакционными операциями резервирования.

===== 2.13.5.2. Применение Event Sourcing

В архитектуре Сервиса используется *паттерн событийного хранения данных* (*Event Sourcing*), при котором источником истины является не текущее состояние объекта, а последовательность событий, приведших к этому состоянию.

Для хранения событий используется *Хранилище событий*, в котором регистрируются:

* изменения остатков товаров,
* создание и отмена резервов,
* истечение срока действия резервов,
* результаты сверки данных,
* корректировки состояния.

Каждое событие является неизменяемой записью и после сохранения не модифицируется. Изменение состояния объекта всегда оформляется созданием нового события.

Хранилище событий используется для:

* сохранения полной истории изменений,
* восстановления состояния объектов,
* формирования производных моделей данных,
* анализа причин возникновения конфликтов,
* аудита выполненных операций.

После регистрации события оно передается через Event Bus и используется для обновления:

* Хранилища модели чтения,
* Гео-индекса, координат,
* Распределенного кеша,
* других производных представлений данных.

Текущее состояние объектов формируется на основании последовательности ранее зарегистрированных событий. При необходимости любое производное хранилище может быть перестроено путем повторной обработки событий из Хранилища событий.

Применение Event Sourcing позволяет:

* обеспечить полную трассируемость изменений,
* исключить потерю истории операций,
* выполнять восстановление состояния после сбоев,
* повторно воспроизводить обработку событий,
* использовать единый источник данных для контура чтения и контура записи.

В рассматриваемом Сервисе *применение Event Sourcing обусловлено* необходимостью надежного хранения истории изменений остатков и резервов, а также возможностью повторного построения Хранилища модели чтения и других производных хранилищ без обращения к внешним источникам данных.

===== 2.13.5.3. Применение CDC

В архитектуре Сервиса используется *паттерн захвата изменений данных* (*CDC*), реализованный компонентом Захват изменений (CDC).

Основной задачей компонента CDC является передача подтвержденных изменений из транзакционного контура в событийный контур без прямой связи между бизнес-транзакциями и потребителями событий.

Источником данных для *Захвата изменений (CDC)* является БД резервирования. После подтверждения транзакции изменения фиксируются в журнале предварительной записи БД резервирования. Захват изменений (CDC) считывает зарегистрированные изменения, преобразует их в события установленного формата и публикует их в Event Bus.

Через механизм CDC передаются:

* события создания, отмены и изменения состояния резервов,
* события изменения доступного остатка,
* другие изменения, влияющие на контур чтения.

Полученные события далее используются:

* Загрузчиком моделей чтения,
* Сервисом сверки,
* другими подписчиками Event Bus.

Применение CDC позволяет:

* *исключить прямую зависимость БД резервирования от Event Bus*,
* передавать только подтвержденные изменения данных,
* сократить количество интеграционной логики в Сервисе резервирования,
* обеспечить асинхронное распространение изменений между компонентами,
* уменьшить влияние отказов потребителей событий на выполнение транзакционных операций.

Использование CDC обеспечивает *надежную синхронизацию транзакционного контура и событийного контура* без необходимости выполнения публикации событий в рамках транзакций резервирования.

===== 2.13.5.4. Применение кеширования

Для обеспечения требуемой производительности операций чтения в архитектуре Сервиса используется *многоуровневое кеширование данных* на основе Распределенного кеша и Кэша лояльности.

Основными потребителями кешированных данных являются:

* Сервис запросов,
* Клиент лояльности.

Сервис запросов использует схему чтения через кеш. При поступлении запроса:

* сначала выполняется поиск данных в Распределенном кеше,
* при отсутствии данных выполняется обращение к Хранилищу модели чтения,
* полученные данные помещаются в кеш для последующих запросов.

Такой подход позволяет *существенно снизить нагрузку на Хранилище модели чтения* и *обеспечить минимальное время ответа для массовых пользовательских запросов*.

Актуализация данных в кеше выполняется путем *инвалидации ключей после изменения данных в контуре записи*. После обработки соответствующего события Загрузчик моделей чтения обновляет Хранилище модели чтения и удаляет связанные записи из Распределенного кеша. При следующем обращении пользователя данные автоматически загружаются из актуальной модели чтения и повторно помещаются в кеш.

Для *снижения времени восстановления после перезапуска* используется БД прогрева кеша. Данный компонент хранит перечень наиболее востребованных объектов и используется для *предварительного наполнения Распределенного кеша* при запуске Сервиса.

Дополнительно применяются механизмы *защиты от лавинообразной нагрузки при массовом истечении срока действия ключей*:

* использование различного времени жизни для однотипных записей,
* предварительное обновление наиболее востребованных данных,
* ограничение количества одновременно выполняемых операций повторного заполнения кеша.

Использование кеширования позволяет:

* сократить количество обращений к Хранилищу модели чтения,
* обеспечить стабильную работу при пиковых нагрузках,
* уменьшить задержки пользовательских запросов,
* снизить нагрузку на MongoDB Cluster,
* обеспечить требуемые показатели производительности Сервиса запросов.

===== 2.13.5.5. Обоснование допустимости шардирования

В архитектуре Сервиса применяется *горизонтальное масштабирование отдельных хранилищ* данных путем *распределения данных между несколькими узлами* хранения.

*Необходимость применения шардирования* обусловлена:

* большим количеством магазинов,
* большим количеством товаров,
* высоким объемом событий изменения остатков,
* постоянным ростом объема модели чтения,
* высокой интенсивностью пользовательских запросов.

Наиболее целесообразным применение шардирования является для:

* Хранилища модели чтения,
* Гео-индекса, координат,
* Event Bus.

Для Хранилища модели чтения и Гео-индекса, координат распределение данных между узлами выполняется по бизнес-ключам:

* регион,
* магазин,
* диапазон SKU.

Такой подход обеспечивает:

* равномерное распределение данных,
* локализацию запросов,
* независимое масштабирование узлов хранения,
* снижение нагрузки на отдельные серверы.

Для Event Bus аналогичную роль выполняет *партиционирование топиков*. Распределение сообщений между партициями выполняется по ключам:

* SKU,
* комбинация SKU и магазин,
* идентификатор резерва.

Это позволяет одновременно обеспечить:

* сохранение порядка обработки связанных событий,
* параллельную обработку потока сообщений,
* увеличение пропускной способности при росте нагрузки.

Для транзакционных хранилищ применение шардирования ограничено.

К таким компонентам относятся:

* БД резервирования,
* Хранилище событий,
* БД конфликтов,
* Метаданные, справочники,
* БД реестра схем.

Для указанных компонентов приоритетом является сохранение транзакционной целостности и консистентности данных. На текущем этапе развития Сервиса требуемая производительность обеспечивается за счет:

* вертикального масштабирования,
* репликации,
* секционирования крупных таблиц,
* использования реплик чтения.

*Отказ от шардирования транзакционных хранилищ* позволяет избежать:

* распределенных транзакций,
* межузловых блокировок,
* усложнения схем резервирования,
* усложнения процедур восстановления.

Выбранный подход обеспечивает *возможность дальнейшего роста объема данных и нагрузки без существенного усложнения архитектуры хранения* и *без снижения консистентности критичных бизнес-данных*.

==== 2.13.6. Ограничения и компромиссы архитектуры хранения

Выбранная архитектура хранения данных обеспечивает требуемые показатели производительности, масштабируемости и отказоустойчивости, однако *предполагает ряд осознанных технических компромиссов*.

Использование CQRS, Хранилища событий, Event Bus и Хранилища модели чтения *приводит к появлению временного рассогласования между контуром записи и контуром чтения*.

После выполнения операции резервирования данные сначала фиксируются в:

* БД резервирования,
* Хранилище событий,

и только затем через Event Bus распространяются в:

* Хранилище модели чтения,
* Распределенный кеш.

В течение короткого промежутка времени пользователь может получить данные предыдущей версии. *Данное ограничение является допустимым* и учитывается при проектировании Сервиса.

Использование Распределенного кеша также связано с рядом ограничений:

* данные кеша не являются источником истины,
* возможна потеря данных при аварийных сценариях,
* требуется управление временем жизни записей,
* требуется контроль актуальности кешированных данных.

Для обеспечения корректности работы дополнительно используются:

* инвалидация кеша после изменения данных,
* прогрев кеша,
* повторное наполнение из Хранилища модели чтения.

Использование Event Bus позволяет масштабировать обработку событий, однако *приводит к дополнительным требованиям по сопровождению*:

* управлению партициями,
* контролю отставания групп потребителей,
* мониторингу репликации,
* сопровождению схем сообщений.

По мере роста количества магазинов, товаров и операций резервирования *увеличивается объем* данных:

* Хранилища событий,
* Хранилища модели чтения,
* архивных данных,
* журналов мониторинга и аудита.

Это требует:

* регулярного архивирования данных,
* секционирования крупных таблиц,
* контроля свободного дискового пространства,
* планирования развития инфраструктуры хранения.

Отдельным ограничением является *сложность повторного воспроизведения событий*. При восстановлении производных хранилищ необходимо обеспечить:

* сохранение порядка обработки событий,
* защиту от повторной обработки,
* контроль версий состояния,
* согласованность результатов повторного построения.

По мере роста объема исторических данных *увеличивается продолжительность* процедур повторного воспроизведения и восстановления производных представлений.

В архитектуре принят сознательный отказ от многорегионального размещения компонентов.

Такое решение упрощает:

* обеспечение консистентности данных,
* управление транзакциями,
* репликацию данных,
* сопровождение инфраструктуры.

Одновременно это означает *отсутствие автоматического переключения на удаленную географическую площадку при полной потере центра обработки данных*.

Использование нескольких специализированных хранилищ данных также *увеличивает эксплуатационную сложность решения*.

Для обеспечения стабильной работы требуется сопровождение:

* Хранилища событий,
* БД резервирования,
* Хранилища модели чтения,
* Гео-индекса, координат,
* Распределенного кеша,
* Event Bus,
* Хранилища временных рядов,
* Архивного хранилища.

Для каждого компонента необходимо выполнять:

* мониторинг состояния,
* резервное копирование,
* контроль репликации,
* обновление программного обеспечения,
* планирование емкости хранения.

Указанные ограничения являются следствием выбранных архитектурных решений и *рассматриваются как приемлемый компромисс для достижения требуемых показателей* производительности, масштабируемости и надежности Сервиса.

==== 2.13.7. Выводы по подсистеме хранения данных

Подсистема хранения данных Сервиса построена с учетом требований к обработке большого количества пользовательских запросов, высокой интенсивности изменений остатков товаров и необходимости выполнения операций резервирования с сохранением консистентности данных.

Для различных типов данных используются специализированные хранилища:

* БД резервирования и Хранилище событий для транзакционных операций,
* Хранилище модели чтения для обслуживания запросов чтения,
* Гео-индекс, координаты для геопространственного поиска,
* Распределенный кеш для ускорения доступа к наиболее востребованным данным,
* Event Bus для передачи событий между компонентами.

Такое разделение позволяет оптимизировать каждое хранилище под свой профиль нагрузки и обеспечивает возможность независимого масштабирования отдельных компонентов.

Масштабируемость подсистемы хранения достигается за счет:

* горизонтального масштабирования Хранилища модели чтения, Гео-индекса, координат,
* использования Redis Cluster, партиционирования Event Bus,
* секционирования крупных таблиц,
* применения реплик чтения.

Отказоустойчивость обеспечивается за счет:

* многоузловых кластеров хранения данных,
* репликации данных,
* резервных узлов ожидания,
* автоматического переключения при отказах,
* резервного копирования и восстановления.

Применение CQRS позволяет разделить контур записи и контур чтения, исключить конкуренцию между транзакционными операциями и пользовательскими запросами, а также обеспечить независимое масштабирование каждого контура.

Применение Event Sourcing обеспечивает сохранение полной истории изменений состояния объектов, возможность повторного воспроизведения событий и восстановление производных хранилищ на основании данных Хранилища событий.

Для синхронизации транзакционного и событийного контуров используется Захват изменений (CDC), обеспечивающий передачу только подтвержденных изменений данных.

Минимизация риска потери данных достигается за счет:

* использования журналов предварительной записи,
* репликации критичных хранилищ,
* подтвержденной записи событий в Event Bus,
* резервного копирования,
* восстановления на произвольный момент времени,
* хранения полной истории событий,
* использования Очереди ошибок для проблемных сообщений.

Выбранная архитектура хранения обеспечивает выполнение требований Сервиса по производительности, масштабируемости, консистентности данных и надежности обработки операций резервирования при прогнозируемом росте объемов данных и нагрузки.
