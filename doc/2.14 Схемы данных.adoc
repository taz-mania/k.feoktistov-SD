=== 2.14. Схемы данных

Настоящий раздел определяет логическую структуру данных Сервиса и описывает схемы данных, используемые различными компонентами архитектуры.

Сервис обрабатывает несколько взаимосвязанных категорий данных:

* данные о товарах и магазинах,
* данные об остатках товаров,
* данные резервирования,
* события изменения состояния,
* данные модели чтения,
* данные кеширования,
* данные мониторинга и трассировки.

Особенностью архитектуры является то, что одна и та же бизнес-сущность может быть представлена несколькими различными моделями данных в зависимости от решаемой задачи. Например, данные резервирования одновременно используются:

* в транзакционной модели для выполнения операций резервирования,
* в событийной модели для передачи изменений через Event Bus,
* в модели чтения для обслуживания пользовательских запросов,
* в кеше для ускорения доступа к данным.

По этой причине описание только физических хранилищ недостаточно для понимания структуры данных и правил их использования различными компонентами Сервиса.

Дополнительной особенностью архитектуры является разделение контура записи и контура чтения. В результате данные, используемые для выполнения транзакционных операций, и данные, используемые для обработки запросов на получение информации о наличии товаров, имеют различную структуру, состав атрибутов и правила обновления.

Изменения состояния объектов распространяются между компонентами Сервиса в виде событий, что требует отдельного описания схем событийных сообщений, правил их формирования и версионирования.

В рамках данного раздела определяются:

* основные сущности предметной области и их взаимосвязи,
* схемы транзакционных данных,
* схемы событий Event Bus,
* структуры модели чтения и кешей,
* схемы хранения данных мониторинга и трассировки,
* правила версионирования схем данных,
* политики жизненного цикла данных,
* требования к консистентности и целостности данных.

Представленные схемы используются в качестве единой логической модели для проектирования хранилищ данных, интерфейсов взаимодействия между компонентами Сервиса и событийных контрактов.

==== 2.14.1. Общая модель данных Сервиса

Сервис предназначен для хранения и обработки данных, связанных с доступностью товаров в магазинах, резервированием товаров и распространением изменений состояния между компонентами архитектуры.

Общая модель данных формируется вокруг ограниченного набора ключевых бизнес-сущностей, которые используются в транзакционном контуре, контуре чтения и событийном контуре.

В качестве основных идентификаторов объектов используются:

* идентификатор товара (`sku_id`),
* идентификатор магазина (`store_id`),
* идентификатор резерва (`reservation_id`),
* идентификатор события (`event_id`),
* идентификатор конфликта (`conflict_id`).

Связь между большинством сущностей строится вокруг комбинации `SKU + магазин`.

Данная комбинация является основным бизнес-ключом Сервиса и используется при:

* хранении информации о доступности товара,
* выполнении резервирования,
* обработке событий изменения остатков,
* построении модели чтения,
* кешировании данных,
* обработке результатов сверки.

Общая модель данных включает три взаимосвязанных уровня:

* транзакционную модель данных, используемую для хранения текущего состояния бизнес-объектов и выполнения операций резервирования,
* событийную модель данных, используемую для фиксации изменений состояния и распространения этих изменений между компонентами Сервиса через Event Bus,
* модель чтения, используемую для предоставления пользователям агрегированных данных о наличии товаров и доступности магазинов без обращения к транзакционным данным.

Переход данных между указанными моделями осуществляется через последовательную обработку событий и обновление производных представлений данных.

Основные сущности и связи между ними представлены в последующих подразделах.

===== 2.14.1.1. Основные сущности Сервиса

Для обеспечения процессов определения доступности товаров, резервирования и распространения изменений состояния в Сервисе используется набор взаимосвязанных бизнес-сущностей.

.Основные сущности Сервиса
[cols="1,3",options="header"]
|===
| Сущность | Назначение

| *Товар* | Описывает товарную позицию, для которой выполняется хранение информации о доступности, резервировании и изменении остатков
| *Магазин* | Описывает торговый объект, в котором осуществляется учет остатков товаров и выполнение операций резервирования
| *Остаток* | Отражает текущее состояние доступности товара в конкретном магазине и содержит информацию о доступном и зарезервированном количестве
| *Резерв* | Описывает факт резервирования определенного количества товара в конкретном магазине на ограниченный период времени
| *Операция резервирования* | Описывает изменение состояния резерва и используется для фиксации жизненного цикла резервирования
| *Событие изменения остатка* | Описывает изменение состояния доступности товара и используется для распространения изменений через Event Bus
| *Конфликт сверки* | Описывает выявленное расхождение между различными источниками данных об остатках товаров
| *Snapshot модели чтения* | Представляет собой зафиксированное состояние модели чтения, используемое для ускорения восстановления производных данных
| *Запрос доступности* | Описывает параметры пользовательского запроса на получение информации о наличии товара в магазинах
|===

Между сущностями существуют следующие основные связи:

* один *Магазин* содержит множество *Остатков*,
* один *Товар* может присутствовать в *Остатках* множества *Магазинов*,
* один *Остаток* может быть связан с несколькими *Резервами*,
* один *Резерв* может содержать несколько *Операций резервирования*,
* один *Остаток* может порождать множество *Событий изменения остатка*,
* один *Конфликт сверки* относится к конкретному *Товару* и *Магазину*,
* один *Snapshot модели чтения* содержит состояние множества *Остатков* на определенный момент времени.

Ключевыми сущностями предметной области являются *Остаток* и *Резерв*.

Сущность *Остаток* определяет текущее количество товара, доступного для продажи и резервирования в магазине.

Сущность *Резерв* определяет количество товара, временно исключенного из доступного остатка до момента подтверждения, отмены или истечения срока действия резерва.

Все остальные сущности используются для хранения истории изменений, обеспечения согласованности данных, распространения изменений между компонентами Сервиса и обслуживания пользовательских запросов.

ER-диаграмма отражает основные сущности Сервиса и связи между ними. Центральной сущностью является Остаток, связывающий конкретный Товар с конкретным Магазином. На основании остатка выполняются операции резервирования, формируются события изменения остатка и фиксируются состояния модели чтения. Snapshot модели чтения содержит зафиксированное состояние множества остатков на определенный момент времени.

===== 2.14.1.2. Логическая модель данных

Логическая модель данных Сервиса строится вокруг связи между товаром, магазином и остатком. Комбинация `sku_id + store_id` используется как основной бизнес-ключ для хранения доступности товара, выполнения резервирования, обработки событий изменения остатка и формирования модели чтения.

ER-диаграмма верхнего уровня представлена ниже.

image::full_er_diagram.svg[Полная ER-диаграмма основных сущностей Сервиса]

В логической модели *Остаток* является центральной сущностью, связывающей *Товар* и *Магазин*. На основании этой сущности выполняются резервирование, формирование событий изменения остатка, сверка данных и построение модели чтения.

*Резерв* описывает временное исключение части доступного остатка из доступного для продажи количества. Изменения состояния резерва фиксируются через *Операции резервирования*.

*Конфликт сверки* фиксирует расхождение по конкретной паре `sku_id + store_id` и используется Сервисом сверки для дальнейшей обработки расхождений.

*Snapshot модели чтения* хранит зафиксированное состояние множества остатков на определенный момент времени и используется для восстановления или проверки согласованности модели чтения.

==== 2.14.2. Схемы транзакционных данных

Транзакционные данные используются для хранения текущего состояния бизнес-объектов и выполнения операций, изменяющих состояние Сервиса. Для таких данных предъявляются повышенные требования к консистентности, целостности и корректности обработки конкурентных изменений.

В рамках Сервиса транзакционные данные используются для:

* управления резервированием товаров,
* хранения состояний резервов,
* фиксации операций изменения резервов,
* регистрации результатов сверки данных и выявленных конфликтов.

Транзакционная модель является источником актуального состояния объектов на момент выполнения бизнес-операций. Изменения в данной модели выполняются в рамках транзакций и сопровождаются контролем версий данных, механизмами идемпотентности и обработкой конкурентного доступа.

Основными объектами транзакционной модели являются:

* резервы,
* операции резервирования,
* конфликты сверки.

В последующих подразделах рассматриваются схемы хранения указанных сущностей, их состав атрибутов, ключевые идентификаторы и взаимосвязи между объектами транзакционной модели данных.

===== 2.14.2.1. Схема хранения резервов

Сущность *Резерв* является основной транзакционной сущностью Сервиса и используется для фиксации факта временного резервирования товара в конкретном магазине. Каждый резерв создается для определенной комбинации `sku_id + store_id` и существует до момента подтверждения, отмены или истечения срока действия.

Для одного остатка может существовать несколько резервов одновременно. Суммарное количество зарезервированного товара учитывается при расчете доступного остатка.

| CONFIRMED | Резерв подтвержден внешней системой

Изменение состояния резерва не приводит к созданию новой записи. Жизненный цикл объекта отслеживается через изменение атрибута `status`, а история выполненных действий фиксируется отдельными записями операций резервирования.

Для защиты от конкурентного изменения данных используется:

* оптимистическая блокировка по полю `version`,
* контроль идемпотентности по `request_id`,
* распределенная блокировка по комбинации `sku_id + store_id` на время выполнения операции резервирования.

===== 2.14.2.2. Схема хранения операций резервирования

Сущность *Операция резервирования* предназначена для хранения истории изменений состояния резерва в течение его жизненного цикла.

В отличие от сущности *Резерв*, которая содержит текущее состояние объекта, сущность *Операция резервирования* фиксирует отдельные действия, выполнявшиеся над резервом. Наличие отдельного журнала операций позволяет выполнять аудит изменений, анализировать причины изменения состояния резерва и восстанавливать последовательность выполненных действий.

* `operation_id`,
* `reservation_id`,
* `request_id`,
* `created_at`.

===== 2.14.2.3. Схема хранения конфликтов сверки

Сущность *Конфликт сверки* используется для регистрации расхождений, выявленных Сервисом сверки при сравнении данных из различных источников.

Конфликт создается только при обнаружении несоответствия между ожидаемым и фактическим состоянием данных по конкретной комбинации `sku_id + store_id`. После устранения причины расхождения запись сохраняется для последующего анализа и аудита выполненных действий.

| IGNORED | Конфликт признан допустимым и не требует исправления
|===

Для обеспечения поиска и обработки конфликтов используются следующие логические ключи доступа:

* `conflict_id`,
* `sku_id + store_id`,
* `conflict_status`,
* `detected_at`.

Каждая запись конфликта относится к конкретной паре `sku_id + store_id` и отражает состояние расхождения на момент его выявления. После регистрации конфликт не удаляется, а используется для анализа качества данных, контроля работы механизмов синхронизации и формирования отчетности по результатам сверки.

==== 2.14.3. Схемы событийных данных

Событийная модель данных используется для передачи изменений состояния между компонентами Сервиса через Event Bus.

Каждое событие отражает факт выполнения определенного бизнес-действия или изменения состояния объекта. После публикации событие становится неизменяемым и используется для синхронизации данных между контуром записи, Хранилищем событий, Хранилищем модели чтения, Распределенным кешем и другими компонентами, обрабатывающими поток изменений.

События являются основой реализации событийного взаимодействия в архитектуре Сервиса и используются для:

* распространения изменений остатков товаров,
* передачи информации о создании, изменении и завершении резервов,
* передачи результатов сверки данных,
* обновления производных представлений данных,
* восстановления состояния компонентов путем повторной обработки событий.

Несмотря на различное бизнес-назначение, все события Event Bus используют единые принципы построения схем данных:

* наличие уникального идентификатора события,
* наличие типа события,
* наличие времени возникновения события,
* наличие идентификатора бизнес-объекта,
* наличие версии схемы сообщения,
* наличие полезной нагрузки с бизнес-данными.

Для обеспечения совместимости между производителями и потребителями сообщений схемы событий подлежат версионированию и централизованному управлению через БД реестра схем.

В последующих подразделах рассматриваются общая структура сообщений Event Bus и схемы основных типов событий, используемых в Сервисе.

===== 2.14.3.1. Общая схема Event Bus событий

Независимо от бизнес-назначения каждое событие содержит:

* служебную информацию о событии,
* информацию о версии схемы,
* идентификаторы бизнес-объектов,
* полезную нагрузку с данными предметной области.

==== 2.14.4. Схемы модели чтения и кешей

Модель чтения и кеши используются для обслуживания пользовательских запросов на получение информации о наличии товаров и доступности магазинов. В отличие от транзакционных данных, данные данного контура не являются первичным источником информации и формируются на основе событий, поступающих через Event Bus.

Основной задачей модели чтения является предоставление данных в форме, оптимизированной для выполнения поисковых запросов и операций чтения. По этой причине структура данных модели чтения отличается от структуры транзакционных данных и ориентирована на минимизацию количества операций, необходимых для формирования ответа пользователю.

Для сокращения времени доступа к наиболее востребованным данным дополнительно используется Распределенный кеш. В кеше хранятся предварительно подготовленные результаты вычислений и агрегированные данные, используемые наиболее нагруженными сценариями работы Сервиса.

Особенностью данного контура является наличие нескольких уровней представления одних и тех же данных:

* основной модели чтения,
* кешированных представлений данных,
* снимков состояния модели чтения.

Все указанные представления формируются на основании событийных данных и могут быть повторно построены при необходимости путем последовательной обработки событий из Event Bus.

В последующих подразделах рассматриваются:

* структура данных Хранилища модели чтения,
* структура ключей Распределенного кеша,
* схемы хранения снимков состояния и производных представлений данных.

===== 2.14.4.1. Структура модели чтения

Хранилище модели чтения содержит производное представление данных, предназначенное для обслуживания пользовательских запросов о наличии товаров в магазинах. Структура модели чтения оптимизирована под операции поиска и получения данных и не повторяет структуру транзакционных данных контура записи.

Основной единицей хранения является агрегированное состояние товара в конкретном магазине. Каждая запись содержит актуальную информацию о доступном и зарезервированном количестве товара, необходимую для формирования ответа без обращения к транзакционным хранилищам.

* получение информации о наличии конкретного товара в магазине,
* поиск магазинов с доступным товаром,
* отображение доступного количества товара,
* получение актуального состояния резервирования.

Все данные модели чтения являются производными и формируются путем последовательной обработки событий, поступающих из Event Bus. Первичным источником данных для формирования модели чтения являются события изменения остатков, резервирования, отмены резервов и результаты сверки данных.

===== 2.14.4.2. Структура Redis-ключей

Распределенный кеш используется для хранения краткоживущих представлений данных, необходимых для быстрого выполнения запросов Сервиса запросов. Ключи Redis формируются по предсказуемым шаблонам, чтобы обеспечить быстрый доступ к данным по основным сценариям чтения.

Redis не является источником истины. Все значения в Распределенном кеше являются производными от Хранилища модели чтения, БД резервирования или технических данных обработки.

Схема основных групп ключей представлена ниже.

ifndef::imagesdir[:imagesdir: /doc/images]

image::read_model_ke_diagram.svg[Структура Redis-ключей]

.Основные шаблоны Redis-ключей
[cols="2,2,3,1",options="header"]
|===
| Шаблон ключа | Тип значения | Назначение | Срок жизни

| `availability:{store_id}:{sku_id}` | JSON | Кешированное состояние доступности товара в магазине | 5–15 секунд
| `hot_sku:{sku_id}` | Счетчик / JSON | Учет популярных товаров для прогрева кеша | 5–10 минут
| `query:{hash}` | JSON | Кеширование результата повторяющегося пользовательского запроса | 5–30 секунд
| `lock:reservation:{store_id}:{sku_id}` | Строка / JSON | Распределенная блокировка на время операции резервирования | До 5 секунд
| `idempotency:{request_id}` | JSON | Защита от повторной обработки одного запроса | По сроку жизни операции
|===

Ключ `availability:{store_id}:{sku_id}` является основным ключом чтения для сценария проверки наличия товара в конкретном магазине. Значение ключа содержит доступное количество, количество в резерве, версию состояния и время обновления.

Ключ `hot_sku:{sku_id}` используется для учета популярных товаров и подготовки данных для прогрева кеша. Информация по таким ключам используется совместно с БД прогрева кеша.

Ключ `query:{hash}` используется для кратковременного кеширования повторяющихся пользовательских запросов. Хеш формируется из параметров запроса, влияющих на результат.

Ключ `lock:reservation:{store_id}:{sku_id}` используется для распределенной блокировки операций, изменяющих один и тот же товар в одном магазине. Значение ключа содержит идентификатор владельца блокировки и срок действия.

Ключ `idempotency:{request_id}` используется для защиты от повторной обработки одного и того же запроса. Значение содержит результат ранее выполненной операции или ее промежуточное состояние.

Инвалидация ключей выполняется при обработке событий, изменяющих состояние доступности товара или резерва. В первую очередь удаляются ключи:

* `availability:{store_id}:{sku_id}`,
* связанные `query:{hash}`,
* технические ключи идемпотентности после завершения срока хранения.

Срок жизни ключей используется как дополнительная защита от устаревания данных и не заменяет событийную инвалидацию.

===== 2.14.4.3. Снимки состояния и материализованные представления

Модель чтения формируется путем последовательной обработки событий, поступающих через Event Bus. Для сокращения времени восстановления производных данных и уменьшения объема повторной обработки событий в Сервисе используются снимки состояния и материализованные представления.

*Снимок состояния* представляет собой зафиксированное состояние части или всей модели чтения на определенный момент времени. *Материализованное представление* представляет собой заранее подготовленную структуру данных, оптимизированную для выполнения определенного типа запросов.

==== 2.14.5. Версионирование схем данных

Версионирование схем данных используется для управляемого изменения структуры транзакционных таблиц, событий Event Bus, документов модели чтения и кешируемых представлений без нарушения работы действующих компонентов Сервиса.

В Сервисе версионируются:

* схемы событий Event Bus,
* схемы таблиц БД резервирования, БД конфликтов, Хранилища событий и других PostgreSQL-хранилищ,
* структура документов Хранилища модели чтения,
* структура снимков состояния модели чтения,
* структура значений Redis-ключей.

Основной акцент делается на версионировании событийных схем, поскольку через Event Bus изменения распространяются между компонентами Сервиса.

ifndef::imagesdir[:imagesdir: /doc/images]

image::schema_registry_diagram.svg[Схема версионирования событийных данных]

.Основные объекты версионирования
[cols="2,2,3",options="header"]
|===
| Объект | Где применяется | Способ версионирования

| События Event Bus | inventory.events, reservation.events, reconciliation.events, error.events | Версия схемы в БД реестра схем и поле `event_version`
| Транзакционные таблицы | БД резервирования, БД конфликтов, Хранилище событий | Версионированные миграции схемы БД
| Документы модели чтения | Хранилище модели чтения | Версия документа в поле `version` или `schema_version`
| Снимки состояния | Snapshot модели чтения | Версия схемы снимка в поле `schema_version`
| Значения Redis-ключей | Распределенный кеш | Версия структуры значения внутри кешируемого объекта
|===

Для событий Event Bus используется формат сериализации Avro. Схемы событий хранятся в БД реестра схем и используются производителями и потребителями сообщений для проверки структуры данных.

.Правила развития схем событий
[cols="2,3",options="header"]
|===
| Изменение схемы | Правило применения

| Добавление необязательного поля | Допускается без создания несовместимой версии
| Удаление обязательного поля | Не допускается без выпуска новой версии события
| Изменение типа поля | Не допускается для действующей версии схемы
| Добавление нового типа события | Выполняется через регистрацию новой схемы в БД реестра схем
| Изменение смысла существующего поля | Не допускается, требуется новое поле или новая версия схемы
|===

Для обеспечения обратной совместимости потребители событий должны продолжать обработку сообщений предыдущих версий в течение переходного периода. Новые поля добавляются как необязательные, а значения по умолчанию задаются на уровне схемы.

Миграции PostgreSQL выполняются версионированно и применяются отдельно для каждого транзакционного хранилища. Изменения схемы БД выполняются по принципу безопасного расширения:

* сначала добавляются новые таблицы, поля или индексы,
* затем обновляются сервисы, использующие новую структуру,
* после завершения переходного периода удаляются устаревшие элементы схемы.

Откат изменений выполняется только для совместимых миграций. Для изменений, затрагивающих данные, заранее предусматривается обратная миграция или сохранение старой структуры до завершения проверки новой версии.

.Стратегия отката изменений
[cols="2,3",options="header"]
|===
| Тип изменения | Подход к откату

| Новое необязательное поле | Откат выполняется отключением использования поля в сервисах
| Новый индекс | Индекс может быть удален без изменения данных
| Новая таблица | Таблица сохраняется до завершения проверки и удаляется отдельной миграцией
| Изменение типа поля | Не применяется напрямую, выполняется через новое поле и перенос данных
| Новая версия события | Потребители временно поддерживают старую и новую версии схемы
|===

Для документов Хранилища модели чтения и снимков состояния используется поле версии схемы. Это позволяет Загрузчику моделей чтения отличать старые документы от новых и выполнять поэтапное обновление структуры без полной остановки обработки событий.
