=== 2.11. Схема взаимодействия

Настоящий раздел описывает: 

* схему взаимодействия компонентов Сервиса проверки доступности товаров, 
* используемые способы обмена данными и сетевые протоколы, 
* принципы маршрутизации запросов, 
* механизм обнаружения сервисов,
* организацию синхронного и асинхронного взаимодействия, 
* подходы к обеспечению отказоустойчивости и наблюдаемости.

В рамках Сервиса используются следующие типы взаимодействия:

* *синхронное* взаимодействие по модели "запрос -- ответ",
* *внутреннее* межсервисное взаимодействие,
* *асинхронное* взаимодействие на основе событий,
* *потоковая передача* обновлений в режиме реального времени.

Сервис реализован по микросервисному принципу с разделением компонентов (микросервисов) на слои и контуры: 

* входно слой,
* контур чтения, 
* контур записи и резервирования, 
* контур обработки событий, 
* контур интеграции,
* слой данных,
* слой инфраструктуры и наблюдаемости.

Такой подход позволяет независимо масштабировать обработку пользовательских запросов и поток обновления остатков.

==== 2.11.1. Общая схема взаимодействия компонентов

Взаимодействие компонентов Сервиса организовано через сочетание API Gateway, внутренних межсервисных вызовов и асинхронного обмена событиями через брокер сообщений Event Bus.

Основные принципы взаимодействия:

* все внешние запросы поступают через API Gateway,
* синхронные клиентские запросы обрабатываются через *REST API*,
* внутренние высокочастотные вызовы между сервисами выполняются через *gRPC*,
* обмен событиями осуществляется через Event Bus на базе *Apache Kafka*,
* Распределеный кеш на базе *Redis* используется как средство быстрого доступа к данным,
* *PostgreSQL* используется для хранения событий и транзакционных данных,
* сервисы взаимодействуют через *механизм обнаружения сервисов*,
* технические события и метрики передаются вспомогательным сервисам на слой инфраструктуры и наблюдаемости.

Все внешние запросы клиентских приложений направляются через API Gateway.

Внутренние компоненты Сервиса не публикуются во внешний контур и взаимодействуют между собой через механизм обнаружения сервисов и внутренние сетевые протоколы.

Для пользовательских сценариев проверки доступности товара используется REST API, а для внутренних высокочастотных взаимодействий между сервисами применяется gRPC поверх HTTP/2.

Для наглядного представления сетевых связей между компонентами ниже приведена диаграмма компонентов Сервиса с указанием основных протоколов взаимодействия. Диаграмма показывает не последовательность выполнения отдельных сценариев, а архитектурные связи между компонентами Сервиса.

На диаграмме выделены следующие типы взаимодействия:

* REST / HTTPS для внешних запросов клиентских приложений через API Gateway,
* gRPC / HTTP/2 для внутренних синхронных вызовов между сервисами,
* Kafka protocol / Avro для асинхронного обмена событиями через Event Bus,
* WebSocket / HTTPS для потоковой передачи обновлений клиентским приложениям,
* протокол Redis для обращения к Распределенному кешу,
* SQL-протокол для обращения к PostgreSQL-хранилищам.

==== 2.11.2. Типы взаимодействия в Сервисе

===== 2.11.2.1. Синхронное взаимодействие "запрос -- ответ"

Синхронное взаимодействие используется в Сервисе для пользовательских сценариев, в которых *клиент ожидает немедленного получения результата обработки запроса*. Основным таким сценарием является *проверка доступности товара в магазинах торговой сети*.

В рамках данного подхода клиентское приложение отправляет запрос и ожидает завершения обработки в течение одного сетевого взаимодействия. Такой режим работы используется для операций, результат которых должен быть предоставлен пользователю непосредственно во время выполнения бизнес-сценария. 

К таким операциям относятся:

* проверка доступности товара,
* поиск ближайших магазинов,
* получение информации об остатках,
* создание резерва,
* отмена резерва,
* получение статуса резерва.

Для реализации внешнего синхронного взаимодействия используется *REST API поверх HTTPS*. Данный подход выбран как основной механизм взаимодействия с клиентскими приложениями, поскольку Сервис должен обеспечивать совместимость одновременно с несколькими типами потребителей:

* МП My eDA,
* Web-клиент My eDA,
* ПМ ТСД eDA,
* внешние интегрируемые системы.

Использование REST API позволяет обеспечить единый и понятный контракт взаимодействия для всех типов клиентов без необходимости использования специализированных клиентских библиотек.

Все внешние запросы поступают в Сервис через API Gateway, который выступает единой точкой входа во внешний контур. Перед передачей запроса во внутренние компоненты API Gateway выполняет:

* проверку подлинности клиента,
* проверку прав доступа,
* ограничение частоты запросов,
* журналирование обращения,
* формирование идентификатора трассировки запроса,
* маршрутизацию запроса в соответствующий контур обработки.

После прохождения API Gateway *запрос проверки доступности товара* передаётся в Сервис запросов. Далее выполняется последовательность внутренних операций:

1. Определяется набор магазинов, участвующих в поиске. +
Если пользователь выполняет поиск товара рядом со своим местоположением, Сервис запросов обращается к Гео-сервису для определения ближайших магазинов в пределах заданного радиуса. Ограничение радиуса поиска и максимального количества магазинов используется для предотвращения чрезмерной нагрузки на контур чтения при массовых запросах.

2. Выполняется пакетное получение данных о доступности товара. +
Основным источником данных для пользовательских запросов является Распределенный кэш, содержащий модель чтения доступности товаров. Использование Распределенного кэша позволяет избежать прямого обращения к транзакционным данным резервирования при каждом пользовательском запросе и обеспечивает стабильное время ответа даже при высокой интенсивности чтения.

3. Если данные в Распределенном кэше отсутствуют либо требуют уточнения, Сервис запросов выполняет обращение к Хранилищу модели чтения. Такой подход позволяет снизить вероятность отказа пользовательского сценария при временной деградации кэша.

Разделение пользовательского контура чтения и контура записи и резервирования является одним из ключевых архитектурных решений Сервиса. Основная нагрузка в рассматриваемой предметной области приходится именно на операции чтения — проверку доступности товара и поиск магазинов. При этом операции резервирования требуют более строгого контроля согласованности данных и выполняются значительно реже. Разделение контуров позволяет независимо масштабировать обработку чтения и операций записи.

Дополнительно синхронное взаимодействие используется для операций резервирования товара. В отличие от проверки доступности, операции резервирования требуют более строгого контроля конкурентного доступа, поскольку несколько пользователей могут одновременно пытаться зарезервировать один и тот же товар в одном магазине. По этой причине внутри контура резервирования используются дополнительные механизмы защиты:

* распределённые блокировки,
* идемпотентность запросов,
* ограниченное время жизни резерва,
* проверка актуального доступного остатка перед подтверждением операции.

Несмотря на то, что пользователь получает результат резервирования синхронно, дальнейшее распространение изменений внутри Сервиса выполняется асинхронно через Event Bus. Такой подход позволяет сократить время ответа клиенту и уменьшить влияние потока внутренних обновлений на пользовательские запросы.

Для всех REST API используется единый подход к версионированию интерфейсов. Версия API указывается в URL запроса:

`/api/v1/...`

Использование явного версионирования позволяет развивать контракты взаимодействия без нарушения совместимости с существующими клиентскими приложениями.

Описание REST API публикуется в формате OpenAPI. Использование OpenAPI позволяет:

* формализовать контракты взаимодействия,
* обеспечить единообразие форматов запросов и ответов,
* упростить интеграцию внешних потребителей,
* автоматизировать публикацию документации API,
* упростить тестирование взаимодействия между компонентами.

===== 2.11.2.2. Внутреннее межсервисное взаимодействие

Помимо внешнего взаимодействия с клиентскими приложениями Сервис использует внутреннее межсервисное взаимодействие между своими компонентами. Данный тип взаимодействия применяется *для передачи данных и выполнения внутренних операций между сервисами* без участия внешнего клиента.

Внутреннее взаимодействие используется в следующих основных сценариях:

* получение данных о доступности товара,
* пакетная загрузка остатков,
* поиск магазинов через Гео-сервис,
* обработка резервирования,
* обновление модели чтения,
* распространение изменений остатков,
* техническая синхронизация компонентов,
* контроль состояния зависимых сервисов.

В отличие от внешнего REST API, ориентированного на универсальность и совместимость с различными типами клиентов, внутреннее взаимодействие оптимизируется прежде всего под производительность, минимизацию задержек и снижение сетевых накладных расходов.

Основной объём внутреннего межсервисного взаимодействия приходится на контур чтения. При выполнении пользовательского запроса Сервис запросов должен за короткое время получить данные сразу по нескольким магазинам и товарам. В случае популярных SKU количество одновременно обрабатываемых запросов может резко возрастать, поэтому эффективность внутреннего взаимодействия становится критически важной для стабильности Сервиса.

Для реализации внутренних вызовов используется *gRPC поверх HTTP/2*. Выбор данного подхода обусловлен несколькими причинами:

1. gRPC использует бинарный формат передачи данных, что уменьшает объём сетевого трафика по сравнению с текстовыми REST-запросами. Это особенно важно при массовом обмене данными между сервисами.

2. gRPC обеспечивает строгую типизацию контрактов взаимодействия. Для Сервиса это особенно важно, поскольку ошибки форматов данных при обработке событий остатков и резервирования могут приводить к расхождению модели чтения и фактического состояния резервов.

Внутренние gRPC-вызовы активно используются в следующих цепочках взаимодействия:

* Сервис запросов → Гео-сервис,
* Сервис запросов → Распределенный кэш,
* Сервис резервирования → Распределенный кэш,
* Загрузчик моделей чтения → Хранилище модели чтения,
* Загрузчик моделей чтения → Распределенный кэш,
* Сервис сверки → Хранилище событий.

При выполнении пользовательского сценария проверки доступности товара Сервис запросов использует пакетную обработку данных. Вместо большого количества отдельных обращений по каждому магазину выполняется агрегированный запрос сразу по набору комбинаций SKU + магазин. Такой подход позволяет существенно сократить количество сетевых вызовов и уменьшить нагрузку на внутреннюю сеть Сервиса.

Дополнительно внутреннее межсервисное взаимодействие используется для обработки операций резервирования. После получения запроса резервирования Сервис резервирования должен:

* проверить актуальный остаток,
* убедиться в отсутствии конфликтующего резерва,
* выполнить захват распределённой блокировки,
* сформировать событие резервирования,
* обновить состояние модели чтения.

Часть этих операций требует синхронного подтверждения результата до возврата ответа клиенту. По этой причине внутренние вызовы в контуре резервирования выполняются с минимальными задержками и ограниченным временем ожидания ответа.

Для предотвращения каскадной деградации Сервиса внутренние взаимодействия используют дополнительные механизмы защиты:

* ограничение времени ожидания ответа,
* повторные попытки только для безопасных операций,
* контроль доступности зависимых компонентов,
* ограничение количества одновременных вызовов,
* изоляцию перегруженных компонентов.

При временной недоступности отдельного компонента допускается частичная деградация функциональности. Например, при недоступности Гео-сервиса Сервис может продолжить обработку запросов проверки доступности товара для конкретного магазина без выполнения геопоиска.

Поскольку внутреннее взаимодействие является критически важным для стабильности Сервиса, все вызовы сопровождаются передачей идентификаторов трассировки. Это позволяет отслеживать полную цепочку обработки запроса между компонентами и упрощает диагностику проблем в распределённой архитектуре.

Для мониторинга внутреннего межсервисного взаимодействия контролируются метрики изложенные в разделе <<section-monitoring-interconnect, 2.11.10>>.

===== 2.11.2.3. Асинхронное взаимодействие на основе событий

Помимо синхронного взаимодействия "запрос -- ответ" в Сервисе активно используется асинхронное взаимодействие на основе событий. Данный подход является одним из ключевых архитектурных решений, поскольку *позволяет разделить обработку пользовательских запросов и поток изменений состояния данных*.

Основная особенность предметной области заключается в том, что изменения остатков происходят непрерывно и поступают одновременно из нескольких источников:

* кассовых операций POS,
* складских систем WMS,
* операций резервирования OMS,
* отмен резервов,
* корректировок остатков,
* результатов сверки данных.

При этом пользовательские запросы проверки доступности товара должны обрабатываться независимо от интенсивности потока обновлений. Если бы обновление остатков и обработка пользовательских запросов выполнялись в рамках единого синхронного процесса, рост количества событий изменения остатков напрямую влиял бы на время ответа клиентских приложений.

Для разделения потоков чтения и обновления данных в Сервисе используется событийная модель взаимодействия через Event Bus. В качестве механизма передачи событий применяется *Apache Kafka*.

Использование Event Bus позволяет:

* снизить связанность компонентов,
* независимо масштабировать обработку событий,
* отделить пользовательские запросы от внутреннего потока обновлений,
* обеспечить повторную обработку событий,
* сохранить последовательность обработки изменений,
* уменьшить влияние временной недоступности отдельных компонентов.

В рамках событийной модели изменение состояния Сервиса рассматривается как последовательность событий, распространяемых между компонентами. После возникновения изменения соответствующий сервис формирует событие и публикует его в Event Bus. Далее событие независимо обрабатывается всеми заинтересованными компонентами.

Наиболее интенсивным потоком событий являются изменения остатков товаров. После поступления обновления выполняется следующая последовательность действий:

1. Источник изменения публикует событие изменения остатка.
2. Событие передаётся в Event Bus.
3. Загрузчик моделей чтения получает событие.
4. Обновляется Хранилище модели чтения.
5. Обновляются данные в Распределенном кэше.
6. Сервис сверки контролирует корректность обновления.
7. Компоненты мониторинга фиксируют параметры обработки события.

Таким образом, пользовательский контур чтения не взаимодействует напрямую с потоком обновления остатков, а использует уже подготовленную модель чтения.

Аналогичный подход используется для операций резервирования товара. После успешного подтверждения резервирования Сервис резервирования формирует событие резервирования и публикует его в Event Bus. Далее событие используется для:

* обновления модели чтения,
* изменения значения зарезервированного остатка,
* распространения информации о резерве,
* запуска процедур контроля согласованности,
* формирования технических метрик и журналов аудита.

Такой подход позволяет сократить время синхронного ответа клиенту, поскольку дальнейшее распространение изменений выполняется независимо от пользовательского запроса.

Для различных типов событий используются отдельные топики Event Bus:

* `inventory.events` -- изменения остатков,
* `reservation.events` -- операции резервирования,
* `reconciliation.events` -- события сверки,
* `monitoring.events` -- технические события мониторинга.

Разделение событий по топикам позволяет независимо масштабировать обработку различных потоков данных и снижает влияние перегрузки одного типа обработки на остальные компоненты.

Для сохранения порядка обработки события секционируются по комбинации SKU + магазин. Данный подход особенно важен для операций резервирования и изменения остатков, поскольку нарушение последовательности обработки может привести к временной неконсистентности модели чтения.

Асинхронное взаимодействие также используется для реализации сценария *обработки задач одним потребителем*. Такой подход применяется для:

* повторной обработки ошибок,
* задач сверки,
* пересчёта модели чтения,
* технических фоновых операций.

В данном случае задача помещается в отдельную очередь обработки, после чего выбирается одним экземпляром обработчика из группы потребителей. Это позволяет избежать параллельной обработки одной и той же задачи несколькими экземплярами сервиса.

Поскольку асинхронное взаимодействие не гарантирует мгновенное обновление модели чтения, в Сервисе используется модель eventual consistency. После фиксации изменения обновление пользовательской модели чтения выполняется с небольшой задержкой относительно исходного события.

В нормальном режиме работы данная задержка минимальна и не влияет на пользовательский сценарий. Однако при высокой нагрузке или временной деградации отдельных компонентов возможно кратковременное расхождение между:

* Хранилищем событий,
* Хранилищем модели чтения,
* Распределенным кэшем,
* внешними системами.

Для обнаружения подобных расхождений используется Сервис сверки, который выполняет:

* контроль согласованности остатков,
* поиск конфликтующих состояний,
* повторную синхронизацию,
* публикацию корректирующих событий.

Для повышения надёжности обработки событий в Event Bus используются:

* подтверждение записи сообщений,
* повторная отправка при временных ошибках,
* контроль смещений чтения,
* идемпотентная обработка,
* отдельные очереди проблемных сообщений,
* повторная обработка неуспешных событий.

Поскольку поток событий является критически важным для корректной работы модели чтения, в Сервисе дополнительно контролируются метрики изложенные в разделе <<section-monitoring-kafka, 2.11.10>>.

===== 2.11.2.4. Потоковые обновления

Помимо синхронного взаимодействия и асинхронного обмена событиями в Сервисе используется механизм потоковой передачи обновлений в режиме реального времени. Данный подход применяется в сценариях, где клиентскому приложению *необходимо получать изменения состояния без постоянного повторного обращения к REST API*.

Основной причиной использования потоковых обновлений является высокая динамика изменения данных в отдельных пользовательских сценариях. В рамках операций резервирования, изменения остатков и обработки заказов состояние товара может изменяться несколько раз за короткий промежуток времени. При использовании только периодического опроса REST API клиентские приложения были бы вынуждены постоянно выполнять повторные запросы для получения актуальной информации.

При высокой пользовательской нагрузке такой подход привёл бы к дополнительной нагрузке на API Gateway, Сервис запросов и Распределенный кэш, особенно в сценариях работы с популярными товарами.

Для решения данной проблемы в Сервисе используется взаимодействие через *WebSocket*. После установки соединения сервер получает возможность самостоятельно передавать обновления клиенту при изменении состояния данных без необходимости постоянного повторного опроса со стороны приложения.

Потоковые обновления используются в следующих сценариях:

* изменение статуса резерва,
* изменение доступности товара,
* обновление операторских интерфейсов,
* отображение технических уведомлений,
* обновление состояния обработки пользовательских операций.

Наиболее важным сценарием является обновление статуса резервирования товара. После создания резерва пользователь может ожидать:

* подтверждение резервирования,
* изменение статуса обработки,
* отмену резерва,
* истечение времени действия резерва.

При использовании стандартного REST-взаимодействия клиентскому приложению пришлось бы регулярно выполнять запросы проверки статуса. Вместо этого после установки WebSocket-соединения изменения состояния резерва автоматически передаются клиенту при поступлении соответствующих событий.

Последовательность взаимодействия выглядит следующим образом:

1. Клиентское приложение устанавливает WebSocket-соединение через API Gateway.
2. API Gateway выполняет проверку подлинности клиента.
3. После успешной аутентификации клиентское соединение регистрируется в API Gateway для последующей передачи обновлений.
4. При изменении состояния резервирования соответствующее событие поступает через Event Bus.
5. Сервис обновления модели чтения получает и обрабатывает событие, актуализируя данные доступности.
6. API Gateway передаёт обновлённые данные клиентскому приложению через активное WebSocket-соединение.

Таким образом, потоковые обновления используют событийную модель Сервиса и не требуют отдельного механизма синхронизации состояния.

Дополнительно потоковые обновления применяются в операторских интерфейсах и внутренних рабочих местах сотрудников. В подобных сценариях требуется отображение изменений доступности товара практически в реальном времени, особенно при работе с ограниченными остатками или высокооборачиваемыми SKU.

Использование WebSocket особенно важно при повышенной нагрузке на пользователей внешнего слоя. Вместо большого количества коротких REST-запросов используется ограниченное количество долгоживущих соединений, через которые передаются только фактические изменения состояния.

Это позволяет:

* уменьшить количество повторных запросов,
* снизить нагрузку на API Gateway,
* сократить количество обращений к Распределенному кэшу,
* уменьшить сетевой трафик,
* повысить оперативность обновления пользовательского интерфейса.

При этом потоковые обновления не используются как основной механизм получения данных о доступности товара. Основным способом взаимодействия с Сервисом остаётся REST API. WebSocket применяется только для сценариев, требующих оперативной доставки изменений состояния.

Для предотвращения чрезмерного потребления ресурсов в Сервисе используются дополнительные ограничения:

* ограничение количества одновременных соединений,
* контроль времени неактивности клиента,
* автоматическое закрытие неиспользуемых соединений,
* ограничение частоты отправки сообщений.
* повторная аутентификация при длительном соединении.

Поскольку механизм потоковых обновлений является частью распределённой архитектуры Сервиса, для него также используются единые механизмы трассировки и мониторинга. Контролируются метрики изложенные в разделе <<section-monitoring-stream, 2.11.10>>.

==== 2.11.3. Сценарии межсервисного взаимодействия

Для описания ключевых процессов Сервиса приведены диаграммы последовательности (sequence-диаграммы) основных сценариев межсервисного взаимодействия. Диаграммы показывают последовательность вызовов между компонентами, используемые протоколы взаимодействия, синхронные и асинхронные участки обработки, а также механизмы обеспечения надежности и согласованности данных.

Рассматриваемые сценарии отражают основные типы нагрузки Сервиса:

* операции чтения данных о наличии товара,
* операции изменения состояния при резервировании товара,
* асинхронную обработку событий обновления остатков.

Каждый сценарий демонстрирует взаимодействие между API Gateway, внутренними микросервисами, Распределенным кешем, Хранилищем модели чтения, Хранилищем событий и Event Bus. Дополнительно в описаниях рассматриваются механизмы обеспечения идемпотентности, ограничения конкурентного доступа, обработка деградированных состояний и распространение событий между компонентами.

Приведенные диаграммы последовательности описывают архитектурный уровень взаимодействия компонентов и не детализируют внутреннюю реализацию отдельных сервисов, сетевые повторы, низкоуровневые механизмы транспортных протоколов и служебные операции брокера сообщений.

===== 2.11.3.1. Проверка наличия товара

Сценарий проверки наличия товара относится к основным сценариям чтения. Его цель — быстро вернуть клиентскому приложению информацию о доступности товара в выбранном магазине или в группе ближайших магазинов.

Основной путь выполнения запроса построен по принципу приоритетного обращения к Распределенному кешу. Это позволяет не нагружать Хранилище модели чтения при каждом пользовательском запросе и выдерживать пиковую нагрузку на операции проверки наличия. Хранилище модели чтения используется как резервный источник данных в случае промаха кеша или временной недоступности части кешированных данных.

image::avalability_check_diagram.svg[Проверка наличия товара. Диаграмма последовательности]

На диаграмме показан основной сценарий проверки наличия товара. Клиентское приложение обращается к API Gateway по внешнему REST API поверх HTTPS. API Gateway выполняет проверку токена доступа, применяет ограничения частоты запросов и передает запрос во внутренний контур Сервиса. Внутреннее взаимодействие между API Gateway и Сервисом проверки доступности выполняется по REST или gRPC поверх HTTP/2.

Сценарий является синхронным и реализует модель «запрос — ответ»: клиентское приложение ожидает результат проверки наличия в рамках одного пользовательского запроса. Event Bus в этом сценарии напрямую не участвует, так как проверка наличия является операцией чтения. Асинхронные механизмы используются в других потоках — при обновлении остатков, обработке резервов и актуализации модели чтения.

Проверка наличия относится к latency-критичным операциям, так как выполняется в пользовательском сценарии просмотра товара, оформления заказа или выбора ближайшего магазина. Поэтому основной путь построен по принципу cache-first: Сервис проверки доступности сначала обращается к Распределенному кешу и только при отсутствии актуальных данных выполняет чтение из Хранилища модели чтения. Такой подход снижает задержку ответа и уменьшает нагрузку на постоянное хранилище.

Идемпотентность в данном сценарии обеспечивается самой природой операции чтения. Повторный запрос с теми же параметрами `skuId` и `storeId` не изменяет состояние Сервиса и не создает новых бизнес-сущностей. Дополнительная запись результата в Распределенный кеш при промахе кеша не меняет бизнес-состояние товара, а только обновляет техническое представление данных для ускорения последующих запросов.

Надежность сценария обеспечивается несколькими механизмами. При недоступности или промахе Распределенного кеша Сервис проверки доступности использует Хранилище модели чтения как резервный источник данных. При отсутствии актуальной записи в модели чтения Сервис возвращает деградированный ответ, не подтверждая наличие товара как гарантированное. Это позволяет сохранить управляемое поведение пользовательского сценария даже при задержках обновления данных.

Актуальность данных в Распределенном кеше и Хранилище модели чтения обеспечивается не самим сценарием проверки наличия, а смежными асинхронными потоками: обработкой событий изменения остатков, событий резервирования и событий отмены резервов. Поэтому данный сценарий не выполняет пересчет остатков, а только читает уже подготовленную модель доступности.

===== 2.11.3.2. Резервирование товара

Сценарий резервирования товара относится к операциям изменения состояния. В отличие от проверки наличия, он не только читает подготовленную модель доступности, но и создает запись о временном резерве товара для конкретного магазина, товара и пользовательского сценария.

Резервирование выполняется синхронно с точки зрения клиентского запроса: клиентское приложение ожидает результат операции в рамках одного вызова. При этом часть последующей обработки выполняется асинхронно через Event Bus: событие о создании резерва используется для обновления модели чтения, кеша, аналитики и последующей сверки состояния.

image::reservation_diagram.svg[Резервирование товара. Диаграмма последовательности]

На диаграмме показан основной сценарий резервирования товара. Клиентское приложение отправляет запрос на создание резерва через API Gateway по REST API поверх HTTPS. API Gateway выполняет проверку токена доступа, проверку прав, применение ограничений частоты запросов и передает команду во внутренний контур Сервиса. Внутреннее взаимодействие между API Gateway и Сервисом резервирования выполняется по REST или gRPC поверх HTTP/2.

Сценарий резервирования является смешанным. Основная операция создания резерва выполняется синхронно по модели «запрос — ответ»: клиентское приложение ожидает результат создания резерва в рамках текущего запроса. При этом последствия создания резерва распространяются асинхронно через Event Bus. Событие `ReservationCreated` используется другими обработчиками для актуализации модели чтения, обновления кешированных представлений и последующей сверки состояния.

Резервирование относится к latency-критичным операциям, так как выполняется в пользовательском сценарии оформления заказа или выбора товара для доставки. Пользователь ожидает быстрый и однозначный ответ: резерв создан, товар недоступен или операцию нужно повторить позже. Поэтому в синхронной части сценария выполняются только операции, необходимые для принятия решения о резервировании: проверка идемпотентности, захват блокировки, проверка доступного остатка, запись резерва, фиксация события и публикация события в Event Bus.

Идемпотентность обеспечивается в Сервисе резервирования на уровне ключа идемпотентности. Клиентское приложение передает `Idempotency-Key` вместе с запросом на создание резерва. Сервис резервирования проверяет, выполнялся ли ранее запрос с таким ключом. Если запрос уже был успешно обработан, Сервис не создает повторный резерв, а возвращает ранее сохраненный результат. Дополнительно идемпотентность должна поддерживаться ограничением уникальности в Хранилище резервов для пары `Idempotency-Key` и идентификатора клиента или заказа.

Защита от конкурентного резервирования одного и того же остатка обеспечивается распределенной блокировкой по сочетанию `skuId` и `storeId`. Перед созданием резерва Сервис резервирования захватывает блокировку в Распределенном кеше. Если блокировка уже удерживается другим запросом, Сервис возвращает клиенту управляемый отказ с кодом `409 Conflict`. Это предотвращает одновременное списание одного и того же доступного остатка несколькими параллельными запросами.

Надежность сценария обеспечивается несколькими механизмами. Запись резерва выполняется в Хранилище резервов как атомарная операция с сохранением статуса, срока действия резерва и ключа идемпотентности. Событие `ReservationCreated` фиксируется в Хранилище событий, после чего публикуется в Event Bus. Это позволяет восстановить распространение события при сбое публикации и не потерять факт созданного резерва.

Распределенная блокировка имеет ограниченное время жизни. Поэтому при сбое экземпляра Сервиса резервирования блокировка не остается навсегда и освобождается автоматически после истечения срока действия. После успешного создания резерва Сервис обновляет технические счетчики в Распределенном кеше и освобождает блокировку явно.

Если доступного остатка недостаточно, резерв не создается, событие не публикуется, а клиент получает управляемый отказ. Если событие создано, но его дальнейшая обработка задерживается, синхронный результат резервирования остается корректным, так как источником факта резерва является Хранилище резервов и Хранилище событий. Актуализация модели чтения и кеша может выполниться с небольшой задержкой в асинхронном контуре обработки событий.

===== 2.11.3.3. Отмена резервирования товара

Сценарий отмены резервирования товара используется при отказе пользователя от оформления заказа, завершении времени жизни резерва или невозможности дальнейшего подтверждения заказа. Операция отмены освобождает ранее зарезервированный остаток и инициирует распространение изменений в модель чтения и связанные подсистемы через Event Bus.

Как и создание резерва, отмена резервирования выполняется синхронно с точки зрения клиентского запроса, однако дальнейшее распространение изменений состояния происходит асинхронно.

ifndef::imagesdir[:imagesdir: /doc/images]

image::reservation_canselling.svg[Отмена резервирования товара. Диаграмма последовательности]

На диаграмме показан сценарий отмены резервирования товара. Клиентское приложение отправляет запрос на отмену ранее созданного резерва через API Gateway по REST API поверх HTTPS. API Gateway выполняет проверку токена доступа, проверку прав и ограничение частоты запросов, после чего передает запрос в Сервис резервирования. Внутреннее взаимодействие между API Gateway и Сервисом резервирования выполняется по REST или gRPC поверх HTTP/2.

Сценарий является смешанным. Изменение состояния резерва выполняется синхронно в рамках пользовательского запроса: клиент получает подтверждение отмены только после фиксации нового состояния резерва. Дальнейшее распространение изменений выполняется асинхронно через Event Bus. Событие `ReservationCancelled` используется для актуализации модели чтения, обновления кешированных представлений, аналитики и сверки состояния.

Операция отмены резервирования относится к latency-критичным операциям, так как влияет на доступность товара для других пользовательских запросов. После отмены резерва товар должен как можно быстрее стать доступным для повторного резервирования и отображения в сценариях проверки наличия. Поэтому в синхронной части сценария выполняются только операции, необходимые для гарантированного изменения состояния: проверка идемпотентности, проверка статуса резерва, захват блокировки, изменение состояния резерва, фиксация события и публикация события в Event Bus.

Идемпотентность обеспечивается использованием `Idempotency-Key`. Если клиент повторно отправляет запрос отмены с тем же ключом, Сервис резервирования не выполняет повторное изменение состояния, а возвращает ранее сохраненный результат. Дополнительно идемпотентность обеспечивается самой моделью состояния резерва: повторная отмена уже отмененного или завершенного резерва не приводит к повторному освобождению остатка.

Для предотвращения конкурентного изменения остатка используется распределенная блокировка по сочетанию `skuId` и `storeId`. Это исключает ситуации, при которых отмена резерва и создание нового резерва одновременно изменяют один и тот же доступный остаток. Если блокировка не может быть получена, клиент получает управляемый отказ с кодом `409 Conflict`.

Надежность сценария обеспечивается фиксацией изменения состояния резерва в Хранилище резервов и записью события `ReservationCancelled` в Хранилище событий. После этого событие публикуется в Event Bus для дальнейшего распространения изменений между компонентами Сервиса. Если публикация события или обработка downstream-компонентами задерживается, факт отмены резерва остается сохраненным в основном хранилище и может быть повторно распространен через механизм повторной обработки событий.

Подтверждение публикации события выполняется до освобождения распределенной блокировки. Это уменьшает риск ситуации, при которой резерв уже отменен, но событие об освобождении остатка еще не распространено в асинхронном контуре. После успешной публикации события Сервис резервирования обновляет технические счетчики в Распределенном кеше и освобождает блокировку.

Таким образом, сценарий отмены резервирования обеспечивает согласованное освобождение ранее зарезервированного остатка с последующим асинхронным распространением изменений в модель чтения и связанные подсистемы.

===== 2.11.3.4. Обновление остатка

Сценарий обновления остатка относится к асинхронным потокам обработки событий. Его задача — принять изменение остатка из внешних учетных систем, зафиксировать событие, распространить его через Event Bus и обновить Хранилище модели чтения и Распределенный кеш, используемые при проверке наличия товара.

ifndef::imagesdir[:imagesdir: /doc/images]

image::inventory_update_diagram.svg[Обновление остатка. Диаграмма последовательности]

На диаграмме показан сценарий обновления остатка товара. Источником изменения выступают внешние учетные системы, например POS или WMS. Они передают событие изменения остатка в Адаптер внешних событий. В зависимости от типа внешней системы интеграция может выполняться по HTTPS, gRPC или через потоковую передачу событий. Внутри Сервиса нормализованное событие публикуется в Event Bus в формате Avro с использованием протокола брокера сообщений.

Сценарий является асинхронным. В отличие от проверки наличия и резервирования, здесь нет клиентского приложения, ожидающего немедленный ответ по пользовательскому запросу. Изменение остатка принимается, фиксируется как событие и далее обрабатывается потребителями Event Bus. Обновление Хранилища модели чтения и Распределенного кеша выполняется независимо от исходной операции во внешней учетной системе.

Критичность задержки в этом сценарии связана не с временем ответа пользователю, а со свежестью данных о наличии товара. Чем больше задержка между изменением остатка во внешней системе и обновлением модели чтения, тем выше риск показать клиенту устаревшую информацию. Поэтому для потока обновления остатков важны задержка доставки события через Event Bus, задержка обработки события и задержка обновления Распределенного кеша.

Идемпотентность обеспечивается на уровне обработки события. Каждое событие изменения остатка содержит уникальный идентификатор `eventId` или исходный идентификатор `sourceEventId`. Обработчик обновления модели чтения проверяет, было ли это событие уже обработано. Если событие поступило повторно, повторное изменение модели чтения не выполняется, а смещение в Event Bus подтверждается. Дополнительно обновление выполняется с учетом версии события или времени события, чтобы более старое событие не перезаписало более новое состояние остатка.

Надежность сценария обеспечивается фиксацией события в Хранилище событий и публикацией в Event Bus с подтверждением записи. Обработчик обновления модели чтения подтверждает обработку сообщения только после успешного обновления Хранилища модели чтения и Распределенного кеша. Если обработчик завершается с ошибкой до подтверждения смещения, событие будет прочитано повторно и обработано заново. За счет идемпотентности повторная обработка не приводит к некорректному изменению состояния.

Распределенный кеш в этом сценарии обновляется после успешной записи в Хранилище модели чтения. В зависимости от выбранной политики может выполняться либо прямая запись нового значения availability-ключа, либо инвалидация устаревшего ключа. В обоих случаях цель одна — чтобы последующие запросы проверки наличия получали актуальные данные из быстрого слоя чтения.

Таким образом, сценарий обновления остатка является асинхронным механизмом актуализации модели чтения. Он не отвечает напрямую на пользовательские запросы, но определяет качество и свежесть данных, которые используются в сценарии проверки наличия товара.

===== 2.11.3.5. Обновление модели чтения и загрузка кеша

Сценарий обновления модели чтения и загрузки кеша обеспечивает подготовку данных, используемых в latency-критичных операциях проверки наличия товара. Основная задача данного потока — преобразовать поток событий изменения состояния в актуальное представление доступности товара, оптимизированное для быстрого чтения.

В отличие от сценариев проверки наличия и резервирования, данный процесс полностью выполняется асинхронно и не обслуживает пользовательские запросы напрямую. Он отвечает за поддержание согласованности между Event Bus, Хранилищем модели чтения и Распределенным кешем.

ifndef::imagesdir[:imagesdir: /doc/images]

image::read_model_update_and_cach_loading.svg[Обновление модели чтения и загрузка Распределенного кеша. Диаграмма последовательности]

На диаграмме показан сценарий обновления модели чтения и загрузки кеша. Источником изменений являются события, опубликованные в Event Bus: события изменения остатка, создания резерва и отмены резервирования. Обработчик модели чтения подписывается на соответствующие потоки событий и последовательно применяет изменения к Хранилищу модели чтения и Распределенному кешу.

Взаимодействие между компонентами в данном сценарии выполняется асинхронно. Обработчик модели чтения получает события из Event Bus по протоколу брокера сообщений. Передача событий между компонентами выполняется в формате Avro. Хранилище модели чтения обновляется через внутренний протокол доступа к выбранному хранилищу данных, а Распределенный кеш обновляется через бинарный протокол кеша.

Сценарий не относится к операциям синхронного пользовательского взаимодействия, однако является latency-критичным с точки зрения актуальности данных. Задержка между публикацией события и обновлением модели чтения напрямую влияет на вероятность показа клиенту устаревшей информации о наличии товара. Поэтому обработка событий выполняется непрерывно, а Хранилище модели чтения и Распределенный кеш обновляются сразу после получения нового события.

Идемпотентность обеспечивается на уровне обработки событий. Каждое событие содержит уникальный идентификатор события и информацию о версии состояния. Перед применением изменения Обработчик модели чтения проверяет, было ли событие уже обработано ранее. Если событие поступает повторно, повторное обновление модели чтения и кеша не выполняется.

Дополнительно защита от некорректного порядка обработки обеспечивается проверкой версии события или времени формирования события. Это предотвращает ситуацию, при которой более старое событие перезапишет более новое состояние доступности товара.

Надежность сценария обеспечивается несколькими механизмами. Event Bus хранит события до подтверждения обработки, поэтому при временной недоступности Обработчика модели чтения события не теряются и могут быть обработаны позднее. Подтверждение обработки события выполняется только после успешного обновления Хранилища модели чтения и выполнения операции с кешем.

Если экземпляр Обработчика модели чтения завершается с ошибкой до подтверждения смещения, событие будет доставлено повторно. За счет идемпотентной обработки повторное применение события не приводит к некорректному изменению состояния.

Для обновления Распределенного кеша могут использоваться две стратегии. Первая стратегия предполагает прямую запись нового availability-ключа сразу после обновления модели чтения. Вторая стратегия предполагает инвалидацию существующего ключа, после чего данные будут повторно загружены при следующем запросе проверки наличия. Выбор конкретной стратегии зависит от допустимого объема фоновой нагрузки и требований к скорости распространения обновлений.

Таким образом, сценарий обновления модели чтения и загрузки кеша формирует подготовленное представление доступности товара, оптимизированное для быстрого чтения в пользовательских сценариях проверки наличия и резервирования.
