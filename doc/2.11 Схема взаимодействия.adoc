=== 2.11. Схема взаимодействия

Настоящий раздел описывает: 

* схему взаимодействия компонентов Сервиса проверки доступности товаров, 
* используемые способы обмена данными и сетевые протоколы, 
* принципы маршрутизации запросов, 
* механизм обнаружения сервисов,
* организацию синхронного и асинхронного взаимодействия, 
* подходы к обеспечению отказоустойчивости и наблюдаемости.

В рамках Сервиса используются следующие типы взаимодействия:

* *синхронное* взаимодействие по модели "запрос -- ответ",
* *внутреннее* межсервисное взаимодействие,
* *асинхронное* взаимодействие на основе событий,
* *потоковая передача* обновлений в режиме реального времени.

Сервис реализован по микросервисному принципу с разделением компонентов (микросервисов) на слои и контуры: 

* входно слой,
* контур чтения, 
* контур записи и резервирования, 
* контур обработки событий, 
* контур интеграции,
* слой данных,
* слой инфраструктуры и наблюдаемости.

Такой подход позволяет независимо масштабировать обработку пользовательских запросов и поток обновления остатков.

==== 2.11.1. Общая схема взаимодействия компонентов

Взаимодействие компонентов Сервиса организовано через сочетание API Gateway, внутренних межсервисных вызовов и асинхронного обмена событиями через брокер сообщений Event Bus.

Основные принципы взаимодействия:

* все внешние запросы поступают через API Gateway,
* синхронные клиентские запросы обрабатываются через *REST API*,
* внутренние высокочастотные вызовы между сервисами выполняются через *gRPC*,
* обмен событиями осуществляется через Event Bus на базе *Apache Kafka*,
* Распределеный кеш на базе *Redis* используется как средство быстрого доступа к данным,
* *PostgreSQL* используется для хранения событий и транзакционных данных,
* сервисы взаимодействуют через *механизм обнаружения сервисов*,
* технические события и метрики передаются вспомогательным сервисам на слой инфраструктуры и наблюдаемости.

Все внешние запросы клиентских приложений направляются через API Gateway.

Внутренние компоненты Сервиса не публикуются во внешний контур и взаимодействуют между собой через механизм обнаружения сервисов и внутренние сетевые протоколы.

Для пользовательских сценариев проверки доступности товара используется REST API, а для внутренних высокочастотных взаимодействий между сервисами применяется gRPC поверх HTTP/2.

==== 2.11.2. Типы взаимодействия в Сервисе

===== 2.11.2.1. Синхронное взаимодействие "запрос -- ответ"

Синхронное взаимодействие используется в Сервисе для пользовательских сценариев, в которых *клиент ожидает немедленного получения результата обработки запроса*. Основным таким сценарием является *проверка доступности товара в магазинах торговой сети*.

В рамках данного подхода клиентское приложение отправляет запрос и ожидает завершения обработки в течение одного сетевого взаимодействия. Такой режим работы используется для операций, результат которых должен быть предоставлен пользователю непосредственно во время выполнения бизнес-сценария. 

К таким операциям относятся:

* проверка доступности товара,
* поиск ближайших магазинов,
* получение информации об остатках,
* создание резерва,
* отмена резерва,
* получение статуса резерва.

Для реализации внешнего синхронного взаимодействия используется *REST API поверх HTTPS*. Данный подход выбран как основной механизм взаимодействия с клиентскими приложениями, поскольку Сервис должен обеспечивать совместимость одновременно с несколькими типами потребителей:

* МП My eDA,
* Web-клиент My eDA,
* ПМ ТСД eDA,
* внешние интегрируемые системы.

Использование REST API позволяет обеспечить единый и понятный контракт взаимодействия для всех типов клиентов без необходимости использования специализированных клиентских библиотек.

Все внешние запросы поступают в Сервис через API Gateway, который выступает единой точкой входа во внешний контур. Перед передачей запроса во внутренние компоненты API Gateway выполняет:

* проверку подлинности клиента,
* проверку прав доступа,
* ограничение частоты запросов,
* журналирование обращения,
* формирование идентификатора трассировки запроса,
* маршрутизацию запроса в соответствующий контур обработки.

После прохождения API Gateway *запрос проверки доступности товара* передаётся в Сервис запросов. Далее выполняется последовательность внутренних операций:

1. Определяется набор магазинов, участвующих в поиске. +
Если пользователь выполняет поиск товара рядом со своим местоположением, Сервис запросов обращается к Гео-сервису для определения ближайших магазинов в пределах заданного радиуса. Ограничение радиуса поиска и максимального количества магазинов используется для предотвращения чрезмерной нагрузки на контур чтения при массовых запросах.

2. Выполняется пакетное получение данных о доступности товара. +
Основным источником данных для пользовательских запросов является Распределенный кэш, содержащий модель чтения доступности товаров. Использование Распределенного кэша позволяет избежать прямого обращения к транзакционным данным резервирования при каждом пользовательском запросе и обеспечивает стабильное время ответа даже при высокой интенсивности чтения.

3. Если данные в Распределенном кэше отсутствуют либо требуют уточнения, Сервис запросов выполняет обращение к Хранилищу модели чтения. Такой подход позволяет снизить вероятность отказа пользовательского сценария при временной деградации кэша.

Разделение пользовательского контура чтения и контура записи и резервирования является одним из ключевых архитектурных решений Сервиса. Основная нагрузка в рассматриваемой предметной области приходится именно на операции чтения — проверку доступности товара и поиск магазинов. При этом операции резервирования требуют более строгого контроля согласованности данных и выполняются значительно реже. Разделение контуров позволяет независимо масштабировать обработку чтения и операций записи.

Дополнительно синхронное взаимодействие используется для операций резервирования товара. В отличие от проверки доступности, операции резервирования требуют более строгого контроля конкурентного доступа, поскольку несколько пользователей могут одновременно пытаться зарезервировать один и тот же товар в одном магазине. По этой причине внутри контура резервирования используются дополнительные механизмы защиты:

* распределённые блокировки,
* идемпотентность запросов,
* ограниченное время жизни резерва,
* проверка актуального доступного остатка перед подтверждением операции.

Несмотря на то, что пользователь получает результат резервирования синхронно, дальнейшее распространение изменений внутри Сервиса выполняется асинхронно через Event Bus. Такой подход позволяет сократить время ответа клиенту и уменьшить влияние потока внутренних обновлений на пользовательские запросы.

Для всех REST API используется единый подход к версионированию интерфейсов. Версия API указывается в URL запроса:

`/api/v1/...`

Использование явного версионирования позволяет развивать контракты взаимодействия без нарушения совместимости с существующими клиентскими приложениями.

Описание REST API публикуется в формате OpenAPI. Использование OpenAPI позволяет:

* формализовать контракты взаимодействия,
* обеспечить единообразие форматов запросов и ответов,
* упростить интеграцию внешних потребителей,
* автоматизировать публикацию документации API,
* упростить тестирование взаимодействия между компонентами.

===== 2.11.2.2. Внутреннее межсервисное взаимодействие

Помимо внешнего взаимодействия с клиентскими приложениями Сервис использует внутреннее межсервисное взаимодействие между своими компонентами. Данный тип взаимодействия применяется *для передачи данных и выполнения внутренних операций между сервисами* без участия внешнего клиента.

Внутреннее взаимодействие используется в следующих основных сценариях:

* получение данных о доступности товара,
* пакетная загрузка остатков,
* поиск магазинов через Гео-сервис,
* обработка резервирования,
* обновление модели чтения,
* распространение изменений остатков,
* техническая синхронизация компонентов,
* контроль состояния зависимых сервисов.

В отличие от внешнего REST API, ориентированного на универсальность и совместимость с различными типами клиентов, внутреннее взаимодействие оптимизируется прежде всего под производительность, минимизацию задержек и снижение сетевых накладных расходов.

Основной объём внутреннего межсервисного взаимодействия приходится на контур чтения. При выполнении пользовательского запроса Сервис запросов должен за короткое время получить данные сразу по нескольким магазинам и товарам. В случае популярных SKU количество одновременно обрабатываемых запросов может резко возрастать, поэтому эффективность внутреннего взаимодействия становится критически важной для стабильности Сервиса.

Для реализации внутренних вызовов используется *gRPC поверх HTTP/2*. Выбор данного подхода обусловлен несколькими причинами:

1. gRPC использует бинарный формат передачи данных, что уменьшает объём сетевого трафика по сравнению с текстовыми REST-запросами. Это особенно важно при массовом обмене данными между сервисами.

2. gRPC обеспечивает строгую типизацию контрактов взаимодействия. Для Сервиса это особенно важно, поскольку ошибки форматов данных при обработке событий остатков и резервирования могут приводить к расхождению модели чтения и фактического состояния резервов.

Внутренние gRPC-вызовы активно используются в следующих цепочках взаимодействия:

* Сервис запросов → Гео-сервис,
* Сервис запросов → Распределенный кэш,
* Сервис резервирования → Распределенный кэш,
* Загрузчик моделей чтения → Хранилище модели чтения,
* Загрузчик моделей чтения → Распределенный кэш,
* Сервис сверки → Хранилище событий.

При выполнении пользовательского сценария проверки доступности товара Сервис запросов использует пакетную обработку данных. Вместо большого количества отдельных обращений по каждому магазину выполняется агрегированный запрос сразу по набору комбинаций SKU + магазин. Такой подход позволяет существенно сократить количество сетевых вызовов и уменьшить нагрузку на внутреннюю сеть Сервиса.

Дополнительно внутреннее межсервисное взаимодействие используется для обработки операций резервирования. После получения запроса резервирования Сервис резервирования должен:

* проверить актуальный остаток,
* убедиться в отсутствии конфликтующего резерва,
* выполнить захват распределённой блокировки,
* сформировать событие резервирования,
* обновить состояние модели чтения.

Часть этих операций требует синхронного подтверждения результата до возврата ответа клиенту. По этой причине внутренние вызовы в контуре резервирования выполняются с минимальными задержками и ограниченным временем ожидания ответа.

Для предотвращения каскадной деградации Сервиса внутренние взаимодействия используют дополнительные механизмы защиты:

* ограничение времени ожидания ответа,
* повторные попытки только для безопасных операций,
* контроль доступности зависимых компонентов,
* ограничение количества одновременных вызовов,
* изоляцию перегруженных компонентов.

При временной недоступности отдельного компонента допускается частичная деградация функциональности. Например, при недоступности Гео-сервиса Сервис может продолжить обработку запросов проверки доступности товара для конкретного магазина без выполнения геопоиска.

Поскольку внутреннее взаимодействие является критически важным для стабильности Сервиса, все вызовы сопровождаются передачей идентификаторов трассировки. Это позволяет отслеживать полную цепочку обработки запроса между компонентами и упрощает диагностику проблем в распределённой архитектуре.

Для мониторинга внутреннего межсервисного взаимодействия контролируются метрики изложенные в разделе <<section-monitoring-interconnect, 2.11.9>>.

===== 2.11.2.3. Асинхронное взаимодействие на основе событий

Помимо синхронного взаимодействия "запрос -- ответ" в Сервисе активно используется асинхронное взаимодействие на основе событий. Данный подход является одним из ключевых архитектурных решений, поскольку *позволяет разделить обработку пользовательских запросов и поток изменений состояния данных*.

Основная особенность предметной области заключается в том, что изменения остатков происходят непрерывно и поступают одновременно из нескольких источников:

* кассовых операций POS,
* складских систем WMS,
* операций резервирования OMS,
* отмен резервов,
* корректировок остатков,
* результатов сверки данных.

При этом пользовательские запросы проверки доступности товара должны обрабатываться независимо от интенсивности потока обновлений. Если бы обновление остатков и обработка пользовательских запросов выполнялись в рамках единого синхронного процесса, рост количества событий изменения остатков напрямую влиял бы на время ответа клиентских приложений.

Для разделения потоков чтения и обновления данных в Сервисе используется событийная модель взаимодействия через Event Bus. В качестве механизма передачи событий применяется *Apache Kafka*.

Использование Event Bus позволяет:

* снизить связанность компонентов,
* независимо масштабировать обработку событий,
* отделить пользовательские запросы от внутреннего потока обновлений,
* обеспечить повторную обработку событий,
* сохранить последовательность обработки изменений,
* уменьшить влияние временной недоступности отдельных компонентов.

В рамках событийной модели изменение состояния Сервиса рассматривается как последовательность событий, распространяемых между компонентами. После возникновения изменения соответствующий сервис формирует событие и публикует его в Event Bus. Далее событие независимо обрабатывается всеми заинтересованными компонентами.

Наиболее интенсивным потоком событий являются изменения остатков товаров. После поступления обновления выполняется следующая последовательность действий:

1. Источник изменения публикует событие изменения остатка.
2. Событие передаётся в Event Bus.
3. Загрузчик моделей чтения получает событие.
4. Обновляется Хранилище модели чтения.
5. Обновляются данные в Распределенном кэше.
6. Сервис сверки контролирует корректность обновления.
7. Компоненты мониторинга фиксируют параметры обработки события.

Таким образом, пользовательский контур чтения не взаимодействует напрямую с потоком обновления остатков, а использует уже подготовленную модель чтения.

Аналогичный подход используется для операций резервирования товара. После успешного подтверждения резервирования Сервис резервирования формирует событие резервирования и публикует его в Event Bus. Далее событие используется для:

* обновления модели чтения,
* изменения значения зарезервированного остатка,
* распространения информации о резерве,
* запуска процедур контроля согласованности,
* формирования технических метрик и журналов аудита.

Такой подход позволяет сократить время синхронного ответа клиенту, поскольку дальнейшее распространение изменений выполняется независимо от пользовательского запроса.

Для различных типов событий используются отдельные топики Event Bus:

* `inventory.events` -- изменения остатков,
* `reservation.events` -- операции резервирования,
* `reconciliation.events` -- события сверки,
* `monitoring.events` -- технические события мониторинга.

Разделение событий по топикам позволяет независимо масштабировать обработку различных потоков данных и снижает влияние перегрузки одного типа обработки на остальные компоненты.

Для сохранения порядка обработки события секционируются по комбинации SKU + магазин. Данный подход особенно важен для операций резервирования и изменения остатков, поскольку нарушение последовательности обработки может привести к временной неконсистентности модели чтения.

Асинхронное взаимодействие также используется для реализации сценария *обработки задач одним потребителем*. Такой подход применяется для:

* повторной обработки ошибок,
* задач сверки,
* пересчёта модели чтения,
* технических фоновых операций.

В данном случае задача помещается в отдельную очередь обработки, после чего выбирается одним экземпляром обработчика из группы потребителей. Это позволяет избежать параллельной обработки одной и той же задачи несколькими экземплярами сервиса.

Поскольку асинхронное взаимодействие не гарантирует мгновенное обновление модели чтения, в Сервисе используется модель eventual consistency. После фиксации изменения обновление пользовательской модели чтения выполняется с небольшой задержкой относительно исходного события.

В нормальном режиме работы данная задержка минимальна и не влияет на пользовательский сценарий. Однако при высокой нагрузке или временной деградации отдельных компонентов возможно кратковременное расхождение между:

* Хранилищем событий,
* Хранилищем модели чтения,
* Распределенным кэшем,
* внешними системами.

Для обнаружения подобных расхождений используется Сервис сверки, который выполняет:

* контроль согласованности остатков,
* поиск конфликтующих состояний,
* повторную синхронизацию,
* публикацию корректирующих событий.

Для повышения надёжности обработки событий в Event Bus используются:

* подтверждение записи сообщений,
* повторная отправка при временных ошибках,
* контроль смещений чтения,
* идемпотентная обработка,
* отдельные очереди проблемных сообщений,
* повторная обработка неуспешных событий.

Поскольку поток событий является критически важным для корректной работы модели чтения, в Сервисе дополнительно контролируются метрики изложенные в разделе <<section-monitoring-kafka, 2.11.9>>.

===== 2.11.2.4. Потоковые обновления

Помимо синхронного взаимодействия и асинхронного обмена событиями в Сервисе используется механизм потоковой передачи обновлений в режиме реального времени. Данный подход применяется в сценариях, где клиентскому приложению *необходимо получать изменения состояния без постоянного повторного обращения к REST API*.

Основной причиной использования потоковых обновлений является высокая динамика изменения данных в отдельных пользовательских сценариях. В рамках операций резервирования, изменения остатков и обработки заказов состояние товара может изменяться несколько раз за короткий промежуток времени. При использовании только периодического опроса REST API клиентские приложения были бы вынуждены постоянно выполнять повторные запросы для получения актуальной информации.

При высокой пользовательской нагрузке такой подход привёл бы к дополнительной нагрузке на API Gateway, Сервис запросов и Распределенный кэш, особенно в сценариях работы с популярными товарами.

Для решения данной проблемы в Сервисе используется взаимодействие через *WebSocket*. После установки соединения сервер получает возможность самостоятельно передавать обновления клиенту при изменении состояния данных без необходимости постоянного повторного опроса со стороны приложения.

Потоковые обновления используются в следующих сценариях:

* изменение статуса резерва,
* изменение доступности товара,
* обновление операторских интерфейсов,
* отображение технических уведомлений,
* обновление состояния обработки пользовательских операций.

Наиболее важным сценарием является обновление статуса резервирования товара. После создания резерва пользователь может ожидать:

* подтверждение резервирования,
* изменение статуса обработки,
* отмену резерва,
* истечение времени действия резерва.

При использовании стандартного REST-взаимодействия клиентскому приложению пришлось бы регулярно выполнять запросы проверки статуса. Вместо этого после установки WebSocket-соединения изменения состояния резерва автоматически передаются клиенту при поступлении соответствующих событий.

Последовательность взаимодействия выглядит следующим образом:

1. Клиентское приложение устанавливает WebSocket-соединение через API Gateway.
2. API Gateway выполняет проверку подлинности клиента.
3. После успешной аутентификации клиентское соединение регистрируется в API Gateway для последующей передачи обновлений.
4. При изменении состояния резервирования соответствующее событие поступает через Event Bus.
5. Сервис обновления модели чтения получает и обрабатывает событие, актуализируя данные доступности.
6. API Gateway передаёт обновлённые данные клиентскому приложению через активное WebSocket-соединение.

Таким образом, потоковые обновления используют событийную модель Сервиса и не требуют отдельного механизма синхронизации состояния.

Дополнительно потоковые обновления применяются в операторских интерфейсах и внутренних рабочих местах сотрудников. В подобных сценариях требуется отображение изменений доступности товара практически в реальном времени, особенно при работе с ограниченными остатками или высокооборачиваемыми SKU.

Использование WebSocket особенно важно при повышенной нагрузке на пользователей внешнего слоя. Вместо большого количества коротких REST-запросов используется ограниченное количество долгоживущих соединений, через которые передаются только фактические изменения состояния.

Это позволяет:

* уменьшить количество повторных запросов,
* снизить нагрузку на API Gateway,
* сократить количество обращений к Распределенному кэшу,
* уменьшить сетевой трафик,
* повысить оперативность обновления пользовательского интерфейса.

При этом потоковые обновления не используются как основной механизм получения данных о доступности товара. Основным способом взаимодействия с Сервисом остаётся REST API. WebSocket применяется только для сценариев, требующих оперативной доставки изменений состояния.

Для предотвращения чрезмерного потребления ресурсов в Сервисе используются дополнительные ограничения:

* ограничение количества одновременных соединений,
* контроль времени неактивности клиента,
* автоматическое закрытие неиспользуемых соединений,
* ограничение частоты отправки сообщений.
* повторная аутентификация при длительном соединении.

Поскольку механизм потоковых обновлений является частью распределённой архитектуры Сервиса, для него также используются единые механизмы трассировки и мониторинга. Контролируются метрики изложенные в разделе <<section-monitoring-stream, 2.11.9>>.

==== 2.11.4. Использование HTTP/2 и HTTP/3

===== 2.11.4.1. Использование HTTP/2

Для внутреннего взаимодействия между компонентами Сервиса используется HTTP/2. Данный протокол применяется в первую очередь для gRPC-вызовов между сервисами контура чтения, резервирования и обработки событий.

Необходимость использования HTTP/2 обусловлена характером нагрузки на Сервис. В процессе обработки пользовательского запроса компоненты выполняют большое количество коротких внутренних обращений.

При использовании традиционного HTTP/1.1 значительная часть времени и сетевых ресурсов расходовалась бы на постоянное создание и обслуживание большого количества отдельных соединений между сервисами. Для Сервиса проверки доступности товара такая модель работы является неэффективной, поскольку основная нагрузка формируется именно множеством коротких высокочастотных запросов.

HTTP/2 позволяет решить данную проблему за счёт мультиплексирования запросов. Несколько параллельных обращений могут выполняться через одно установленное соединение без необходимости повторного открытия сетевых подключений. Это особенно важно для сценариев пакетной обработки данных, когда Сервис запросов одновременно получает информацию сразу по нескольким магазинам и товарам.

Использование HTTP/2 обеспечивает следующие преимущества для Сервиса:

* уменьшение сетевых накладных расходов,
* снижение задержек межсервисного взаимодействия,
* более эффективное использование соединений,
* повышение пропускной способности прикладного слоя и слоя данных,кеш
* уменьшение влияния сетевых задержек при высокой нагрузке.

Дополнительно HTTP/2 позволяет более эффективно использовать gRPC в сочетании с бинарным форматом передачи данных. Это уменьшает объём передаваемого трафика и снижает нагрузку на внутреннюю сеть Сервиса.

Использование HTTP/2 особенно важно при работе с популярными товарами, когда количество одновременных пользовательских запросов может резко возрастать. В подобных сценариях Сервис должен выполнять большое количество внутренних обращений с минимальными задержками, не создавая чрезмерную нагрузку на сетевую инфраструктуру.

Дополнительно HTTP/2 улучшает устойчивость внутреннего взаимодействия при временных сетевых перегрузках. За счёт повторного использования соединений уменьшается количество операций установления новых подключений, что снижает вероятность деградации внутреннего контура при пиковых нагрузках.

Поскольку HTTP/2 используется только для внутреннего межсервисного взаимодействия, данный протокол не предъявляет дополнительных требований к внешним клиентским приложениям. Внешний контур Сервиса продолжает использовать REST API поверх HTTPS через API Gateway, а переход на HTTP/2 происходит внутри контура обработки запросов.

Для контроля эффективности использования HTTP/2 в Сервисе отслеживаются следующие показатели, изложенные в разделе <<section-monitoring-http2, 2.11.9>>.

===== 2.11.4.2. Использование HTTP/3

В рамках перспективного развития внешнего контура взаимодействия в Сервисе рассматривается возможность использования HTTP/3 для клиентских подключений через API Gateway. Основной областью применения HTTP/3 являются мобильные клиентские приложения, работающие *в условиях нестабильных сетевых подключений и переменного качества канала связи*.

HTTP/2 использует TCP в качестве транспортного протокола. *HTTP/3 работает поверх QUIC* и ориентирован на снижение влияния сетевых задержек и потери пакетов на пользовательские запросы. Данный подход особенно актуален для мобильных сценариев использования Сервиса, поскольку пользователи могут выполнять проверку доступности товара при нестабильном качестве связи, переключении между сетями или временном ухудшении сигнала.

Для Сервиса проверки доступности товара это имеет практическое значение по причинам:

1. Пользовательский сценарий предполагает большое количество коротких запросов:

* проверка доступности товара,
* поиск ближайших магазинов,
* обновление информации о резерве,
* повторные проверки остатков.

    При нестабильном соединении задержки установления новых подключений могут существенно влиять на время ответа клиентского приложения. HTTP/3 уменьшает данные накладные расходы за счёт более эффективного механизма установки соединения.

2. HTTP/3 снижает влияние потери отдельных сетевых пакетов на остальные запросы, передаваемые через соединение. Для мобильных сетей это особенно важно, поскольку потеря пакетов и кратковременные обрывы соединения являются типичной ситуацией.

3. HTTP/3 позволяет быстрее восстанавливать соединение при смене сетевого интерфейса. Например, пользователь может переключаться между мобильной сетью и Wi-Fi без необходимости полного повторного установления соединения. Для клиентских приложений это уменьшает вероятность ошибок при выполнении операций проверки доступности товара и резервирования.

Несмотря на указанные преимущества, HTTP/3 не рассматривается как обязательный базовый механизм взаимодействия для всех компонентов Сервиса. Основным протоколом внешнего взаимодействия продолжает оставаться HTTPS/1.1 с поддержкой REST API через API Gateway.

Использование HTTP/3 рассматривается, как дополнительный механизм оптимизации пользовательского контура, прежде всего для мобильных клиентов. Такой подход позволяет сохранить совместимость с существующими клиентскими приложениями и не требует изменения внутренних механизмов обработки запросов.

Поскольку HTTP/3 внедряется на уровне API Gateway, внутренние компоненты Сервиса продолжают использовать существующую схему взаимодействия без изменения прикладной логики сервисов. Таким образом, внедрение HTTP/3 не влияет на внутреннюю архитектуру Сервиса и не требует изменения контрактов взаимодействия между сервисами.

Использование HTTP/3 рассматривается прежде всего как способ повышения качества пользовательского взаимодействия при неблагоприятных сетевых условиях и высокой мобильности клиентов. Наибольший эффект ожидается в сценариях:

* нестабильного мобильного соединения,
* высокой задержки сети,
* частой потери пакетов,
* большого количества коротких пользовательских запросов.

Для оценки эффективности использования HTTP/3 в Сервисе могут дополнительно контролироваться следующие метрики, изложенные в разделе <<section-monitoring-http3, 2.11.9>>.

Использование HTTP/3 рассматривается как одно из направлений дальнейшего развития внешнего API Сервиса и позволяет повысить устойчивость пользовательского контура без усложнения внутреннего взаимодействия между компонентами Сервиса.

==== 2.11.5. Обнаружение сервисов

===== 2.11.5.1. Назначение механизма обнаружения сервисов

Сервис реализован как распределённая микросервисная архитектура, в которой обработка пользовательских запросов и внутренних операций выполняется несколькими независимыми компонентами. В процессе работы экземпляры сервисов могут динамически запускаться, останавливаться, масштабироваться или перемещаться между вычислительными узлами. В таких условиях использование статически заданных сетевых адресов становится неэффективным и существенно усложняет сопровождение Сервиса.

Для решения данной проблемы в Сервисе используется механизм обнаружения сервисов. Его основное назначение заключается в *автоматическом определении доступных экземпляров сервисов и предоставлении другим компонентам актуальной информации* о том, каким образом с ними необходимо взаимодействовать.

Механизм обнаружения сервисов является одним из *базовых элементов внутренней инфраструктуры* Сервиса и используется для организации взаимодействия между микросервисами. 

Основная причина использования механизма обнаружения сервисов -- необходимость горизонтального масштабирования. +
При использовании фиксированных адресов любое изменение количества экземпляров потребовало бы ручного обновления конфигурации взаимодействия между компонентами. Для высоконагруженного Сервиса такой подход является неприемлемым, поскольку существенно усложняет масштабирование и снижает устойчивость архитектуры.

Наибольшая нагрузка в Сервисе приходится на пользовательские сценарии проверки доступности товара. При росте количества запросов может потребоваться запуск дополнительных экземпляров Сервиса запросов или компонентов обработки событий. Механизм обнаружения сервисов позволяет подключать новые экземпляры автоматически без изменения конфигурации клиентских компонентов.

Дополнительно данный механизм используется для повышения отказоустойчивости Сервиса. +
При отказе отдельного экземпляра сервиса его сетевой адрес автоматически исключается из списка доступных узлов. После этого запросы начинают перенаправляться только на работоспособные экземпляры. Такой подход *уменьшает вероятность каскадной деградации* обработки внешних запросов и повышает устойчивость внутреннего взаимодействия.

Особенно важную роль механизм обнаружения сервисов играет для компонентов контура чтения. Сервис запросов выполняет большое количество внутренних вызовов к Гео-сервису, Распределенному кэшу и другим компонентам. При высокой интенсивности пользовательских запросов количество экземпляров отдельных сервисов может динамически изменяться. Использование автоматического обнаружения сервисов позволяет выполнять балансировку нагрузки между экземплярами без ручного обновления сетевой конфигурации.

Аналогичный подход используется и в контуре обработки событий. При росте интенсивности поступления событий изменения остатков могут запускаться дополнительные экземпляры Загрузчика моделей чтения. После регистрации в механизме обнаружения сервисов новые экземпляры автоматически начинают участвовать в обработке потока событий.

*Дополнительно* механизм обнаружения сервисов используется для:

* централизованного контроля доступности компонентов,
* упрощения маршрутизации внутренних вызовов,
* автоматического исключения недоступных экземпляров,
* сокращения времени восстановления после отказов,
* упрощения развёртывания новых версий сервисов.

Использование механизма обнаружения сервисов особенно важно для Сервиса проверки доступности товара, поскольку архитектура решения ориентирована на работу в условиях переменной нагрузки. В течение суток интенсивность пользовательских запросов и потоков обновления остатков может существенно изменяться. Возможность динамического масштабирования компонентов без изменения конфигурации взаимодействия является необходимым условием обеспечения стабильной работы Сервиса.

Дополнительным преимуществом механизма обнаружения сервисов является снижение связанности компонентов. Внутренние сервисы взаимодействуют не с конкретными сетевыми адресами, а с логическими именами сервисов. Это упрощает сопровождение архитектуры и позволяет изменять внутреннюю инфраструктуру без влияния на прикладную логику компонентов.

Для контроля работы механизма обнаружения сервисов в Сервисе отслеживаются метрики, изложенные в разделе <<section-monitoring-srvdiscovery, 2.11.9>>.

Использование механизма обнаружения сервисов позволяет обеспечить: 

* динамическое масштабирование, 
* автоматическую балансировку нагрузки, 
* устойчивое взаимодействие между компонентами Сервиса,
* исключение ручного управления сетевой конфигурацией.

===== 2.11.5.2. Принцип работы

Механизм обнаружения сервисов обеспечивает автоматическое определение доступных экземпляров компонентов Сервиса и используется для организации внутреннего взаимодействия между сервисами без применения статически заданных сетевых адресов.

После запуска экземпляра сервиса выполняется следующая последовательность действий:

1. Экземпляр сервиса регистрируется в реестре сервисов.
2. В реестр передаются:

* логическое имя сервиса,
* сетевой адрес,
* используемый порт,
* техническая информация о состоянии экземпляра.

3. Сервис начинает периодически отправлять сигналы доступности, подтверждаетя своё состояние.
4. После успешной регистрации экземпляр становится доступным для внутренних вызовов.
5. API Gateway и внутренние сервисы получают информацию о доступных экземплярах из реестра сервисов и используют её для маршрутизации запросов.
6. Запросы направляются только на работоспособные экземпляры.

Например, при поступлении запроса проверки доступности товара API Gateway определяет список доступных экземпляров Сервиса запросов и перенаправляет запрос на один из них. Аналогичным образом Сервис запросов получает информацию о доступных экземплярах Гео-сервиса и выполняет внутренние вызовы без необходимости хранения статических сетевых адресов.

Дополнительно механизм обнаружения сервисов используется для обработки отказов компонентов.

Если экземпляр сервиса:

* перестаёт отправлять сигналы доступности,
* завершает работу аварийно,
* становится недоступным по сети,

то механизм обнаружения сервисов автоматически исключает его из списка доступных экземпляров и маршрутизации.

После этого:

* новые запросы перестают направляться на недоступный экземпляр,
* нагрузка перераспределяется между оставшимися экземплярами,
* API Gateway и внутренние сервисы получают обновлённую информацию о составе сервисов.

Такой подход позволяет уменьшить вероятность каскадной деградации обработки внешних запросов и сократить время восстановления после отказов.

Для предотвращения нестабильной маршрутизации в Сервисе используются дополнительные механизмы контроля состояния экземпляров:

* проверка доступности сервиса,
* контроль времени ответа,
* исключение перегруженных экземпляров,
* ограничение количества неуспешных обращений,
* повторное подключение после восстановления работоспособности.

Механизм обнаружения сервисов используется как для REST-взаимодействия, так и для gRPC-вызовов.

==== 2.11.6. API Gateway

===== 2.11.6.1. Назначение API Gateway

API Gateway используется в Сервисе как единая точка входа для внешних запросов, точка интеграции с внешнии по отношению к Сервису информационными системами и обеспечивает централизованное управление взаимодействием клиентских приложений и интегрируемых систем с внутренними компонентами Сервиса.

Основной задачей API Gateway является изоляция внутренней структуры Сервиса от внешних потребителей и поставщиков. Клиентские приложения взаимодействуют только с единым публичным интерфейсом, не имея прямого доступа к внутренним сервисам и не завися от особенностей внутренней маршрутизации запросов.

Такой подход особенно важен для Сервиса проверки доступности товара, поскольку внутренняя архитектура включает несколько независимых контуров обработки, недоступных снаружи:

* контур чтения,
* контур записи и резервирования,
* контур обработки событий,
* компоненты модели чтения.

API Gateway скрывает данную внутреннюю структуру и обеспечивает единые правила взаимодействия для всех типов клиентов.

Дополнительно API Gateway используется как центральная точка применения инфраструктурных политик. Через API Gateway реализуются:

* маршрутизация запросов,
* аутентификация и авторизация,
* ограничение частоты запросов,
* централизованное журналирование,
* управление версиями API,
* трассировка распределённых запросов.

Централизация данных механизмов позволяет избежать дублирования одинаковой инфраструктурной логики во внутренних сервисах и упрощает сопровождение архитектуры.

API Gateway также выполняет роль точки применения политик безопасности (PEP) во внешнем контуре взаимодействия Сервиса.

В данной роли API Gateway обеспечивает централизованное применение правил доступа и инфраструктурных ограничений до передачи запроса во внутренние сервисы. Это позволяет исключить дублирование одинаковых механизмов контроля в каждом компоненте Сервиса и обеспечить единообразное применение политик безопасности.

На уровне API Gateway выполняются:

* проверка подлинности клиента,
* проверка прав выполнения операции,
* контроль допустимых маршрутов доступа,
* ограничение частоты запросов,
* проверка технических ограничений запросов,
* фильтрация некорректных или потенциально опасных обращений.

Использование API Gateway как точки применения политик позволяет отделить бизнес-логику внутренних сервисов от инфраструктурных механизмов безопасности и централизованно управлять правилами доступа ко всем внешним API Сервиса.

Дополнительно такой подход упрощает изменение политик безопасности, поскольку обновление правил выполняется на уровне API Gateway без необходимости изменения внутренней логики отдельных сервисов.

Для Сервиса особое значение имеет возможность централизованного ограничения нагрузки. Наибольшая интенсивность запросов возникает при массовой проверке доступности популярных товаров. В таких сценариях API Gateway используется как первый уровень защиты внутренних компонентов от перегрузки.

Дополнительно API Gateway обеспечивает единый механизм публикации REST API и позволяет независимо развивать внутреннюю архитектуру Сервиса без изменения внешних контрактов взаимодействия.

Поскольку API Gateway является критически важным компонентом внешнего взаимодействия, его работоспособность и производительность контролируются отдельными средствами мониторинга и трассировки.

На уровне архитектуры Сервиса различают:

* API Gateway *входного слоя* -- для подключеня пользователей,
* API Gateway *внешних интеграций* -- для подключения внутренних ИС компании eDA, интегрируемых с Cервисом,
* API Gateway *служб доставки* -- для подключения к Cервису внешних ИС независимых служб доставки: Яндекс.Еда, Самокат, Купер и др.

===== 2.11.6.2. Правила маршрутизации

API Gateway выполняет маршрутизацию внешних запросов к внутренним компонентам Сервиса на основании набора правил, определяющих целевой контур обработки и соответствующий сервис.

Основной задачей маршрутизации является разделение различных типов операций между специализированными компонентами архитектуры. Такой подход позволяет независимо масштабировать обработку чтения, резервирования и фоновых операций, а также уменьшает влияние отдельных категорий нагрузки друг на друга.

В рамках Сервиса используются следующие основные правила маршрутизации:

[cols="1,1,1,1", options="header"]
|===
| Внешний путь | Тип запроса | Назначение | Целевой компонент

| `/api/v1/availability` | Проверка доступности товара | Получение информации об остатках и доступности товара | Сервис запросов
| `/api/v1/stores/nearby` | Поиск ближайших магазинов | Географический поиск магазинов | Сервис запросов / Гео-сервис
| `/api/v1/reservations` | Создание резерва | Обработка резервирования товара | Сервис резервирования
| `/api/v1/reservations/ +
{reservation_id}` | Получение статуса резерва | Проверка состояния резервирования | Сервис резервирования
| `/api/v1/reservations/ +
{reservation_id}` | Отмена резерва | Освобождение зарезервированного остатка | Сервис резервирования
| `/api/v1/health` | Технические проверки | Контроль доступности компонентов | Соответствующий внутренний сервис
|===

Маршрутизация выполняется с учётом версии API и типа операции. Использование версионирования позволяет одновременно поддерживать несколько версий внешних контрактов взаимодействия без изменения внутренней структуры Сервиса.

Дополнительно правила маршрутизации учитывают текущее состояние внутренних компонентов. При временной недоступности отдельных экземпляров API Gateway автоматически исключает их из обработки запросов и перенаправляет обращения на доступные экземпляры сервиса.

Для операций проверки доступности товара используется маршрутизация в контур чтения, оптимизированный под обработку большого количества пользовательских запросов. Операции резервирования направляются в контур записи и резервирования, где применяются дополнительные механизмы контроля согласованности данных.

Такое разделение маршрутов обработки позволяет:

* уменьшить влияние операций резервирования на производительность чтения,
* независимо масштабировать различные контуры обработки,
* локализовать деградацию отдельных компонентов,
* повысить устойчивость Сервиса при высокой нагрузке.

Отдельное внимание уделяется маршрутизации WebSocket-соединений. После установления соединения API Gateway поддерживает привязку соединения к соответствующему экземпляру компонента обработки потоковых уведомлений. Это необходимо для корректной доставки обновлений состояния резервирования и других событий в режиме реального времени.

Для предотвращения перегрузки внутренних компонентов API Gateway может применять дополнительные правила маршрутизации:

* ограничение количества одновременных запросов,
* ограничение запросов по отдельным SKU,
* приоритизацию критически важных операций,
* временное ограничение части функциональности при деградации отдельных компонентов.

===== 2.11.6.3. Проверка доступа

Проверка доступа в Сервисе выполняется централизованно на уровне API Gateway и применяется ко всем внешним запросам до их передачи во внутренние компоненты.

Основной задачей механизма проверки доступа является разграничение допустимых операций для различных категорий клиентов и исключение несанкционированного использования API Сервиса.

API Gateway выполняет:

* проверку токенов доступа,
* разграничение прав клиентов,
* передачу контекста пользователя во внутренние сервисы,
* фильтрацию недопустимых запросов,
* защиту от несанкционированного доступа.

В рамках Сервиса используются различные категории потребителей, включая:

* клиентские приложения,
* внутренние рабочие места сотрудников,
* внешние интеграционные системы,
* внутренние технические компоненты.

Для каждой категории клиентов определяются:

* доступные маршруты API,
* допустимые операции,
* ограничения на интенсивность обращений,
* доступ к отдельным функциям Сервиса.

После успешной проверки доступа API Gateway передаёт во внутренние сервисы контекст запроса, содержащий:

* идентификатор клиента,
* идентификатор пользователя,
* тип клиента,
* correlation_id,
* информацию о разрешённых операциях.

Внутренние сервисы используют уже проверенный контекст запроса и не выполняют повторную аутентификацию внешнего клиента.

Для внутреннего межсервисного взаимодействия применяется отдельный механизм сервисной аутентификации, позволяющий разделить внешние пользовательские обращения и внутренние технические вызовы между компонентами архитектуры.

Дополнительно механизм проверки доступа используется для ограничения обращений:

* к недопустимым маршрутам,
* к неподдерживаемым операциям,
* к техническим API,
* к операциям, запрещённым для конкретного типа клиента.

События отказа в доступе фиксируются средствами централизованного журналирования и используются для мониторинга попыток некорректного или несанкционированного обращения к API Сервиса.

===== 2.11.6.4. Ограничение частоты запросов

Для защиты Сервиса от перегрузки и неконтролируемого роста нагрузки на внешнем контуре взаимодействия используется механизм ограничения частоты запросов.

Необходимость применения данного механизма обусловлена особенностями предметной области. Проверка доступности товара относится к высокочастотным операциям, количество которых может резко возрастать в периоды повышенной пользовательской активности, проведения маркетинговых кампаний или появления популярных товаров с ограниченным остатком.

Дополнительным фактором риска является возможность многократного повторного обращения клиентских приложений к одним и тем же данным. Например, при обновлении пользовательского интерфейса или автоматическом повторе запросов количество обращений по одному SKU может многократно увеличиваться за короткий промежуток времени.

Без ограничения частоты запросов подобные сценарии могут приводить к таким последствиям:

* перегрузке Сервиса запросов,
* увеличению нагрузки на Распределенный кэш,
* росту количества внутренних вызовов,
* деградации времени ответа,
* вытеснению критически важных операций менее приоритетными запросами.

Ограничение частоты запросов применяется на уровне API Gateway до передачи запроса во внутренние компоненты Сервиса.

При превышении лимитов API Gateway возвращает ошибку ограничения запросов:

* 429   Too Many Requests,
* 509   Bandwidth Limit Exceeded.

В Сервисе используются несколько уровней ограничения нагрузки:

* по клиентскому приложению,
* по пользователю,
* по IP-адресу,
* по внешней интеграционной системе,
* по отдельным маршрутам API.

Дополнительно могут применяться отдельные ограничения для операций, создающих повышенную нагрузку на внутренние компоненты. В первую очередь это относится к сценариям массовой проверки доступности популярных товаров.

Для подобных операций могут использоваться:

* уменьшенные лимиты частоты запросов,
* ограничение количества параллельных обращений,
* временное ограничение повторных запросов,
* ограничение глубины поиска магазинов,
* ограничение максимального размера ответа.

Разделение лимитов по типам операций позволяет более гибко управлять нагрузкой на различные контуры обработки. Например, операции проверки доступности товара могут иметь более высокую интенсивность обращений, чем операции резервирования, для которых важнее сохранение согласованности данных и устойчивость обработки.

При превышении установленных ограничений API Gateway отклоняет запрос до его передачи во внутренние сервисы. Это позволяет предотвратить распространение перегрузки на компоненты контура чтения и контур записи и резервирования.

Дополнительно механизм ограничения частоты запросов используется как элемент защиты от аномальной активности и ошибочного поведения внешних интеграций. Резкий рост количества однотипных запросов может свидетельствовать о таких проблемах, как:

* ошибки клиентского приложения,
* некорректная интеграция,
* попытки массового автоматизированного обращения к API,
* лавинообразный повтор запросов после временной деградации.

Использование централизованного ограничения нагрузки позволяет локализовать подобные ситуации на уровне API Gateway без влияния на внутренние компоненты Сервиса.

Для контроля эффективности механизма ограничения запросов отслеживаются метрики, изложенные в разделе <<section-monitoring-ratelim, 2.11.9>>.

==== 2.11.7. Асинхронное взаимодействие через Apache Kafka

===== 2.11.7.1. Назначение брокера сообщений

В Сервисе проверки доступности товара брокер сообщений в составе Event Bus используется как центральный механизм передачи событий между компонентами распределённой архитектуры. В качестве брокера сообщений применяется *Apache Kafka*, обеспечивающий обработку большого количества событий при высокой интенсивности обновления данных.

Основной задачей брокера сообщений является передача изменений состояния между независимыми компонентами без необходимости их прямого синхронного взаимодействия. Такой подход позволяет компонентам обрабатывать события независимо друг от друга и уменьшает связанность архитектуры.

Использование брокера сообщений особенно важно для Сервиса, поскольку изменения состояния данных происходят непрерывно и одновременно затрагивают несколько контуров обработки. Одно изменение остатка или операция резервирования могут требовать:

* обновления модели чтения,
* обновления Распределенного кэша,
* фиксации события в Хранилище событий,
* запуска проверки согласованности,
* формирования технических метрик,
* передачи данных на уровень мониторинга.

При прямом синхронном взаимодействии между компонентами каждая операция изменения состояния была бы жёстко связана с доступностью и скоростью обработки всех зависимых сервисов. В условиях высокой нагрузки это привело бы к существенному росту времени обработки операций и увеличению вероятности каскадных отказов.

Использование брокера сообщений позволяет разделить момент фиксации изменения и дальнейшую обработку события различными компонентами. После публикации события дальнейшая обработка выполняется независимо отдельными потребителями.

Дополнительной причиной использования Apache Kafka является необходимость устойчивой обработки интенсивного потока событий изменения остатков. Для Сервиса проверки доступности товара характерно постоянное поступление изменений из различных источников, включая:

* кассовые операции через адаптер POS,
* складские системы через адаптер WMS,
* операции резервирования и отмены резервов,
* результаты сверки данных.

Apache Kafka позволяет обрабатывать подобный поток событий с сохранением порядка обработки внутри разделов и возможностью горизонтального масштабирования потребителей.

Брокер сообщений также используется как механизм временной буферизации нагрузки. При кратковременном росте интенсивности событий компоненты обработки могут продолжать получать сообщения из очереди с собственной скоростью без остановки публикации новых событий.

Дополнительно использование Apache Kafka позволяет реализовать повторную обработку событий. Это особенно важно для операций резервирования и обновления модели чтения, поскольку временные ошибки отдельных компонентов не должны приводить к потере изменений состояния.

Для различных категорий событий в Сервисе используются отдельные топики брокера сообщений. Такой подход позволяет независимо масштабировать обработку различных потоков данных и локализовать влияние перегрузки отдельных компонентов.

Поскольку брокер сообщений является критически важным элементом архитектуры, в Сервисе контролируются отдельные показатели его работы, изложенные в разделе <<section-monitoring-kafka, 2.11.9>>.

===== 2.11.7.2. Сценарий публикации и подписки

Сценарий публикации и подписки используется в Сервисе для распространения изменений состояния между несколькими независимыми компонентами. В рамках данного подхода один компонент публикует событие в Event Bus, после чего событие может быть обработано несколькими потребителями одновременно и независимо друг от друга.

Данный механизм применяется в ситуациях, когда одно изменение должно инициировать несколько параллельных процессов обработки. Наиболее характерным примером является изменение остатка товара.

1. После поступления нового значения остатка формируется событие изменения состояния товара, содержащее информацию:

* о SKU,
* магазине,
* новом значении остатка,
* времени изменения,
* источнике обновления.

2. Событие публикуется в топик `inventory.events`.
3. После публикации события его начинают независимо обрабатывать различные компоненты Сервиса:
.. Загрузчик моделей чтения использует событие для обновления Хранилища модели чтения и последующего обновления данных в Распределенном кэше.
.. Сервис сверки использует тот же поток событий для контроля корректности изменения состояния и выявления возможных расхождений между различными источниками данных.
.. Компоненты мониторинга используют события для расчёта технических метрик, включая:

* скорость изменения остатков,
* задержку распространения событий,
* интенсивность обновлений по отдельным магазинам или SKU.

Аналогичный сценарий используется и для операций резервирования. После подтверждения резервирования событие из топика `reservation.events` может одновременно использоваться для:

* обновления модели чтения,
* расчёта зарезервированного остатка,
* фиксации операций аудита,
* формирования технических уведомлений,
* последующего анализа конфликтов резервирования.

Ключевой особенностью модели публикации и подписки является независимость потребителей друг от друга. Ошибка или временная недоступность одного из компонентов не блокирует обработку события остальными сервисами.

Например, временная недоступность Сервиса сверки не влияет на обновление модели чтения или распространение изменений в Распределенный кэш. Такой подход повышает устойчивость распределённой архитектуры и уменьшает вероятность каскадной деградации.

Дополнительно модель публикации и подписки упрощает развитие Сервиса. При необходимости новый компонент может быть подключён к обработке событий без изменения логики сервисов-публикаторов. Для этого достаточно подписать новый компонент на соответствующий топик Event Bus.

Для предотвращения нарушения последовательности обработки событий используется секционирование по комбинации SKU + магазин. Это особенно важно для потоков изменения остатков и резервирования, поскольку изменение порядка обработки может приводить к временному расхождению состояния модели чтения.

Поскольку различные категории событий могут иметь существенно разную интенсивность обработки, в Сервисе используются независимые группы потребителей для:

* обновления модели чтения,
* сверки данных,
* мониторинга,
* аналитической обработки,
* технических задач обработки ошибок.

Такое разделение позволяет масштабировать отдельные направления обработки независимо друг от друга.

Для контроля корректности работы сценария публикации и подписки дополнительно отслеживаются метрики, изложенные в разделе <<section-monitoring-kafka, 2.11.9>>.

===== 2.11.7.3. Сценарий обработки задачи одним потребителем

Помимо модели публикации и подписки в Сервисе используется сценарий обработки задач одним потребителем. Данный подход применяется для операций, которые должны быть выполнены только одним экземпляром обработчика и не допускают параллельного исполнения одной и той же задачи несколькими компонентами одновременно.

Основной целью данного механизма является предотвращение дублирующей обработки и снижение риска возникновения конфликтующих изменений состояния.

Сценарий обработки одним потребителем используется для:

* повторной обработки ошибочных событий,
* фоновой сверки данных,
* пересчёта модели чтения,
* обработки задач восстановления состояния,
* выполнения технических служебных операций.

В отличие от модели публикации и подписки, где событие может независимо обрабатываться несколькими компонентами, в данном сценарии каждая задача должна быть назначена только одному экземпляру обработчика из группы потребителей.

Наиболее характерный пример -- *задача повторной синхронизации состояния модели чтения после обнаружения расхождения данных*. После фиксации проблемы Сервис сверки:

1. Формирует задачу на повторную обработку.
2. Публикует задачу в отдельный поток обработки в соответствующий топик Event Bus.

Далее задача последовательно проходит следующие этапы:

[start=3]
3. Получение задачи одним экземпляром обработчика.
4. Выполнение операции обработки.
5. Подтверждение успешного завершения задачи.
6. Фиксация результата обработки.

Если обработчик завершает работу аварийно до подтверждения выполнения задачи, сообщение может быть повторно передано другому экземпляру обработчика. Такой подход позволяет избежать потери задач при отказе отдельных компонентов.

Отдельное значение данный сценарий имеет для операций, связанных с восстановлением согласованности данных. При обнаружении конфликтующего состояния важно исключить ситуацию, при которой несколько экземпляров сервиса одновременно пытаются изменить одно и то же состояние модели чтения.

Использование модели обработки одним потребителем позволяет:

* избежать дублирования обработки,
* уменьшить вероятность конфликтующих изменений,
* контролировать последовательность выполнения операций,
* локализовать обработку фоновых задач,
* управлять скоростью выполнения ресурсоёмких операций.

Для различных категорий задач в Сервисе могут использоваться отдельные очереди обработки. Это позволяет разделить:

* задачи повторной обработки,
* задачи сверки,
* операции восстановления,
* технические служебные операции.

Разделение потоков обработки уменьшает влияние длительных или ресурсоёмких задач на остальные процессы обработки.

Дополнительно для задач обработки используются:

* ограничение количества повторных попыток,
* контроль времени выполнения,
* отдельные очереди ошибочных задач,
* фиксация состояния обработки,
* контроль зависших операций.

Это особенно важно для операций восстановления модели чтения и обработки конфликтов резервирования, поскольку неконтролируемое повторное выполнение задач может привести к дополнительной деградации состояния данных.

Для контроля работы сценария обработки одним потребителем отслеживаются показатели, изложенные в разделе <<section-monitoring-single, 2.11.9>>.

Использование сценария обработки задач одним потребителем позволяет обеспечить контролируемое выполнение фоновых операций, уменьшить вероятность конфликтующей обработки и повысить устойчивость Сервиса при выполнении технических операций восстановления и синхронизации состояния данных.

===== 2.11.7.4. Надёжность доставки сообщений

Для Сервиса проверки доступности товара потеря событий недопустима, поскольку события используются для обновления модели чтения, изменения состояния резервов и синхронизации данных между компонентами. Ошибки доставки или пропуск событий могут приводить к расхождению фактических остатков, некорректному отображению доступности товара и конфликтам резервирования.

Для обеспечения надёжности обработки событий в Event Bus используются несколько уровней защиты.

1. На уровне публикации сообщений используется *подтверждение успешной записи* события в брокер сообщений. +
Событие считается опубликованным только после подтверждения его сохранения. Это уменьшает вероятность потери сообщений при временных сетевых ошибках или отказе отдельных узлов брокера сообщений.

2. Для критически важных операций, связанных с резервированием товара и изменением остатков, используется *повторная отправка сообщений* при временной недоступности брокера сообщений. +
Повторные попытки публикации выполняются ограниченное количество раз с контролем времени ожидания между попытками.

3. Дополнительно в Сервисе применяется *идемпотентная обработка событий*. +
Один и тот же экземпляр события может быть получен повторно при восстановлении после ошибки или повторной доставке сообщения. Для предотвращения повторного применения изменений каждое событие содержит *уникальный идентификатор*, позволяющий обнаруживать дублирующую обработку.

4. Отдельное внимание уделяется *сохранению порядка обработки изменений состояния*. +
Для потоков изменения остатков и резервирования нарушение последовательности событий может привести к временному неконсистентному состоянию модели чтения. Для предотвращения подобных ситуаций события секционируются по комбинации SKU + магазин, что обеспечивает последовательную обработку изменений одного товара в рамках конкретного магазина.

5. В процессе обработки событий используется *контроль смещений чтения*. +
Подтверждение обработки сообщения выполняется только после успешного завершения операции обработки. Если экземпляр обработчика завершает работу аварийно до подтверждения смещения, событие может быть повторно передано другому экземпляру потребителя.

6. Для обработки нештатных ситуаций используются *отдельные очереди проблемных сообщений*. +
В такие очереди помещаются события, которые не удалось обработать после установленного количества повторных попыток. Это позволяет:

* изолировать ошибочные события от основного потока обработки,
* предотвратить остановку обработки очереди,
* выполнять последующий анализ проблемных сообщений,
* запускать повторную обработку после устранения причины ошибки.

+
Для задач повторной обработки используются отдельные обработчики, не влияющие на основной поток обновления модели чтения.

7. Дополнительно *контролируется время распространения событий* между публикацией изменения и обновлением модели чтения. +
Рост задержки обработки может свидетельствовать о:

* перегрузке потребителей,
* накоплении необработанных сообщений,
* деградации брокера сообщений,
* проблемах отдельных обработчиков событий.

8. Для контроля надёжности доставки сообщений отслеживаются показатели, изложенные в разделе <<section-monitoring-delivery, 2.11.9>>.

==== 2.11.8. Обработка отказов и деградаций

===== 2.11.8.1. Отказ экземпляра сервиса

Поскольку Сервис реализован как распределённая микросервисная архитектура, отказ отдельных экземпляров компонентов рассматривается как штатная эксплуатационная ситуация. Архитектура Сервиса должна обеспечивать продолжение обработки запросов и событий без полной остановки функциональности при выходе из строя отдельных узлов.

Наиболее вероятными причинами отказа экземпляра сервиса являются:

* аварийное завершение процесса,
* исчерпание доступных ресурсов,
* потеря сетевой доступности,
* ошибки внутреннего программного состояния,
* перегрузка экземпляра,
* отказ вычислительного узла.

Для уменьшения влияния подобных ситуаций критически важные компоненты Сервиса разворачиваются в нескольких экземплярах. В первую очередь это относится к:

* Сервису запросов,
* Сервису резервирования,
* Загрузчику моделей чтения,
* компонентам обработки событий,
* API Gateway.

При отказе экземпляра механизм обнаружения сервисов фиксирует потерю доступности и исключает проблемный экземпляр из списка активных узлов. После этого новые запросы перестают направляться на недоступный сервис.

Перераспределение нагрузки между оставшимися экземплярами выполняется автоматически без изменения клиентских настроек и без перезапуска других компонентов Сервиса.

Для внешних запросов это позволяет сохранить доступность API даже при частичном отказе внутренней инфраструктуры. Для внутренних сервисов — продолжить обработку событий и межсервисных вызовов без полной остановки контура обработки.

Дополнительно для предотвращения каскадной деградации используются ограничения времени ожидания ответа и механизмы контроля состояния зависимых сервисов. Если компонент начинает отвечать с критическими задержками или увеличивается количество ошибок взаимодействия, экземпляр может быть временно исключён из обработки запросов до восстановления работоспособности.

Для операций резервирования отказ экземпляра во время выполнения запроса требует отдельного контроля состояния незавершённых операций. В подобных ситуациях используются:

* ограниченное время жизни распределённых блокировок,
* повторная проверка состояния резерва,
* повторная обработка неподтверждённых событий,
* контроль зависших операций.

Это позволяет избежать длительного удержания блокировок и уменьшает вероятность конфликтов резервирования после восстановления компонента.

При отказе экземпляра обработчика событий необработанные сообщения не теряются и могут быть переданы другому экземпляру группы потребителей после восстановления обработки.

Для ускорения восстановления работоспособности используются автоматические механизмы перезапуска экземпляров сервисов. После повторного запуска экземпляр заново проходит регистрацию и автоматически включается в обработку запросов или событий.

Для контроля устойчивости Сервиса при отказах компонентов отслеживаются метрики, изложенные в разделе <<section-monitoring-foultcontrol, 2.11.9>>.

===== 2.11.8.2. Недоступность Распределенного кеша

Распределенный кэш используется в Сервисе как основной источник быстрого доступа к данным модели чтения при обработке пользовательских запросов. Недоступность кэша не приводит к полной остановке Сервиса, однако существенно влияет на производительность обработки операций проверки доступности товара.

Наиболее вероятными *причинами деградации или отказа Распределенного кэша* являются:

* перегрузка узлов кэша,
* потеря сетевой доступности,
* отказ отдельных экземпляров,
* превышение допустимого объёма памяти,
* массовое вытеснение данных,
* всплеск запросов по популярным SKU.

При возникновении подобных ситуаций Сервис запросов *переключается на использование Хранилища модели чтения* как резервного источника данных.

Поскольку Хранилище модели чтения не оптимизировано под такую же интенсивность запросов, как Распределенный кэш, в период деградации возможно:

* увеличение времени ответа,
* уменьшение пропускной способности контура чтения,
* рост нагрузки на внутренние сервисы,
* ограничение части некритичных операций.

Для предотвращения перегрузки Хранилища модели чтения в период деградации могут дополнительно применяться:

* ограничение количества параллельных запросов,
* временное ограничение глубины поиска магазинов,
* снижение частоты обновления отдельных данных,
* ограничение запросов по популярным SKU.

При частичной недоступности кэша допускается использование данных с ограниченной степенью актуальности. В подобных ситуациях ответ может формироваться с признаком `stale=true`, указывающим на возможное отставание состояния модели чтения от последних событий изменения остатков.

Такой подход позволяет сохранить доступность операций проверки товара даже при временной деградации инфраструктуры кэширования.

Для предотвращения *лавинообразной перегрузки после восстановления кэша* используется постепенное восстановление нагрузки и повторное заполнение данных модели чтения. Это уменьшает вероятность ситуации, при которой большое количество запросов одновременно инициирует повторную загрузку отсутствующих данных.

Дополнительно контролируется состояние синхронизации между Распределенным кэшем и Хранилищем модели чтения. После восстановления работоспособности выполняется проверка корректности актуальных данных и устранение возможных расхождений состояния.

Для контроля деградации Распределенного кэша отслеживаются метрики, изложенные в разделе <<section-monitoring-redis, 2.11.9>>.

===== 2.11.8.3. Недоступность Event Bus

Event Bus используется в Сервисе для передачи событий изменения состояния между компонентами архитектуры. Недоступность брокера сообщений не приводит к мгновенной остановке обработки пользовательских запросов, однако влияет на распространение изменений состояния и синхронизацию данных между компонентами.

Наиболее критичными последствиями недоступности Event Bus являются:

* задержка обновления модели чтения,
* накопление необработанных изменений,
* рост отставания между источниками данных,
* временное нарушение актуальности данных о доступности товара.

При недоступности Event Bus новые события не могут быть немедленно переданы обработчикам. В таких ситуациях компоненты публикующие события *временно сохраняют события во внутренних буферах повторной отправки* до восстановления соединения с брокером сообщений.

Для предотвращения бесконтрольного накопления данных используются:

* ограничение размера буферов,
* контроль времени хранения событий,
* ограничение количества повторных попыток публикации,
* отдельная обработка критически важных событий.

*Наибольший приоритет* сохраняется для событий резервирования и изменения остатков, поскольку потеря данных событий может приводить к *нарушению согласованности состояния модели чтения*.

Во время деградации Event Bus обработка пользовательских запросов продолжает выполняться с использованием последнего согласованного состояния модели чтения и Распределенного кэша. При этом возможна временная задержка распространения новых изменений состояния.

После восстановления работоспособности Event Bus выполняется *публикация накопленных событий* и *постепенная синхронизация состояния между компонентами*. Скорость восстановления обработки контролируется отдельно для предотвращения перегрузки потребителей при массовой догрузке событий.

Дополнительно контролируется *порядок обработки накопленных событий*. Для потоков изменения остатков и резервирования важно сохранить последовательность изменений одного товара в рамках конкретного магазина даже после временной деградации брокера сообщений.

Если время недоступности Event Bus превышает допустимые значения, могут *временно ограничиваться отдельные операции*, создающие повышенную нагрузку на поток событий. В первую очередь это относится к массовым операциям обновления данных и техническим задачам фоновой синхронизации.

Для контроля деградации Event Bus отслеживаются параметры, изложенные в разделе <<section-monitoring-Kafkafoult, 2.11.9>>.

===== 2.11.8.4. Ошибки операций резервирования

Операции резервирования *относятся к наиболее чувствительным сценариям* Сервиса, поскольку связаны с изменением доступного остатка товара и требуют повышенного контроля согласованности данных. Ошибки обработки резервирования могут приводить к двойному резервированию, превышению доступного остатка или расхождению состояния между различными компонентами архитектуры.

Наиболее вероятными причинами ошибок резервирования являются:

* одновременное резервирование одного товара несколькими пользователями,
* использование устаревшего состояния модели чтения,
* повторная отправка пользовательского запроса,
* временная недоступность внутренних компонентов,
* нарушение последовательности обработки событий,
* ошибки синхронизации между контурами обработки.

Для предотвращения конфликтующих изменений в контуре записи и резервирования используется *распределённая блокировка по комбинации SKU + магазин*. Захват блокировки выполняется до изменения состояния резерва и позволяет исключить параллельное подтверждение нескольких операций для одного и того же остатка.

Поскольку аварийное завершение обработки может приводить к зависшим блокировкам, для них используется *ограниченное время жизни*. После истечения установленного интервала блокировка автоматически освобождается и операция может быть выполнена повторно.

Дополнительно операции резервирования *выполняются с повторной проверкой актуального доступного остатка* перед подтверждением изменения состояния. Это позволяет уменьшить вероятность использования устаревших данных модели чтения при высокой интенсивности обновлений.

Для предотвращения повторного применения одной и той же операции используется *идемпотентность запросов резервирования*. Повторная отправка одинакового запроса не должна приводить к созданию нескольких резервов или повторному уменьшению доступного остатка.

Отдельное внимание уделяется ситуациям частичного завершения операции. Например, событие резервирования может быть успешно сформировано, но обновление модели чтения ещё не завершено. В подобных случаях возможна *временная задержка отображения актуального состояния доступности товара*.

Для обнаружения и устранения подобных расхождений используются:

* повторная обработка событий резервирования,
* контроль неподтверждённых операций,
* сверка состояния резервов,
* повторная синхронизация модели чтения.

При невозможности корректного завершения операции резервирования выполняется *откат изменений и освобождение захваченных блокировок*.

Дополнительно контролируются ситуации *превышения допустимого времени обработки резервирования*. Длительное выполнение операции может свидетельствовать о:

* перегрузке внутренних компонентов,
* проблемах синхронизации состояния,
* накоплении очередей обработки событий,
* конфликтующих изменениях остатков.

В подобных ситуациях операция может быть завершена с ошибкой до фиксации изменения состояния.

Для контроля ошибок резервирования отслеживаются показатели, изложенные в следующем разделе <<section-monitoring-reserverror, 2.11.9>>.

==== 2.11.9. Мониторинг взаимодействий

Мониторинг взаимодействий используется для контроля состояния распределённой архитектуры Сервиса, обнаружения деградаций и анализа производительности межсервисного взаимодействия.

Поскольку обработка пользовательских запросов зависит от большого количества взаимосвязанных компонентов, ухудшение работы одного сервиса или канала взаимодействия может приводить к росту задержек в других частях архитектуры. По этой причине мониторинг охватывает не только состояние отдельных компонентов, но и параметры взаимодействия между ними.

Основное внимание уделяется следующим категориям взаимодействий:

* внешние REST-запросы,
* внутренние gRPC-вызовы,
* обработка событий Event Bus,
* операции резервирования,
* обновление модели чтения,
* потоковые WebSocket-соединения.

1. Для внешнего взаимодействия контролируются:

* время ответа API,
* количество запросов,
* количество ошибок,
* интенсивность обращений по маршрутам,
* количество отклонённых запросов,
* количество ошибок проверки доступа.

[#section-monitoring-interconnect]
2. Для внутреннего межсервисного взаимодействия отслеживаются:

* задержки gRPC-вызовов,
* количество ошибок внутренних вызовов,
* количество повторных подключений,
* процент неуспешных запросов,
* загрузка отдельных компонентов,
* количество активных соединений,
* количество недоступных экземпляров сервисов,
* время ответа зависимых компонентов,
* время обновления модели чтения после поступления события.

[#section-monitoring-kafka]
3. Отдельно контролируется состояние потоков обработки событий. Для Event Bus отслеживаются:

* скорость публикации событий,
* скорость обработки сообщений,
* отставание групп потребителей,
* количество необработанных сообщений,
* задержки распространения событий,
* задержка обновления модели чтения,
* загрузка разделов топиков,
* количество повторных обработок,
* времени обновления модели чтения после поступления события.

[#section-monitoring-Kafkafoult]
Дополнительно для контроля деградации Event Bus отслеживаются

* доступность брокеров сообщений,
* количество неподтверждённых публикаций,
* размер внутренних буферов повторной отправки,
* задержка распространения событий,
* объём накопленных необработанных сообщений,
* скорость восстановления обработки после деградации,
* отставание групп потребителей.

[#section-monitoring-stream]
4. Для потоков обновления отслеживаются:

* количество активных WebSocket-соединений,
* задержка доставки обновлений,
* количество разорванных соединений,
* количество повторных подключений,
* интенсивность отправки сообщений,
* доля неуспешной доставки уведомлений.

[#section-monitoring-reserv]
5. Для операций резервирования контролируются:

* количество конфликтов резервирования,
* количество отклонённых операций,
* время обработки резервирования,
* количество зависших блокировок,
* количество неподтверждённых операций.

6. Для Распределенного кэша контролируются:

* количество обращений,

* процент вытеснения данных,
* количество промахов кэша,
* интенсивность запросов по отдельным SKU.

[#section-monitoring-redis]
Дополнительно для контроля деградации Распределенного кэша отслеживаются:

15. Для контроля устойчивости Сервиса при отказах компонентов отслеживаются:

* количество отказов экземпляров,
* время восстановления после отказа,
* количество недоступных экземпляров,
* количество перенаправленных запросов,
* количество ошибок межсервисного взаимодействия,
* длительность деградации отдельных компонентов.

[#section-monitoring-reserverror]
16. Для контроля ошибок резервирования отслеживаются:

* количество конфликтов резервирования,
* количество повторных запросов,
* количество отклонённых операций,
* количество зависших блокировок,
* время обработки резервирования,
* количество неподтверждённых операций,
* количество повторных обработок событий резервирования.

Полученные данные мониторинга используются для:

* обнаружения деградаций,
* анализа производительности,
* поиска узких мест,
* выявления перегруженных компонентов,
* анализа причин ошибок,
* контроля выполнения целевых показателей производительности.

Для визуализации состояния Сервиса используются централизованные панели мониторинга, отображающие текущее состояние компонентов, потоков событий и межсервисного взаимодействия.
