==== 3.2.3 Мониторинг безопасности

===== 3.2.3.1 Метрики безопасности

Мониторинг безопасности Сервиса использует специализированные метрики для выявления нарушений политик безопасности, контроля административных действий и обнаружения аномального поведения пользователей и сервисов.

Метрики безопасности дополняют технические метрики observability, разработанные в разделе 2.9, и используются совместно с журналами аудита безопасности и трассировкой запросов.

*1. Метрики изменения политик безопасности*

Контролируются изменение политик безопасности Сервиса и параметры административных изменений.

Контролируется количество:

* изменений политик безопасности, политик API Gateway, HashiCorp Vault,
* изменений ограничений Event Bus,
* ошибок применения политик,
* неуспешных административных изменений.

Рост количества изменений политик безопасности рассматривается как потенциальный признак:

* несанкционированной административной активности,
* ошибочной конфигурации,
* попыток обхода ограничений безопасности.

Дополнительно контролируется:

* частота изменений,
* инициатор изменений,
* источник административного запроса,
* доля неуспешных изменений.

*2. Метрики изменения ролей*

Контролируются операции изменения пользовательских и сервисных ролей.

Контролируются количество:

* назначений и удалений ролей,
* изменений привилегированных ролей,
* ошибок изменения ролей,
* операций изменения сервисных полномочий.

Дополнительно анализируются:

* изменение количества привилегированных ролей,
* рост числа административных учетных записей,
* аномальное изменение сервисных полномочий,
* повторяющиеся ошибки изменения ролей.

Для привилегированных ролей дополнительно контролируется:

* частота использования,
* количество административных операций, отказов авторизации, 
* количество попыток выполнения запрещенных операций.

*3. Метрики изменения конфигурации сервисов*

Контролируются изменения конфигурации компонентов Сервиса.

Контролируются количество:

* изменений конфигурации API Gateway, Event Bus, параметров Kubernetes,
* изменений параметров сервисов, сетевых политик,
* ошибок применения конфигурации.

Дополнительно анализируются:

* частота изменений конфигурации,
* количество аварийных изменений, откатов конфигурации,
* изменение параметров ограничения запросов, подключения к хранилищам данных.

Рост количества изменений конфигурации рассматривается как *потенциальный риск нарушения стабильности или безопасности* Сервиса.

*4. Метрики изменения криптографических настроек*

Контролируются изменения параметров TLS, mTLS и криптографических механизмов Сервиса.

Контролируются количество:

* изменений наборов шифров, TLS-конфигурации, параметров mTLS,
* отзывов сертификатов, 
* ошибок проверки сертификатов, JWT access-токенов,
* изменений доверенных ключей проверки JWT access-токенов.

Дополнительно анализируются:

* рост ошибок проверки сертификатов, количества невалидных JWT access-токенов,
* изменение частоты ротации сертификатов,
* количество отказов сервисной аутентификации, ошибок проверки mTLS.

Данные метрики дополнительно сопоставляются с техническими метриками:

* `error_rate`,
* `p95/p99 latency`,
* `consumer lag`,
* количеством ошибок API Gateway, межсервисного взаимодействия.

Это позволяет выявлять *инциденты безопасности, влияющие на стабильность и доступность* Сервиса.

===== 3.2.3.2 Выявление аномальной активности

Мониторинг безопасности Сервиса выполняет автоматизированное выявление аномальной активности пользователей, сервисных учетных записей и интеграционных компонентов.

Выявление аномалий выполняется на основе:

* журналов аудита безопасности, метрик безопасности, корреляции событий,
* анализа пользовательских и сервисных сессий, сетевой активности, обращений к данным и API.

Мониторинг безопасности выявляет события, потенциально указывающие на:

* компрометацию учетных записей,
* попытки обхода механизмов авторизации,
* несанкционированное использование сервисных ролей,
* массовый сбор данных,
* нарушение политик безопасности Сервиса.

*1. Подбор учетных данных*

Мониторинг безопасности выявляет признаки автоматизированного подбора учетных данных пользователей и сервисов.

Контролируются:

* повторяющиеся ошибки аутентификации,
* большое количество запросов аутентификации с одного IP-адреса,
* попытки аутентификации для множества учетных записей,
* высокая частота ошибок проверки JWT access-токенов,
* повторное использование недействительных refresh-токенов,
* аномальная частота запросов аутентификации.

Дополнительно анализируются:

* география запросов,
* изменение IP-адресов,
* время между попытками аутентификации,
* изменение пользовательского агента,
* использование заблокированных учетных записей.

При выявлении признаков подбора учетных данных мониторинг безопасности может инициировать:

* блокировку учетной записи,
* ограничение частоты запросов,
* завершение пользовательской сессии,
* формирование уведомления безопасности повышенного уровня критичности.

*2. Массовые отказы авторизации*

Мониторинг безопасности выявляет аномальное количество отказов авторизации при обращении к API и внутренним сервисам Сервиса.

Контролируются:

* рост количества ошибок доступа, запрещенных операций,
* повторяющиеся обращения к административным API,
* массовые ошибки проверки полномочий,
* рост количества нарушений политик доступа,
* повторяющиеся ошибки доступа к Event Bus.

Дополнительно анализируются:

* тип вызываемых API,
* используемые роли,
* сервисные учетные записи,
* источник запросов,
* последовательность операций пользователя или сервиса.

Массовые отказы авторизации рассматриваются как потенциальный признак:

* компрометации учетной записи,
* ошибочной конфигурации ролей,
* попытки обхода ограничений доступа,
* использования устаревших сервисных полномочий.

*3. Аномальная активность сервисных учетных записей*

Мониторинг безопасности анализирует активность сервисных учетных записей и выявляет отклонения от ожидаемого поведения сервисов.

Контролируются:

* изменение характера межсервисных запросов,
* обращение сервисов к нехарактерным API,
* рост количества отказов сервисной аутентификации,
* использование неразрешенных топиков Event Bus,
* аномальная активность сервисов в HashiCorp Vault,
* резкий рост количества операций доступа к данным,
* изменение частоты публикации событий.

Дополнительно анализируются:

* сервисная роль,
* используемые учетные данные Kafka,
* тип выполняемых операций,
* частота запросов,
* объем передаваемых данных,
* изменение шаблонов сетевого взаимодействия.

Аномальная активность сервисных учетных записей рассматривается как потенциальный признак:

* компрометации сервисного секрета,
* нарушения сервисной политики,
* ошибочной конфигурации сервиса,
* несанкционированного использования сервисной роли.

*4. Необычные обращения к данным*

Мониторинг безопасности выявляет аномальные операции доступа к данным Сервиса.

Контролируются:

* массовое чтение данных,
* резкий рост количества запросов к хранилищам,
* нетипичный экспорт данных,
* обращения к административным данным, к данным вне стандартного сценария работы роли,
* аномальная активность в PostgreSQL, MongoDB, Redis Cluster.

Дополнительно анализируются:

* объем выгружаемых данных,
* тип данных,
* источник запросов,
* используемая роль,
* время выполнения операций,
* последовательность обращений к данным.

Мониторинг безопасности дополнительно сопоставляет аномальные обращения к данным с:

* изменением ролей, политик доступа, конфигурации сервисов, сетевой активности,
* использованием привилегированных функций.

Это позволяет выявлять сложные инциденты безопасности, связанные с *компрометацией учетных записей или несанкционированным сбором данных* Сервиса.

===== 3.2.3.3 Контроль доступа к критичным функциям

Мониторинг безопасности Сервиса обеспечивает дополнительный контроль операций, связанных с использованием привилегированных полномочий и доступом к критичным функциям Сервиса.

Контроль выполняется для:

* административных операций, изменения ролей, политик безопасности,
* операций управления секретами, сертификатами, изменения конфигурации, параметров защищенных соединений.

Для критичных операций дополнительно контролируются:

* идентификатор пользователя или сервиса,
* используемая роль,
* источник запроса,
* идентификатор запроса,
* затронутый компонент Сервиса,
* последовательность выполняемых действий,
* результат выполнения операции.

*1. Контроль административных операций*

Механизмы мониторинга безопасности Сервиса контролируют выполнение административных операций, влияющих на безопасность и стабильность работы компонентов Сервиса.

Контролируются:

* административные операции API Gateway, Kubernetes,
* операции управления Event Bus, HashiCorp Vault,
* операции изменения параметров PostgreSQL, MongoDB, Redis Cluster.

Дополнительно анализируются:

* использование привилегированных ролей,
* выполнение административных операций вне регламентного времени,
* выполнение последовательности административных изменений,
* выполнение операций из нетипичных источников,
* аномальная частота административных действий.

Для критичных административных операций *формируются события повышенного уровня критичности*.

*2. Контроль операций управления ролями*

Мониторинг безопасности Сервиса контролирует операции назначения, изменения и удаления пользовательских и сервисных ролей.

Контролируются:

* назначение привилегированных ролей, ролей интеграционных сервисов,
* изменение политик RBAC, ограничений доступа, сервисных полномочий,
* удаление ролей.

Дополнительно выявляются:

* массовое изменение ролей,
* повторяющиеся ошибки изменения ролей,
* аномальное изменение сервисных полномочий,
* использование ролей вне стандартного сценария работы,
* попытки назначения избыточных привилегий.

Особое внимание уделяется операциям, затрагивающим:

* административные роли,
* сервисные роли Event Bus,
* роли доступа к HashiCorp Vault,
* роли доступа к хранилищам данных.

*3. Контроль изменения политик безопасности*

Мониторинг безопасности Сервиса контролирует изменение политик безопасности и параметров авторизации.

Контролируются изменения:

* политик API Gateway, HashiCorp Vault,
* ограничений межсервисного взаимодействия,
* сетевых политик Kubernetes, политик доступа Event Bus,
* параметров проверки JWT access-токенов,
* параметров mTLS.

Дополнительно анализируются:

* частота изменений,
* последовательность административных действий,
* ошибки применения политик,
* изменение параметров ограничений доступа, криптографической защиты.

Изменение политик безопасности рассматриваются как операции *повышенного уровня критичности и подлежит обязательной регистрации* в журналах аудита безопасности.

*4. Контроль операций с секретами*

Мниторинг безопасности Сервиса контролирует использование сервисных секретов, сертификатов и криптографических ключей.

Контролируются:

* получение секретов из HashiCorp Vault,
* изменение сервисных секретов,
* ротация учетных данных,
* получение сервисных сертификатов,
* отзыв сертификатов,
* изменение политик доступа к секретам,
* операции с ключами проверки JWT access-токенов.

Дополнительно выявляются:

* обращение сервисов к нехарактерным секретам,
* рост количества запросов к HashiCorp Vault,
* повторяющиеся ошибки доступа к секретам,
* аномальная частота ротации секретов,
* использование запрещенных сервисных ролей,
* попытки доступа к административным секретам.

Операции с секретами и сертификатами рассматриваются как события *повышенного уровня критичности и подлежат дополнительному контролю* со стороны механизмов мониторинга безопасности Сервиса.

===== 3.2.3.4 Оповещение о событиях безопасности

Механизмы мониторинга безопасности Сервиса обеспечивают автоматическое формирование уведомлений при выявлении событий, указывающих на нарушение политик безопасности, компрометацию учетных записей или аномальную активность компонентов Сервиса.

Оповещения используются для:

* оперативного выявления инцидентов безопасности,
* контроля критичных административных операций, использования привилегированных ролей,
* выявления компрометации сервисных учетных записей, аномального поведения пользователей и сервисов.

Для уведомлений безопасности дополнительно фиксируются:

* уровень критичности события,
* источник события,
* затронутый компонент Сервиса,
* идентификатор пользователя или сервиса,
* идентификатор запроса,
* время возникновения события,
* описание выявленного нарушения.

*1. Пороговые значения*

Мониторинг безопасности Сервиса используют пороговые значения для автоматического выявления аномальной активности и формирования уведомлений безопасности.

Контролируется количество:

* ошибок аутентификации, проверки JWT access-токенов, mTLS, сервисной аутентификации,
* ошибок доступа к HashiCorp Vault,
* отказов авторизации,
* попыток обращения к запрещенным API,
* операций изменения политик безопасности, административных операций,
* аномальных обращений к данным.

Для различных категорий событий используются отдельные пороговые значения с учетом:

* критичности компонента,
* типа учетной записи,
* характера операции,
* уровня привилегий,
* времени возникновения события,
* частоты повторения события.

Превышение пороговых значений рассматривается как *потенциальный инцидент безопасности и инициирует формирование уведомления* безопасности.

*2. Критичность событий*

События безопасности классифицируются по уровням критичности в зависимости от влияния на безопасность и работоспособность Сервиса.

К событиям повышенного уровня критичности относятся:

* компрометация административной учетной записи,
* попытки обхода авторизации,
* изменение политик безопасности, криптографических параметров,
* отзыв сервисных сертификатов,
* аномальная активность HashiCorp Vault, Event Bus,
* массовые ошибки сервисной аутентификации,
* попытки доступа к административным секретам.

Дополнительно повышенный уровень критичности присваивается событиям, связанным с:

* привилегированными ролями,
* административными API,
* изменением сервисных полномочий,
* массовым экспортом данных,
* изменением конфигурации Kubernetes, параметров TLS и mTLS.

Уровень критичности события влияет на:

* приоритет обработки инцидента,
* маршрут уведомления,
* необходимость эскалации,
* скорость реагирования.

*3. Маршрутизация уведомлений*

Механизмы мониторинга безопасности Сервиса выполняют маршрутизацию уведомлений в зависимости от типа события и уровня критичности.

Уведомления направляются:

* специалистам сопровождения,
* администраторам безопасности,
* администраторам инфраструктуры,
* ответственным за эксплуатацию компонентов Сервиса.

Маршрутизация выполняется с учетом:

* затронутого компонента,
* типа события,
* категории инцидента,
* уровня критичности,
* времени возникновения события.

Для различных категорий инцидентов используются отдельные маршруты уведомлений.

Дополнительно обеспечивается:

* предотвращение дублирования уведомлений,
* объединение связанных событий,
* группировка повторяющихся ошибок,
* контроль доставки уведомлений.

*4. Эскалация инцидентов*

При отсутствии подтверждения обработки уведомления или при развитии инцидента выполняется автоматическая эскалация события безопасности.

Эскалация выполняется при:

* длительном отсутствии реакции на инцидент,
* повторении критичных событий,
* росте количества связанных нарушений,
* распространении инцидента на несколько компонентов Сервиса,
* нарушении доступности критичных компонентов.

Для событий повышенного уровня критичности дополнительно контролируются:

* время реакции,
* время устранения,
* статус обработки инцидента,
* последовательность действий при расследовании.

Механизмы мониторинга безопасности Сервиса обеспечивают сохранение истории эскалации и последующего расследования инцидентов безопасности.

==== 3.2.4 Хранение и расследование событий безопасности

===== 3.2.4.1 Защищенное хранение журналов аудита

Журналы аудита безопасности Сервиса хранятся централизованно и используются для: 

* расследования инцидентов безопасности, 
* анализа действий пользователей и сервисов, 
* подтверждения соблюдения политик безопасности Сервиса.

Хранение журналов аудита безопасности обеспечивает:

* защиту от несанкционированного изменения,
* контроль доступа к журналам,
* сохранность критичных событий,
* резервное копирование,
* долговременное хранение событий безопасности,
* возможность последующего расследования инцидентов.

Для журналов аудита безопасности дополнительно контролируются:

* целостность записей,
* последовательность событий,
* полнота поступления журналов,
* наличие попыток удаления или изменения записей аудита.

*1. Неизменяемость журналов аудита*

Журналы аудита безопасности Сервиса защищаются от несанкционированного изменения и удаления.

Для критичных событий безопасности обеспечивается:

* запись только на добавление,
* запрет изменения ранее сохраненных записей,
* контроль целостности журналов,
* фиксация попыток удаления или модификации событий,
* контроль последовательности записей аудита.

Дополнительно регистрируются операции:

* очистки журналов,
* изменения параметров хранения,
* экспорта журналов,
* административного доступа к журналам аудита.

Для критичных журналов безопасности дополнительно контролируется:

* целостность архивных копий,
* непрерывность последовательности событий,
* корректность временных меток,
* наличие пропущенных событий.

*2. Контроль доступа к журналам аудита*

Доступ к журналам аудита безопасности ограничивается политиками RBAC Сервиса.

Доступ предоставляется только:

* администраторам безопасности,
* ограниченному перечню учетных записей сопровождения,
* сервисам анализа журналов аудита,
* механизмам мониторинга безопасности Сервиса.

Контролируются:

* операции чтения журналов,
* экспорт журналов,
* административный доступ,
* изменение параметров хранения,
* изменение политик доступа к журналам.

Дополнительно регистрируются:

* использование привилегированных ролей,
* массовое чтение журналов,
* попытки доступа к запрещенным журналам,
* обращения к архивным журналам безопасности.

*3. Резервное копирование журналов аудита*

Журналы аудита безопасности Сервиса подлежат регулярному резервному копированию.

Резервное копирование используется для:

* сохранения истории событий безопасности,
* восстановления журналов после отказов,
* расследования инцидентов,
* долговременного хранения критичных событий.

Контролируются:

* успешность резервного копирования,
* целостность резервных копий,
* полнота архивирования,
* доступность архивных журналов,
* успешность восстановления журналов аудита.

Для резервных копий дополнительно обеспечивается:

* защита от несанкционированного доступа,
* контроль операций восстановления, удаления архивных копий, срока хранения архивов.

*4. Сроки хранения журналов аудита*

Срок хранения журналов аудита определяется категорией события и уровнем критичности инцидента безопасности.

Длительное хранение журналов используется для:

* расследования инцидентов безопасности,
* анализа повторяющихся нарушений,
* контроля административных операций,
* анализа действий привилегированных ролей,
* расследования компрометации учетных записей.

Для различных категорий журналов используются отдельные политики хранения.

Дополнительно контролируются:

* автоматическое архивирование,
* удаление устаревших журналов,
* корректность переноса архивов,
* сохранение целостности архивных данных,
* фиксация операций удаления журналов аудита.

Удаление журналов аудита безопасности выполняется только регламентированными процедурами с *обязательной регистрацией административной операции в журналах аудита* Сервиса.

===== 3.2.4.2 Обеспечение целостности журналов

Механизмы хранения журналов аудита безопасности Сервиса обеспечивают контроль целостности записей аудита и выявление попыток несанкционированного изменения событий безопасности.

Контроль целостности используется для:

* подтверждения достоверности событий аудита,
* выявления попыток модификации журналов,
* контроля полноты последовательности событий,
* предотвращения удаления критичных записей,
* подтверждения корректности расследования инцидентов безопасности.

Для журналов аудита безопасности дополнительно контролируются:

* последовательность событий,
* непрерывность временных меток,
* целостность архивных копий,
* наличие пропущенных событий,
* попытки изменения или удаления записей.

*1. Контроль изменений журналов аудита*

Журналы аудита безопасности Сервиса защищаются от несанкционированного изменения административными и сервисными учетными записями.

Контролируются изменение:

* записей аудита,
* параметров хранения, архивирования,
* политик хранения журналов,
* конфигурации журналирования,

и операции очистки журналов.

Дополнительно регистрируются:

* идентификатор пользователя или сервиса,
* тип административной операции,
* источник изменения,
* идентификатор запроса,
* результат выполнения операции.

Любые *операции изменения параметров хранения журналов аудита* рассматриваются как *события повышенного уровня критичности*.

*2. Криптографическая защита журналов*

Для критичных журналов аудита безопасности используются механизмы криптографической защиты целостности записей.

Контролируются:

* контрольные суммы журналов,
* целостность архивных файлов,
* корректность криптографических подписей,
* соответствие временных меток последовательности событий.

Для архивных журналов дополнительно обеспечивается контроль:

* неизменности архивных копий,
* целостности резервных копий,
* повреждения архивных файлов,
* корректности восстановления журналов аудита.

Механизмы мониторинга безопасности Сервиса дополнительно фиксируют:

* ошибки проверки целостности,
* повреждение архивных журналов,
* несоответствие контрольных сумм,
* ошибки проверки подписей журналов.

*3. Контроль удаления записей аудита*

Удаление журналов аудита безопасности выполняется только регламентированными административными процедурами.

Контролируются:

* удаление журналов, архивных копий,
* изменение сроков хранения,
* операции очистки журналов, массового удаления событий.

Дополнительно выявляются:

* попытки удаления критичных журналов,
* удаление журналов вне регламентного периода,
* повторяющиеся операции удаления,
* удаление журналов привилегированными ролями,
* аномальная активность при работе с архивами журналов.

Для всех операций удаления дополнительно фиксируются:

* инициатор операции,
* причина удаления,
* перечень удаляемых журналов,
* время выполнения операции,
* результат выполнения операции.

*4. Проверка достоверности журналов*

Журналы аудита безопасности Сервиса периодически проверяются на полноту и достоверность записей.

Проверка достоверности включает:

* контроль последовательности событий, непрерывности идентификаторов запросов, соответствия временных меток,
* сопоставление журналов различных компонентов Сервиса,
* выявление отсутствующих или поврежденных записей.

Дополнительно выполняется сопоставление событий:

* API Gateway,
* сервисной аутентификации,
* Event Bus,
* HashiCorp Vault,
* Kubernetes,
* доступа к хранилищам данных.

Механизмы мониторинга безопасности Сервиса используют результаты проверки достоверности журналов для выявления:

* попыток сокрытия инцидентов,
* удаления событий безопасности,
* нарушения последовательности событий,
* компрометации административных учетных записей,
* повреждения архивных журналов аудита.

Сопоставление выполняется по:

Дополнительно анализируются:

* изменения политик безопасности,

Временная линия используется для:

* восстановления последовательности действий,
* определения момента компрометации,
* анализа распространения инцидента,
* определения затронутых компонентов,
* анализа реакции на инцидент.

Временная линия включает:

Статистика событий безопасности используется для анализа общей активности механизмов безопасности Сервиса и выявления изменений в характере событий безопасности.

Анализируется количество:

* событий аутентификации,
* отказов авторизации,
* административных операций,
* событий изменения конфигурации,
* операций с секретами, изменения ролей,
* ошибок сервисной аутентификации,
* нарушений политик безопасности,
* аномальных обращений к данным.

Дополнительно контролируются:

* распределение событий по уровням критичности, по компонентам Сервиса,
* динамика изменения количества событий,
* рост количества критичных нарушений,
* повторяемость инцидентов безопасности.

Результаты анализа статистики событий безопасности используются для: 

* корректировки политик безопасности, 
* настройки механизмов мониторинга безопасности Сервиса
* выявления новых сценариев нарушений безопасности.
