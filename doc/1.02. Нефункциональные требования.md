### 1.2. Нефункциональные требования

#### 1.2.1. Производительность и Масштабируемость

Сервис должен: 

1. обеспечивать базовую нагрузку 10,000 RPS с учётом концентрации запросов на популярных SKU, которые могут составлять до 80% трафика;
2. выдерживать пиковую нагрузку 50,000 RPS на операции чтения;
3. обрабатывать потоковые обновления со скоростью не менее 5,000 событий в секунду в пиковые часы продаж при интеграции с сетью магазинов;
4. обеспечивать метрику p99 latency не более:
* 50 ms для запроса наличия конкретного SKU в конкретном магазине при попадании в кэш;
* 200 ms для операций геопоиска в радиусе 5 км с включением до 20 магазинов в результат;
5. обеспечивать задержка доставки события от момента фиксации продажи на POS до момента доступности в API не более 5 секунд (метрика p95 latency).

#### 1.2.2. Доступность

Сервис должен: 

1. обеспечивать доступность на уровне 99.95%, что соответствует допустимому времени простоя не более 19 минут в месяц;
2. предоставлять данные из кэша с маркировкой статуса устаревших данных без отказа в обслуживании при недоступности WMS или других внешних источников; 
3. обеспечивать RTO не более 30 секунд путем переключения на active-реплику при отказе компонентов чтения. 

#### 1.2.3. Консистентность

Сервис должен: 

1. приоритизировать доступность над консистентностью для операций чтения текущего наличия SKU;
2. допускать временную разницу между реальными и системными остатками в пользу непрерывности обслуживания;
3. обеспечивать строгую консистентность для предотвращения оверселлинга (продажи одной единицы нескольким клиентам) для операций резервирования и списания остатков под заказы; 
4. поддерживать eventual consistency для основных данных с максимальным лагом синхронизации между источником событий и API чтения не более 5 секунд;
5. следовать модели BASE для операций чтения;
6. следовать модели ACID для транзакций резервирования SKU.

#### 1.2.4. Наблюдаемость

Сервис должен:

1. собирать метрики RPS, Latency (p50/p95/p99), Cache Hit Ratio, Data Freshness Lag и Error Rate с периодом хранения исторических данных не менее 30 дней;
2. инициировать оповещение администратора при:
* превышении p99 latency более 100 ms,
* падении Cache Hit Ratio ниже 85 процентов,
* превышении лага данных более 10 секунд;
3. поддерживать распределенный трейсинг для сквозной трассировки запросов от POS через шину данных и кэш до API с корреляционным ID для всех операций;
4. добавлять корреляционный идентификатор, связывающий запросы по конкретному SKU и магазину для всех логов операций;
5. хранить логи операций в течение 30 дней.

#### 1.2.5. Безопасность и Compliance

Сервис должен обеспечивать:

1. для защиты персональных данных:
* защиту всего трафика по TLS 1.3;
* анонимизацию гео-запросов клиентов;
2. ограничение до 100 запросов в минуту на клиента для предотвращения несанкционированного web-скрепинга ассортиментных данных конкурентами.
