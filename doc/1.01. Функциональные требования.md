## 1. Требования

### 1.1. Функциональные требования

#### 1.1.1. Для клиента сервис должен:

1. предоставлять возможность через мобильное приложение проверять наличие товара на полках конкретного магазина с привязкой к торговому залу, исключая складские запасы;
2. обеспечивать функцию геопоиска ближайших магазинов в заданном радиусе, где товар физически присутствует на полке и не зарезервирован под другие заказы;
4. позволять фильтровать с учетом актуальных остатков результаты поиска по дополнительным параметрам: наличие определенного объема упаковки, срок годности, органический/неорганический статус;
5. предоставлять функцию сравнения наличия выбранного товара одновременно в нескольких магазинах с отображением расстояния до каждой точки;
6. реализовать механизм резервирования товара непосредственно на полке при оформлении клиентом заказов на самовывоз (Click&Collect) или экспресс-доставку;
7. реализовать функцию подписки на push-уведомления о поступлении конкретного товара на полку выбранного магазина;
8. поддерживать создание персонального списка отслеживаемых товаров с автоматической отправкой уведомлений при появлении товара на полке в предпочитаемом магазине или при снижении остатка до оределенного критического уровня;
9. обеспечивать доступ к кешированным данным о наличии при временном отсутствии интернет-соединения с индикацией возможной неактуальности информации;
10. предоставлять push-уведомления об истечении срока резерва товара, забронированного для самовывоза;
11. позволять проверять наличие сопутствующих товаров в том же магазине, где есть в наличии основной товар.

#### 1.1.2. Для сотрудника магазина сервис должен:

1. предоставлять сотрудникам торгового зала через handheld-терминалы возможность сканирования SKU для подтверждения фактического наличия на полке и оперативной коррекции остатков при обнаружении пересорта;
2. обеспечивать сотрудникам функцию массового пересчета остатков в рамках плановой инвентаризации с пакетной загрузкой корректировок в систему;
3. предоставлять персоналу магазина доступ к истории изменений остатков по конкретному SKU за выбранный период с указанием источника корректировки: продажа, инвентаризация, перемещение, ручная правка;
4. позволять сотруднику торгового зала временно блокировать товар на полке (маркировка как "в пересчете", "брак", "ожидает выкладки") с автоматическим исключением из доступности для клиентов;
5. поддерживать сценарий перемещения товара между зонами торгового зала (основная полка, промо-стойка, холодильник) с сохранением общего остатка по магазину, но изменением локации для поиска размещения внутри магазина;
6. предоставлять функцию печати ценников и этикеток только для товаров, фактически присутствующих на полке, с автоматической проверкой наличия перед печатью.

#### 1.1.3. Для администратора системы сервис должен:

1. предоставлять администратору инструменты управления порогами безопасного запаса по категориям товаров или индивидуально по SKU для автоматической корректировки отображаемого клиенту наличия;
2. обеспечивать возможность настройки параметров кэширования через административный интерфейс без остановки системы, включая время жизни записей, политика invalidation, размеры сегментов;
3. предоставлять функции управления географическими зонами покрытия, включая радиусы поиска, исключение отдельных магазинов из общей выдачи, настройка приоритетов магазинов;
4. поддерживать управление интеграционными коннекторами, включая включение/отключение потоков данных от конкретных POS-систем или WMS, настройку таймаутов и retry-политик;
5. обеспечивать управление пользователями и ролевой моделью, включая создание учетных записей работников, назначение прав доступа к конкретным магазинам или функциям, блокировка утерянных handheld-устройств;
6. предоставлять возможность ручного запуска ребилда поисковых индексов и полной пересинхронизации кэша с основным хранилищем при обнаружении масштабных расхождений данных;
7. позволять настраивать пороги и правила оповещения, включая latency, процент ошибок, lag синхронизации, и каналы уведомлений через email, SMS, мессенджеры для групп ответственных;
8. предоставлять инструменты мониторинга и управления активными резервами: просмотр списка забронированных товаров, принудительное освобождение резерва при отмене заказа, массовая чистка "зависших" резервов;
9. обеспечивать функцию аудита всех изменений критичных параметров в неизменяемом логе действий: кто и когда изменил остатки вручную, изменил настройки безопасного остатака, заблокировал товар; 
10. предоставлять возможность через единый интерфейс включать в "черный" или "белый" списки отдельные SKU (исключение из отображения наличия) или магазины (техническое обслуживание, закрытие на реновацию);
11. поддерживать настройку сезонных коэффициентов корректировки наличия, включая увеличение виртуального safety stock в пиковые периоды, изменение правил резервирования в ночные часы;
12. предоставлять инструменты диагностики связности с внешними системами (проверка статуса подключения к POS и WMS конкретного магазина, просмотр очереди необработанных событий, ручная переотправка сообщений)

### 1.2. Требования интеграции (API & Интеграции)

#### Для интеграции с другими системами сервис должен:

1. предоставлять API (REST/gRPC) для синхронного запроса наличия товара по SKU и идентификатору магазина;
2. поддерживать batch-запросы для проверки нескольких позиций SKU одновременно;
3. обеспечивать потоковую интеграцию с кассовыми терминалами (POS) для приема событий продаж, возвратов и аннулирований чеков в реальном времени;
4. интегрироваться с системой управления складом (WMS) для учета перемещений товаров со склада на торговую точку, а также данных проведенных инвентаризаций;
5. реализовывать алгоритм разрешения конфликтов между источниками данных: POS, WMS, ручные корректировки, приоритизируя POS-данные, как наиболее актуальные отражение фактических продаж;
6. обеспечивать управление распределенным кэшем с настраиваемыми параметрами и механизмом проактивной инвалидации при получении событий изменения остатков;
7. поддерживать пространственный индекс для выполнения геопоиска магазинов с фильтрацией по фактическому наличию товара на полке.
